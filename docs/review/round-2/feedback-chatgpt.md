## Round 1 지적 해결 여부

| # | 지적사항                                                             | 판정        | 근거                                                                                                                                                                                                                                                                                                       |
| - | ---------------------------------------------------------------- | --------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1 | 핵심 계약 계층 부재 (Canonical Model, Spec DSL Schema, OpenAPI Contract) | **부분 해결** | v3는 **Phase 1에서 API Server를 제거**하고 CLI가 Plumb를 직접 subprocess로 호출하게 바꿔, 당장 필요한 OpenAPI 계약 부담은 줄였습니다. 또 **Phase 2에서 OpenAPI 3.1 계약을 선작성**하겠다고 명시했습니다. 하지만 **Canonical Model**과 **Spec DSL Schema** 자체는 여전히 문서에 정식 계약으로 등장하지 않습니다. 즉, “Phase 1 범위 축소로 계약 필요성을 줄인 것”이지, 계약 계층을 실제로 만든 것은 아닙니다. ([GitHub][1]) |
| 2 | Git SSOT vs 자연어 협업 논리 충돌                                         | **해결됨**   | v3는 이 충돌을 상당히 잘 정리했습니다. **Git이 항상 승리**한다고 선언하고, DB는 메타데이터만 저장하며, 자연어 참여는 **Phase 2로 이관**했고, 그때도 **LLM 생성 Spec은 사람 리뷰/승인 후에만 Git commit**되도록 했습니다. “자연어가 바로 SSOT를 바꾸지 않는다”는 규칙이 생겼습니다. ([GitHub][1])                                                                                                      |
| 3 | NL → DSL 변환 계층 미정의                                               | **부분 해결** | v3는 Round 1보다 진전됐습니다. **NL→Spec Layer를 Phase 2에 명시**했고, 흐름도 `자연어 입력 → LLM → Spec Markdown 초안 → 사람 리뷰/승인 → Git commit → SDD 검증`으로 적었습니다. 다만 이것은 아직 **개념 흐름 정의**이고, DSL 스키마나 변환 규칙, 충돌 머지 규칙까지는 아닙니다. ([GitHub][1])                                                                                        |
| 4 | 서비스 경계 불일치                                                       | **해결됨**   | v3는 이 문제를 구조적으로 줄였습니다. **Phase 1은 CLI + Git + Plumb만**, **Phase 2에서 API Server/Web/PostgreSQL/Redis 추가**로 경계를 시간축으로 분리했습니다. 또한 **멀티리포 대신 모노리포**로 시작해 초기 범위를 단순화했습니다. 적어도 v2에서 보이던 “현재 시점 서비스 수/역할 불일치”는 많이 정리됐습니다. ([GitHub][1])                                                                         |

## 새로운 이슈 (있다면)

* **Phase 1은 엄밀히 말해 “협업 플랫폼”이라기보다 “SDD CLI + 하네스 도구”에 가깝습니다.**
  v3 스스로도 Phase 1의 범위를 `init / sync / status` 3개 CLI와 Plumb 연동으로 한정하고, **협업 워크스페이스와 에이전트 오케스트레이션은 Phase 2**로 미뤘습니다. 따라서 “최종 비전”으로는 협업 플랫폼이 맞지만, **Phase 1 산출물의 제품 정체성은 플랫폼보다 개발자용 워크플로 도구**에 더 가깝습니다. 이건 결함이라기보다 **포지셔닝 리스크**입니다. 외부 설명에서는 “협업 플랫폼 MVP”보다 “개발자용 SDD 하네스 CLI”로 부르는 편이 더 정확합니다. ([GitHub][1]) |

* **계약 계층 문제를 Phase 2로 미루는 것은 ‘해결’이 아니라 ‘범위 축소를 통한 회피’에 가깝습니다.**
  이건 나쁜 선택은 아닙니다. 오히려 3개월 Go/Kill 실험에서는 합리적입니다. 다만 의미를 정확히 써야 합니다. v3는 OpenAPI를 Phase 2 선행 조건으로 적었지만, **Phase 1에서도 최소한의 내부 계약**—예를 들면 `.plumb/` 출력 형식, `decisions.jsonl` 필드, spec markdown 규약, hook 실패 코드—은 필요합니다. 지금 문서는 그 내부 계약이 충분히 명문화되진 않았습니다. ([GitHub][1]) |

* **모노리포 전환은 방향상 개선이지만, 기존 멀티리포 검토 문서와의 문서 정합성 충돌은 남습니다.**
  v3는 `pnpm workspace`와 `turbo.json`을 둔 **모노리포 구조**를 채택했고, 필요 시 **Phase 3에서 멀티리포 분리**를 검토하겠다고 했습니다. 이 선택 자체는 소규모 팀의 초기 실행성 측면에서 더 낫습니다. 다만 조직 안에 이전 멀티리포 기준 문서가 계속 살아 있으면, **아키텍처 기준점이 둘이 되는 운영 리스크**가 생깁니다. 따라서 v3를 채택한다면, 예전 멀티리포 문서는 반드시 “폐기/대체됨”으로 정리해야 합니다. ([GitHub][1]) |

* **Python subprocess 의존은 여전히 실행 리스크입니다.**
  v3는 이를 숨기지 않고, `pip install plumb-dev`와 **Track A/Track B 전략**을 명시했으며, Month 2에 전환 기준도 두었습니다. 이건 성숙한 대응입니다. 하지만 여전히 설치/버전/OS 차이, CI 환경 일관성, 성능 병목은 남습니다. 즉, 리스크는 **관리 가능해졌지만 제거되진 않았습니다.** ([GitHub][1]) |

* **Phase 1의 “Git pre-commit 훅 강제”는 사용자 채택 실패의 단일 장애점이 될 수 있습니다.**
  v3도 이를 알고 `--no-verify` 비율을 KPI로 잡았습니다. 좋은 신호입니다. 다만 훅이 느리거나 오탐이 많으면 CLI 자체보다 훅 회피 문화가 먼저 생길 수 있습니다. 이 경우 제품 가치 검증이 아니라 **개발자 마찰도만 측정하는 실험**이 될 위험이 있습니다. ([GitHub][1]) |

## 착수 판단

**판정: Conditional**

**Round 1 대비 변화: 개선됨**

**이유:**
v3는 Round 1의 핵심 문제 중 두 가지를 실질적으로 정리했습니다. 첫째, **Git SSOT와 자연어 협업의 충돌을 시간축 분리(Phase 1/2)**와 **human-in-the-loop**로 정리했습니다. 둘째, **서비스 경계와 범위를 MVP 수준으로 과감히 축소**해, “지금 무엇을 만드는가”가 훨씬 선명해졌습니다. ([GitHub][1])

다만 여전히 **Ready**는 아닙니다. 이유는 두 가지입니다.

1. **계약 계층이 아직 제품 계약으로 존재하지 않습니다.**
   v3는 OpenAPI를 Phase 2로 미뤘고, Canonical Model과 Spec DSL Schema는 여전히 문서상 실체가 없습니다. Phase 1에서도 내부 계약은 필요합니다. ([GitHub][1])

2. **Phase 1의 산출물은 플랫폼보다 도구에 가깝습니다.**
   그래서 기술 착수는 가능하지만, 조직 내 기대치는 낮춰야 합니다. “협업 플랫폼을 만든다”가 아니라 “3개월 안에 SDD 하네스 CLI의 채택성과 마찰도를 검증한다”로 정의해야 맞습니다. ([GitHub][1])

제가 내리는 최종 평가는 이렇습니다.

* **제품 비전**: 좋아졌음
* **MVP 정의**: 많이 좋아졌음
* **실행 가능성**: 좋아졌음
* **계약 완결성**: 아직 부족
* **따라서 판정은 Conditional 유지, 하지만 Round 1보다 한 단계 전진**

실제 착수 조건은 단 3개면 충분합니다.

* `.plumb` 출력과 `decisions.jsonl`의 **내부 계약 문서화**
* pre-commit 훅의 **최대 허용 시간**과 실패 처리 규칙 명시
* v3가 **이전 멀티리포 문서를 대체한다는 ADR 1건** 작성

이 3개가 채워지면, v3는 **소규모 파일럿 착수용으로는 충분히 타당**합니다.
