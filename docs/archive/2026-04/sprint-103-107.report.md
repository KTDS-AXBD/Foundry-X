## §6 핵심 알고리즘 요약

### 6.1 TF-IDF 시맨틱 검색 (F275)
- 필드 가중치: name ×3.0, description ×1.5, category ×1.0
- 토큰 정규화 + 불용어 제거
- skill_search_index 테이블에 사전 계산된 TF-IDF 벡터

### 6.2 안전성 6규칙 100점 감점제 (F275)
- prompt-injection(-20), external-url(-15), filesystem-access(-15), env-secrets(-20), code-execution(-20), infinite-loop(-10)
- Grade: A(90+), B(70+), C(50+), D(30+), F(0+)

### 6.3 Wilson Score Lower Bound (F276)
- 95% CI (z=1.96) 기반 통계적 신뢰도
- 샘플 수 적을수록 보수적 추정 → 과적합 방지

### 6.4 체이닝 패턴 감지 (F276)
- 30분 이내 연속 실행 스킬 쌍 감지 (`julianday` 비교)
- 크로스-아이템 집계 (biz_item_id를 GROUP BY에서 제외)

### 6.5 워크플로우 시퀀스 추출 (F277)
- workflow_executions × skill_executions JOIN
- 시퀀스 패턴 → 메타 스킬 후보 생성

### 6.6 Cold Start vs Warm Run 분류 (F278)
- `ROW_NUMBER() OVER (PARTITION BY skill_id ORDER BY executed_at ASC)`
- cold_threshold(기본 3) 이하 → Cold, 초과 → Warm
- savings_pct = (cold_avg - warm_avg) / cold_avg × 100

### 6.7 BD_ROI 공식 (F278)
```
BD_ROI = (Total_Savings + Signal_Value) / Total_Investment × 100

Total_Savings = Σ (cold_avg_cost - warm_avg_cost) × warm_executions
Signal_Value  = Go_count × $50K + Pivot_count × $10K
Total_Investment = Σ cost_usd from skill_executions
```

## §8 교훈 (Lessons Learned)

### 좋았던 점

1. **레이어 일관성**: 5개 Sprint 모두 동일한 D1→Type→Schema→Service→Route→Test 패턴 → Plan/Design 작성이 빠르고 구현 예측 정확
2. **의존성 체인 설계**: READ-only 참조로 Sprint 간 충돌 없이 병렬 실행 가능
3. **HITL 패턴 재사용**: F276과 F277 모두 extract→generate→review 3단계 워크플로우 → 코드 구조 유사하여 학습 곡선 최소화
4. **Wilson Score**: 통계적 신뢰도로 패턴 추출 품질 보장 → 과적합 방지
5. **BD_ROI 공식**: Warm Run 절감 + 신호등 달러 환산의 통합으로 단일 지표 제공

### 개선할 점

1. **Sprint 106 Plan/Design 부재**: Plan/Design 없이 직접 구현 → 갭 분석 기준이 Report 내 자체 비교에 의존. 향후 모든 Sprint에 Plan+Design 선행 권장
2. **체이닝 패턴 GROUP BY**: biz_item_id 포함 시 독립 그룹 문제 → 초기 Design에서 쿼리 레벨까지 명시 필요
3. **Shared 타입 파일 분리**: types.ts 인라인이 500행 이상 → 도메인별 분리 (types/skill.ts, types/roi.ts) 검토 필요
