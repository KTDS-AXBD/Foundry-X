## 1. Overview

### 1.1 Design Goals

1. **단일 보드 가시성** — KTDS-AXBD org에 통합 Projects 보드 1개, 프로젝트별 필터 뷰로 분리
2. **최소 마찰 프로세스** — 소규모 팀(1~3개 리포)에 맞는 가벼운 리뷰 체계
3. **Git SSOT 유지** — 모든 문서는 Git MD, GitHub Projects는 뷰/추적 레이어
4. **점진적 강화** — 초기에는 기본 규칙만, 팀 적응 후 단계적으로 강화

### 1.2 Design Constraints

| 제약 | 설명 |
|------|------|
| GitHub Free Plan | KTDS-AXBD org 기준, GitHub Free에서 사용 가능한 기능만 활용 |
| 소규모 팀 | 1~3개 리포, 소수 팀원 — 과도한 프로세스 지양 |
| HTTPS 인증 | PAT 기반 인증 (ktds.axbd@gmail.com) |
| 기존 체계 호환 | Foundry-X의 SPEC.md/PDCA 체계와 공존 |

## 2. GitHub Projects 보드 설계

### 2.1 보드 구성

**통합 보드 1개** (Org-level Project)

| 항목 | 값 |
|------|-----|
| **이름** | `AX BD팀 — 프로젝트 관리` |
| **위치** | KTDS-AXBD org → Projects |
| **범위** | org 내 모든 리포의 Issues 연결 가능 |
| **접근** | org 멤버 전원 읽기/쓰기 |

### 2.2 커스텀 필드 스키마

| 필드명 | 타입 | 값 | 용도 |
|--------|------|-----|------|
| `Status` | Single Select | Backlog / Sprint / In Review / Done | WBS 상태 추적 (기본 필드) |
| `Priority` | Single Select | 🔴 P0 / 🟠 P1 / 🟡 P2 / ⚪ P3 | 우선순위 |
| `Project` | Single Select | Foundry-X / {기타 리포명} | 프로젝트 분류 |
| `Sprint` | Iteration (2주) | v0.3, v0.4, ... | 스프린트/마일스톤 단위 |
| `Type` | Single Select | Feature / Bug / Improvement / Chore / Doc | 작업 유형 |
| `REQ Code` | Text | FX-REQ-021, ... | 요구사항 코드 추적 |
| `Deliverable` | Text | docs/01-plan/..., src/... | 산출물 경로 (Git 파일 경로) |
| `Assignee` | Assignees | (GitHub 사용자) | 담당자 |

### 2.3 뷰 설계

#### 2.3.1 Board 뷰 (기본) — "WBS 보드"

칸반 보드, `Status` 필드 기준 컬럼 분리:

```
┌─────────────┬─────────────┬─────────────┬─────────────┐
│   Backlog   │   Sprint    │  In Review  │    Done     │
│             │             │             │             │
│ [F20] P2    │ [F21] P0 🔴│ [F19] P1    │ [F12] ✅    │
│ non-TTY     │ PM 체계     │ eslint      │ ADR-000     │
│             │             │             │             │
│ [F18] P1    │ [F16] P1    │             │ [F13] ✅    │
│ sync TUI    │ status TUI  │             │ Plumb 계약  │
└─────────────┴─────────────┴─────────────┴─────────────┘
```

**필터**: `Project = Foundry-X` (프로젝트별 뷰 생성)

#### 2.3.2 Table 뷰 — "요구사항 목록"

| 제목 | REQ Code | Priority | Status | Sprint | Assignee | Deliverable |
|------|----------|----------|--------|--------|----------|-------------|
| (GitHub Issues 데이터) | | | | | | |

**그룹**: `Project` 기준 그룹핑
**정렬**: `Priority` ASC → `Status` 순

#### 2.3.3 Timeline 뷰 — "일정"

- **날짜 필드**: `Sprint` (Iteration) 기반 타임라인
- **그룹**: `Project` 기준
- **용도**: 마일스톤별 작업 일정 시각화

#### 2.3.4 프로젝트별 필터 뷰

각 리포별로 Board 뷰를 복제하고 `Project = {리포명}` 필터 적용:

| 뷰 이름 | 필터 | 타입 |
|---------|-------|------|
| `[Foundry-X] WBS` | Project = Foundry-X | Board |
| `[전체] 요구사항` | (없음) | Table |
| `[전체] 일정` | (없음) | Timeline |
| `[P0/P1] 긴급` | Priority in (P0, P1) | Board |

## 3. GitHub Issues 연동 설계

### 3.1 Issue 템플릿

각 리포의 `.github/ISSUE_TEMPLATE/` 에 생성:

#### feature-request.yml

```yaml
name: Feature Request
description: 새 기능 또는 개선 요청
labels: ["feature"]
body:
  - type: input
    id: req-code
    attributes:
      label: REQ 코드
      description: "SPEC.md의 요구사항 코드 (예: FX-REQ-021)"
      placeholder: "FX-REQ-XXX"
  - type: input
    id: f-item
    attributes:
      label: F-Item 번호
      description: "SPEC.md의 F번호 (예: F21)"
      placeholder: "F##"
  - type: dropdown
    id: priority
    attributes:
      label: 우선순위
      options:
        - "🔴 P0 (즉시)"
        - "🟠 P1 (이번 마일스톤)"
        - "🟡 P2 (다음)"
        - "⚪ P3 (백로그)"
  - type: dropdown
    id: type
    attributes:
      label: 유형
      options:
        - Feature
        - Bug
        - Improvement
        - Chore
        - Doc
  - type: textarea
    id: description
    attributes:
      label: 상세 설명
      placeholder: "구현 목표, 배경, 제약 사항 등"
  - type: input
    id: deliverable
    attributes:
      label: 예상 산출물 경로
      placeholder: "docs/01-plan/features/..., src/..."
```

#### bug-report.yml

```yaml
name: Bug Report
description: 버그 리포트
labels: ["bug"]
body:
  - type: textarea
    id: description
    attributes:
      label: 버그 설명
      placeholder: "무엇이 문제인가요?"
    validations:
      required: true
  - type: textarea
    id: reproduce
    attributes:
      label: 재현 방법
      placeholder: "1. ... 2. ... 3. ..."
  - type: textarea
    id: expected
    attributes:
      label: 기대 동작
  - type: dropdown
    id: priority
    attributes:
      label: 심각도
      options:
        - "🔴 Critical"
        - "🟠 High"
        - "🟡 Medium"
        - "⚪ Low"
```

### 3.2 SPEC.md F-item → Issue 매핑 규칙

| SPEC.md | GitHub Issue |
|---------|-------------|
| F번호 (F21) | Issue title prefix: `[F21]` |
| REQ코드 (FX-REQ-021) | Issue body의 REQ 코드 필드 |
| 상태 (📋/🔧/✅) | Projects Status 컬럼 |
| 버전 (v0.3) | Sprint iteration |
| Priority (P0~P3) | Priority 커스텀 필드 |

**Issue 제목 규칙**: `[F{N}] {제목}` (예: `[F21] 프로젝트 관리 점검 및 개선`)

**양방향 동기화 규칙**:
1. **새 F-item 등록 시**: SPEC.md에 추가 → Issue 생성 → Projects 보드에 자동 추가
2. **상태 변경 시**: Issue 상태 변경 → SPEC.md 수동 갱신 (또는 `/ax-10-req status`)
3. **완료 시**: Issue close → SPEC.md ✅ → Projects Done 컬럼

> ⚠️ **주의**: 완전 자동 동기화는 Phase 2(Foundry-X CLI/봇). 현재는 수동 + `/ax-10-req` 스킬 보조.

## 4. Branch Protection 규칙

### 4.1 대상 브랜치

| 브랜치 | Protection | 비고 |
|--------|:----------:|------|
| `main` / `master` | ✅ | 프로덕션/메인 브랜치 |
| `develop` | 선택 | 존재 시 적용 |
| `feature/*` | — | 자유 |

### 4.2 `main` / `master` Branch Protection 상세

```
Branch protection rule: main (또는 master)
─────────────────────────────────────────────

✅ Require a pull request before merging
   ├── ✅ Required approving reviews: 1
   ├── ❌ Dismiss stale pull request approvals (초기에는 미적용)
   ├── ❌ Require review from Code Owners (초기에는 미적용)
   └── ✅ Require approval of the most recent push

✅ Require status checks to pass before merging
   ├── ✅ Require branches to be up to date before merging
   └── Required checks: (프로젝트별 CI 존재 시)
       ├── typecheck (있을 때)
       ├── build (있을 때)
       └── test (있을 때)

❌ Require signed commits (초기에는 미적용)

✅ Require linear history (squash merge 권장)

❌ Require deployments to succeed

✅ Lock branch: ❌ (push는 PR을 통해서만)

✅ Do not allow bypassing the above settings
   └── 관리자도 동일 규칙 적용 (예외: 긴급 hotfix 시 임시 해제)
```

### 4.3 점진적 강화 로드맵

| 시기 | 추가 규칙 | 트리거 |
|------|----------|--------|
| 초기 (현재) | 위 기본 규칙 | 즉시 적용 |
| 1개월 후 | Dismiss stale approvals 활성화 | 팀 적응 완료 시 |
| 안정화 후 | Code Owners 파일 추가, 2명 리뷰 | 팀원 증가 또는 필요 시 |

## 6. 리뷰 프로세스 설계

### 6.1 리뷰 워크플로우

```
담당자                          리뷰어
  │                               │
  ├── feature branch 생성          │
  ├── 작업 수행                    │
  ├── PR 생성 (템플릿 기반)        │
  │   ├── 리뷰어 지정 ────────────→│
  │   └── 라벨 부착                │
  │                               ├── 코드/문서 리뷰
  │                               ├── 코멘트 작성
  │   ←────────────────── 수정 요청│
  ├── 수정 반영                    │
  ├── Re-request review ──────────→│
  │                               ├── Approve
  ├── Squash merge                 │
  └── Issue close (Done)           │
```

### 6.2 리뷰 기준 (산출물 유형별)

| 산출물 유형 | 리뷰 의무 | 리뷰어 수 | SLA (응답 기한) |
|------------|:--------:|:--------:|:--------------:|
| 코드 변경 (src/) | 필수 | 1명 | 1 영업일 |
| 핵심 문서 (SPEC, PRD, Blueprint) | 필수 | 1명 | 2 영업일 |
| PDCA 문서 (plan, design) | 필수 | 1명 | 1 영업일 |
| 설정 파일 (CI, 린트 등) | 선택 | 0~1명 | — |
| 기타 문서 (가이드, 참고 등) | 선택 | 0~1명 | — |

### 6.3 Merge 전략

| 전략 | 적용 | 이유 |
|------|------|------|
| **Squash and merge** | 기본 | 커밋 히스토리 깔끔 유지 |
| Merge commit | 대규모 feature | 커밋 이력 보존 필요 시 |
| Rebase and merge | 미사용 | 히스토리 복잡도 증가 |

GitHub 리포 설정에서:
- ✅ Allow squash merging (기본)
- ✅ Allow merge commits (선택적)
- ❌ Allow rebase merging (비활성)
- ✅ Automatically delete head branches

## 9. 구현 순서 (Do Phase 가이드)

### Step 1: Foundry-X 파일럿 — GitHub 설정

