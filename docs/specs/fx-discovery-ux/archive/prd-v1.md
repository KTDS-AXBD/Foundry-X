**버전:** v1

**작성자:** AX BD팀 (Sinclair Seo)
**상태:** 🔄 검토 중

## 1. 요약 (Executive Summary)

**한 줄 정의:**
AX BD팀이 사업 아이템 발굴 프로세스를 Foundry-X에서 직관적으로 시작하고, AI 개인 비서(Help Agent)의 안내를 받으며 단계별로 완료할 수 있는 UX를 구축한다.

**배경:**
Foundry-X는 76개 스킬 카탈로그, 프로세스 가이드(2-0~2-10), 사업성 신호등 등 발굴 관련 기능을 갖추고 있으나, 팀원들은 CC Cowork을 주력으로 사용 중이다. 핵심 원인은 **진입 장벽** — 메뉴가 나열형으로 과다하고, 어디서 시작해야 할지 불명확하며, 단계별 인터랙션이 없어 도구로서의 가치를 체감하지 못한다.

**목표:**
- 초보 사용자가 5분 안에 발굴 프로세스를 시작할 수 있는 직관적 경험
- AI Help Agent가 현재 컨텍스트를 인식하고 다음 단계를 안내하는 협력자 경험
- 각 단계에서 AI 산출물을 사용자가 검토·수정·승인하는 HITL 흐름

---

## 2. 문제 정의

### 2.1 현재 상태 (As-Is)
- 발굴 메뉴(사이드바)가 **나열형 10+ 항목**으로 구성 — 계층/흐름 없이 평면 배치
- 76개 스킬 카탈로그 페이지가 있으나 "어떤 스킬을 언제 써야 하는지" 안내 부재
- 프로세스 가이드(2-0~2-10) 텍스트 설명은 있으나, 실제 작업과 연결되지 않음
- 사업성 체크포인트/신호등은 구현되어 있으나, 단계별 자연스러운 흐름에 통합되지 않음
- **결과: CC Cowork이 주력 도구, Foundry-X 발굴 기능은 미사용**

### 2.2 목표 상태 (To-Be)
- 사이드바 → **단계별 흐름 뷰**: 현재 단계가 하이라이트되고, 다음 할 일이 명확
- **Help Agent(개인 비서)**: 챗 인터페이스로 "다음 뭐 하면 돼?", "이 스킬 어떻게 써?" 등 질문 가능
- **HITL 결과물 패널**: AI 산출물(BMC 초안, 분석 결과 등)을 인라인으로 검토·수정·승인
- **온보딩**: 처음 방문 시 3~5 스텝 가이드로 첫 발굴 아이템 생성까지 안내

### 2.3 시급성
- AX BD팀 7명 대상 온보딩 완료(Sprint 87~88) 후 **실사용 전환이 안 되고 있음**
- 실사용 없이 추가 기능 개발은 의미 없음 — UX 개선이 선행되어야 모든 기존 기능의 가치 실현
- 2주 내 MVP로 팀 내부 데모 → 실사용 전환 목표

## 4. 기능 범위

### 4.1 핵심 기능 (Must Have)

| # | 기능 | 설명 | 우선순위 |
|---|------|------|----------|
| 1 | **발굴 프로세스 단계별 안내 UI** | 현재 2-0~2-10 나열형 메뉴를 **위저드/스텝퍼** 형태로 재구성. 각 단계에서 (a) 이 단계의 목적, (b) 사용할 스킬/도구, (c) 예상 산출물, (d) 다음 단계 안내를 표시. biz-item별 현재 진행 단계 자동 추적 | P0 |
| 2 | **Help Agent (개인 비서)** | 페이지 하단 또는 사이드패널 챗 UI. OpenRouter 경유 LLM API + SSE 스트리밍. 컨텍스트: 현재 biz-item + 현재 단계 + 사용자 역할. 질문 예시: "다음 뭐 해야 돼?", "BMC가 뭐야?", "이 스킬 어떻게 써?" | P0 |
| 3 | **온보딩 투어 개선** | 발굴 페이지 첫 방문 시 **인터랙티브 투어** (3~5 스텝): (1) 아이템 등록, (2) 유형 분류, (3) 첫 분석 실행, (4) 결과 확인, (5) 다음 단계 안내. 기존 온보딩(F252)과 연계하되 발굴 특화 | P0 |
| 4 | **HITL 인터랙션 + 결과물 확인** | 스킬 실행(F260) 결과물을 인라인 패널에서 표시. 사용자 액션: ✅ 승인 / ✏️ 수정 / 🔄 재생성 / ❌ 거부. 승인된 산출물은 다음 단계 입력으로 자동 연결 | P0 |

### 4.2 부가 기능 (Should Have)

| # | 기능 | 설명 | 우선순위 |
|---|------|------|----------|
| 1 | 프로세스 커스터마이징 | 단계별 Depth 조절 (핵심/보통/간소), 사용자별 선호 스킬 저장 | P1 |
| 2 | 사용 패턴 대시보드 | 팀원별 발굴 진행 현황, 단계별 소요 시간, 완료율 시각화 | P1 |
| 3 | Help Agent 컨텍스트 메모리 | 이전 대화 히스토리 기반 추천 개선 (세션 간 연속성) | P2 |

### 4.3 제외 범위 (Out of Scope)
- **외부 연동**: Discovery-X API 연동, 외부 시장 데이터 API 실시간 연동 (2차)
- **자동 프로세스 조절**: 사용 패턴 기반 AI 자동 Depth 조절 (2차)
- **모바일 최적화**: PC 웹 우선, 모바일 레이아웃은 이후

### 4.4 외부 연동
| 시스템 | 연동 방식 | 필수 여부 |
|--------|-----------|-----------|
| OpenRouter | REST API + SSE | 필수 (Help Agent LLM) |
| Foundry-X API (Workers) | 기존 Hono API | 필수 (biz-item, 스킬 실행 등) |
| D1 Database | 기존 D1 | 필수 (대화 이력, 산출물 저장) |

## 6. 제약 조건

### 6.1 일정
- 목표 완료일: 2026-04-14 (2주, ~Sprint 92~97)
- 마일스톤 1 (1주): 단계별 안내 UI + 온보딩 투어
- 마일스톤 2 (2주): Help Agent + HITL 인터랙션

### 6.2 기술 스택
- 프론트엔드: Vite 8 + React 18 + React Router 7 (기존)
- 백엔드: Hono on Cloudflare Workers (기존)
- LLM: OpenRouter API (SSE 스트리밍)
- DB: Cloudflare D1 (대화 이력, 산출물 메타데이터)
- 기존 의존: F258~F262 (BD 스킬 통합), F57 (SSE 인프라), F252 (온보딩 시스템)

### 6.3 인력/예산
- 투입: 1명 (Sinclair) + Claude Code
- LLM 비용: OpenRouter pay-per-use (팀 7명 규모, 월 예상 $10~30)

### 6.4 컴플라이언스
- 사내 데이터(사업 아이템 정보)가 외부 LLM에 전송됨 — OpenRouter 경유 시 데이터 처리 정책 확인 필요
- KT DS 내부 정책: 현재 CC Cowork도 외부 LLM 사용 중이므로 동일 수준
