## 기술 검토 의견: fx-discovery-ux PRD

### 1. **기술적 실현 가능성**
**기본적으로 실현 가능하나 몇 가지 중요한 제약이 존재합니다.**

- **긍정적 요소**:
  - 기존 기술 스택(Vite/React/Hono/D1)과의 연속성이 좋아 개발 부담이 상대적으로 낮음
  - OpenRouter API + SSE 스트리밍은 검증된 패턴으로 Help Agent 구현에 적합
  - F57(SSE 인프라), F252(온보딩) 등 기존 모듈 재사용 가능

- **주요 기술적 제약/병목**:
  1. **컨텍스트 관리의 복잡성**: Help Agent가 "현재 biz-item + 현재 단계 + 사용자 역할"을 실시간으로 인식해야 함. 이는 세션 상태를 서버와 클라이언트 간에 정확히 동기화해야 하며, 다중 탭 환경에서 일관성 유지가 어려울 수 있음.
  2. **LLM 응답 품질 의존성**: OpenRouter를 통한 모델 성능이 CC Cowork 대비 현저히 낮을 경우 전체 UX가 실패할 수 있음. 프롬프트 엔지니어링과 컨텍스트 주입 전략이 매우 중요.
  3. **데이터 전송 보안**: 사내 데이터(사업 아이템)가 OpenRouter로 전송되는데, 이는 PRD에서 언급된 대로 컴플라이언스 리스크. 암호화나 익명화 처리가 반드시 필요.

### 2. **아키텍처 적합성**
**현재 제안된 구조는 MVP에는 적합하나 확장성 관점에서 재고가 필요합니다.**

- **문제점**:
  1. **단일 의존성 리스크**: Help Agent가 OpenRouter에 완전히 의존. API 장애 시 전체 기능 마비. 폴백 메커니즘(로컬 규칙 기반 응답)이 필요.
  2. **상태 관리 복잡성**: 클라이언트(React 상태) + 서버(D1) + SSE 스트림 간의 상태 동기화가 복잡해질 수 있음. 특히 HITL 승인/수정 시 여러 시스템에 걸친 데이터 일관성 보장이 어려움.
  3. **데이터베이스 설계**: 대화 이력, 산출물 메타데이터, 프로세스 단계 추적을 D1에 저장할 경우 스키마 설계가 중요. 현재 PRD에는 데이터 모델이 명시되지 않아 추후 마이그레이션 부담 가능성.

- **개선 제안**:
  - **이벤트 소싱 패턴 도입**: 사용자 상호작용(단계 이동, 승인, 수정)을 이벤트로 기록하여 상태 복원과 감사 로그를 용이하게 함.
  - **커맨드/쿼리 분리(CQRS)**: Help Agent 쿼리와 프로세스 상태 업데이트를 분리하여 성능과 확장성 개선.
  - **로컬 캐시 계층**: 자주 사용되는 프로세스 가이드, 스킬 정보를 클라이언트에 캐싱하여 LLM 의존성 감소.

### 3. **구현 복잡도**
**과소평가된 기술적 난이도가 다수 존재합니다.**

- **숨겨진 복잡도**:
  1. **동시성 문제**: 동일 biz-item을 여러 사용자가 동시에 수정할 경우(예: BD와 팀장) 데이터 충돌 발생 가능. 낙관적/비관적 동시성 제어 전략 필요.
  2. **에지 케이스 처리**:
     - 사용자가 단계를 건너뛰거나 역행할 때의 상태 관리
     - Help Agent가 잘못된 컨텍스트를 인식했을 때의 복구 메커니즘
     - SSE 연결 끊김 후 재연결 시 상태 동기화
  3. **성능 고려사항**: 76개 스킬 카탈로그 + 프로세스 가이드를 컨텍스트로 주입할 경우 토큰 수가 급증하여 비용 증가와 응답 지연 초래. 스킬 정보를 벡터 DB에 임베딩하여 유사도 기반 동적 주입 고려.
  4. **온보딩 투어와 기존 시스템 통합**: F252(기존 온보딩)와의 충돌 가능성. 투어 상태를 영구 저장할지 세션 기반으로 할지 명확하지 않음.

### 4. **기술 리스크**
**가장 크게 우려되는 요소 3가지:**

1. **LLM 품질 및 일관성 리스크**(높음): OpenRouter를 통한 모델 응답이 CC Cowork 대비 낮을 경우 사용자 이탈 불가피. 프롬프트 튜닝과 평가에 상당한 시간 투자 필요.
2. **데이터 동기화 실패 리스크**(중간): 클라이언트 상태, 서버 DB, SSE 스트림 간 불일치 발생 시 사용자 경험 심각하게 저하. 특히 HITL 승인 후 다음 단계로 자동 연결 실패하면 전체 흐름이 끊김.
3. **보안/컴플라이언스 리스크**(중간): 사내 데이터 외부 전송에 대한 명확한 정책 부재. GDPR 또는 내부 규정 위반 가능성.
