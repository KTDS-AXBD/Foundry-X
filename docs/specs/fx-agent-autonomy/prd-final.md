## 1. 요약 (Executive Summary)

**한 줄 정의:**
Sprint autopilot이 생성한 Feature의 E2E 테스트를 자동 생성하고, Gap Analysis에 E2E 결과를 통합하여 "코드 구조 일치 + 실제 동작 검증"을 하나의 품질 점수로 보장하는 시스템.

**배경:**
Foundry-X는 Phase 39까지 39개 Phase, 523+ F-items를 AI 에이전트 기반 autopilot으로 개발해왔다. Gap Analysis Match Rate(Design↔Code 일치율)가 90%+ 기준으로 품질을 판단하지만, 이는 "코드 구조가 설계와 일치하는가"만 검증할 뿐 "실제로 브라우저에서 동작하는가"는 보장하지 않는다. 현재 E2E 273건은 수동으로 작성/유지되며, 새 Feature 추가 시 자동 확장되지 않아 커버리지 격차가 누적되고 있다.

<!-- R1: 시장 트렌드와 기술 배경, Shift-Left, DevOps, AI-Driven 개발 트렌드 명시 -->
**시장 및 기술 트렌드 배경:**
- **AI 기반 개발 및 코드 자동화:** GitHub Copilot, Google Gemini Code Assistant 등 AI 기반 코드 생성 도구가 보편화됨에 따라, 생성 코드의 품질 검증 및 유지보수 자동화의 필요성이 커지고 있음.
- **Shift-Left Testing:** 개발 초기 단계에서 품질 확보를 강조하는 Shift-Left Testing 패러다임의 부상. 본 시스템은 Design 문서에서 E2E를 추출하여 Sprint 단계에서 자동 실행함으로써 조기 품질 보증을 실현.
- **DevOps 및 CI/CD 자동화:** 테스트 자동화와 품질 지표 통합이 DevOps 성숙도에 핵심 역할을 함.
- **로우코드/노코드 및 멀티모달 개발 확산:** 비전문가 개발 환경 및 텍스트·시각 정보 혼합 문서 등장으로 자동 품질 검증 시스템의 중요성 증가.

**목표:**
Feature 개발 시 Design 문서에서 E2E 시나리오를 자동 추출하고, Gap Analysis에 E2E 결과를 포함하여 종합 품질 점수를 산출함으로써, autopilot의 품질 보장을 "구조 일치"에서 "동작 검증"까지 확장한다.

<!-- R1: AI 에이전트 자율성 및 피드백 루프 강조 -->
**AI 에이전트 자율성 강조:**
AI 에이전트(Sprint autopilot)가 단순 코드 생성 단계를 넘어서, 스스로 품질 검증 및 개선 피드백 루프를 완성하는 핵심 요소로 본 시스템이 역할함.

## 2. 문제 정의

### 2.1 현재 상태 (As-Is)
- Gap Analysis는 Design 문서의 파일 매핑 vs 실제 코드 존재 여부만 비교 (구조적 일치)
- E2E 테스트는 수동으로 작성하며, `/ax:e2e-audit`가 커버리지 갭을 감지하지만 자동 생성하지 않음
- Sprint autopilot 완료 후 E2E 추가는 별도 수동 작업으로, 종종 누락됨
- Gap 97% PASS → 프로덕션 배포 → E2E 미커버 영역에서 리그레션 발생 가능

<!-- R1: Design 문서 포맷 다양성 및 파싱 난이도, 오류 추적/디버깅의 어려움 언급 -->
- Design 문서 §4 기능 명세 포맷이 Phase마다 다르며, E2E 추출에 필요한 정보(사용자 액션, 예상 결과 등)가 불충분한 경우가 많음. 파싱 전략 수립 및 문서 표준화 필요.
- 자동 생성된 E2E에서 오류 발생 시 디버깅, 오류 원인 추적이 복잡하여, 개발자의 생산성 저해 우려.

### 2.2 목표 상태 (To-Be)
- Design 문서의 §5 파일 매핑 + §4 기능 명세에서 E2E 시나리오를 자동 추출
- Sprint autopilot의 Verify 단계에서 E2E 생성 + 실행이 자동 포함
- Gap Analysis 보고서에 E2E PASS/FAIL 결과가 반영되어 종합 Match Rate 산출
- 새 Feature = 자동으로 E2E 커버리지 확장
<!-- R1: 자동 생성 E2E의 assertion depth, 실패 시 롤백/수동 개입, 오류 리포팅 프로세스 명시 -->
- 자동 생성 E2E는 smoke 수준(기본 동작 확인)에서 functional 수준(주요 시나리오 검증)까지 assertion depth를 명확하게 정의하여, 생성 품질을 보장함.
- 자동 생성된 E2E 실패 또는 Flaky 다발 시, 수동 개입 및 롤백(재생성, 커밋 보류 등) 플로우를 운영하며, 실패 원인 자동 리포팅 프로세스 도입.

### 2.3 시급성
- Phase 39 완료로 10개 패키지, 11 routes, 30 services 규모 — 수동 E2E 관리의 한계점 도달
- MSA 분리(fx-gateway, fx-discovery)로 크로스 서비스 E2E 필요성 증가
- 누적 기능이 많아질수록 리그레션 리스크 가속

### 5.2 MVP 최소 기준
- [ ] Design 문서에서 E2E 시나리오 3개 이상 자동 추출 (PoC 성공률 70% 이상)
- [ ] 추출된 시나리오로 Playwright spec 파일 생성 → 실행 → PASS
- [ ] Gap Analysis 보고서에 E2E 결과 행이 포함됨
- [ ] 자동 생성 E2E의 assertion이 smoke 수준 이상이며, 실패 시 원인 리포트가 자동 생성됨

### 5.3 실패/중단 조건
- Design 문서 포맷이 너무 다양해서 파싱 신뢰도 60% 미만인 경우

## 8. 자동 생성 E2E의 검증 및 유지보수 프로세스

<!-- R1: 자동 생성 E2E 검증·유지보수, 실패 시 롤백/수동 개입, 유지보수 전략, 오류 리포팅, flaky 관리 등 신규 섹션 추가 -->
### 8.1 검증 프로세스
- 자동 생성된 E2E는 smoke 및 주요 functional 시나리오에 대해 자동 assertion depth 평가를 거침.
- 최초 생성 후 3회 반복 실행하여 flaky성 여부 및 오류 발생 시 자동 리포트 생성.
- 실패 또는 미흡시 자동 롤백(해당 E2E 커밋 보류), 운영자는 오류 리포트 확인 후 수동 개입(재생성, 수정, 무시 등) 선택.

### 8.2 유지보수 전략
- Design 문서 포맷 변화 감지 시, 파싱 모듈 자동 업데이트(버전 관리).
- 자동 생성 E2E의 실패/플레이크 이력 데이터베이스화, 반복 실패 시 자동 알림 및 유지보수 우선순위 상향.
- E2E 생성/실패/수정 내역은 모두 로그로 추적·분석되어, 향후 AI 에이전트의 개선 입력 데이터로 활용.

### 8.3 오류 원인 추적 및 리포팅
- 실패한 E2E는 로그, 스택트레이스, 실행환경 캡처 등과 함께 자동 리포트 생성.
- 리포트 결과는 Gap Analysis 보고서에 연동되며, 운영자가 원클릭으로 재실행/수정/무시를 선택 가능.

### 8.4 테스트 환경 및 데이터 관리
- 각 E2E 실행 전후, 테스트 데이터베이스 및 환경 변수를 스냅샷 또는 초기화하여 테스트 간 간섭 방지.
- 테스트 컨테이너 기반 격리 실행, 필요 시 mock service 자동 구성.

### 8.5 Flaky Test 관리
- 자동 생성 E2E는 실행당 최소 3회 반복하여 flaky 여부 판단, flaky 발생 시 자동 알림 및 재생성/수정 플로우 실행.
- flaky 비율 10% 이상이면 자동 생성 기능 일시 중단, 진단 리포트 작성 필수.
