## 검토 의견: PRD → Prototype 자동 생성 & 배포 (v2)

### 1. 기술적 실현 가능성: **부분적 가능 (중요한 제약 존재)**

**긍정적 요소:**
- 기존 Foundry-X 인프라(Cloudflare Workers, D1, API 레이어) 활용은 현명함
- Claude Code CLI를 구독 계정으로 실행하여 API 비용 제로 접근법은 경제적
- Dry-run으로 O-G-D 루프 기본 동작 검증은 실증적 접근

**중요한 기술적 제약/병목:**

**🔴 CLI 운영 환경의 불확실성**
- **핵심 리스크**: Anthropic의 Claude Code CLI가 **프로덕션 서버/자동화 환경에서의 사용을 공식적으로 허용하는지 명확하지 않음**. 현재는 개발자 도구로 제공되며, 대규모 자동화에 대한 사용 제한이나 정책 변경 가능성이 높음.
- **Rate Limit 미확인**: Max 구독 계정의 시간당/일일 호출 제한이 공개되지 않아 용량 계획 불가능. 30턴/건 × 다수 Job 시 빠르게 제한에 도달할 수 있음.

**🔴 리소스 격리 부재로 인한 보안/안정성 위험**
- 단일 서버/디렉토리에서 모든 Job 실행 시:
  - **파일 충돌**: 동시성 관리 실패 시 작업 디렉토리 오염
  - **보안 취약**: 한 Job의 악성 코드가 전체 서버 파일 시스템 접근 가능
  - **개인정보 노출**: PRD에 포함된 민감정보가 다른 Job에 노출될 수 있음
- Docker 컨테이너 격리가 "향후 확장"으로 언급되었으나, **운영 시작 전 필수 요건**으로 승격 필요

**🔴 상태 관리의 취약성**
- Polling(30초) + 단순 status 필드 방식은:
  - **분산 트랜잭션 문제**: Builder Server 장애 시 Job이 영구 "building" 상태 고착
  - **재시도/복구 메커니즘 부재**: 빌드 타임아웃 시 자동 롤백 없음
  - **Dead-letter queue 미구성**: 반복 실패 Job이 큐를 막을 수 있음

**🔴 생성 품질 검증의 표면성**
- "빌드 성공"만으로 품질 판단은 위험:
  - 기능 정합성 미검증 (PRD 요구사항 실제 구현 여부)
  - UI/UX, 접근성, 모바일 대응 등 사용성 요소 전무
  - Dry-run에서도 PRD 정합성 문제(기능 누락, 항목 불일치) 확인됨

### 2. 아키텍처 적합성: **개선 필요**

**현재 아키텍처의 문제점:**

**🟡 단일 장애점(Single Point of Failure) 구조**
- Prototype Builder Server가 단일 인스턴스로 설계됨
- 해당 서버 장애 시 전체 Prototype 생성 파이프라인 중단
- **제안**: Active-Standby 구성 또는 무상태 워커 + 중앙 큐 패턴으로 재설계

**🟡 강한 결합(Tight Coupling)**
- Foundry-X API ↔ Builder Server가 직접 통신(polling)
- 변경/확장 시 양쪽 모두 수정 필요
- **제안**: 메시지 큐(RabbitMQ, Redis Queue) 도입으로 느슨한 결합

**🟡 데이터 일관성 보장 미흡**
- Job 상태가 DB(status) ↔ 실제 실행 상태 불일치 가능성 높음
- 장애 복구 시 정합성 복구 메커니즘 없음
- **제안**: 상태 머신(State Machine) 패턴 도입 + 보상 트랜잭션

**대체 아키텍처 제안:**

```
┌─────────────┐    ┌──────────────┐    ┌─────────────────────┐
│ Foundry-X   │───▶│   Job Queue  │───▶│  Worker Pool        │
│   API       │    │  (Redis)     │    │ (Docker Containers) │
└─────────────┘    └──────────────┘    │  • 각 Job 독립 실행  │
        ▲                              │  • 리소스 격리 보장  │
        │                              └─────────────────────┘
        │                                      │
        │                              ┌─────────────────────┐
        └──────────────────────────────│  Result Aggregator  │
                                       │  • 상태 동기화       │
                                       │  • 재시도 관리       │
                                       └─────────────────────┘
```

### 3. 구현 복잡도: **현저히 과소평가됨**

**숨겨진 복잡성 요소:**

**🔴 Claude Code CLI의 프로덕션 운영 복잡성**
- CLI 업데이트 관리: 자동 업데이트 시 호환성 문제
- 인증 토큰 갱신: 구독 계정 세션 만료 시 자동 갱신 필요
- 로그/모니터링: subprocess 실행의 상세 로그 수집과 분석

**🔴 에지 케이스 처리**
- **빌드 무한 루프**: Claude Code가 수정-빌드 실패 사이클에 빠질 경우
- **대형 PRD 처리**: 1000줄 이상 PRD의 컨텍스트 한계(Claude 토큰 제한)
- **네트워크 불안정**: 배포 중 Cloudflare API 실패 시 부분 배포 상태 처리

**🔴 동시성 관리의 실제 난이도**
- 단순 "순차 실행"으로는 실제 운영 요구사항 충족 불가:
  - 우선순위 Job 처리 (긴급 데모)
  - 장시간 Job 선점 시 다른 Job 대기 시간 과다
  - 리소스 사용량 모니터링과 스로틀링

**🔴 템플릿/프롬프트 유지보수 부담**
- React, Vite, Tailwind, shadcn/ui 버전 업데이트 시:
  - 템플릿 정기적 업데이트 필요
  - 프롬프트 최적화 지속적 실험 필요
  - 하위 호환성 보장 어려움

### 4. 기술 리스크: **상당히 높음**

**가장 우려되는 실패 요인:**

1. **CLI 정책 변경 리스크 (가장 치명적)**
   - Anthropic이 서버 자동화용 CLI 사용 금지 시 전체 시스템 무용지물
   - Fallback으로 Claude API 전환 시 비용 급증($0 → $2-5/건)

2. **생성 품질 불확실성으로 인한 신뢰도 상실**
   - BD팀이 생성된 Prototype을 실제 데모에 사용할 수 없는 수준
   - 품질 편차가 커서 일관된 결과 기대 어려움

3. **보안 사고 가능성**
   - PRD에 포함된 고객 민감정보가 생성된 Prototype에 그대로 노출
   - 악성 코드 삽입 가능성(현재 템플릿 검증 없음)

4. **운영 부담 과다**
   - 서버 모니터링, 장애 대응, 로그 분석에 상시 인력 필요
   - 예상보다 높은 유지보수 비용 발생
