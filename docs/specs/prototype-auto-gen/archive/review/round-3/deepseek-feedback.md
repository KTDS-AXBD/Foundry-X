## 검토 의견: PRD → Prototype 자동 생성 & 배포 (v3)

### 1. **기술적 실현 가능성**
**부분적으로 가능하나 중대한 제약 존재**

**긍정적 요소:**
- Dry-Run으로 핵심 기술(Claude Code CLI)의 기본 동작은 검증됨.
- Cloudflare 스택(Workers, Pages, D1)은 이 시스템의 요구사항(서버리스, 빠른 배포)에 잘 부합.
- Docker 격리 실행은 보안과 환경 일관성을 보장하는 올바른 접근.

**주요 기술적 제약 및 병목:**
1. **Claude Code CLI의 서버 자동화 정책 불확실성 (치명적 리스크):**
   - `--dangerously-skip-permissions` 플래그에 의존하는 설계는 근본적으로 취약. Anthropic이 언제든 이 기능을 제한하거나 CLI의 무허가 자동화를 금지할 수 있음.
   - **병목:** 모든 생성 파이프라인이 단일 벤더(Anthropic)의 비공식 인터페이스에 종속됨. 이는 시스템의 존재 근간을 위협.

2. **생성 품질과 안정성의 편차 예상:**
   - PRD의 구조화 수준에 따라 생성 결과가 극심하게 달라질 것. 비개발자(BD팀)가 작성한 자연어 PRD를 일관되게 파싱하는 것은 NLP 자체의 난제.
   - **병목:** "기능 체크리스트 전처리"가 핵심이지만, 이를 구현하는 것 자체가 복잡한 AI 문제임. 실패 시 전체 파이프라인이 무너짐.

3. **빌드/배파이프라인의 복잡성 과소평가:**
   - `npm run build` 실패 시 "자동 수정"은 현실적으로 매우 제한적. TypeScript 에러, 의존성 충돌, Tailwind CSS 클래스 오류 등을 자동으로 고치는 것은 간단한 lint fix 이상의 난이도.
   - **병목:** 빌드 실패 → 자동 수정 루프가 예상보다 많이 발생하면 전체 처리 시간이 10분을 쉽게 초과.

### 2. **아키텍처 적합성**
**전반적으로 합리적이나, 단일 장애점(SPOF)과 결합도 문제가 있음**

**개선이 필요한 부분:**
1. **Builder Server의 SPOF (Single Point of Failure):**
   - 모든 생성 작업을 처리하는 단일 Node.js 서버는 명백한 SPOF. 이 서버가 다운되면 전체 서비스 중단.
   - **제안:** 최소한 Worker Queue(예: Cloudflare Queues)를 도입해 작업 수신/상태 관리와 실제 실행기(Builder)를 분리. Builder는 Stateless하게 여러 개 증설 가능하도록 설계 변경을 고려.

2. **과도한 결합도:**
   - Orchestrator, Generator, Discriminator, 빌드, 배포 로직이 하나의 모놀리식 프로세스(Docker 컨테이너 내)에 강하게 결합됨. 한 단계 실패 시 전체 재시작해야 함.
   - **제안:** 각 단계를 독립적인 서비스 또는 함수로 분리(예: Generator Service, Validator Service). 이렇게 하면 단계별 확장, 개별 재시도, 기술 스택 변경이 용이해짐.

3. **상태 관리의 복잡성:**
   - D1을 사용한 Job 상태 관리와 실시간 로그(SSE) 스트리밍을 동시에 구현하는 것은 워커 환경에서 추가적인 복잡도를 유발할 수 있음.
   - **제안:** Job 상태는 D1으로 관리하되, 실시간 로그는 더 간단한 메커니즘(예: Cloudflare Durable Objects의 스트리밍 또는 Polling 기반 로그 조회)으로 우선 구현하는 것을 권장.

### 3. **구현 복잡도**
**현저히 과소평가됨**

1. **O-G-D 루프의 숨겨진 복잡도:**
   - "max 3 rounds" 내에 품질 점수 0.85 이상으로 수렴한다는 가정은 매우 낙관적. Discriminator의 평가 기준을 설정하고, 평가 결과를 다음 Generator 프롬프트에 효과적으로 반영하는 피드백 루프 설계는 미해결된 AI 엔지니어링 문제에 가까움.
   - **Edge Case:** Discriminator가 일관되지 않은 평가를 하거나, 개선 불가능한 Critical 이슈를 지적할 경우 루프가 무한 반복되거나 저품질 결과로 수렴할 수 있음.

2. **동시성 및 리소스 관리:**
   - 단일 Builder Server에서 여러 Docker 컨테이너를 동시에 실행할 때의 리소스(CPU, 메모리, 디스크 I/O) 경합이 고려되지 않음. 메모리 부족으로 인한 전체 서버 다운 가능성 있음.
   - **제안:** 동시 실행 Job 수를 엄격히 제한하고, 큐잉 시스템을 도입해 부하를 관리해야 함.

3. **에러 처리 및 복구의 복잡성:**
   - "dead_letter → manual_review"로 간단히 기술되었지만, 실패 원인 진단(CLI 실패? 빌드 실패? 네트워크 타임아웃?), 부분 롤백(Cloudflare Pages에 반쯤 배포된 프로젝트 정리), 재시도 정책 구현은 상당한 개발 비용이 듬.

### 4. **기술 리스크**
**가장 크게 우려되는 요인 3가지:**

1. **벤더 종속성 및 정책 리스크 (최고 위험도):**
   - 시스템의 핵심(Claude Code CLI)이 비공식 인터페이스이며, Anthropic의 정책 변경에 완전히 노출됨. 이는 **프로젝트 단일 최대 실패 요인**.

2. **생성 품질 불확실성으로 인한 사용자 기대치 관리 실패:**
   - BD팀(비개발자)은 "동작하는 데모"를 기대하지만, AI가 생성한 코드는 논리적 결함, 불완전한 상호작용, 부자연스러운 데이터 흐름을 가질 가능성이 높음. 이로 인해 도구에 대한 신뢰도가 급격히 떨어질 수 있음.

3. **운영 복잡성 과소평가:**
   - Builder Server, Docker 데몬, Cloudflare 연동, AI API 호출을 모니터링하고 유지보수하는 운영 부담이 PRD에 비해 크게 저평가됨. 이는 장기적인 기술 부채로 이어질 수 있음.

### 5. **착수 판단: Conditional**

**Conditional (조건부 승인) 이유:**
이 프로젝트는 기술적으로 매력적이고 비즈니스 가치가 높지만, **핵심 종속성인 Claude Code CLI의 서버 자동화 공식 허용 여부가 확인되지 않은 상태에서는 착수하기에 위험성이 너무 큽니다.**

**착수를 위한 선결 조건:**
1.  **Anthropic 공식 확인 필수:** BD팀은 Phase A 착수 전에 반드시 Anthropic에 공식 채널로 서버 측 자동화(헤드리스 실행) 및 `--dangerously-skip-permissions` 플래그의 장기 사용 가능성을 **문서로 확인**해야 합니다.
2.  **Fallback 전략의 실질적 검증:** CLI 사용이 불가능해질 경우를 대비한 Fallback 1(Claude API) 경로에 대한 **비용 추정과 PoC(Proof of Concept)가 선행**되어야 합니다. 현재 예상 비용($2-5/건)으로는 운영이 지속 가능한지 불분명합니다.
3.  **MVP 범위 재조정:** Phase A의 목표를 "E2E 자동화"에서 **"위험성 검증 PoC"** 로 축소하는 것을 강력히 권고합니다.
    -   목표: 단일 PRD로 CLI를 통한 코드 생성 → 로컬 빌드 성공까지를 자동화 스크립트로 확인.
    -   이 단계에서 생성 품질, 안정성, 실행 시간을 평가한 후, 본격적인 파이프라인 개발(Builder Server, 배포 등)에 대한 Go/No-Go를 결정해야 합니다.

**결론적으로, 기술적 야심은 인정하지만, 근본적인 불확실성(벤더 정책)을 해소하지 않고 모든 Phase를 진행하는 것은 큰 기술적 및 비즈니스적 리스크를 초래할 수 있습니다. 선행 조건이 충족된다면, 재설계된 범위와 리스크 인지 하에 착수할 수 있습니다.**
