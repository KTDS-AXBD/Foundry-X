# AI Foundry × BeSir 아키텍처 정합성 분석 v1

> **본 문서의 위상**
>
> 2026-04-29 BeSir(서버키트) × KT DS AX컨설팅팀 미팅에서 공유된 BeSir 솔루션 아키텍처와, 사용자가 진행 중인 **AI Foundry 정의서 v0.2** 간의 기술 정합성을 분석하는 PM 운영 문서입니다. **사업적 플랫폼 정의는 의도적으로 후순위로 두고, 기술 아키텍처·로드맵 정합성에만 집중**합니다.
>
> 5월 2주차 BeSir 차기 미팅 전에 정의서 v0.3 변경 안건을 확정하는 것이 본 문서의 직접 목적입니다.

## 0. 한 페이지 요약

### 0.1 BeSir 솔루션의 본질 (3줄)

- 온톨로지 = "사람을 복제하는 시멘틱 레이어" (현행만, 워크플로우+용어+룰 포함)
- Graph/Vector DB **의도적 미사용** — 메타는 파일(Git), 인스턴스는 PostgreSQL, 데이터 원본은 직결
- CQ(Critical Question) 5축 100점 + 90점 핸드오프로 검증, AI 80-20-80 + 사람 최종

### 0.2 AI Foundry v0.2 vs BeSir — 한 줄 진단

| AI Foundry v0.2 입장 | BeSir 입장 | 우선 처리 |
|---|---|---|
| Graph DB(Neo4j) 명시 | Graph DB 명시 거부 | **§3.4 재정의 P0** |
| 12 노드 타입 | 7 타입 (Fact·Dimension·Workflow·Event·Actor·Policy·Support) | 재매핑 P0 |
| 4-Asset Model (Policy/Ontology/Skill/Log) | 5번째 자산: System Knowledge (암묵지 파일) | 자산 모델 보강 P0 |
| 4대 진단(Missing/Duplicate/Overspec/Inconsistency) | CQ 5축 평가 | **공존 검토 P1** |
| HITL Reviewer (모호) | 80-20-80 + 사람 최종 (구체) | HITL 흐름 보강 P0 |
| 자동화 명확하지 않음 | 원본 DB 직결, 상태 변경까지 | 자동화 정책 명시 P0 |
| Layer 5 단일 Agent | Agent-to-Agent (6~7월) + Agent Brain (12월) | Multi-Agent 단계화 P1 |

## 1. BeSir 솔루션 핵심 기술 인사이트 12개

> 회의록(2026-04-29)에서 추출. 각 인사이트는 AI Foundry 반영 매트릭스로 §2에서 매핑됩니다.

### 1.1 온톨로지 = 사람을 복제하는 시멘틱 레이어

> "온톨로지는 단순한 데이터 구조나 스키마가 아니라 실제 그 데이터를 가지고 일하는 사람의 워크플로우, 용어, 업무 방식, 룰/규칙을 담은 시멘틱 레이어다."

- 온톨로지 = 사람의 **워크플로우 + 용어 + 룰**의 묶음
- **현행만 보유** (과거 비교는 의미 없음)
- 데이터 표준화·정규화·현행화는 "이상" — 현실은 담당자 머릿속의 암묵지

### 1.2 7-타입 객체 자동 추출

| BeSir 타입 | 정의 |
|---|---|
| Fact | 숫자로 나타나는 객체 (매출액, 고객만족도 비율) |
| Dimension | Fact를 가르는 기준 (프로젝트별, 고객별 등) |
| Workflow | 업무 흐름 |
| Event | 사건 |
| Actor | 주체/담당자 |
| Policy | 룰/규칙 |
| Support | 위 6개를 보조하는 정보 |

→ AI가 인터뷰 트랜스크립트에서 **7타입 + 객체 간 관계 + 속성을 자동 추출**.

### 1.3 인터뷰 필수 (엑셀 설문 X)

- 엑셀 설문은 답만 적힘 → 맥락이 사라짐
- 대면 인터뷰는 한 개념 설명에 이전 맥락·연관 개념까지 함께 → **관계가 자연스럽게 녹아듦**
- 이 관계가 온톨로지 자동 추출의 원천 소스

### 1.4 80-20-80 검수 룰

1. AI가 80% 수준 결과물 생성 + 자체 평가 (리소스 충실도/할루시네이션)
2. 집중 검수: 나머지 20%에 다시 80% 채움
3. 사람 최종 체크 → 100% 완성

### 1.5 CQ (Critical Question) — 검증의 시작점이자 종점

- CQ = 시스템에 던지는 핵심 질문 + 그에 대한 정답지 쌍
- **사람이 직접 작성** (AI 금지). End User만이 진짜 요구사항을 안다는 철학
- 출처: C-Level + 경영분석실 + 도메인 PM (다층 수집)
- 정답지는 **사람이 시스템 직접 조회**, 특정 시점에 고정
- 매뉴얼 보유 (좋은 CQ vs 나쁜 CQ 가이드)

### 1.6 5축 100점 벤치마크 + 90점 핸드오프

별도 평가 에이전트(문맥 분리)가 100점 만점으로 채점. 평균 90점 이상이면 핸드오프(자동화).

| 축 | 평가 내용 |
|---|---|
| 1. 온톨로지 활용도 | 질문의 의미·맥락을 온톨로지로 잘 이해했는가 |
| 2. 툴 선택 | 적합한 툴(API)을 선택했는가 |
| 3. 코드 품질 | 툴을 가공하는 코드를 잘 만들었는가 |
| 4. 결과값 일치 | 사용자 정답과 일치하는가 |
| 5. 거버넌스 | 권한 없는 정보를 답하지 않았는가 |

90점 미만은 (a) 휴먼 에러 / (b) 인프라 이슈로 분리 분석.

### 1.7 Graph DB·Vector DB 의도적 미사용

> "그래프 DB의 목적 자체가 관계 탐색인데, LLM이 그래프 쿼리를 만드는 게 오히려 성능이 떨어집니다."

- LLM이 Cypher 같은 그래프 쿼리 생성하는 것보다 **파일 retrieve가 더 빠르고 정확**
- **Claude Code의 접근 방식과 동일** — 파일 기반 탐색
- Vector DB도 거부: 임베딩으로 압축하면 시멘틱 의미 손실

### 1.8 저장 아키텍처 — 파일 + Git + PostgreSQL

| 저장 대상 | 저장소 | 비고 |
|---|---|---|
| 온톨로지 메타 객체 (스키마·타입·관계 정의) | **파일 (Knowledge Map)** | Git 버전관리 |
| 인스턴스 데이터 (객체-객체 관계, 가중치) | **PostgreSQL** | 운영 DB |
| 운영 로그 / 도구 호출 이력 | **PostgreSQL** | Browser 콘솔 조회 |
| 캐시 | **Redis** | 데이터 원천 캐싱 |
| 데이터 원천(Truth) | **고객사 원본 DB (Connect)** | 복제하지 않음 |

### 1.9 데이터 원본 직결 — 자동화까지 가기 위해

> "분석만이면 Snowflake로 복제해서 퍼포먼스를 높일 수 있지만, 자동화(상태 변경)까지 가야 하기 때문에 느리더라도 원본 DB를 바라보는 정책."

- 자동화 = AI가 상태를 직접 변경 (예: PM 프로젝트 상태를 "리스크"로 변경, 구매 금액 1억 → 1.5억)
- **분석만이면 OK / 자동화면 원본 직결 필수**

### 1.10 System Knowledge — 암묵지 별도 파일

- RDB에 매핑되지 않는 도메인 암묵지를 별도로 관리
- 예시: "DSBR(대시보드) 테이블"이 다른 솔루션이 만든 결과 데이터인데, 이 테이블을 신뢰해서 직접 쿼리해야 한다는 룰
- AI가 retrieve해서 적절한 API 생성

### 1.11 Browser 콘솔 — 운영환경 가시성

- 사용자별 대화 이력 조회
- 도구(API) 호출 성공/실패율 (현재 ~99% 성공)
- 에러 케이스 분석 (어떤 파라미터가 잘못)
- 온톨로지 시각화 / Heating(자주 접근하는 객체)
- **MCP 툴 파라미터 단위 권한 부여** (PoC 미적용)

### 1.12 BeSir 로드맵

| 시점 | 기능 |
|---|---|
| 2026-06~07 | **Agent-to-Agent 오케스트레이션** — 도메인별 에이전트 코워킹 |
| 2026-12 | **Agent Brain** — 요청 X 미션 부여 → 자율 가중치 탐색 |
| 추가 | **전용 하드웨어 모델** — 토큰 비용 부담 회피 |

## 2. AI Foundry × BeSir 매핑 매트릭스 (12개 인사이트별)

> Overlap = 양 제품에 모두 존재 / Complement = BeSir이 보강할 수 있음 / Gap = AI Foundry가 보강 필요

### 2.1 인사이트 1 — 온톨로지 = 시멘틱 레이어 (사람 복제)

- **AI Foundry v0.2 §2.2.2 / §3.4**: "도메인 지식의 그래프 표현. 12 노드 타입"
- **상태**: **Gap** — 본질 정의가 다름. AI Foundry는 데이터 구조 관점, BeSir은 사람 복제 관점
- **반영 안**:
  - §2.2.2 정의를 **"사람의 워크플로우·용어·룰을 담은 시멘틱 레이어"** 로 재작성
  - "현행만 보유" 원칙 명시
  - 데이터 표준화/정규화에 대한 입장 표명 ("이상에 가깝다")
- **우선순위**: **P0**

### 2.2 인사이트 2 — 7-타입 객체

- **AI Foundry v0.2 §3.4**: 12 노드 타입 (Domain·Process·Policy·Entity·Actor·Event·Outcome·Constraint·Reference·Metric·Artifact·Decision)
- **상태**: **충돌 + Complement** — 12 vs 7. BeSir의 7타입이 더 압축적·실용적
- **매핑 가설**:

| BeSir 7타입 | AI Foundry 12노드 매핑 | 비고 |
|---|---|---|
| Fact | Metric + Outcome | 숫자로 나타나는 객체 |
| **Dimension** | (없음 — 신규) | Fact를 가르는 기준. **AI Foundry에 누락** |
| Workflow | Process | 업무 흐름 |
| Event | Event | 사건 |
| Actor | Actor | 주체 |
| Policy | Policy + Constraint | 룰·규칙 |
| Support | Reference + Artifact | 보조 정보 |

- **반영 안**:
  - 12 노드 → **7 타입 + Domain + Decision = 9 타입**으로 압축 재정의
  - **Dimension 추가** (가장 중요한 신규)
  - §3.4 + §12.1 용어 사전 + §12.3 데이터 스키마 일괄 갱신
- **우선순위**: **P0**

### 2.3 인사이트 3 — 인터뷰 필수

- **AI Foundry v0.2 §3.3 Data Layer**: HTTP Upload / DB Connector / Stream / API
- **상태**: **Gap** — 인터뷰 입력 채널 명시 안 됨
- **반영 안**: §3.3에 **Interview Connector** 추가 (음성 녹취 → 트랜스크립트 → 7타입 자동 추출)
- **우선순위**: **P1**

### 2.4 인사이트 4 — 80-20-80 검수 룰

- **AI Foundry v0.2 §3.6 Workflow Layer**: HITL Reviewer/Approver, accept/reject/modify (구체 룰 없음)
- **상태**: **Complement** — BeSir의 구체 룰을 흡수
- **반영 안**: §3.6 + §8.4에 80-20-80 룰 명시:
  - Stage 1: AI 1차 80% + 자체 평가 (충실도/할루시네이션)
  - Stage 2: AI 2차 (남은 20%에 집중 → 다시 80%)
  - Stage 3: 사람 최종 체크 (100%)
- **우선순위**: **P0**

### 2.5 인사이트 5 — CQ (Critical Question)

- **AI Foundry v0.2 §4 4대 진단 + §9.2 측정지표**: 진단·정밀도·재현율 중심
- **상태**: **Complement (큰 보강)** — CQ는 **요구사항 → 검증 → 운영**의 통합 신호 체계. AI Foundry에 부재
- **반영 안**:
  - §4 또는 신규 §4.5 "CQ 기반 검증 체계" 절 추가
  - 5축 평가 + 90점 핸드오프 룰 명시
  - §9.2 측정지표를 CQ 5축으로 연결
- **우선순위**: **P1** (4대 진단과의 관계 정리 필요)

### 2.6 인사이트 6 — 5축 100점 벤치마크 + 90점 핸드오프

- **AI Foundry v0.2 §3.5.1 Multi-Evidence Triangulation (E1·E2·E3)**: 신뢰도 가중
