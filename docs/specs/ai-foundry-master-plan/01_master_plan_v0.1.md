# Decision Foundry

> **이 문서는 무엇인가**

## 0. Executive Summary

### 0.1 한 문장 정의

> **Decision Foundry는, 기업의 매일 수천 번 일어나는 의사결정을 데이터·온톨로지·LLM·Workflow·도메인 Agent로 "설명가능하게" 만들어주는 KT의 Agentic AI OS입니다.**

### 0.2 두 트랙, 하나의 제품

| 트랙 | 목표 | 데드라인 | 본질 |
|---|---|---|---|
| **트랙1 — 플랫폼 구축** | "플랫폼화 했다"고 말할 수 있는 가시적 산출물 | 2026-08 | 임원/사업본부/고객사에 보여줄 **제품의 모양** |
| **트랙2 — 사업화** | KT 공공·금융사업본부와 합의된 **사업기획안 (공공 1부 + 금융 1부)** | 2026-06~ | 트랙1을 어떻게 팔고 자산화할지의 **GTM 청사진** |

두 트랙은 동일 프로젝트의 두 면입니다. 트랙1이 데모/PoC라면 트랙2는 GTM이고, 트랙1의 플랫폼 모양이 트랙2 사업기획안의 첫 페이지에서 그대로 설명되어야 합니다.

### 0.3 핵심 의사결정 (인터뷰 1라운드 결과)

1. **포지셔닝**: AI Foundry OS의 5개 repo를 백엔드로 두고, 외부에는 **Decision Foundry**라는 신규 제품 브랜드 단일 면으로 패키징합니다.
2. **8월 데모의 첫 인스턴스**: 신규 Lighthouse 도메인이 아니라, **Decode-X에서 이미 진행 중인 퇴직연금 + 온누리상품권 두 도메인** 위에 Decision Foundry 옷을 입힙니다. 8월 데드라인을 지키기 위한 현실적 선택입니다.
3. **본부-도메인 매핑**: **퇴직연금 = KT 금융사업본부 / 온누리상품권 = KT 공공사업본부**. 양 본부 부록은 이 매핑을 기반으로 작성합니다.
4. **단계적 청중**: 8월 임원 Show & Tell → 9~10월 사업본부 공조 → 4Q Lighthouse(메리츠/포스코/현대차/경기도청 중 1) 계약.
5. **Decode-X와의 관계**: Decode-X는 **Decision Foundry의 기술 백본**으로 위치시키고, 임원 시연에서는 "5단계 파이프라인"의 검증 케이스로 활용합니다.

### 0.4 임원 보고용 한 슬라이드 메시지 (8월 Show & Tell)

## 1. 전략적 위치 — 왜 Decision Foundry인가

### 1.1 KT Enterprise 2026 경영계획과의 정합

2026년 경영계획에서 Enterprise부문의 전략목표는 "**CT 기반의 공공·산업 특화 AX로 B2B AX 시장 선도**"이고, 이를 받치는 세 축은 다음과 같습니다.

| 경영계획 축 | 핵심 액션 | Decision Foundry 기여 |
|---|---|---|
| **B2B의 기본과 원칙** | 정도영업·공정준수·안전·정보보안 | HITL 거버넌스 5 RBAC 역할로 의사결정 감사 추적 (Decode-X svc-policy 활용) |
| **B2B의 확실한 성장** | 전략사업 확장(BigPJT·AX인프라), 산업별 AX 확산, CT+AX 시너지 | 양본부 사업기획안으로 **공공/금융 표준 모델** 즉시 라인업화 |
| **B2B의 미래** | 고객業 니즈 기반 공공·산업 CT+AX 표준 모델 구현 + B2B 사업수행체계 혁신 | **Decision Foundry = 그 표준 모델의 인스턴스화 도구**, To-Be의 Factory→Racing→PitStop→Finish 4단계 릴레이 구조에서 "Factory" 단계 생산성 극대화 |

특히 경영계획에서 이미 명시된 **산업특화 AX 발굴 라인업** 다섯 개와 정확히 매칭됩니다.

| 산업 | 경영계획상 발굴 영역 | Decision Foundry 첫 인스턴스(예정) |
|---|---|---|
| 금융 | Agentic AICC + 데이터플랫폼/차세대 구축 | 퇴직연금 의사결정(지급·심사·통제) → 메리츠 Sales Agent로 확장 |
| 공공 | 경기도청 생성형 AI → 타 지자체 확산 | 온누리상품권 정책 진단 → 행안부·소상공인진흥공단·울산시청·경남/전남도청 확산 |
| 제조 | 포스코 설비관리 특화 LLM | (4Q Lighthouse 후보) 포스코 설비 의사결정 |
| 의료 | 아산병원/연세의료원 컨설팅 → 의료 AX 모델 사업화 | (2027년 후보) 의료 의사결정(진단/처방 가드레일) |
| 교육 | 교수학습 플랫폼 + K-에듀파인 | (2027년 후보) 교육정책 의사결정 |

### 1.2 AI Foundry OS 안에서의 위치

KTDS-AXBD가 운영 중인 AI Foundry OS는 5개 repo로 구성되어 있고, Decision Foundry는 이 위에서 **수직 도메인용 외부 GTM 브랜드**로 자리잡습니다.

```
[AI Foundry OS — 백엔드 / 내부 자산]
  ├── Foundry-X      : AX 사업개발 라이프사이클 OS (KTDS 내부용)
  ├── Decode-X       : 기존 시스템·SI 산출물을 해독하여 정책·온톨로지·스킬로 변환 (Input Plane)
  ├── Discovery-X    : AX 신사업 실험·발굴 시스템 (Discovery Plane)
  ├── AXIS-DS        : Design System (UI Layer, agentic-ui 별도 패키지)
  └── ax-plugin      : 24개 스킬 + 15개 표준 (Skill Layer)

       │
       ▼  (도메인-인스턴스 패키징 + 외부 브랜드)

[Decision Foundry — 외부 GTM 제품]
  ├── Pension Edition         (금융본부 GTM, 1차 인스턴스)
  ├── Voucher Edition         (공공본부 GTM, 1차 인스턴스)
  ├── Sales-Agent Edition     (4Q Lighthouse: 메리츠 등)
  └── (확장 도메인 ...)
```

핵심 원칙은 **"Git이 진실, Decision Foundry는 의사결정 컨텍스트"** 입니다. Foundry-X의 "Git이 진실, Foundry-X는 렌즈" 철학을 도메인으로 한 단계 내려서, 고객사의 의사결정 근거(데이터·정책·룰)를 git처럼 추적가능한 산출물로 남기는 것을 약속합니다.

### 1.3 외부 고객 입장에서의 가치 제안

| 고객 페르소나 | 현재의 고통 | Decision Foundry가 약속하는 것 |
|---|---|---|
| 금융 — 보험 설계사 (메리츠) | 상품 추천이 개인 경험에 의존, 감독·내부통제 추적 불가 | 추천 결정의 데이터·정책 근거를 자동 기록, 감독원 응답 자료 즉시 출력 |
| 금융 — 연금 운용 담당자 | 가입자 인출 신청 사유별 룰이 사람 머릿속에만 | 인출 정책을 `POL-PENSION-WD-*` 형태로 코드화, 변경 이력 추적 |
| 공공 — 정책 담당자 (행안부) | 정책 시행 후 효과·문제점 진단이 수개월 소요 | 4대 진단(missing/duplicate/overspec/inconsistency) 자동화로 정책 일관성 진단 |
| 공공 — 지자체 (경기도청) | 도청 생성형 AI가 단일 시스템에 종속 | Decision Foundry로 도메인별 인스턴스 추가 가능, 타 지자체 확산 표준화 |

## 2. 제품 정의 — Decision Foundry란 무엇인가

### 2.1 5-Layer 아키텍처

Decision Foundry는 **Data → Ontology → LLM → Workflow → 특화Agent**의 5층 구조이고, 각 층은 AI Foundry OS의 기존 자산을 재사용합니다.

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 5  특화 Agent           Skill Package (.skill.json)  │
│           도메인 의사결정 자동화   POL-{DOMAIN}-{TYPE}-{SEQ}    │
├─────────────────────────────────────────────────────────────┤
│  Layer 4  Workflow             HITL via Cloudflare DO+Queue │
│           사람 검토·승인·수정     5 RBAC roles                  │
├─────────────────────────────────────────────────────────────┤
│  Layer 3  LLM                  Tier1 Opus(policy)           │
│           정책 추론·문장화        Tier2 Sonnet(structure)       │
│                                Tier3 Haiku/Workers AI       │
├─────────────────────────────────────────────────────────────┤
│  Layer 2  Ontology             Neo4j Aura, 12 node types    │
│           도메인 지식의 관계도     Domain·Process·Policy·Entity   │
├─────────────────────────────────────────────────────────────┤
│  Layer 1  Data                 svc-ingestion, svc-extraction│
│           SI 산출물·문서·DB 수집  5 D1 DBs                       │
└─────────────────────────────────────────────────────────────┘
```

**5-Stage Pipeline (Decode-X에서 가져옴)**: Document Ingestion → Structure Extraction → **Policy Inference (HITL 코어)** → Ontology Normalization → Skill Packaging.

여기서 가장 중요한 것은 **Layer 3의 Policy Inference**입니다. 의사결정의 본질인 "조건(Condition) — 기준(Criteria) — 결과(Outcome) 트리플"을 명시화해서, "이 결정의 근거가 무엇인가"라는 질문에 항상 답할 수 있게 합니다.

### 2.2 Foundry-X와 Decision Foundry — 무엇이 다른가

자주 받게 될 질문이라 명확하게 정리합니다.

| 항목 | Foundry-X | Decision Foundry |
|---|---|---|
| **누구의 의사결정** | KTDS 자신의 AX 사업개발 라이프사이클 | 고객사의 도메인 업무 의사결정 |
| **사용자** | KTDS-AXBD 내부 | 외부 고객사 (보험설계사·정책담당자·운용역 등) |
| **출력물** | 사업기획·요구사항·테스트·GTM 자료 | 의사결정 결과 + 근거 추적 |
| **GTM** | 내부 효율화 (비공개) | **외부 제품 (이 문서가 다루는 대상)** |
| **공유 자산** | ax-plugin, AXIS-DS, Workers/D1, Anthropic+OpenRouter, HITL 거버넌스 |

> 임원 보고용 한 줄: "Foundry-X가 KTDS의 의사결정 OS라면, Decision Foundry는 우리가 고객사에 파는 의사결정 OS입니다. 같은 엔진, 다른 도메인."

### 2.3 4대 진단 (제품 차별화 핵심)

Decode-X Phase 2-E에서 검증된 4대 진단을 **Decision Foundry의 시그니처 기능**으로 전면화합니다. 임원·고객사 시연에서 가장 강한 인상을 남길 수 있는 부분입니다.

| 진단 | 의미 | 예시 (퇴직연금) | 예시 (온누리상품권) |
|---|---|---|---|
| **Missing** | 룰이 누락된 케이스 | 60세 이상·한도 초과 신청에 대한 정책 부재 | 비대면 환전 한도 정책 누락 |
| **Duplicate** | 동일 케이스에 충돌하는 룰 | 동일 사유 인출에 부서별 다른 한도 | 동일 가맹점 유형에 다른 수수료 |
| **Overspec** | 과도하게 세분화된 룰 | 1만원 단위로 세분화된 인출 한도 | 지역별 세분화된 정책 (효과 없음) |
| **Inconsistency** | 정책 간 모순 | 가입약관과 인출규정의 충돌 | 정책 시점별 충돌 (개정 미반영) |

각 진단은 `severity (critical/warning/info)` + `confidence (0-1)` + `hitl_status (pending/accept/reject/modify)`로 추적되어, 4Q 외부 고객 데모에서 "당신의 정책에는 N개의 critical inconsistency가 있고, AI가 제안한 수정안은 다음과 같습니다"라는 메시지를 즉시 전달할 수 있습니다.

### 2.4 Cross-Org Comparison (확장의 근거)

같은 도메인을 **여러 조직에 적용할 때 무엇이 표준이고 무엇이 차별화인지** 자동 분류하는 기능이 이미 Decode-X Phase 2-E에 구현되어 있습니다. 이게 트랙2 사업화의 핵심 BM 근거입니다.

| 그룹 | 의미 | 사업적 의미 |
|---|---|---|
| **common_standard** | 모든 조직이 공유하는 표준 | KT Decision Foundry 기본 패키지로 자산화 |
| **org_specific** | 조직별로 명확히 다른 부분 | 고객사 커스터마이즈 매출 |
| **tacit_knowledge** | 명문화 안 된 암묵지 | 컨설팅 매출 + 가치 입증 포인트 |
| **core_differentiator** | 그 조직만의 핵심 차별화 | 고객사 IP 보호 약속 영역 (절대 표준화 금지) |

> 트랙2 사업기획안의 한 문장 약속: "우리는 당신 조직의 core_differentiator는 절대 표준화하지 않습니다. common_standard만 자산화해서 산업 전체의 비용을 낮춥니다."

## 3. 트랙1 — 2026년 8월 플랫폼 데모 (Show & Tell)

### 3.1 8월 데모의 목적과 성공 기준

| 항목 | 내용 |
|---|---|
| **목적** | "기업 의사결정 업무 Agentic AI 플랫폼을 만들었다"고 말할 수 있는 가시적 산출물 확보 |
| **1차 청중** | KT Enterprise부문 + AX사업부문 임원 (Show & Tell) |
| **성공 기준** | (1) 두 도메인이 **단일 UI**에서 동작 (2) 5-Layer 아키텍처가 화면 위에서 식별가능 (3) 4대 진단 결과가 즉시 출력 (4) 임원 질의응답에서 "이걸 메리츠/포스코에 넣으면?"이라는 질문이 자연스럽게 나옴 |
| **불이행 정의** | 임원이 "이게 Foundry-X와 뭐가 다른가"라는 질문을 받게 되는 경우 = 포지셔닝 실패 |

### 3.2 데모 시나리오 (15분 시연 시나리오)

**Scene 1 — 도메인 선택 (1분)**
- Decision Foundry 단일 UI에서 좌측 도메인 메뉴: `Pension Edition` / `Voucher Edition` 토글
- 임원 멘트: "두 도메인이 같은 엔진 위에 있다는 점이 핵심"

**Scene 2 — Layer 1 Data: 원본 문서 투입 (2분)**
- 퇴직연금 약관 PDF + 인출 신청서 양식 + 운영 매뉴얼을 드래그앤드롭
- svc-ingestion이 자동 분류 → svc-extraction이 구조화
- 화면: 좌(원본) — 우(추출된 구조) 분할 뷰

**Scene 3 — Layer 2 Ontology: Neo4j 시각화 (3분)**
- Neo4j Aura에서 12 node types 그래프 시각화 (Domain → Process → Policy → Entity)
- 두 도메인 토글 시 ontology 노드 색상이 도메인별로 구분되어 같은 화면에 공존
- 임원 멘트: "도메인 추가 = 새 색상 추가, 엔진은 동일"

**Scene 4 — Layer 3 LLM: Policy Inference (4분, 핵심)**
- "60세 가입자가 주택구입 사유로 연금 인출을 신청" 입력
- Tier 1 Opus가 Policy Triple 추출: `Condition: age>=60 AND reason=housing — Criteria: 한도 5억 — Outcome: 승인 + 세제 혜택`
- POL-PENSION-WD-HOUSING-001 코드 자동 부여
- 임원 멘트: "정책이 코드가 됩니다"

**Scene 5 — Layer 4 Workflow: HITL (2분)**
- Reviewer 역할로 로그인 → 추출된 정책에 `hitl_status: pending` 표시
- 한 정책에 `modify` 클릭 → 수정안 입력 → audit log 자동 기록
- 임원 멘트: "AI 단독 결정이 아니라 사람의 책임이 항상 추적됩니다"

**Scene 6 — Layer 5 Skill Package + 4대 진단 (3분, 클라이맥스)**
- 두 도메인 모두에 4대 진단 실행
- 결과: 퇴직연금 — Critical 3건 / Warning 12건 / Info 28건
- 결과: 온누리상품권 — Critical 5건 / Warning 9건 / Info 17건
- 진단 카드 클릭 시 "근거 문장 → 충돌 정책 → 수정 제안" 트레이스
- Cross-Org 비교 토글 → "같은 인출 정책에 대해 A 운용사·B 운용사가 어떻게 다르게 운영하나" 비교
- 임원 멘트 (마무리): "오늘 우리는 KT의 의사결정을 설명할 수 있습니다. 4Q에는 메리츠·포스코의 의사결정도 같은 방식으로 설명합니다. 이게 Decision Foundry입니다."

### 4.1 공통 본문 (Part A) — 양본부 동일 적용

#### 4.1.1 제품 정의 1쪽

`Decision Foundry는 KT Enterprise부문이 공공·산업 고객의 의사결정 업무를 자동화·설명가능화하는 Agentic AI OS입니다. Data·Ontology·LLM·Workflow·Agent 5개 층으로 구성되며, 도메인별 인스턴스를 빠르게 추가할 수 있는 표준 패키징 구조(.skill.json, POL-{DOMAIN}-{TYPE}-{SEQ})를 채택합니다. KT가 보유한 5개 AI 자산 repo(Foundry-X / Decode-X / Discovery-X / AXIS-DS / ax-plugin) 위에 구현되어, 신규 도메인 추가 시 Time-to-Market을 기존 SI 대비 1/3로 단축합니다.`

#### 4.1.2 BM (Business Model)

| 매출 유형 | 단위 | 가격 모델 (예시) | 책임 본부 |

| **컨설팅 매출** | 워크샵 / 프로젝트 | tacit_knowledge 영역 — 정액 패키지 | AX컨설팅팀 주도 |
