## 검토 의견: QSA PoC PRD — Prototype 품질 검증 + 개선

### 1. 기술적 실현 가능성
**기본적으로 실현 가능하나 몇 가지 기술적 제약이 존재합니다.**

- **Workers AI 모델의 일관성 문제**: Llama-3.1-8b는 QSA의 5차원 판별(시각적 일관성, 접근성, 반응형 등)에 적합할 수 있으나, **판별 기준의 일관성**이 주요 리스크입니다. 동일한 HTML에 대해 다른 시간에 실행하면 다른 점수가 나올 수 있어, 개선 전후 비교의 신뢰도가 떨어질 수 있습니다.
- **D1 데이터 접근 및 업데이트**: Prototype HTML을 D1에서 조회하는 것은 단순하지만, **Guard 적용 후 수정된 HTML을 다시 저장하는 메커니즘**이 명확하지 않습니다. 원본 유지 vs 업데이트 선택에 따라 데이터 버전 관리 전략이 필요합니다.
- **CSS Guard의 한계**: `guardCss()`가 AI 기본 폰트/흑백을 제거할 수 있으나, **시맨틱한 디자인 개선(예: 레이아웃, 색상 대비)** 까지 보장하지는 않습니다. 이는 QSA 점수 향상에 제한적 영향을 미칠 수 있습니다.

### 2. 아키텍처 적합성
**현재 구조는 PoC 수준에는 적합하나, 확장성을 고려한 개선이 필요합니다.**

- **단일 Workers AI 의존성**: 모든 QSA 판별을 하나의 AI 모델에 의존하면, **성능 병목**이나 모델 업데이트 시 전체 시스템 영향이 큽니다. 장기적으로는 차원별 전용 모델/룰 기반 평가를 고려해야 합니다.
- **결과 보고서 생성의 자동화 부재**: 보고서를 수동 문서화한다면 확장 불가능합니다. **구조화된 JSON 출력 + 자동 리포트 생성기**를 PoC에 포함시키는 것이 향후 운영화에 유리합니다.
- **DesignToken(F465) 인터페이스 테스트**: P1 기능이지만, **토큰 적용이 CSS Guard와 충돌**할 수 있습니다(예: Guard가 폰트를 제거했는데 토큰이 새 폰트를 적용). 실행 순서와 롤백 전략이 정의되어야 합니다.

### 3. 구현 복잡도
**과소평가된 복잡도가 있습니다.**

- **HTML 정규화 문제**: D1에서 조회한 HTML이 **불완전하거나 외부 리소스에 의존**할 경우, QSA 평가가 왜곡될 수 있습니다. 로컬 테스트 환경과의 차이를 최소화하는 전처리 단계가 필요합니다.
- **개선 전후 비교의 정확성**: 단순 점수 차이(+0.10)로 개선 효과를 판단하기에는 **통계적 유의성**이 부족합니다. 동일한 입력/모델 조건에서의 재현성 검증이 선행되어야 합니다.
- **에지 케이스**: CSS Guard가 HTML 구조를 손상시킬 수 있습니다(예: 인라인 스타일 제거로 레이아웃 붕괴). **Guard 적용 전후의 시각적 레그레션 테스트**가 빠져 있습니다.

### 4. 기술 리스크
**가장 큰 우려 요인은 다음과 같습니다.**

- **AI 판별의 불확실성**: Workers AI의 출력 변동성이 높다면, PoC 결과 자체가 무의미해질 수 있습니다. **확률적 출력 대신 결정적 룰 기반 검증을 보완**해야 합니다.
- **보안 이슈의 오탐/미탐**: QSA-R1(보안)이 내부 정보 노출을 정확히 탐지할지 검증되지 않았습니다. **민감 정보 마스킹 등의 가상 데이터로 선검증**이 필요합니다.
- **PoC의 국한된 일반화**: 3건의 Prototype만으로 품질 체계의 전반적 효과를 입증하기 어렵습니다. **다양한 산업/복잡도의 샘플을 추가**해 변별력을 확인해야 합니다.
