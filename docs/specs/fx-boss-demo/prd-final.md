## 1. 요약 (Executive Summary)

**한 줄 정의:**
SI 프로젝트에서 추출한 Spec(온누리상품권)을 기반으로, AI가 새로운 사업(전자화폐상품권 플랫폼)의 발굴→형상화→Prototype까지 자동 생성하는 과정을 본부장에게 15분 라이브 시연.

**배경:**
Foundry-X + Decode-X 파이프라인이 "SI 코드 → Spec 추출 → 반제품화"를 자동화할 수 있음을 Phase 44~46까지 구현 완료. 이 능력을 경영층에 시연한 적이 없음. 4/17 오전 대표 보고에서 Foundry-X의 핵심 가치를 증명해야 함.

**가치 전달:**
SI 프로젝트 종료 후 코드에 잠든 도메인 지식을 AI가 자동 추출 → 신규 사업의 MVP를 자동 생성. 기존 3~6개월 → 2시간, 비용 80% 절감이라는 ROI를 직접 시연으로 증명.

**목표:**
(1) 본부장이 "Spec이 현업 수준이다"라고 인정 (2) "실제 사업에 쓸 수 있다"고 판단 (3) 후속 액션(PoC 확장, 투자 검토) 연결

## 3. 시연 시나리오 설계 (15분)

### 전체 구조

```
[슬라이드] Scene 1: 컨텍스트 ─────── 2분
    ↓
[라이브]   Scene 2: Spec 품질 확인 ── 3분
    ↓
[혼합]     Scene 3: 신규 시나리오 발굴 3분
    ↓
[혼합]     Scene 4: 형상화→Prototype ─ 5분  ⭐ 핵심
    ↓
[슬라이드] Scene 5: 임팩트 + Q&A ──── 2분
```

### Scene 1: 컨텍스트 (슬라이드, 2분)

| 슬라이드 | 내용 | 시간 |
|----------|------|------|
| S1 | **문제**: SI 완료 후 도메인 지식이 코드에 묻힘 | 30초 |
| S2 | **해법**: Decode-X Spec 추출 → Foundry-X 반제품 자동 생성 | 60초 |
| S3 | **오늘 시연**: 온누리상품권 → 전자화폐상품권 플랫폼 | 30초 |

**핵심 메시지:** "SI 프로젝트 하나가 끝나면, 그 안의 도메인 지식이 다음 사업의 씨앗이 됩니다"

### Scene 2: Spec 품질 확인 (라이브, 3분)

| 단계 | 화면 | 설명 |
|------|------|------|
| 2-1 | `/ai-foundry-os/demo/lpon` | AI-Ready Score 87점, 3-Pass 분석 |
| 2-2 | Scoring Pass 탭 | 6개 기준 프로그레스 바 — "코드 구조 92, 도메인 규칙 85…" |
| 2-3 | Diagnosis Pass 탭 | Gap findings — "부분취소 테스트 미흡까지 자동 감지" |
| 2-4 | Comparison Pass 탭 | Spec Coverage 87%, Code Alignment 92% |
| 2-5 | 실제 Spec 열기 (선택) | 01-business-logic.md → "47개 BL 자동 추출" |

**예상 Q&A:**
- "87점 기준?" → 6축 AI-Ready 지표 (코드 구조/테스트/도메인 규칙/데이터 모델/API/화면)
- "실제 SI 코드?" → "온누리상품권 실제 코드. Decode-X가 AST 파싱 + LLM으로 추출"
- "정확도?" → "파일럿 3건 평균 87% 일치. 복합 트랜잭션 등 엣지 케이스는 현업 보완 필요"

### Scene 3: 신규 시나리오 발굴 (혼합, 3분)

| 단계 | 화면 | 설명 |
|------|------|------|
| 3-1 | 슬라이드 | "온누리상품권 Spec → 전자화폐상품권 플랫폼 구상" |
| 3-2 | 사전 생성 결과 | Discovery: 비즈니스 아이템 분석, 5단계 평가, Harness |
| 3-3 | `/ai-foundry-os/ontology` | KG 시각화 — 도메인 구조 확인 |
| 3-4 | `/ai-foundry-os/harness` | KT연계성 100, 사업성 82, AI-Ready 90 |

**스토리텔링:** "충전→결제→환불 흐름은 전자화폐에도 동일 적용. AI가 재활용 가능성을 87점으로 평가 — 반제품화 가능"

**Spec 변환 구체성:** 단순 이름 치환이 아니라, 전자화폐 특유의 발급/정산 로직 차별화를 반영한 시나리오

### Scene 4: 형상화 → Prototype (혼합, 5분) ⭐ 핵심

| 단계 | 화면 | 행동 | 시간 |
|------|------|------|------|
| 4-1 | 슬라이드 | Spec→Design→Code→Test 자동 생성 흐름 | 30초 |
| 4-2 | Design 문서 | PRD → API → DB 스키마 자동 생성 결과 | 60초 |
| 4-3 | 코드 (working-version) | `domain/charging.ts` 174줄 — BL 참조 주석 확인 | 60초 |
| 4-4 | **라이브: `bun test`** | 3개 테스트 ALL PASS | 60초 |
| 4-5 | 서버 기동 (선택) | `bun run serve.ts` → API 200 확인 | 60초 |

**본부장 관심사 정조준:**
1. **Spec→Code 매핑**: BL-001~047이 domain/*.ts에 주석으로 추적 가능
2. **반제품 수준**: 에러 핸들링, 타임아웃, 부분취소 등 엣지 케이스 구현 (단순 스켈레톤 아님)
3. **테스트**: 518줄 자동 생성 — 충전/결제/취소별 정상+예외 케이스
4. **DB**: 315줄 migration 자동 생성 — PoC 환경 즉시 배포 가능

**예상 Q&A:**
- "프로덕션에 쓸 수 있나?" → "비즈니스 로직은 즉시 활용 가능. 프로덕션 전환 시 RDBMS + 보안 레이어 추가 필요"
- "얼마나 걸렸나?" → "Spec 추출 30분 + 반제품 생성 2시간. 사람이 하면 3~6개월"
- "다른 도메인?" → "KOAMI, 금융API, 물류 파일럿 진행 중"

**포지셔닝:** "인간-AI 협업으로 개발 기간 80% 단축하는 반제품 생성기. 완전 자동화가 아닌 '씨앗 코드' 생성 → 현업이 프로덕션으로 육성"

### Scene 5: 임팩트 + Q&A (슬라이드, 2분)

| 슬라이드 | 내용 |
|----------|------|
