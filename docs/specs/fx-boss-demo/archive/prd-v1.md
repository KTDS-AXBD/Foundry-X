## 1. 요약 (Executive Summary)

**한 줄 정의:**
SI 프로젝트에서 추출한 Spec(온누리상품권)을 기반으로, AI가 새로운 사업(전자화폐상품권 플랫폼)의 발굴→형상화→Prototype까지 자동 생성하는 과정을 본부장에게 15분 라이브 시연.

**배경:**
Foundry-X + Decode-X 파이프라인이 "SI 코드 → Spec 추출 → 반제품화"를 자동화할 수 있음을 Phase 44~46까지 구현 완료했으나, 이 능력을 경영층에 시연한 적이 없음. 내일(4/17) 오전 대표 보고 시간에 본부장이 참석하며, 이 자리에서 Foundry-X의 핵심 가치를 증명해야 함.

**목표:**
본부장이 (1) "Spec이 현업 수준이다"라고 인정하고 (2) "이걸로 실제 사업을 만들 수 있겠다"고 판단하는 것.

## 2. 문제 정의

### 2.1 현재 상태 (As-Is)
- SI 프로젝트 완료 후 코드/문서에 묻혀 있는 도메인 지식을 재활용하지 못함
- 새로운 사업 기회(전자화폐상품권 등)가 있어도 기획→설계→구현에 수개월 소요
- 경영층이 AI 자동화 능력을 직접 본 적이 없어 투자 판단 근거가 부족

### 2.2 목표 상태 (To-Be)
- SI 코드에서 추출한 Spec으로 "반제품"을 자동 생성 → 수개월 → 수일로 단축
- 본부장이 이 파이프라인을 직접 목격하고 실전 활용 가능성을 인정

### 2.3 시급성
- 내일(4/17) 오전 대표 보고 — D-1, 준비 시간은 오늘 오후~내일 아침

## 4. 시연 시나리오 설계

### 4.0 전체 구조 (15분)

```
[슬라이드] 컨텍스트 (2분)
    ↓
[라이브] Spec 품질 확인 (3분)
    ↓
[슬라이드+라이브] 신규 시나리오 발굴 (3분)
    ↓
[슬라이드+라이브] 형상화 → Prototype (5분)
    ↓
[슬라이드] 임팩트 + Q&A (2분)
```

### 4.1 Scene 1: 컨텍스트 세팅 (슬라이드, 2분)

**목적:** "왜 이걸 보여주는지" 30초 안에 설득

| 슬라이드 | 내용 | 시간 |
|----------|------|------|
| S1 | **문제**: SI 완료 후 도메인 지식이 코드에 묻힘. 새 사업 기획 시 처음부터 다시. | 30초 |
| S2 | **해법**: Decode-X가 SI 코드에서 Spec 6종 자동 추출 → Foundry-X가 Spec으로 반제품 자동 생성 | 60초 |
| S3 | **오늘 시연**: 온누리상품권(실제 SI 코드) → 전자화폐상품권 플랫폼(신규 사업) | 30초 |

**핵심 메시지:** "SI 프로젝트 하나가 끝나면, 그 안의 도메인 지식이 다음 사업의 씨앗이 됩니다"

### 4.2 Scene 2: Spec 품질 확인 (라이브, 3분)

**목적:** "추출된 Spec이 현업 수준임"을 증명

| 단계 | 화면 | 행동 | 설명 |
|------|------|------|------|
| 2-1 | fx.minu.best/ai-foundry-os/demo/lpon | 페이지 열기 | AI-Ready Score 87점, 3-Pass 분석 결과 |
| 2-2 | Scoring Pass 탭 | 6개 기준 프로그레스 바 확인 | "코드 구조 92, 도메인 규칙 추출도 85…" |
| 2-3 | Diagnosis Pass 탭 | Gap findings 확인 | "부분취소 시나리오 테스트 미흡 — 이런 것까지 찾아냅니다" |
| 2-4 | Comparison Pass 탭 | Spec↔Code 정렬 매트릭스 | "Spec Coverage 87%, Code Alignment 92%" |
| 2-5 | 실제 Spec 파일 열기 (선택) | 01-business-logic.md 일부 보여주기 | BL-001~008 충전 로직 → "47개 비즈니스 룰이 자동 추출됨" |

**본부장 예상 질문 대비:**
- Q: "이 87점은 어떤 기준으로 매긴 건가?" → 6축 AI-Ready 지표 설명 (코드 구조, 테스트 커버리지, 도메인 규칙 추출도, 데이터 모델, API 스펙 명확성, 화면 정의)
- Q: "실제 현업 코드에서 추출한 건가?" → "네, 온누리상품권 실제 SI 코드 기반. Decode-X가 AST 파싱 + LLM 분석으로 추출"
- Q: "정확도는?" → "현재 파일럿 수준이고, KOAMI 프로젝트 등에서 실증 중"

### 4.3 Scene 3: 신규 시나리오 — 발굴 (슬라이드+라이브, 3분)

**목적:** "추출한 Spec으로 새로운 사업 기회를 발굴하는 과정"

| 단계 | 화면 | 설명 |
|------|------|------|
| 3-1 | 슬라이드: 전환 시나리오 | "온누리상품권 Spec을 기반으로 '전자화폐상품권 플랫폼'이라는 새 사업을 구상합니다" |
| 3-2 | 사전 생성된 발굴 결과 | Discovery 단계 결과물: 비즈니스 아이템 분석, 5단계 평가, Harness 점수 |
| 3-3 | KG (Knowledge Graph) 보여주기 | fx.minu.best/ai-foundry-os/ontology → 도메인 구조를 시각적으로 확인 |
| 3-4 | Harness 대시보드 | fx.minu.best/ai-foundry-os/harness → KT연계성 100, 사업성 82, AI-Ready 90 |

**스토리텔링:** "온누리상품권의 충전→결제→환불 흐름은, 전자화폐상품권에도 거의 동일하게 적용됩니다. AI가 이 재활용 가능성을 자동으로 평가한 결과, AI-Ready 87점으로 '반제품화 가능'이라고 판단했습니다."

### 4.4 Scene 4: 형상화 → Prototype (슬라이드+라이브, 5분) ⭐ 핵심

**목적:** "Spec에서 실동작 코드가 자동 생성되는 것"을 보여줌

| 단계 | 화면 | 행동 | 시간 |
|------|------|------|------|
| 4-1 | 슬라이드: 형상화 파이프라인 | Spec→Design→Code→Test 자동 생성 흐름 | 30초 |
| 4-2 | 사전 생성된 Design 문서 | PRD → API 설계 → DB 스키마 자동 생성 결과 보여주기 | 60초 |
| 4-3 | 사전 생성된 코드 (working-version) | 실제 도메인 로직 열기 — `domain/charging.ts` 174줄 | 60초 |
| 4-4 | **라이브: 테스트 실행** | `cd working-version && bun test` — 3개 테스트 파일 ALL PASS | 60초 |
| 4-5 | 실제 서버 기동 (선택) | `bun run serve.ts` → API 호출 데모 | 60초 |

**핵심 포인트 (본부장 관심사 정조준):**
1. **Spec 완성도**: "BL-001~047 비즈니스 룰이 코드에 그대로 반영되어 있습니다" → domain/*.ts 에서 BL 참조 주석 보여주기
2. **반제품 수준**: "단순 스켈레톤이 아니라, 에러 핸들링, 타임아웃 처리, 부분취소 같은 엣지 케이스까지 구현됨"
3. **테스트 커버리지**: "518줄의 테스트가 자동 생성됨. 충전/결제/취소 시나리오별 정상+예외 케이스"
4. **DB migration**: "315줄의 스키마 자동 생성 — 즉시 배포 가능"

**본부장 예상 질문 대비:**
- Q: "이 코드가 실제로 프로덕션에 쓸 수 있나?" → "현재는 D1(SQLite) 기반이고, 프로덕션 RDBMS 전환 + 보안 레이어 추가가 필요하지만, 비즈니스 로직 자체는 즉시 활용 가능"
- Q: "얼마나 걸렸나?" → "Spec 추출 30분 + 반제품 생성 2시간. 사람이 하면 3~6개월"
- Q: "다른 도메인에도 적용 가능한가?" → "KOAMI(과학관 티켓팅), 금융API, 물류 등 파일럿 진행 중"
