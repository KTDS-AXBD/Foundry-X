> 원칙: 솔직 + 한계 투명 고지 + 로드맵으로 연결

## Scene 2: Spec 품질 관련

### Q1. AI-Ready 87점은 어떤 기준으로 매긴 건가?
**A:** 6축 AI-Ready 지표를 자동 산출해요.
- 코드 구조 명확성 (92점) — AST 파싱으로 함수/모듈 분리도 측정
- 테스트 커버리지 (85점) — 기존 테스트 파일 존재 여부 + 커버리지율
- 도메인 규칙 추출도 (78점) — BL로 변환 가능한 조건문/분기 비율
- 데이터 모델 완성도 (95점) — ERD 자동 역추출 정확도
- API 스펙 명확성 (88점) — 라우트/파라미터/응답 추출 비율
- 화면 정의 (미적용) — SI 프로젝트 특성상 백엔드 중심

### Q2. 실제 SI 코드에서 추출한 건가?
**A:** 네, 온누리상품권 실제 SI 코드 기반이에요. Decode-X가 2단계로 추출해요:
1. **코드 분석**: AST 파싱 + 호출 그래프 + 스키마 역공학
2. **문서 분석**: LLM + RAG로 정책 문서에서 비즈니스 룰 추출
추출 결과를 교차 검증(Code↔Docs consistency)하여 Spec 품질을 높여요.

### Q3. 정확도는 어느 정도인가?
**A:** 파일럿 3건(온누리상품권, KOAMI, 금융API) 기준 **평균 87% 일치율**이에요.
- 강점: 테이블 구조, API 시그니처, 기본 비즈니스 흐름은 95%+ 정확
- 한계: 복합 트랜잭션(출금 타임아웃→5분 후 재확인 등) 같은 암묵적 룰은 현업 보완 필요
- 개선 방향: Tacit Knowledge Interview(현업 인터뷰) 레이어 추가 예정

## Scene 3: 발굴/재활용 관련

### Q4. 온누리상품권과 전자화폐상품권은 다른 도메인 아닌가?
**A:** 핵심 비즈니스 로직의 83%가 동일해요. 차이는 3가지:
1. **발급 정책**: 정부 보조금 vs 민간 발행자 → 8개 BL 신규 작성
2. **한도/규제**: 고정 한도 vs 가변 한도 + KYC → 파라미터화로 대응
3. **정산 주기**: D+2 고정 vs 계약별 가변 → 기존 BL 확장

단순 이름 치환이 아니라, 도메인 차이를 AI가 자동 분석하고 변환 범위를 제시해요.

### Q5. 다른 도메인에도 적용 가능한가?
**A:** 현재 파일럿 3건 진행 중이에요:
- **KOAMI** (과학관 티켓팅) — 예약→발권→입장→정산 흐름
- **금융API** — 계좌 조회→이체→알림 흐름
- **물류** — 주문→배송→추적→반품 흐름 (준비 중)

공통점: **"충전→사용→취소→정산" 같은 트랜잭션 패턴**을 가진 도메인에서 재활용률이 높아요.

### Q6. AI가 재활용 가능성을 어떻게 판단하는가?
**A:** 3단계 자동 평가예요:
1. **구조 매칭**: 원천 Spec의 엔티티/프로세스 구조와 대상 도메인 매핑
2. **규칙 호환성**: BL의 조건-판단-처리 패턴이 대상에 적용 가능한지
3. **데이터 모델 재활용도**: 테이블/컬럼의 의미적 유사도 (LLM 기반)

## Scene 4: 반제품/코드 품질 관련

### Q7. 이 코드가 실제 프로덕션에 쓸 수 있나?
**A:** **비즈니스 로직은 즉시 활용 가능**해요. 프로덕션 전환 시 추가 필요한 것:
- RDBMS 전환 (SQLite → PostgreSQL/MySQL)
- 보안 레이어 (인증/인가, 데이터 암호화)
- 성능 튜닝 (인덱스, 커넥션 풀)
- CI/CD 파이프라인

포지셔닝: **"씨앗 코드"** — AI가 비즈니스 로직의 80%를 생성하고, 현업이 프로덕션으로 육성.

### Q8. 얼마나 걸렸나?
**A:** 
- Spec 추출 (Decode-X): **30분**
- 반제품 생성 (Foundry-X): **2시간**
- 합계: **약 2.5시간**
- 사람이 동일 범위를 하면: **3~6개월, 3~5명**

### Q9. 에러 핸들링이나 엣지 케이스는?
**A:** 24개 테스트 케이스가 자동 생성되었고, 다음을 커버해요:
- 정상 흐름 (충전/결제/취소 각 1건)
- 에러 케이스 (404/409/422/502 — 존재하지 않음, 이미 처리됨, 유효성 실패, 외부 API 실패)
- 엣지 케이스 (부분취소, 취소 기간 초과, 타인 결제 취소 시도, 관리자 권한)
- 비즈니스 규칙 (5만원 이상 SMS 발송, 비활성 가맹점 거절)

**한계**: 동시성 제어, 분산 트랜잭션, 부하 테스트는 미포함. 이건 프로덕션 전환 시 현업이 추가.

### Q10. 테스트가 의미 있는 테스트인가? (형식적 PASS 아닌지)
**A:** 각 테스트는 Spec의 BL 번호를 참조하며, 다음을 검증해요:
- **DB 상태 변화**: 잔액 증감, 상태값 전환 (PENDING→COMPLETED 등)
- **에러 코드**: HTTP 상태 + 에러 메시지 정확성
- **비즈니스 규칙 적용**: BL-005(한도 초과), BL-015(5만원 SMS), BL-016(취소 잔액 복구)

직접 실행해서 확인하실 수 있어요. (`npm test` → 24/24 PASS, 292ms)

## 사업적/전략적 질문

### Q11. ROI는 어떻게 산출했나?
**A:**
- **Before**: 도메인 분석 4주 + 설계 2주 + 구현 8주 = 14주, 5명 = 17.5인월
- **After**: AI 자동 생성 2.5시간 + 현업 보완 1주 = ~1.2인월
- **절감**: 시간 95%, 비용 약 90%
- 단, 프로덕션 품질까지 올리는 데는 추가 2~4주 필요 (보안, 성능, 운영)

### Q12. 경쟁사 대비 차별점은?
**A:** GitHub Copilot 같은 코딩 어시스턴트와는 레벨이 달라요:
1. **코딩 어시스턴트**: 함수 단위 자동완성
2. **Foundry-X**: SI 코드에서 도메인 Spec 추출 → 새 사업의 전체 반제품 자동 생성

차별점: "**기존 코드에 잠든 도메인 지식을 자동 추출**"하는 Decode-X + "**Spec→반제품 파이프라인**"

### Q13. 실제 사업화 로드맵은?
**A:**
1. **현재**: 파일럿 3건 (온누리, KOAMI, 금융)
2. **Q3 2026**: 내부 확산 — KT DS SI 프로젝트 자동 전환 5건+
3. **Q4 2026**: 외부 사업화 — 반제품 마켓플레이스 + SaaS
4. **2027**: 파트너 생태계 — ISV/SI 업체에 파이프라인 라이선스

### Q14. 보안은 어떻게 되나?
**A:** 현재 반제품은 PoC 수준이에요. 프로덕션 적용 시:
- 인증/인가: JWT + RBAC (Foundry-X 기존 패턴 재활용)
- 데이터 암호화: 전자금융업 기준 AES-256
- API 보안: Rate limiting, Input validation (Zod)
- 시연 데이터: 실제 개인정보 없음, 모두 가상 데이터

### Q15. 이 기술로 뭘 더 할 수 있나?
**A:** 3가지 확장 방향:
1. **역방향**: 레거시 SI 프로젝트 자동 문서화 (Spec 추출만 활용)
2. **순방향**: Spec 기반 신규 사업 자동 기획 + 제안서 생성
3. **순환**: 완료된 프로젝트의 Spec이 다음 프로젝트의 반제품 씨앗 → **지식 복리 효과**
