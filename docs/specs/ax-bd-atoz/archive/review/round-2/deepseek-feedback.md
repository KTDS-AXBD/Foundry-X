## 기술 검토 의견: AX BD A-to-Z PRD v2

### 1. 기술적 실현 가능성: **부분적 우려**

**긍정적 요소:**
- Cloudflare 스택(Workers, D1, R2)은 MVP 및 초기 단계에 적합한 빠른 프로토타이핑과 비용 효율성을 제공합니다.
- 모노리포 구조는 팀 규모(5명)와 초기 개발 속도에 부합합니다.

**주요 기술적 제약 및 병목:**

**A. 데이터 계층의 근본적 취약성 (심각도: 높음)**
- **Cloudflare D1(SQLite)의 운영 데이터베이스로서의 부적합성:**
    - **동시성 한계:** SQLite는 쓰기 시 전체 DB 잠금 방식으로, 다수 BD 담당자가 동시에 BMC 편집/평가 입력 시 심각한 성능 저하 및 데드락 발생 가능성이 높습니다.
    - **확장성 부재:** 데이터 볼륨(아이디어, BMC 버전 히스토리, 평가 데이터)이 증가함에 따라 쿼리 성능이 급격히 저하될 것입니다. 복잡한 포트폴리오 분석 쿼리는 사실상 불가능합니다.
    - **신뢰성 및 복구:** 관리형 DB가 아닌 파일 기반 DB로, 장애 시 복구 메커니즘이 취약합니다. Cloudflare의 백업/복구 정책이 명확하지 않아 데이터 손실 리스크가 큽니다.
    - **마이그레이션 부담:** "향후 PostgreSQL 전환" 계획은 기술 부채로 작용하며, 마이그레이션 중 서비스 중단과 데이터 불일치 리스크를 초래합니다.

**B. AI 에이전트 오케스트레이션의 숨겨진 복잡도 (심각도: 중간-높음)**
- **다중 AI API(Anthropic, OpenAI, Gemini, DeepSeek) 연동:**
    - 각 API의 응답 형식, 레이턴시, 오류 처리 방식이 상이하여 통합 레이어의 복잡도가 급증합니다.
    - 비용 제어 메커니즘(월 $200-500 예상)이 구체적이지 않습니다. 사용량 폭주 시 예산 초과 또는 서비스 차단 가능성이 있습니다.
- **"AI 초안 생성 → 사람 검토" 워크플로우의 데이터 일관성:**
    - AI가 생성한 BMC/PRD 초안과 사람이 수정한 최종본 간의 버전 관리와 변경 이력 추적이 Git만으로는 부족할 수 있습니다. 구조화된 델타(delta) 추적 메커니즘이 필요합니다.

**C. 시스템 통합 및 데이터 흐름의 복잡성 (심각도: 중간)**
- **Discovery-X API 연동:** 단순한 REST API 연동 이상으로, 수집 데이터의 품질 검증, 중복 제거, 실시간 동기화 실패 시 대체 수단(Fallback)이 정의되지 않았습니다.
- **AI Foundry 기능 흡수:** "기능 이관"이 단순 코드 이동이 아닙니다. 기존 AI Foundry의 독립적인 상태 관리, UI 로직, 데이터 모델을 Foundry-X의 아키텍처에 맞게 재설계해야 하며, 이는 PRD에서 과소평가된 작업량입니다.

### 2. 아키텍처 적합성: **개선 필요**

**A. 모노리틱 구조의 중장기적 리스크:**
- "베이스캠프"라는 포지셔닝으로 Foundry-X가 모든 기능(아이디어 관리, BMC, 프로토타입 생성, 평가)을 포함하게 되면서, 시스템이 빠르게 비대화되고 결합도가 높아질 것입니다.
- 이는 향후 **특정 기능(예: 평가 엔진)만의 독립적 스케일링이나 업데이트를 어렵게 만듭니다.** 마이크로서비스나 모듈형 마이크로프론트엔드 접근법을 고려하지 않은 점이 아쉽습니다.

**B. 평가관리 프레임워크의 아키텍처 위치 불명확:**
- 핵심 기능(P0)으로 지정되었으나, 이 프레임워크가 단순한 "KPI 입력/현황판"을 넘어 **의사결정 지원 시스템**으로 발전할 경우, 현재의 단일 D1 DB와 모노리틱 백엔드로는 분석 쿼리 부하와 비즈니스 로직 복잡도를 견디기 어렵습니다. 초기부터 분석용 읽기 전용 복제본이나 별도 데이터 마트 설계를 고려해야 합니다.

**C. 이벤트 기반 아키텍처의 부재:**
- 7단계 라이프사이클은 본질적으로 상태 기계(State Machine)입니다. 단계 간 전환, 승인, 자동화 트리거는 이벤트 드리븐 아키텍처(예: 메시지 큐 활용)에 더 적합합니다. 현재의 동기식 REST API 호출 중심 설계는 유연성과 확장성이 떨어집니다.

### 3. 구현 복잡도: **상당 부분 과소평가됨**

**A. 데이터 일관성과 동시성 문제:**
- 여러 BD 애널리스트가 동일한 BMC를 동시에 편집할 때의 충돌 해결(Conflict Resolution) 전략이 전혀 언급되지 않았습니다. "Git 기반"이라고 하지만, 비개발자 사용자를 위한 실시간 협업 편집 경험은 훨씬 복잡한 문제입니다.
- 아이디어 → BMC → 평가 데이터 간의 **참조 무결성**을 D1에서 어떻게 보장할지 구체적인 설계가 필요합니다.

**B. 에이전트 안전성 및 오류 처리:**
- AI 에이전트가 자동으로 프로토타입 코드를 생성하고 데모 환경을 프로비저닝할 때, **잘못된 코드로 인한 보안 취약점 생성, 인프라 비용 폭주, 불필요한 외부 API 호출** 등의 위험을 차단할 안전장치(Sandbox, Budget Alert, 실행 전 승인 게이트)에 대한 기술적 설계가 부족합니다.

**C. 운영 환경 통합 복잡도:**
- GitHub, Slack 등 외부 시스템과의 연동은 Webhook/API 수준으로 간단해 보이지만, **실제 운영에서의 장애 전파, 자동화 실패 시 수동 개입 절차, 자격 증명 관리** 등은 상당한 운영 복잡도를 추가합니다.

### 4. 기술 리스크: **가장 크게 우려되는 요인**

**1위: Cloudflare D1의 운영 DB 사용**
- 이는 프로젝트의 가장 큰 단일 실패 지점(Single Point of Failure)이 될 가능성이 높습니다. 성능 병목, 동시성 문제, 데이터 손실 리스크가 MVP 출시 후 사용량 증가와 함께 현실화될 것입니다.

**2위: AI 의존도에 따른 비용 및 품질 변동성 리스크**
- 핵심 가치 제안이 AI 자동화에 집중되어 있습니다. AI API 비용 급등, 특정 제공자(예: Anthropic)의 정책 변경, 생성 품질의 불안정성이 전체 서비스의 경제성과 신뢰성을 좌우할 수 있습니다.

**3위: 모노리틱 아키텍처의 기술 부채 누적**
- 빠른 출시를 위해 선택한 모노리틱 구조는 중장기적으로 유지보수성과 기능 추가 속도를 저하시켜, 팀이 "기능 개발"보다 "시스템 유지"에 더 많은 리소스를 쏟게 만들 수 있습니다.

**4위: 보안 및 컴플라이언스 미정의**
- KT DS 내부 보안 정책 검토가 '오픈 이슈'로 남아있습니다. 특히 AI API로 전송되는 데이터(시장 정보, 사업 아이디어 초안 등)가 내부 정보 유출 정책에 저촉되지 않는지 확인되지 않았습니다. 이는 배포 자체를 차단할 수 있는 치명적 리스크입니다.
