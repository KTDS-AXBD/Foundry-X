# KOAMI 산업 공급망 의사결정 지원 사업화 PRD

## 1. 개요 (Overview)

- **사업 아이템명:** KT 연계 산업 공급망 의사결정 지원 사업화
- **한줄 요약:** 한국기계산업진흥회(KOAMI) chatGIVC 후속 사업으로, Ontology/Knowledge Graph 기반 산업 공급망 의사결정 지원 시스템을 구축하고, KT 연계 반복 가능한 AX Offering으로 확장
- **타겟 고객:** 한국기계산업진흥회(기진회) — 산업부 산하 산업정책 지원기관, 소부장넷 운영, GIVC 기반 2,443개 품목 데이터 보유
- **타겟 도메인:** 산업정책 기관의 공급망 분석 및 정책 의사결정 지원 (B2G/B2B)
- **추진 주체:** KT DS AX사업개발본부 AX BD담당 AX BD팀 + 협력사 E8(에잇인사이트)

## 2. 문제 정의 (Problem Statement)

### 해결하려는 핵심 문제

산업정책 기관이 보유한 대량의 산업 데이터(무역, 산업통계, R&D, EWS 등)가 도메인별로 사일로화되어 있어, **품목 간 인과관계 추론, 시나리오 예측, 설명 가능한 의사결정 지원**이 불가능한 상태.

### 현재 Pain Points

1. **AI 의사결정 지원의 신뢰성 우려** — LLM 활용 시 Hallucination, 판단 오류, 근거 설명/재현성 한계. 업무 암묵지의 명시화 및 의사결정 기준 표준화 필요
2. **분산된 기관 데이터 구조** — 시스템/부서별 데이터 분산으로 데이터 간 의미 연결/관계 분석 한계. 무역, 산업, R&D, EWS, GIVC, 보고서 등 6개 도메인이 각각 독립
3. **의사결정 AI 적용 시 책임/리스크** — AI 도입 니즈는 있으나 의사결정 책임 부담으로 실제 활용 제한적
4. **chatGIVC 1차 사업의 범위 한계** — 데이터 조회/분류는 가능하나, 품목 간 인과관계 연결, "만약 ~하면?" 시나리오 예측, 영향 경로 근거 설명, 경보 기반 선제적 대응 추천은 미구현

## 3. 제안 솔루션 (Proposed Solution)

### 솔루션 개요

chatGIVC 인프라 위에 **Ontology Layer**를 추가하여, '개별 데이터 조회'에서 '데이터 간 관계 기반 인과 추론'으로 전환. 품목을 중심 엔티티로 두고, 산업/무역/R&D/EWS/보고서/GIVC 6개 도메인을 관계(생산한다, 교역된다, 연구된다, 경보된다, 분석한다, 분류된다)로 연결.

### 핵심 기능 목록

| # | 기능 | 설명 |
|---|------|------|
| 1 | **품목 간 인과관계 연결** | 원자재 -> 중간재 -> 완성품 등 공급망 관계를 Ontology로 구조적 연결 |
| 2 | **시나리오 예측 (What-if)** | 외부 충격 발생 시 어떤 산업까지 영향이 퍼지는지 KG 기반 시뮬레이션 |
| 3 | **영향 경로 근거 설명** | A->B->C 인과 경로를 투명하게 제시 (설명가능성) |
| 4 | **경보 기반 선제적 대응 추천** | EWS 경보 발생 시 관련 품목/산업 자동 식별 + 대응안 제시 |
| 5 | **취약 품목 진단** | 특정국 의존도, 대체 가능성, 전략산업 연관도 기준 구조적 취약성 분석 |
| 6 | **대응 옵션 비교 분석** | 수입 다변화 / 국산화 / R&D 집중 / 모니터링 등 옵션별 효과 시뮬레이션 |

### 시나리오 설계

- **Scenario A (충격 대응형):** 외부 충격(이란 사태, 철강 급등, 물류 차질 등) 발생 시 산업 영향 분석 및 대응 옵션 제시
  - 처리: 충격 해석 -> 영향 경로 탐색(KG) -> 산업 영향도 산출(H/M/L) -> 의사결정안 도출 -> 최종 권고
  - 산출물: 영향 경로맵 + 산업별 영향도 + 인과 근거 + 대응 옵션 비교표
- **Scenario B (취약성 대응형):** 공급망 취약 품목 진단과 대응 옵션 비교 분석
  - 처리: 취약 품목 식별 -> 취약 원인 분해 -> 선택지 제시 -> 의사결정안 도출 -> 선택 효과 시뮬레이션
  - 산출물: 취약 품목 진단 + 대응 옵션 비교 + 의존도/자립도 변화 시뮬레이션

### 기술 아키텍처 개요

| 레이어 | 기술 | 비고 |
|--------|------|------|
| 기존 인프라 (레버리지) | AI:ON-U (NextJS + Flask + Redis), sLLM, Agent 구조, GIVC 공통플랫폼 (GPU/DBMS/vectorDB) | chatGIVC 1차 사업에서 구축 완료 |
| Ontology Layer (신규) | Ontology 모델링 + Knowledge Graph 구축 | 협력사 E8 기반 |
| Graph DB (신규) | Graph DB 추가 | 기존 인프라 위에 추가 |
| AI 추론 | 도메인 특화 Agent + KG 추론 + Agentic Workflow | chatGIVC Agent 구조 확장 |

### LLM+RAG vs RDB vs Ontology+KG 비교 (선택 근거)

| 항목 | LLM+RAG | RDB (SQL) | Ontology+KG (채택) |
|------|---------|-----------|---------------------|
| 인과관계 추적 | 문서에 있으면 답함, 없으면 hallucination | 미리 설계한 JOIN만 가능 | 관계를 따라가며 인과 경로 추적 + 근거 제시 |
| 시나리오 예측 | 불가 (과거 문서 기반) | 새 질문마다 재설계 필요 | 관계 그래프에서 영향 범위 자동 추적 |
| 설명가능성 | 블랙박스 | 쿼리 투명하나 복잡 관계 표현 한계 | A->B->C 인과 경로 그대로 제시 |
| 재현성 | 같은 질문에 다른 답 가능 | 동일 쿼리 = 동일 결과 | 동일 그래프 = 동일 경로 |
| 확장성 | 문서 추가 시 재색인 | 스키마 변경 = 전체 재설계 | 엔티티/관계 추가만으로 확장 |

### 차별화 요소 (비대칭 우위 3가지)

1. **GIVC 데이터 완전 파악** — chatGIVC 직접 구축으로 2,443개 품목 데이터 구조를 이미 파악. 경쟁사는 파악에만 3개월+ 필요

### 확장 전략

- **Horizontal (공공기관):** KOTRA (공급망 조기경보 + KG), 관세청 (데이터 AI -> Ontology 확장), KIAT (산업AI 솔루션 실증 내 KG 과제) — 동일 'KG 기반 의사결정' 패턴을 공공기관에 복제
- **Vertical (산업 도메인):** 보험/금융 (심사 업무 의사결정), 제조 (설비 이상감지, 공급망 최적화), 보안/유통 (리스크 분석, 수요 예측) — 복잡한 의사결정이 필요한 산업 도메인으로 5-Layer 이식
