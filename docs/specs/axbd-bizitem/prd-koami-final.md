# KOAMI 산업 공급망 의사결정 지원 PRD (최종)

> **한줄 요약:** KOAMI chatGIVC 후속 사업으로, Ontology/KG 기반 산업 공급망 의사결정 지원 시스템을 구축하고, KT 연계 반복 가능한 AX Offering으로 확장하는 사업

## 1. 사업 개요

| 항목 | 내용 |
|------|------|
| **사업 아이템명** | KT 연계 산업 공급망 의사결정 지원 사업화 |
| **타겟 고객** | 한국기계산업진흥회(KOAMI) — 산업부 산하, 소부장넷 운영, GIVC 2,443개 품목 데이터 보유 |
| **타겟 도메인** | 산업정책 기관의 공급망 분석 및 정책 의사결정 지원 (B2G/B2B) |
| **추진 주체** | KT DS AX BD팀 + 협력사 E8(에잇인사이트) |
| **현재 단계** | 제안 PoC (약 3억, 약 8주~) |
| **전략 포지셔닝** | 기존 고객 후속 수주 + KT 연계 사업 모델 확보 (신규 개척이 아닌 확장) |

### 1.1 핵심 문제

산업정책 기관의 대량 산업 데이터(무역/산업통계/R&D/EWS 등)가 도메인별 사일로화되어 있어, **품목 간 인과관계 추론, 시나리오 예측, 설명 가능한 의사결정 지원**이 불가능한 상태.

### 1.2 Pain Points

| # | 문제 | 영향 |
|---|------|------|
| 1 | AI 의사결정의 신뢰성 우려 | Hallucination, 근거 설명/재현성 한계로 실무 활용 제한 |
| 2 | 분산된 기관 데이터 (6개 도메인 독립) | 데이터 간 의미 연결/관계 분석 불가 |
| 3 | 의사결정 AI 책임/리스크 부담 | AI 도입 니즈 있으나 실제 활용 저조 |
| 4 | chatGIVC 1차 사업 범위 한계 | 조회/분류만 가능, 인과추론/시나리오/경보대응 미구현 |

### 1.3 솔루션 개요

chatGIVC 인프라 위에 **Ontology Layer**를 추가하여 '개별 데이터 조회'에서 '데이터 간 관계 기반 인과 추론'으로 전환. 품목을 중심 엔티티로 두고, 6개 도메인(산업/무역/R&D/EWS/보고서/GIVC)을 관계로 연결.

**기술 선택 근거 (Ontology+KG):**

| 항목 | LLM+RAG | RDB (SQL) | Ontology+KG (채택) |
|------|---------|-----------|---------------------|
| 인과관계 추적 | 문서 기반, 없으면 hallucination | JOIN 설계 범위 내만 | 관계 따라 경로 추적 + 근거 제시 |
| 시나리오 예측 | 불가 (과거 문서 기반) | 재설계 필요 | 영향 범위 자동 추적 |
| 설명가능성 | 블랙박스 | 복잡 관계 표현 한계 | A->B->C 경로 그대로 제시 |
| 재현성 | 같은 질문에 다른 답 가능 | 동일 결과 | 동일 그래프 = 동일 경로 |
| 확장성 | 재색인 필요 | 스키마 전체 재설계 | 엔티티/관계 추가만으로 확장 |

### 2.2 사용자 스토리 및 수용 기준

#### US-01: 충격 영향 분석 (Scenario A)

**As a** 정책 분석관(P1), **I want to** 외부 충격 이벤트를 입력하면 관련 산업/품목의 영향 경로와 심각도를 자동 분석받고 싶다, **So that** 수작업 대비 80%+ 시간 절약 + 누락 없는 영향 범위 파악이 가능하다.

| AC | 수용 기준 |
|----|----------|
| AC-01a | 자연어로 충격 이벤트 입력 가능 |
| AC-01b | 30초 이내 영향 경로맵(A->B->C) 시각화 |
| AC-01c | 각 품목/산업에 H/M/L 영향도 + 산출 근거 표시 |
| AC-01d | 각 엣지에 관계 유형(생산/교역/연구 등) + 데이터 출처 명시 |
| AC-01e | 동일 입력 -> 동일 결과 재현 (결정적 추론) |
| AC-01f | PDF/Excel 내보내기 |

#### US-02: 취약 품목 진단 (Scenario B)

**As a** 정책 분석관(P1), **I want to** 조건(의존도/대체가능성/전략산업 연관도)으로 취약 품목을 자동 스크리닝하고 싶다, **So that** 2,443개 품목 중 전략적 주의 필요 품목을 체계적으로 식별할 수 있다.

| AC | 수용 기준 |
|----|----------|
| AC-02a | 의존도/대체가능성/전략산업 연관도 3축 필터링 |
| AC-02b | 위험도순 정렬 |
| AC-02c | 품목별 취약 원인 분해 (특정국 의존/단일 공급원/대체재 부재 등) |
| AC-02d | 대응 옵션(수입 다변화/국산화/R&D 집중/모니터링) 제시 |

#### US-03: What-if 시나리오 시뮬레이션

**As a** 의사결정자(P2), **I want to** "만약 A국 B 원자재 수출 50% 감소하면?" 시나리오를 입력하여 파급효과를 시뮬레이션하고 싶다, **So that** 선제적 대응 전략을 사전 수립할 수 있다.

## 4. 기술 아키텍처

### 4.1 전체 아키텍처

```
사용자 레이어
  분석 대시보드 | 시나리오 UI | 관리자 콘솔
        |              |            |
API Gateway (인증/인가 | Rate Limiting | 감사 로그)
        |
서비스 레이어
  충격 분석 Service | 취약성 진단 Service | What-if 시뮬레이션 Service
        |
  KG 추론 엔진 (경로 탐색 | 영향도 산출 | 시뮬레이션 | 설명 생성)
        |
데이터 레이어
  Graph DB (Neo4j) | Vector DB (기존) | RDB (기존) | Cache (Redis)
        |
데이터 수집/동기화
  GIVC ETL | 공개API 수집기 | 수동 업로드 파이프라인
```

### 4.2 기존 인프라 레버리지

| 레이어 | 기술 | 비고 |
|--------|------|------|
| 기존 (chatGIVC) | AI:ON-U (NextJS + Flask + Redis), sLLM, Agent 구조, GPU/DBMS/vectorDB | 1차 사업에서 구축 완료 |
| **신규 — Ontology Layer** | Ontology 모델링 + KG 구축 | 협력사 E8 기반 |
| **신규 — Graph DB** | Neo4j Enterprise (On-premise) | 기존 인프라 위에 추가 |
| **확장 — AI 추론** | 도메인 특화 Agent + KG 추론 + Agentic Workflow | chatGIVC Agent 확장 |

### 4.3 Graph DB 선정: Neo4j Enterprise

선정 사유: On-premise 필수 충족, Cypher 가독성, 경로 탐색 네이티브 최적화, 국내 공공기관 레퍼런스 존재. Amazon Neptune은 AWS 전용(데이터 주권 불가), JanusGraph은 레퍼런스 부족.

### 4.4 Ontology 모델링

#### 핵심 엔티티 (7종)

| 엔티티 | 속성 | 데이터 출처 |
|--------|------|-----------|
| **품목(Item)** | HS코드, GIVC코드, 명칭, 카테고리, 전략품목 여부 | GIVC DB |
| **산업(Industry)** | KSIC코드, 명칭, 상위산업, GDP 비중 | 산업통계 |
| **국가(Country)** | ISO코드, 명칭, 교역 규모, 리스크 등급 | 무역통계 |
| **기업(Company)** | 사업자번호(비식별), 명칭, 업종, 매출 규모 | 소부장넷 |
| **연구과제(RnD)** | 과제번호, 명칭, 수행기관, TRL | R&D DB |
| **경보(Alert)** | EWS ID, 유형, 심각도, 발생일, 대상품목 | EWS |
| **보고서(Report)** | 문서ID, 제목, 발행기관, 발행일, 키워드 | 보고서 DB |

#### 핵심 관계 (7종)

| 관계 | 출발 -> 도착 | 속성 | 예시 |
|------|-------------|------|------|
| `PRODUCES` | 산업 -> 품목 | 생산량, 점유율 | 석유화학 -> 에틸렌 |
| `TRADES` | 국가 -> 품목 | 수출입량, 금액, 의존도(%) | 사우디 -> 원유 |
| `SUPPLIES` | 품목 -> 품목 | 투입비율, 대체가능성 | 나프타 -> 에틸렌 |
| `RESEARCHES` | 연구과제 -> 품목 | TRL, 예산 | 고순도 소재 R&D -> 포토레지스트 |
| `ALERTS` | 경보 -> 품목 | 심각도, 유형 | EWS-2026-042 -> 희토류 |
| `ANALYZES` | 보고서 -> 품목 | 분석유형, 결론 | 공급망 보고서 -> 반도체 소재 |
| `CLASSIFIED_BY` | 품목 -> GIVC분류 | 등급 | 에틸렌 -> HS코드 |

#### 관리 규칙

- **ID 체계:** `{도메인접두어}_{고유코드}` (예: `ITEM_HS2917`, `IND_KSIC20`)
- **스키마 버전:** SemVer v1.0.0~
- **관계 추가 프로세스:** 도메인 전문가 검증 -> 스키마 변경 PR -> 테스트 데이터 검증 -> 배포

### 4.5 AI Agent 구조

| Agent | 역할 | 입력 | 출력 |
|-------|------|------|------|
| **Shock Interpreter** | 자연어 -> 구조화 파라미터 | 자연어 텍스트 | {대상국가, 품목, 변동유형, 변동률} |
| **Path Finder** | KG 영향 경로 탐색 | 구조화된 파라미터 | 영향 경로 그래프 + 노드별 영향도 |
| **Impact Scorer** | 영향도 정량화(H/M/L) | 영향 경로 그래프 | 영향도 매트릭스 |
| **Option Generator** | 대응 옵션 생성 + 효과 추정 | 영향도 + 품목 속성 | 대응 옵션 비교표 |
| **Explainer** | 자연어 분석 보고서 생성 | 경로 + 영향도 | 자연어 보고서 |
| **Vulnerability Scanner** | 취약성 스크리닝 | 필터 조건 | 취약 품목 목록 + 원인 분해 |

**오케스트레이션:** 입력 -> Shock Interpreter -> Path Finder -> Impact Scorer -> Option Generator -> Explainer -> 결과 반환

### 4.6 추론 엔진

| 추론 유형 | 방법 | 정확도 목표 |
|----------|------|-----------|
| 경로 탐색 | Cypher shortest_path + 가중치 전파 | 재현율 100% (결정적) |
| 영향도 산출 | 투입비율 x 의존도 x 대체가능성 가중 합산 | 전문가 대비 75%+ 일치 |
| 시나리오 시뮬레이션 | Monte Carlo 1,000회 반복 | 95% 신뢰구간 |
| 자연어 해석 | sLLM + 프롬프트 엔지니어링 | 구조화 정확도 90%+ |
| 설명 생성 | 템플릿 + KG 패스 기반 | 사실 정확도 100% (KG 기반) |

### 10.2 PoC 범위 (한정 시나리오)

**Scenario A' (석유화학):**
- 범위: 원유->나프타->에틸렌->PE/PP->완성품 (약 20~30개 품목)
- KG: 엔티티 약 50개 / 관계 약 100개
- 시연 예시: "이란 원유 수출 30% 감소" 입력
  - 영향 경로: 원유 -> 나프타(생산 20%↓) -> 에틸렌(15%↓) -> PE/PP(10%↓) -> 자동차 부품(5%↓)
  - 영향도: 석유화학 H, 플라스틱 가공 M, 자동차 L
  - 대응 옵션: 사우디/UAE 대체 확대 / 에탄 크래커 전환 / 전략 비축
- 검증: KOAMI 분석관 수작업 분석과 비교 -> 일치율 측정

**Scenario B' (반도체 소재):**
