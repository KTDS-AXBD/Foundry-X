## 1. 사용자 스토리 및 수용 기준 (User Stories & Acceptance Criteria)

### 1.1 페르소나 정의

| ID | 페르소나 | 역할 | 핵심 니즈 |
|----|---------|------|----------|
| P1 | **정책 분석관** | KOAMI 산업분석실 연구원 | 품목 간 인과관계를 빠르게 파악하여 정책 보고서 작성 시간 단축 |
| P2 | **의사결정자** | KOAMI 국장/본부장급 | 외부 충격 발생 시 30분 내 영향 범위 파악 및 대응 옵션 확인 |
| P3 | **데이터 운영자** | KOAMI 정보시스템팀 | KG 데이터의 정기 갱신, 품질 모니터링, 시스템 안정성 확보 |
| P4 | **외부 이해관계자** | 산업부 정책 담당자 | KOAMI로부터 받은 분석 결과의 근거 투명성 확인 |

### 1.2 사용자 스토리

#### US-01: 충격 영향 분석 (Scenario A)

**As a** 정책 분석관(P1),
**I want to** 외부 충격 이벤트(예: "이란 원유 수출 중단")를 입력하면 관련 산업/품목의 영향 경로와 심각도를 자동으로 분석받고 싶다,
**So that** 수작업 분석 대비 80% 이상 시간을 절약하고, 누락 없는 영향 범위를 파악할 수 있다.

**수용 기준(AC):**
- [ ] 자연어로 충격 이벤트를 입력할 수 있다
- [ ] 30초 이내에 영향 경로맵(A→B→C 형태)이 시각화된다
- [ ] 각 품목/산업에 H/M/L 영향도가 부여되며, 산출 근거가 표시된다
- [ ] 영향 경로의 각 엣지에 관계 유형(생산, 교역, 연구 등)과 데이터 출처가 명시된다
- [ ] 동일 입력에 동일 결과가 재현된다 (결정적 추론)
- [ ] 분석 결과를 PDF/Excel로 내보내기할 수 있다

#### US-02: 취약 품목 진단 (Scenario B)

**As a** 정책 분석관(P1),
**I want to** 특정 조건(예: "중국 의존도 70% 이상, 대체 공급국 2개 이하")으로 취약 품목을 자동 스크리닝하고 싶다,
**So that** 2,443개 품목 중 전략적 주의가 필요한 품목을 체계적으로 식별할 수 있다.

**수용 기준(AC):**
- [ ] 의존도, 대체 가능성, 전략산업 연관도 3가지 축으로 필터링할 수 있다
- [ ] 취약 품목 목록이 위험도 순으로 정렬된다
- [ ] 각 품목의 취약 원인이 분해되어 표시된다 (특정국 의존, 단일 공급원, 대체재 부재 등)
- [ ] 품목별 대응 옵션(수입 다변화/국산화/R&D 집중/모니터링)이 제시된다
- [ ] 대응 옵션별 예상 효과(의존도 변화율, 소요 기간, 비용 추정)가 비교표로 제공된다

#### US-03: What-if 시나리오 시뮬레이션

**As a** 의사결정자(P2),
**I want to** "만약 A국의 B 원자재 수출이 50% 감소하면?"과 같은 가상 시나리오를 입력하여 파급 효과를 시뮬레이션하고 싶다,
**So that** 선제적 대응 전략을 사전에 수립할 수 있다.

**수용 기준(AC):**
- [ ] 시나리오 파라미터(국가, 품목, 변동률, 기간)를 조합하여 입력할 수 있다
- [ ] 1차/2차/3차 파급 효과가 단계별로 구분되어 표시된다
- [ ] 복수 시나리오를 동시에 비교할 수 있다 (최대 3개)
- [ ] 시뮬레이션 결과에 신뢰도 등급(High/Medium/Low)이 표시된다
- [ ] 과거 유사 사례가 있을 경우 참고 사례로 함께 표시된다

#### US-04: 경보 기반 선제 대응

**As a** 의사결정자(P2),
**I want to** EWS(조기경보시스템) 경보가 발생하면 관련 품목/산업이 자동으로 식별되고 대응 옵션이 추천되길 원한다,
**So that** 경보 발생 후 1시간 이내에 초기 대응 방향을 결정할 수 있다.

**수용 기준(AC):**
- [ ] EWS 경보 발생 시 자동으로 관련 KG 경로가 활성화된다
- [ ] 경보 심각도(Critical/Warning/Watch)별로 대응 긴급도가 차등 표시된다
- [ ] 이전 유사 경보의 실제 영향과 대응 이력이 참고로 제공된다
- [ ] 대응 추천안에 실행 주체(KOAMI/산업부/기업)가 구분된다

#### US-05: KG 데이터 관리

**As a** 데이터 운영자(P3),
**I want to** KG의 엔티티/관계를 추가·수정·삭제하고, 데이터 품질을 모니터링하고 싶다,
**So that** 분석 결과의 신뢰성을 지속적으로 유지할 수 있다.

**수용 기준(AC):**
- [ ] 관리자 UI에서 엔티티/관계를 CRUD할 수 있다
- [ ] 외부 데이터 소스(UNIPASS, KOSIS 등)의 동기화 상태를 대시보드에서 확인할 수 있다
- [ ] 데이터 신선도(최종 갱신일) 경고가 자동 발생한다 (7일 이상 미갱신 시)
- [ ] KG 변경 이력(audit log)이 기록된다

#### US-06: 분석 결과 근거 검증

**As a** 외부 이해관계자(P4),
**I want to** 분석 결과의 인과 경로를 단계별로 추적하고, 각 단계의 데이터 출처를 확인하고 싶다,
**So that** 정책 제안의 근거를 투명하게 검증할 수 있다.

**수용 기준(AC):**
- [ ] 인과 경로의 각 노드를 클릭하면 원천 데이터(통계치, 보고서 원문 등)로 이동한다
- [ ] 데이터 출처별 신뢰도 등급이 표시된다 (공식 통계 > 보고서 > 추정치)
- [ ] 분석 보고서에 "데이터 출처 및 제한사항" 섹션이 자동 포함된다

### 2.2 보안

| 요구사항 | 상세 | 근거 |
|---------|------|------|
| **데이터 주권** | 모든 데이터를 국내 On-premise에 저장·처리. 해외 클라우드 전송 금지 | 산업부 산하기관 데이터 정책 |

| **감사 로그** | 모든 쿼리/분석 요청에 대한 감사 추적 (사용자, 시간, 입력, 결과 해시) | 공공기관 AI 활용 감사 요건 |

### 3.2 Graph DB 선정 기준

| 기준 | Neo4j (권장) | Amazon Neptune | JanusGraph |
|------|-------------|---------------|------------|
| On-premise 배포 | O (Enterprise) | X (AWS only) | O (OSS) |
| Cypher/SPARQL 지원 | Cypher (네이티브) | SPARQL + Gremlin | Gremlin |
| 경로 탐색 성능 | 우수 (네이티브 그래프) | 양호 | 양호 |
| Ontology 연동 | OWL/RDF 임포트 지원 | RDF 네이티브 | 플러그인 |
| 국내 레퍼런스 | 금융/공공 다수 | AWS 의존 | 제한적 |
| 라이선스 비용 | Community(무료) / Enterprise(유료) | 사용량 과금 | 무료(OSS) |

**권장**: Neo4j Enterprise — On-premise 필수, Cypher의 가독성, 경로 탐색 네이티브 최적화, 국내 공공기관 레퍼런스 존재.

### 3.4 AI Agent 구조 상세

| Agent | 역할 | 입력 | 출력 |
|-------|------|------|------|
| **Shock Interpreter** | 자연어 충격 이벤트를 구조화된 파라미터로 변환 | 자연어 텍스트 | {대상국가, 품목, 변동유형, 변동률} |
| **Path Finder** | KG에서 영향 경로를 탐색하고 depth별 전파 계산 | 구조화된 충격 파라미터 | 영향 경로 그래프 + 각 노드 영향도 |
| **Impact Scorer** | 각 품목/산업의 영향도를 H/M/L로 정량화 | 영향 경로 그래프 | 영향도 매트릭스 |
| **Option Generator** | 대응 옵션을 생성하고 효과를 추정 | 영향도 매트릭스 + 품목 속성 | 대응 옵션 비교표 |
| **Explainer** | 인과 경로를 자연어 설명으로 변환 | 영향 경로 + 영향도 | 자연어 분석 보고서 |
| **Vulnerability Scanner** | 취약성 조건에 따른 품목 스크리닝 | 필터 조건 | 취약 품목 목록 + 원인 분해 |

**Agent 오케스트레이션 흐름:**
1. 사용자 입력 → Shock Interpreter (구조화)
2. Path Finder (KG 탐색, Cypher 쿼리)
3. Impact Scorer (정량화 — KG 관계 가중치 + 도메인 룰)
4. Option Generator (대응안 도출)
5. Explainer (자연어 보고서 생성)
6. 사용자에게 결과 반환 (시각화 + 보고서)

### 3.5 추론 엔진 상세

| 추론 유형 | 방법 | 정확도 목표 |
|----------|------|-----------|
| **경로 탐색** | Cypher shortest_path + 가중치 전파 | 재현율 100% (결정적) |
| **영향도 산출** | 투입비율 × 의존도 × 대체가능성 가중 합산 | 전문가 평가 대비 75%+ 일치 |
| **시나리오 시뮬레이션** | Monte Carlo 기반 변동 전파 (1,000회 반복) | 95% 신뢰구간 제공 |
| **자연어 해석** | sLLM (chatGIVC 기존) + 프롬프트 엔지니어링 | 구조화 정확도 90%+ |
| **설명 생성** | 템플릿 + KG 패스 기반 자연어 변환 | 사실 정확도 100% (KG 데이터 기반) |

### 7.3 PoC 시연 시나리오 (구체화)

**Scenario A' (석유화학):**
1. 입력: "이란 원유 수출 30% 감소"
2. 시스템 출력:
   - 영향 경로: 원유 → 나프타(생산 20% 감소) → 에틸렌(15% 감소) → PE/PP(10% 감소) → 자동차 부품(5% 감소)
   - 영향도: 석유화학 H, 플라스틱 가공 M, 자동차 L
   - 대응 옵션: (1) 사우디/UAE 대체 수입 확대, (2) 에탄 크래커 전환 가속, (3) 전략 비축 확대
3. 검증: KOAMI 분석관이 수작업 분석 결과와 비교 → 일치율 측정

**Scenario B' (반도체 소재):**

## 10. 추가 고려사항

### 10.1 경쟁 대응 전략

현재 국내 산업정책 기관에 KG/Ontology를 도입한 사례가 없는 **초기 시장**이므로:
- **선점 전략**: PoC 성공 → 레퍼런스 → 타 기관 제안의 선순환을 6개월 내 확립
- **방어 해자**: GIVC 데이터 구조 파악 + Ontology 스키마는 도메인 특화 자산 → 경쟁사 진입에 3~6개월 리드타임
- **글로벌 대비**: Blue Yonder/o9 등의 한국 시장 진입 시, "국내 데이터 주권 + On-premise + 한국어" 차별화

### 10.2 기술 부채 관리

| 항목 | PoC 허용 부채 | 본사업 해소 방안 |
