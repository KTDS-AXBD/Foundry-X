## 검토 의견: FX-BD-V2 PRD

### 1. 기술적 실현 가능성
**제안된 스택으로 핵심 기능 구현은 가능하나, 심각한 아키텍처 리스크 존재**

- **긍정적 요소**: 기존 Foundry-X(Next.js/Hono/Cloudflare) 스택을 확장하는 접근은 합리적. 단계별 워크플로우, 문서 생성, 기본 알림은 구현 가능.
- **주요 제약 및 병목**:
    1. **Cloudflare D1의 프로덕션 부적합성**: 문서 버전 히스토리, 워크플로우 상태 추적, 액션 로그 등 **상태 기반 데이터가 많은 본 프로젝트에 D1은 위험**. 트랜잭션 보장, 백업/복구, 성능 측면에서 한계가 명확. 장기적으로 PlanetScale, Neon(PostgreSQL) 등으로의 마이그레이션 계획이 PRD에 포함되었으나, **초기 설계부터 이중 구조를 고려해야 함**.
    2. **이벤트 기반 아키텍처 미비**: 단계 전환 알림, 리뷰 요청, 문서 자동 생성 등 비동기 작업 처리를 위한 큐/워커 시스템이 명시되지 않음. Cloudflare Queues나 별도 메시지 브로커(RabbitMQ, Redis) 도입이 필수적이나, 현재 스택에는 포함 안 됨.
    3. **문서 패키징 성능**: Offering Pack/ORB 문서 생성 시 여러 산출물(BMC, PRD, 프로토타입)을 실시간으로 병합·압축할 경우, Cloudflare Workers의 CPU 시간/메모리 제한(10ms~50ms CPU)을 초과할 위험. **비동기 배치 작업으로 전환해야 함**.

### 2. 아키텍처 적합성
**현재 제안된 구조는 확장성과 유지보수성에 심각한 결함**

- **데이터 계층 설계 부재**: "사업 아이템" 하나에 메타데이터, 단계 상태, 문서 버전, 리뷰 코멘트, 액션 로그가 모두 D1 단일 테이블에 혼재될 가능성 높음. **도메인 주도 설계(DDD) 접근이 필요**: `Item`, `WorkflowState`, `DocumentVersion`, `ActivityLog` 등 Aggregates 분리.
- **워크플로우 엔진 없음**: "단계 전환 자동화"가 상태 머신 수준으로 기술됨. 조건부 전환(예: 모든 리뷰어 승인 후 Go), 롤백, 타임아웃 처리 등 복잡한 비즈니스 룰을 하드코딩하면 유지보수 불가능. **유한 상태 머신(FSM) 라이브러리(XState)나 워크플로우 엔진 도입 검토 필요**.
- **모놀리식 서비스 경향**: 310개 엔드포인트를 가진 기존 Foundry-X에 모든 기능을 추가하면 결합도가 급증. **BD 파이프라인 전용 마이크로서비스(또는 별도 Worker) 분리를 고려해야 함**. 특히 문서 생성/압축 같은 고부하 작업은 분리 필수.

### 3. 구현 복잡도
**다음 요소들이 심각하게 과소평가됨**

- **동시성 및 데이터 일관성**: 여러 팀원이 동일한 BDP 문서를 편집할 때의 충돌 처리. **실시간 협업은 Out-of-scope지만, 낙관적 락/버전 충돌 감지는 Must-have**. 현재 PRD에는 언급 없음.
- **권한 및 공유 모델 복잡성**: "공유 링크 생성 시 인증 필수"는 간단해 보이지만, Offering Pack을 외부에 공유할 때 **부분 정보 마스킹(가격 정보 제외), 링크 만료/회수, 다운로드 횟수 추적** 등 요구사항이 확장될 것. RBAC 설계가 복잡해짐.
- **외부 시스템 연동의 숨겨진 복잡성**: "KT DS ORB/PRB 문서 생성 (수동 제출)"로 기재되었으나, 실제로는 사내 포털의 업로드 API 호출, 양식 검증, 제출 상태 동기화 등이 필요할 수 있음. **연동 범위가 불명확하여 스코프 크리프 위험 높음**.
- **알림 시스템 설계**: "통합 알림/Notification 설계(P0)"는 단순 이메일 발송이 아님. **사용자별 알림 설정(이메일/인앱), 읽음 상태 동기화, 대량 발송 최적화** 등 미니 포털 수준의 복잡도.

### 4. 기술 리스크
**가장 우려되는 3가지 실패 요인**

1. **D1 데이터 손실/성능 저하로 인한 신뢰성 붕괴**: 프로덕션에서 워크플로우 상태가 소실되거나, 문서 버전이 꼬이면 팀의 업무 자체가 마비. **초기부터 D1과 외부 PostgreSQL을 이중화하는 하이브리드 아키텍처가 안전**.
2. **워크플로우 엔진 부재로 인한 스파게티 코드**: 단계 전환 로직이 여러 서비스에 흩어지고, 조건이 변경될 때마다 수동 디버깅 필요. **유지보수 비용이 폭발하며, 6개월 내 기술 부채로 프로젝트 실패 가능성**.
3. **1인 개발자 병목 및 아키텍처 결정권 한계**: Sinclair 개발자 한 명이 아키텍처, 프론트, 백엔드, 인프라, 보안을 모두 결정·구현해야 함. **중요한 아키텍처 결정(FSM 도입, 데이터 계층 분리)에 대한 검토 체계가 없음**.
