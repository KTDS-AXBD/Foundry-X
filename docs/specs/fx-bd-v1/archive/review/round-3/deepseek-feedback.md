## 기술 검토 의견: FX-BD-V2 PRD v3

### 1. 기술적 실현 가능성: **제한적 가능성 (중대한 제약 존재)**

**긍정적 요소:**
- 기존 Foundry-X 인프라(Cloudflare Workers, D1, Pages) 활용으로 빠른 프로토타이핑 가능
- 단일 개발자 환경에 적합한 단순한 시작점 제공

**중요한 기술적 제약:**

**A. 데이터 계층의 치명적 취약성**
- **Cloudflare D1의 프로덕션 부적합성**: D1은 현재 실험적 단계로, 데이터 손실 가능성과 성능 한계가 명시됨. BD 파이프라인의 핵심 비즈니스 데이터(사업 아이템, 의사결정 이력)를 D1 단독으로 저장하는 것은 **수용 불가능한 리스크**.
- **트랜잭션 및 관계형 데이터 요구**: 워크플로 상태 관리, 문서 버전관리, 권한 체계는 복잡한 관계형 쿼리와 트랜잭션을 필요로 함. D1의 제한된 기능으로는 구현이 매우 어렵거나 불안정함.

**B. 이벤트 기반 아키텍처 부재**
- "통합 알림", "단계 전환 자동화", "리뷰 요청"은 본질적으로 비동기 이벤트 드리븐 아키텍처가 필요.
- 현재 제안된 Hono+Workers 스택은 요청-응답 모델에 최적화되어 있음. 이벤트 큐, 재시도, 배치 처리 없이는 알림 누락, 상태 불일치가 빈번히 발생할 것.

**C. 문서 처리 및 패키징의 확장성 문제**
- "ORB/PRB 문서 자동 생성", "Offering Pack 패키징"은 동시에 여러 문서(BMC, PRD, 프로토타입)를 병합/변환하는 고부하 작업.
- Cloudflare Workers의 CPU 시간 제한(10ms~50ms의 엣지 런타임) 내에서 안정적으로 수행하기 어려움. 장시간 실행 작업은 별도 처리 서비스가 필수.

### 2. 아키텍처 적합성: **부적절 - 재설계 필요**

**현재 제안된 모놀리식 확장 방식의 문제점:**
- 기존 Foundry-X(310개 엔드포인트)에 BD 전용 복잡한 워크플로를 직접 결합하면 **시스템 복잡도가 기하급수적으로 증가**.
- BD 모듈의 장애가 전체 Foundry-X 서비스에 영향을 미칠 수 있음.

**제안하는 아키텍처 개선 방향:**

```
[프론트엔드: Next.js]
        |
[API Gateway / BFF] - (기존 Foundry-X API와 분리)
        |
        |-----------------------|
        |                       |
[BD 워크플로 서비스]      [기존 Foundry-X 서비스]
        |                       |
[이벤트 버스/큐]           [공유 데이터 계층]
        |                       |
[문서 생성/패키징 워커]    [프로덕션급 RDBMS]
        |                       |
[알림 서비스]              (PlanetScale/Neon/PostgreSQL)
```

**핵심 변경 권고사항:**
1. **데이터 계층 이중화**: D1을 캐시/임시 저장소로만 사용하고, **PlanetScale 또는 Neon(Serverless Postgres)** 을 주요 데이터 저장소로 도입.
2. **BD 전용 마이크로서비스 분리**: 워크플로 엔진, 문서 처리, 알림 서비스를 별도 Cloudflare Worker 또는 별도 서버리스 함수로 분리.
3. **이벤트 드리븐 코어**: 상태 변경, 리뷰 완료 등 주요 이벤트는 큐(Cloudflare Queues, Kafka)를 통해 비동기 처리.
4. **워크플로 엔진 도입**: XState와 같은 유한 상태 머신(FSM) 라이브러리를 사용해 `Go/Hold/Drop` 및 단계 전환 로직을 하드코딩하지 않고 선언적으로 관리.

### 3. 구현 복잡도: **심각하게 과소평가됨**

**숨겨진 복잡성 요소:**

**A. 상태 관리와 동시성 문제**
- 동일 아이템에 대해 팀원이 문서를 수정하는 동시에 팀장이 `Hold` 결정을 내리는 경우의 **낙관적 동시성 제어** 필요.
- 워크플로 상태(단계)와 문서 버전, 의사결정 이력 간의 **트랜잭션 일관성** 보장이 필수적.

**B. 문서 버전관리 및 델타 처리**
- "BDP 편집/버전관리"는 단순 마크다운 저장이 아닌, 변경 이력 추적, diff 비교, 특정 버전으로의 롤백 기능을 요구함.
- 대용량 문서(프로토타입 빌드 결과물)의 스토리지 관리 전략이 전혀 정의되지 않음.

**C. 권한과 공유 모델의 복잡성**
- "외부 공유 링크 생성/만료/회수"는 정교한 토큰 기반 인가 시스템 필요.
- ORB 심사위원(외부), 경영진(내부), BD팀원 간의 **계층적 권한 모델** 설계가 간단하지 않음.

**D. 오프라인/동기화 고려사항**
- PRD에서 오프라인 작업을 Out-of-scope로 명시했지만, BD팀원의 현장 업무(고객 미팅) 중 데이터 입력 필요성을 고려할 때, 오프라인 우선(Offline-first) 설계 고려가 장기적으로 필요할 수 있음.

### 4. 기술 리스크: **상위 3개 우선순위**

1. **데이터 손실 리스크 (Critical)**: Cloudflare D1 단독 사용으로 인한 비즈니스 크리티컬 데이터(사업 아이템, 의사결정) 유실 가능성. **가장 큰 실패 요인**.
2. **워크플로 신뢰성 리스크 (High)**: 이벤트 기반 아키텍처 부재로 인한 알림 누락, 상태 전환 실패, 데이터 불일치가 발생하면 시스템에 대한 신뢰도가 즉시 붕괴.
3. **성능 및 확장성 리스크 (Medium)**: 문서 패키징 및 동시 사용자 처리 시 Cloudflare Workers의 제한된 런타임 환경에서 성능 저하 및 타임아웃 장애 발생 가능성.

### 5. 종합 평가 및 권고사항

**현재 상태로의 착수는 기술적 무모함.**

PRD는 비즈니스 요구사항을 잘 정리했으나, 이를 구현할 기술 기반과 아키텍처 설계가 현실적이지 않습니다. 특히 데이터 저장소 선택과 이벤트 처리 방식은 프로덕션 시스템으로서 **근본적인 결함**을 가지고 있습니다.
