# AX BD팀 사업개발 프로세스 정의

## §1 개요

AX BD팀 사업개발 업무의 전체 라이프사이클을 6단계로 정의한 프로세스 체계안.
AI Agent(Discovery-X, Foundry-X)와 Human-In-The-Loop(HITL)을 결합한 하이브리드 접근.

**출처**: Figma Board — AX-Discovery-Process v0.8 (2026)
**목적**: Foundry-X 플랫폼에 사업개발 워크플로우를 반영하기 위한 기초 프로세스 정의

## §2 목표(안)

| 목표 | 기간 |
|------|------|
| 2주에 1개 아이템 선정 및 팀 내 공유 | 상시 |
| 4주 내 사업기획서 및 Prototype 제작 완료 | 상시 |
| 매월 1개 아이템 본부 내 보고 | 월간 |
| '26년 연간 Q개 아이템 선제안 | 연간 |
| '26년 연간 연판 매출 OO억원 | 연간 |

## §3 6단계 프로세스 체계

### 전체 흐름

```
1. 수집 → 2. 발굴 → 3. 형상화 → 4. 검증 및 공유 → 5. 제품화 → 6. GTM
```

### 범례

- 🤖 : AI Agent
- 👤 : AX BD팀 담당자
- 👥 : 타 조직 협업
- ℹ️ : 사업화 의사결정 (Go / Hold / Drop)

---

### 1단계: 수집

사업 아이템을 다양한 채널에서 수집하는 단계.

| 항목 | 주체 | 설명 |
|------|------|------|
| 1-1. Agent 자동 수집 | 🤖 Discovery-X | Agent가 자동으로 사업 아이템 수집 |
| 1-2. Field-driven 수집 | 👤 AX BD팀 | 자체, 고객, 영업, 벤치마크 등 현장 기반 수집 |
| 1-3. 전사 Bottom-up 수집 | 👥 IDEA Portal | 전사 임직원 아이디어 수집 (IDEA Portal) |

**산출물**: 사업 아이템 목록 (예: "Ontology 기반 손해사정 의사결정 지원")
**후처리**: AX BD팀 담당자 자체 Screening

---

### 2단계: 발굴 (Discovery-X 핵심)

수집된 아이템을 AI + HITL로 분석·평가하는 핵심 단계.

#### 2-0. 사업 아이템 분류 (Item Classification Agent)

AI Agent가 **자연어 대화 3턴**으로 아이템을 유형 분류하고, 해당 유형에 맞는 분석 경로 제안.

| Turn | 질문 예시 |
|------|----------|
| Turn 1 (출처 파악) | "이 아이템은 어디서 시작됐나요? 참고한 서비스·URL이 있나요, 아니면 기술 키워드나 고객 요청인가요?" |
| Turn 2 (핵심 강점) | "현재 갖고 계신 자료가 어느 수준인가요? 레퍼런스 자료/시장 데이터/고객 인터뷰 중 무엇이 있나요?" |
| Turn 3 (초점 검증) | "KT DS 관점 수익 등가는 무엇인가요? 신사업 개척/기존 고객 확장/수익모델 전환 중 어디에 가깝나요?" |

#### 3가지 유형 (Type A / B / C)

| 유형 | 이름 | 설명 | 핵심 분석 |
|------|------|------|----------|
| **Type A** | 서비스 벤치마크형 | 실존하는 서비스·플랫폼·사업모델을 KT DS형으로 전환 | 2-1 레퍼런스(핵심), 2-4 도출(KT DS 전환 로직) |
| **Type B** | 기술·트렌드 탐색형 | 기술 키워드·트렌드·리포트에서 기회를 포착 | 2-2 시장검증(핵심), 2-5 선정(가설 기반 스코어링) |
| **Type C** | 고객 Pain 기반형 | 영업·현장·고객 인터뷰에서 구체적 수요가 출발점 | 2-6 고객정의(핵심, 페르소나/JTBD), 2-7 BM(핵심, BM 실제 심화) |

#### 2-1 ~ 2-7: 유형별 분석 경로

각 단계는 **HITL** 방식 — AI가 분석 초안을 제공하지만, 담당자가 직접 프롬프트를 조정하고 대화하며 내용을 수정·검증하는 것이 핵심.

| 단계 | 분석 항목 | Type A | Type B | Type C | 방법론·프레임워크 |
|------|----------|--------|--------|--------|-----------------|
| 2-1 | 레퍼런스 분석 | **핵심** (서비스 3레이어 해부) | 간소 (트렌드 리서치) | 스킵 가능 | Competitive Analysis, 3-Layer Deconstruction |
| 2-2 | 수요 시장 검증 | 간소 | **핵심** (가설 수립·검증 루프) | 간소 (Pain 체감인) | TAM/SAM/SOM, Market Sizing, Association Mapping |
| 2-3 | 경쟁·자사 분석 | **핵심** (포지셔닝 공간 발굴) | 보통 | 보통 | SWOT, Porter's 5 Forces, Blue Ocean (ERRC) |
| 2-4 | 사업 아이템 도출 | **핵심** (KT DS형 전환 로직) | **핵심** (기획 탐색 아이디에이션) | 간소 (이해 반영) | Opportunity Solution Tree, BMC, JTBD |
| 2-5 | 핵심 아이템 선정 | 보통 | **핵심** (가설 기반 스코어링) | 간소 | Opportunity Scoring Matrix, ICE Scoring |
| 2-6 | 타겟 고객 정의 | 보통 | 보통 | **핵심** (페르소나/JTBD) | User Personas, Customer Journey Map, Value Proposition |
| 2-7 | 비즈니스 모델 정의 | 보통 | 보통 | **핵심** (BM 실제 심화) | BMC, Value Canvas, Unit Economics, KTO 라이선스 M/C |

#### 2-8. 발굴 결과 패키징 (Discovery Output Report)

2-1~2-7 결과를 통합하여 **3단계 형상화의 Input Deck**으로 완성.
GTM 초안과 Beachhead 시장을 확인하는 단계.

- 방법론: GTM Strategy, Beachhead Segment, Executive Summary, Pyramid Structure

#### 2-9. AI 멀티 페르소나 사전 평가

**8개 KT DS 내부 페르소나**를 AI Agent로 구현하여 발굴 결과를 자동 평가.
담당자가 AI 평가 결과를 직접 확인하고 각 페르소나의 논점을 파악·조정하는 것이 핵심 (HITL).

| # | 페르소나 | 핵심 평가 관점 |
|---|---------|--------------|
| 1 | 전략기획팀장 | 전략적합성, 시장규모, 성장잠재력 |
| 2 | 영업총괄부장 | 수주확보 가능성, 기존고객 확장성, 영업난이도 |
| 3 | AP사업본부장 | 기술실현 가능성, 자원투입비, 타임라인 |
| 4 | AI기술본부장 | 기술차별성, AI 활용 적합성, 데이터 확보 |
| 5 | 경영기획팀장 | 재무타당성, ROI, 투자회수기간 |
| 6 | 보안전략팀장 | 보안위험, 컴플라이언스, 데이터 거버넌스 |
| 7 | 대외협력팀장 | 파트너십, 규제환경, 시장진입 장벽 |
| 8 | 기술사업화PM | 사업화 실행력, 리스크, MVP 빌드 가능성 |

**평가 기준 (8축 레이더 차트)**:
사업성/사업타당성, 전략적합성, 고객가치, 기술시장성, 실행력/리스크, 재무타당성, 경쟁차별화, 확장가능성

**의사결정 기준**:
- 🟢 **G (Green)**: 점수 상위, 핵심 쟁점 없음 → 형상화 진행
- 🟡 **K (Keep)**: 일부 쟁점 존재, 보완 후 재평가
- 🔴 **R (Red)**: 핵심 쟁점 다수, Drop 또는 피봇 권장

#### 2-10. AX BD팀 공유 및 검토

2-1~2-9 전체를 팀 검토 Deck으로 통합. 질문·토론 안건을 도출해 3단계 형상화 Handoff.

**팀 의사결정**: Go → 3단계 형상화 진입 / Hold → 보완 / Drop → 종료

---

### 3단계: 형상화

| 항목 | 주체 | 설명 |
|------|------|------|
| 3-1. 사업계획서 초안 생성 | 🤖 Discovery-X | AI 기반 초안 자동 생성 |
| 3-2. 사업계획서 작성 | 👤 AX BD팀 | B2B 의사결정 지원 사업계획서 참고하여 최종 작성 |
| 3-3. PRD 생성 | 🤖 Foundry-X | PRD 자동 생성 (HITL) |
| 3-4. Prototype 생성 | 🤖 Foundry-X | 디자인시스템 기반 Prototype 자동 생성 |

**산출물**: 사업계획서 + Prototype

### 4단계: 검증 및 공유

```
고객·도메인 전문가 인터뷰
        ↓
내부 유관부서 미팅 (영업·경영기획 등)
        ↓
본부 내 보고 및 검토
        ↓
ℹ️ 본부 의사결정: Go / Hold / Drop

kt ds 전사: 비용·투자 심의 → Pre-PRB
        ↓
임원 보고 및 검토
        ↓
ℹ️ 임원 의사결정: Go / Hold / Drop
```

## §4 Pain Points 및 자동화 기회

| 단계 | Pain Point | 자동화 방향 |
|------|-----------|-----------|
| 1. 수집 | 사업 아이템 수집 채널 부족 (고객·영업 접점 부족, Bottom-up 활성화 안 됨) | AI 자동 수집 채널 다각화 (자동/자체/전사 연계) |
| 2. 발굴 | 담당자별 분석 방법론 부재, 사업성 판단 어려움, RNP 500개 Use Case 선별 불가 | AX BD팀 방법론 기반 사업성 분석·평가 자동화 (Skill.md 구성) |
| 3. 형상화 | 문서 작성해도 전달 안 됨, '계리', 'B2C', 'Physical AI' 등 설득력 부족 | 설득력 있는 사업기획서 및 프로토타입 기반 공유·보고 |
| 4. 검증 | "AX BD팀이 뭐하는 팀?" — 가시성·신뢰도 부족 | 근거와 형상화 자료 기반 상위 보고 및 사내 공유 |

## §6 Discovery-X / Foundry-X 역할 분담
