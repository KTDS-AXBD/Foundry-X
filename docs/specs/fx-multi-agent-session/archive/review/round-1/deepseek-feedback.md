## 기술 검토 의견

### 1. 기술적 실현 가능성
**부분적 가능성 있으나 상당한 기술적 제약 존재**

**주요 제약사항:**
- **Claude Squad의 미성숙도**: 오픈소스 도구로서 프로덕션 안정성 검증 부족. WSL2 + tmux 3.5a 특이 환경에서의 호환성은 높은 리스크
- **이중 lifecycle 관리**: 기존 `task-daemon.sh`와 cs의 병행 운영 설계가 명확하지 않음. 데몬 간 상태 충돌 가능성 높음
- **상태 동기화 문제**: cs CLI 출력 파싱 vs 파일 기반 vs D1 연동 중 선택이 미정. 각 방식 모두 단점 존재:
  - CLI 파싱: 출력 형식 변경에 취약
  - 파일 기반: 실시간성 보장 어려움
  - D1 연동: cs 수정 필요 (오픈소스 의존성)

### 2. 아키텍처 적합성
**외부 도구 의존성으로 인한 아키텍처 취약점 도입**

**문제점:**
- **Vendor Lock-in 위험**: cs에 대한 깊은 의존성 형성. cs 개발 중단 시 전체 시스템 영향
- **관심사 분리 위반**: 세션 관리(infra)와 워크플로우(business logic)가 cs에 결합됨
- **모니터링 격차**: cs 내부 상태를 외부에서 모니터링할 표준 인터페이스 부재

**대안 제안:**
```
[현재] sprint → cs → tmux/worktree
[제안] sprint → fx-session-manager(thin wrapper) → cs/tmux/worktree
```
- cs를 추상화하는 얇은 레이어 도입
- 세션 상태를 표준 포맷(D1/파일)으로 추출하는 인터페이스 강제
- cs 실패 시 fallback 메커니즘(기존 방식) 보유

### 3. 구현 복잡도
**과소평가된 복잡도 다수 존재**

**숨겨진 복잡성:**
1. **동시성 제어**: 3개 cs 세션 + 기존 데몬 + 수동 tmux 세션 간 리소스 경합
   - 공유 리소스: git index, worktree 목록, tmux 서버
2. **에지 케이스**:
   - cs 세션 비정상 종료 시 orphaned worktree 정리
   - 네트워크 불안정 시 Claude API 호출 실패 처리
   - 세션 상태 전이(busy→idle→done)의 경계 조건 정의
3. **데이터 일관성**: 웹 polling(5초)과 실제 세션 상태 간 지연으로 인한 불일치

### 4. 기술 리스크
**우선순위별 기술 리스크:**

**P0 (치명적):**
- cs의 WSL2 호환성 실패 → 전체 접근법 무효화
- 기존 task-daemon과의 상태 충돌로 데이터 손실 가능성

**P1 (고위험):**
- cs 프로젝트 유지보수 중단 시 기술 부채 급증
- 세션 격리 실패로 인한 환경 오염(모델, 계정, 설정)

**P2 (중위험):**
- 웹 API와 cs 간 상태 동기화 지연으로 잘못된 의사결정 유발
- git-workflow.md 규칙 우회 가능성(--autoyes 우회 방법 존재)
