## 1. 요약 (Executive Summary)

**한 줄 정의:**
AX BD팀 7명을 위해, 발굴(2-0~2-10)→형상화(Phase A~F) 전체 파이프라인을 자동화하여 수동 개입을 최소화하고 E2E 데모가 가능한 상태로 만든다.

**배경:**
F270 1차 정비(Sprint 101)에서 사이드바 투명도, 대시보드 통합, 다음 단계 네비게이션, 404 에러, Agent 스킬 실행 UI 등 6가지 기초 이슈를 해결했다. 그러나 핵심 문제인 **파이프라인 단절**(단계 간 수동 전환, 발굴→형상화 미연결, E2E 테스트 부재)은 여전히 남아 있다.

**목표:**
- 발굴 시작(2-0)부터 형상화 완료(Phase F)까지 자동 파이프라인 동작
- HITL 체크포인트에서만 사용자 개입, 나머지는 자동
- Discovery 위저드 전체 흐름 E2E 테스트로 회귀 방지
<!-- CHANGED: E2E 데모의 구체적 정의 추가 -->
- 실제 업무 시나리오(실제 BD팀 사용 데이터셋) 기반으로, 자동화 파이프라인이 '아이디어 분류 → 발굴 2-0~2-10 → 형상화 Phase A~F' 전체를 자동으로 실행하며, HITL 체크포인트에서는 사용자 승인/거부/타임아웃 처리가 UI상에서 명확히 동작하는 데모 환경을 제공

## 2. 문제 정의

### 2.1 현재 상태 (As-Is)
- **수동 스킬 실행**: 각 단계(2-0~2-10)의 스킬을 사용자가 하나씩 클릭해서 실행해야 함. 11단계 × 복수 스킬 = 수십 회 수동 클릭
- **발굴→형상화 단절**: 2-10 최종보고서 완료 후 형상화(Stage 3)로의 자동 전환 없음. 사용자가 별도로 형상화 프로세스를 수동 시작해야 함
- **E2E 테스트 부재**: Discovery 위저드, 아이템 상세 페이지, 스킬 실행 흐름에 대한 E2E가 없어 UI 변경 시 회귀 감지 불가
- **현재 E2E 커버리지**: Discovery 관련 테스트 = 기본 네비게이션 수준만 존재

### 2.2 목표 상태 (To-Be)
- 2-0 아이디어 분류 완료 → 2-1~2-10 자동 순차 실행 (HITL 체크포인트에서만 정지)
- 2-10 최종보고서 완료 → 형상화 Phase A~F 자동 트리거
- Discovery 위저드 E2E 테스트 커버리지 확보 (위저드 흐름 + 상세 페이지 + 스킬 실행)

<!-- CHANGED: 실패/예외 플로우 및 상태 관리 목표 추가 -->
- 파이프라인 각 단계에서 실패(스킬 오류, 네트워크 장애 등) 발생 시, 사용자는 재시도/건너뛰기/전체 중단/롤백 등 명확한 선택지를 UI상에서 제공받음
- HITL 체크포인트에서 승인 지연/거부/실수 발생 시, 자동 타임아웃(예: 24시간) 및 알림, 승인/거부 이력 추적, 재요청 처리 정책이 명확히 동작

### 2.3 시급성
- 발굴만 완료되고 형상화로 이어지지 않으면 BD 프로세스 전체가 미완성
- 팀원들이 수동 반복 작업에 시간을 소비하면 도구 채택률 저하
- F270 1차 정비 직후가 후속 개선의 최적 타이밍 (코드 컨텍스트가 신선)

## 6. 제약 조건

### 6.1 일정
- 목표 완료: **5 Sprint 이내** (Sprint 132~136 예상)
- 마일스톤:
  - M1 (Sprint 132~133): P0 형상화 연동
  - M2 (Sprint 134): P1 Agent 파이프라인
  - M3 (Sprint 135~136): P2 E2E 테스트 + 통합 검증

### 6.2 기술 스택
- 프론트엔드: Vite 8 + React 18 + React Router 7 (기존)
- 백엔드: Hono + Cloudflare Workers + D1 (기존)
- CLI 스킬: `.claude/skills/ax-bd-shaping/`, `.claude/skills/ax-bd-discovery/`
- 에이전트: `.claude/agents/` (ogd-*, shaping-*, six-hats-*, expert-*)

## 9. 데이터 및 아키텍처 설계

<!-- CHANGED: 데이터 흐름, 상태 머신, 이벤트 기반 오케스트레이션 등 신규 섹션 신설 -->
### 9.1 데이터 흐름 및 계약

- **발굴 산출물 → 형상화 입력**: 발굴 단계(2-0~2-10) 산출물은 JSON Schema로 명시된 데이터 계약(예: `{ "ideaId": string, "summary": string, "details": object, ... }`)에 따라 D1 DB에 저장, 형상화 단계(Phase A~F) 진입 시 해당 산출물 전체를 입력값으로 전달
- **중간 상태 저장**: 파이프라인 각 단계의 입력/출력/에러/메타 데이터는 `bd_pipeline_runs`, `shaping_*` 등 테이블에 단계별 버전+타임스탬프와 함께 저장
- **데이터 백업/복구**: 주요 산출물 및 상태는 버전 이력 관리를 통해 백업, 복구/재시작/재실행 시 해당 버전 복원

### 9.2 상태 머신/오케스트레이션

- **상태 머신 기반 파이프라인**: 각 단계(2-0~2-10, Phase A~F)는 '대기'→'실행중'→'성공'/'실패'/'수동 대기' 등 명확한 상태 전이 모델로 관리
- **이벤트 기반 트리거**: 사용자 승인, 스킬 완료, 실패, 네트워크 복구 등 이벤트에 따라 파이프라인이 자동 전이
- **중간 실패/에러 발생 시, 예외 상태로 전이하며, 수동 개입(재시도/건너뛰기/취소) 옵션 및 알림 제공

### 9.3 CLI ↔ Web 통합 계층

- CLI 스킬 실행 결과(stdout, stderr, exit code)는 WebSocket 등으로 Web UI에 실시간 전송
- 로그 파싱, 상태 매핑, 에러 표준화(예: {step, status, message, detail}) 어댑터 계층 설계
- 장기 실행 프로세스(O-G-D 루프 등)는 외부 배치 워커/Queue로 분리, 진행상황 폴링/알림 지원
