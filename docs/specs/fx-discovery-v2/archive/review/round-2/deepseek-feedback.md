## 검토 의견: fx-discovery-v2 PRD

### 1. 기술적 실현 가능성
**제안된 기술 스택은 기본적으로 요구사항을 충족하지만, 심각한 기술적 제약이 존재합니다.**

- **Cloudflare Workers의 실행 시간 한계**: O-G-D Agent Loop(Phase C)와 같은 장기 실행 워크플로우는 Workers의 30초/10분 제한을 초과할 가능성이 매우 높습니다. Queue/Batch Worker 아키텍처 전환은 필수적이며, 이는 기존 아키텍처의 근본적인 변경을 의미합니다.
- **CLI ↔ Web 통합의 복잡성**: CLI 스킬의 stdout/stderr를 실시간으로 Web UI에 표시하려면 WebSocket + 로그 파싱 계층이 필요합니다. 기존 CLI 스킬들이 구조화된 출력을 제공하지 않는다면, 파싱 로직이 취약하고 유지보수 비용이 높아질 것입니다.
- **다중 외부 시스템 간 일관성 보장 불가**: BdSkillExecutor(REST), CLI subprocess, D1 Database 간 원자적 트랜잭션을 보장할 수 없습니다. "best-effort 일관성"은 데이터 불일치와 복구 난이도를 크게 증가시킵니다.

### 2. 아키텍처 적합성
**이벤트 기반 상태 머신 접근법은 적절하지만, 구체적인 설계가 부족합니다.**

- **상태 관리의 복잡성**: 발굴(11단계) + 형상화(6단계) + HITL 체크포인트를 통합한 상태 머신은 상태 전이 그래프가 매우 복잡해질 것입니다. 상태 수가 기하급수적으로 증가하여 디버깅이 어려울 수 있습니다.
- **데이터 흐름 설계 미비**: "발굴 산출물 → 형상화 입력" 데이터 계약(JSON Schema)이 구체적으로 정의되지 않았습니다. 기존 스킬들의 출력 형식이 이 계약을 따르지 않는다면, 대규모 변환 레이어가 필요합니다.
- **모놀리식 파이프라인 오케스트레이터 위험**: 모든 단계를 하나의 오케스트레이터가 관리하면 단일 장애점(SPOF)이 됩니다. 단계별로 독립적인 워커/마이크로서비스로 분리하는 것이 확장성에 유리합니다.

### 3. 구현 복잡도
**과소평가된 복잡도가 다수 존재합니다.**

- **에지 케이스 처리**: 네트워크 일시적 장애, CLI 스킬의 비정상 종료(exit code ≠ 0), 외부 API rate limiting, 사용자 HITL 응답 지연/거부 등 수십 가지 에지 케이스에 대한 명확한 처리 로직이 설계에 포함되어야 합니다.
- **동시성 제어**: 동일한 아이템에 대해 사용자가 수동 개입(재시도/건너뛰기)을 시도하는 동안 파이프라인이 백그라운드에서 진행되는 상황에 대한 동시성 제어 메커니즘이 필요합니다.
- **데이터 버전 관리 및 롤포워드**: "완전한 롤백은 불가"하지만, 실패 단계부터 재시작하기 위해서는 각 단계의 입력/출력 데이터를 버전 관리하고 저장해야 합니다. 이는 기존 DB 스키마에 큰 부담을 줄 수 있습니다.

### 4. 기술 리스크
**가장 우려되는 기술적 실패 요인은 다음과 같습니다.**

1. **장기 실행 워크플로우와 서버리스 환경의 불일치**: O-G-D 루프가 Cloudflare Workers 환경에서 안정적으로 실행될지 검증되지 않았습니다. 이는 P0 기능의 핵심 위험 요소입니다.
2. **CLI 의존성으로 인한 취약성**: CLI 스킬 실행은 프로세스 생성, 권한, 환경 변수 등 플랫폼 의존성이 높아 배포와 운영을 복잡하게 만듭니다. Web API 기반으로의 점진적 마이그레이션이 장기적으로 필요합니다.
3. **E2E 테스트의 유지보수 비용**: Playwright를 이용한 15건 이상의 E2E 테스트는 UI 변경에 매우 취약합니다. 테스트 스위트의 유지보수에 상당한 리소스가 지속적으로 필요할 것입니다.
4. **운영 인프라 및 모니터링 부재**: "AI+1명" 인력으로 설계, 개발, QA, 운영까지 담당하기에는 운영 복잡도가 너무 높습니다. 24/7 파이프라인 모니터링, 알림, 장애 대응 체계가 구축되지 않으면 프로덕션 신뢰도를 확보할 수 없습니다.

### 5. 착수 판단
**착수 판단: Conditional**
