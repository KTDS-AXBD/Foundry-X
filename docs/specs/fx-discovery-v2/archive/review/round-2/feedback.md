### 1. 논리적 완결성

**강점**
- 현재 문제(수동 프로세스, E2E 부재, 프로세스 단절) → 목표(자동화, HITL 최소화, E2E 회귀 방지) → 성공기준(KPI, MVP 기준)으로 논리 흐름이 비교적 명확하게 이어집니다.
- 실패/예외 플로우(재시도/건너뛰기/복구 등)와 상태 관리 목표도 구체적으로 언급되어 있습니다.

**개선 포인트**
- “파이프라인 자동화”의 구체적 유저 경험 흐름(예: 에러 발생 시 실제 BD 담당자/팀장은 어떤 액션을 하게 되는지)이 서술적으로는 있지만, 비즈니스 플로우 다이어그램, 액티비티 다이어그램 등 시각적 모델이 없습니다. 실무 협업 시 오해 소지가 큽니다.
- HITL의 의사결정 포인트(HITL이 실제 어떤 품질 보증/리뷰를 하는지)의 구체적 사례가 부족해서, QA/실사용자 입장에서 품질 보증의 실질적 맥락이 약합니다.
- 주요 성공 기준이 “E2E 데모 성공” 등 기능 동작 위주에 치우쳐 있고, 실제 현업 BD팀의 작업 소요 시간 단축, 실질적 오류 감소 등 사용성/성과 개선 효과가 정량적으로 더 들어가면 좋겠습니다.

---

### 2. 실행 가능성

**강점**
- 일정, 마일스톤, 기술스택, 필요 인력 및 리스크가 명확히 정의되어 있습니다.
- 외부 연동/상태 머신/CLI-Web 통합 등 기술적 난점에 대한 인식이 상세합니다.

**개선 포인트**
- “AI+1명”으로 설계/개발/QA/운영/배포/교육까지 처리한다는 계획은 명백히 비현실적입니다. (별도 인력 3~4명 필요 명시가 있으나 ‘배정 계획 별도 마련’으로 미루고 있음)
    - QA/테스트, 실제 E2E 환경 구축, 운영 모니터링/알림까지 커버하려면 실투입 인력 및 역할 분장이 착수 전 구체화되어야 합니다.
- 외부 시스템 연동(특히, CLI ↔ Web, long-running agent, 상태 동기화 등)은 설계 의존도가 높고, 실제 통합/테스트/운영에서 예상보다 더 많은 장애와 디버깅 리소스가 소요될 수 있음.
- “기존 인프라 활용, 추가 예산 없음”이 강조되지만, 실질적으로 신규 배치 워커, Queue, 실시간 통합 계층 등은 추가 인프라/운영/모니터링이 필요할 가능성이 높음.
- QA와 운영(운영 책임자, 장애/알림 대응 등) 역할이 구체적 인명, 시간, 책임 단위로 명시되어야 함.

### 3. 누락된 핵심 요소

- **상세 UX 플로우/와이어프레임**  
  HITL UI, 실패/복구 UI 등 실제 사용자의 액션 플로우 설계가 명확하지 않음. 특히 비개발자 대상이라면 UX 시나리오/와이어프레임/프로토타입이 반드시 필요.
- **테스트/QA 전략**  
  “E2E 15건+” 등 목표는 있으나, 실제로 어떤 시나리오를 어떻게(실제/Mock 데이터, 자동/수동) 테스트할지 구체적 테스트 전략/계획이 누락됨.  
- **운영/배포/Hotfix 룰북**  
  장애 발생 시 구체적 핫픽스/롤백 절차, 담당자, 실행 플로우(예: 운영중단 시나리오, Dev/QA/Prod 분리 등)가 실무적으로 정리되어야 함.
- **권한/접근 제어 상세 정책**  
  승인자 권한, 오남용 방지, 이력관리, 긴급 중단 플로우 등 정책이 테이블 수준으로 구체화되어야 함(예: 승인자 변경/위임, 승인 만료, 관리자 override 등).
- **장애/지연 대응 정책**  
  장기 대기, 승인 지연, 대규모 실패(다수 아이템 실패 등) 등 운영상 경고/알림/자동 중단/관리자 개입 정책이 구체적으로 정의돼야 함.
- **외부 시스템 계약/테스트 환경**  
  연동하는 외부 시스템(BdSkillExecutor 등)의 테스트 환경, 오류 주입, Mocking, 장애 복구 등 사전 검증 계획이 필요.

### 4. 리스크

- **인력/운영 리소스 부족**  
  실투입 1~2명으로는 무리. 실제 일정, 품질, 운영 대응에서 장애 발생 가능성 높음.
- **외부 시스템 연동 복잡성**  
  CLI/REST/DB 등 이질적 시스템 간 상태 동기화, 장애 복구, 롤백 등에서 예측 불가 장애/버그 다수 예상.
- **사용자 UX 불만 및 도구 미채택**  
  자동화가 실제 업무 플로우와 어긋나거나, 실패/중단/알림 등이 불명확할 경우, 실 사용자들이 도구 사용을 중단하거나 우회할 가능성 높음.
- **데이터 무결성 및 복구 실패**  
  자동화 과정에서 산출물 누락, 중복, 손상, 상태 불일치 등이 발견될 경우, 신뢰도 하락 및 롤백 불가 상황 야기.
- **테스트 커버리지 미흡**  
  실제 E2E 시나리오, 예외 케이스, 롱런 플로우, 실패 복구 등 QA 리소스 부족으로 회귀/품질 문제 가능성.
- **운영 중 장애/알림 미흡**  
  장애 탐지/알림/핫픽스/롤백 프로세스 미흡시, 운영 리스크(업무중단, 데이터 유실) 가중.

PRD를 면밀히 검토했습니다. AX BD팀의 핵심 업무 프로세스를 자동화하여 효율성을 극대화하려는 본 프로젝트는 IT 시장 트렌드에 매우 부합하며, 잘 정의된 목표와 문제 인식을 가지고 있습니다. 특히 v2로 업데이트되면서 실패/예외 처리, 운영 계획, 아키텍처 고려사항 등 현실적인 부분이 대폭 보강된 점이 인상적입니다.

다음은 각 항목별 검토 의견입니다.

---

### 1. 시장 적합성

본 솔루션은 현재 IT 시장의 주요 트렌드에 매우 잘 맞닿아 있습니다.

*   **하이퍼오토메이션(Hyperautomation) 및 AI 에이전트:** 반복적이고 수동적인 업무를 AI와 자동화 기술로 대체하여 생산성을 높이는 하이퍼오토메이션은 기업 IT 전략의 핵심입니다. 특히 LLM 기반 AI 에이전트들이 복잡한 워크플로우를 스스로 판단하고 실행하는 능력은 BD 프로세스와 같은 지식 노동 분야에서 큰 잠재력을 가집니다. 발굴(Discovery)부터 형상화(Shaping)까지의 파이프라인 자동화는 이러한 AI 에이전트의 실제 적용 사례로 볼 수 있습니다.
*   **AI 기반 코파일럿/워크플로우 오케스트레이션:** 기업들은 내부 업무 시스템에 AI 코파일럿 기능을 통합하여 직원들의 의사결정을 돕고, 복잡한 다단계 워크플로우를 효율적으로 관리하기 위한 오케스트레이션 도구에 투자하고 있습니다. 본 프로젝트는 BD 담당자를 위한 코파일럿 역할과, 전체 BD 프로세스를 조율하는 오케스트레이션 시스템의 성격을 동시에 가집니다.
*   **데이터 기반 의사결정 및 가치 사슬 최적화:** 발굴부터 형상화까지의 모든 단계가 데이터로 연결되고, HITL(Human-in-the-Loop) 체크포인트를 통해 지속적으로 개선될 수 있다면, BD 프로세스 전반의 가치 사슬을 최적화하고 데이터 기반 의사결정을 강화할 수 있습니다.

**유사 서비스 및 차별점:**

*   **유사 서비스/제품:**
    *   **RPA(Robotic Process Automation) 툴 (UiPath, Automation Anywhere 등):** 반복적인 규칙 기반 작업을 자동화하지만, AI 에이전트처럼 복잡한 의사결정이나 비정형 데이터 처리에는 한계가 있습니다.
    *   **워크플로우 오케스트레이션 플랫폼 (AWS Step Functions, Azure Logic Apps, Google Cloud Workflows 등):** 복잡한 분산 시스템의 워크플로우를 정의하고 실행하지만, 비즈니스 로직 자체를 AI 에이전트에 위임하는 것은 별도의 개발이 필요합니다.
    *   **내부 AI 코파일럿/자동화 플랫폼:** Salesforce Einstein, Microsoft Copilot Studio, ServiceNow의 AI 기능 등은 각 플랫폼에 특화된 AI 기반 자동화 및 의사결정 지원 기능을 제공합니다. 대기업들은 종종 이러한 상용 솔루션을 커스터마이징하거나, 사내 특정 업무에 최적화된 자체 AI 자동화 시스템을 구축합니다. 본 프로젝트는 후자에 가깝습니다.
    *   **LLM 오케스트레이션 프레임워크 (LangChain, AutoGen 등):** 여러 LLM 기반 에이전트와 도구를 연결하여 복잡한 작업을 수행하는 데 사용됩니다. 본 프로젝트의 `.claude/agents/` 및 스킬 실행 방식과 유사한 기술 스택을 활용하고 있습니다.

*   **차별점 (내부 솔루션 관점):**
    이 솔루션은 외부 판매를 위한 제품이 아닌, KT DS 내부 AX BD팀의 **고유한 발굴→형상화 프로세스**에 최적화된 도구라는 점에서 차별점을 가집니다.
    1.  **도메인 특화된 지식 및 방법론 내재화:** KT DS의 BD 전문성과 노하우(2-0~2-10, Phase A~F)가 AI 에이전트 스킬과 파이프라인에 직접 반영되어, 일반적인 자동화 툴로는 달성하기 어려운 깊이 있는 자동화를 제공합니다.
    2.  **기존 내부 시스템 및 스킬과의 긴밀한 통합:** `ax-bd-shaping 스킬`, `BdSkillExecutor 서비스`, `D1 Database` 등 기존 인프라와 스킬셋을 활용하여 효율적인 구축이 가능합니다.
    3.  **HITL 기반의 신뢰성 확보:** AI의 한계를 인식하고 HITL 체크포인트를 통해 중요한 의사결정 단계에서 전문가의 검토를 거치도록 설계하여, 자동화의 효율성과 인간의 전문성을 결합합니다.

만약 이 솔루션을 외부 서비스로 확장할 경우, 'BD 프로세스 자동화 SaaS' 형태로 발전할 수 있으며, 이때는 특정 산업군(IT 서비스, 컨설팅 등)에 특화된 템플릿, 커스터마이징 가능한 워크플로우 빌더, AI 에이전트의 학습 및 개선 기능 등이 핵심적인 차별점이 될 것입니다.

### 2. 사실 확인

PRD는 v2로 업데이트되면서 많은 현실적인 고려사항을 반영했지만, 여전히 몇 가지 가정이 검증이 필요합니다.

*   **인력 구성 및 할당의 현실성:**
    *   **가정:** "투입: Claude Code Agent + 사용자 1명 (HITL 검토)"
    *   **PRD 자체의 반박:** "실제 설계, 개발, QA, 운영/배포/교육을 위한 추가 인력(최소 3~4명) 필요성 및 인력 배정 계획 별도 마련"
    *   **검증 필요:** 이 프로젝트는 단순히 코드 에이전트와 한 명의 사용자로 진행하기에는 복잡도가 매우 높습니다. 백엔드(오케스트레이션, DB 스키마, 외부 연동), 프론트엔드(UI/UX, 실시간 알림), DevOps(배포, 모니터링), QA(E2E, 통합 테스트) 등 다양한 전문성이 필요합니다. 최소 3~4명의 인력 배정 계획이 **실제 확정되고 인력이 투입 가능해야** 착수할 수 있습니다. 현재의 'AI + 1명' 가정은 극도로 비현실적입니다.

*   **5 Sprint 이내 완료 일정의 타당성:**
    *   **가정:** "목표 완료: 5 Sprint 이내 (Sprint 132~136 예상)"
    *   **검증 필요:** 위에 언급된 인력 구성의 문제와 더불어, "오픈 이슈" 섹션에 나열된 수많은 핵심 설계 결정(CLI ↔ Web 통합 계층, 장기 실행 워크플로우 관리, 데이터 스키마/계약, 실패 복구 전략 등)이 아직 "설계 단계"에 있습니다. 이러한 근본적인 아키텍처 및 통합 문제를 5 sprints 내에 설계, 구현, 테스트, 배포, 그리고 안정화까지 완료하는 것은 매우 도전적입니다. 특히 복잡한 외부 연동과 예외 처리가 많은 시스템의 경우 초기 안정화에 상당한 시간이 소요될 수 있습니다.

*   **기존 스킬셋의 자동화 실행 적합성 및 안정성:**
    *   **가정:** "새로운 스킬 개발: 기존 68개 스킬을 활용하며..."
    *   **검증 필요:** 기존에 수동으로 실행되던 68개 스킬이 파이프라인으로 묶여 자동 실행될 때, 각 스킬의 입력/출력 인터페이스가 다음 스킬로의 자동 연결에 적합한지, 그리고 비정상 종료나 예상치 못한 에러를 얼마나 안정적으로 처리하는지 검증이 필요합니다. 수동 실행 시에는 사람이 판단하고 보완했지만, 자동화 환경에서는 작은 오류가 전체 파이프라인을 중단시킬 수 있습니다.

*   **Cloudflare Workers의 장기 실행 에이전트 처리:**
    *   **가정:** "장기 실행 단계(특히 O-G-D Agent 루프)는 Cloudflare Workers 한계(30초/10분)에 맞춰, 외부 Queue/Batch Worker(예: Cloudflare Queues, 별도 Worker 서비스) 연동 검토"
    *   **검증 필요:** PRD에서 리스크를 명확히 인지하고 대안을 제시한 점은 긍정적입니다. 하지만 '연동 검토' 수준이며, 실제 아키텍처 설계와 구현이 필요합니다. Cloudflare Workers의 한계를 우회하는 Queue 기반 비동기 처리나 외부 워커 연동은 단순한 작업이 아니며, 이에 대한 기술 검증(PoC)과 구현이 일정에 반영되어야 합니다.

### 3. 사용자 관점

AX BD팀 담당자들은 이 솔루션을 통해 엄청난 효율성 향상을 체감할 수 있을 것입니다. 수십 번의 수동 클릭이 HITL 체크포인트 7회로 줄어든다면, 반복 작업에 소모되던 시간을 전략적 사고와 고부가가치 업무에 집중할 수 있게 됩니다. E2E 데모가 실제 업무 시나리오와 데이터셋 기반으로 제공된다는 점은 사용자 신뢰 확보에 매우 중요합니다.

**도입 장벽:**

*   **신뢰성 및 안정성:** 가장 큰 장벽은 시스템의 **신뢰성**입니다. 자동화된 파이프라인이 자주 실패하거나, 예상치 못한 결과를 내거나, 데이터 무결성 문제를 일으킨다면 사용자들은 결국 수동 프로세스로 회귀할 것입니다. PRD에서 실패/예외 관리, 데이터 백업/복구, 모니터링, 알림 등을 강조한 점은 매우 적절합니다. 초기 UAT(사용자 인수 테스트)와 지속적인 피드백 반영이 중요합니다.
*   **투명성 및 통제력 상실 우려:** AI가 자동으로 많은 단계를 처리하면서, 사용자들은 "블랙박스"처럼 느껴질 수 있습니다. HITL 체크포인트는 이러한 우려를 완화하지만, 각 단계에서 AI가 어떤 판단을 했고 어떤 산출물을 생성했는지 명확하게 시각화하고, 필요 시 사용자가 개입하여 수정하거나 재시도할 수 있는 유연한 UI/UX가 필수적입니다.
*   **학습 곡선:** 새로운 시스템에 적응하는 데 필요한 학습 시간과 노력입니다. 직관적인 UI/UX, 명확한 가이드라인, 그리고 충분한 교육이 필요합니다.
*   **성능:** 파이프라인 실행 속도가 너무 느리거나 UI 반응성이 떨어진다면 사용자 경험이 저하될 수 있습니다. 특히 장기 실행 에이전트의 처리와 실시간 상태 업데이트가 중요합니다.

### 4. 누락된 시장 관점

PRD는 내부 솔루션으로서의 기능과 기술적 고려사항을 잘 다루고 있지만, 시장 및 트렌드 관점에서 추가로 고려하면 좋을 점은 다음과 같습니다.

*   **AI 윤리 및 책임 있는 AI (Responsible AI):** BD 프로세스는 사업의 방향을 결정하는 중요한 과정입니다. AI 에이전트가 아이디어 분류, 보고서 작성, 형상화 등의 과정에 개입할 때, 특정 편향(Bias)이 학습되어 반영되거나, 의도치 않은 결과를 도출할 위험이 있습니다.
    *   **고려사항:** AI 모델의 편향성 검토, 산출물의 공정성 확보 방안, AI의 의사결정 과정을 설명할 수 있는 XAI(Explainable AI) 기능의 필요성 (최소한 HITL 체크포인트에서라도), AI 책임자 지정 등의 가이드라인이 필요합니다.
*   **지속적인 AI 에이전트 성능 개선 및 학습:** 현재는 기존 스킬셋을 활용하지만, 장기적으로 AI 에이전트의 성능을 어떻게 개선하고 학습시킬 것인지에 대한 로드맵이 필요합니다.
    *   **고려사항:** HITL 체크포인트에서 사용자의 승인/거부/수정 피드백 데이터를 수집하여 AI 모델을 재학습하거나 프롬프트 엔지니어링을 개선하는 메커니즘, 새로운 비즈니스 트렌드나 시장 변화에 맞춰 스킬셋을 업데이트하는 관리 프로세스 등이 고려될 수 있습니다.
*   **보안 및 데이터 거버넌스:** BD 과정에서 생성되는 아이디어, 사업 계획, 시장 분석 등은 매우 민감한 기업 정보입니다.
    *   **고려사항:** 데이터 암호화(저장/전송), 엄격한 접근 제어(권한별 접근 범위), 감사 로그, 데이터 유출 방지(DLP) 솔루션 연동, 내부 정보보호 정책 준수 여부 등이 명확히 다뤄져야 합니다.
*   **확장성 및 온보딩:** 현재는 AX BD팀 7명을 위한 솔루션이지만, 향후 다른 팀이나 더 많은 사용자에게 확장될 가능성을 고려해야 합니다.
    *   **고려사항:** 시스템의 아키텍처가 확장성을 충분히 지원하는지, 새로운 사용자 그룹을 위한 온보딩 및 교육 프로그램, 커스터마이징 옵션(예: 파이프라인 템플릿) 등이 필요할 수 있습니다.
*   **경쟁 우위 확보 전략 (장기적 관점):** 이 내부 솔루션이 KT DS의 BD 역량 강화를 통해 시장에서 어떤 경쟁 우위를 가져올 것인지에 대한 비전이 있으면 좋습니다. 단순히 효율성 증대를 넘어, 더 빠르고 정확한 아이디어 발굴, 혁신적인 사업 기회 포착 등으로 연결될 수 있음을 강조할 수 있습니다.

### 5. 착수 판단

**착수 판단: Conditional**

**이유:**

PRD v2는 프로젝트의 목표, 문제 정의, 기능 범위, 주요 리스크 식별 등 대부분의 측면에서 매우 상세하고 현실적인 내용을 담고 있습니다. 특히 v1에서 v2로 업데이트되면서 실패/예외 처리, 아키텍처 고려사항, 운영 계획 등 시스템의 견고성과 실용성을 위한 중요한 요소들이 대폭 보강된 점은 매우 긍정적입니다.

그러나, 몇 가지 **핵심적인 전제 조건과 설계 결정**이 여전히 "오픈 이슈" 상태이거나, 현실적인 자원 배정 계획이 수립되지 않아 즉시 착수하기에는 위험이 따릅니다.

**착수를 위한 필수 조건:**

1.  **인력 확보 및 명확한 역할 정의:** "최소 3~4명"의 인력 배정 계획이 **실제 확정되고 투입되어야** 합니다. 이 인력들은 백엔드, 프론트엔드, DevOps/인프라, QA 등의 명확한 역할을 가지고 전담되어야 합니다. "Claude Code Agent + 사용자 1명"으로는 결코 달성할 수 없는 프로젝트입니다.
2.  **핵심 아키텍처 설계 완료:** "오픈 이슈" 섹션에 나열된 P0/P1 우선순위의 설계 단계 이슈들(CLI ↔ Web 통합 계층, 장기 실행 워크플로우 관리, 파이프라인 상태 저장 스키마, 실패 복구 전략 등)이 **개발 착수 전 선행 설계 단계에서 완료되어야** 합니다. 이는 단순한 기능 구현을 넘어선 시스템의 근간을 이루는 부분입니다. 필요하다면 1~2 Sprint의 별도 아키텍처/설계 스파이크(spike)를 먼저 진행해야 합니다.
3.  **Cloudflare Workers 장기 실행 솔루션 검증:** 장기 실행 에이전트(O-G-D 루프)가 Cloudflare Workers 환경에서 안정적으로 동작할 수 있는 Queue/Batch Worker 아키텍처에 대한 **기술 검증(PoC) 및 구체적인 설계가 확정**되어야 합니다. 이는 전체 파이프라인의 안정성에 직결되는 핵심 요소입니다.
4.  **발굴→형상화 데이터 계약 확정:** 발굴 산출물을 형상화 입력으로 전달하기 위한 JSON Schema 기반의 **데이터 계약이 명확하게 정의**되어야 합니다. 이는 두 시스템 간의 인터페이스를 견고하게 만들고, 향후 유지보수 및 확장성을 보장합니다.

이러한 조건들이 충족된다면, 본 프로젝트는 AX BD팀의 업무 혁신에 크게 기여할 수 있는 성공적인 사례가 될 것으로 기대됩니다.

## DeepSeek 검토의견

**착수 판단:** Conditional
**역할:** 기술적 실현 가능성, 아키텍처 리뷰
**소요:** 32.2초

## 검토 의견: fx-discovery-v2 PRD

### 1. 기술적 실현 가능성
**제안된 기술 스택은 기본적으로 요구사항을 충족하지만, 심각한 기술적 제약이 존재합니다.**

- **Cloudflare Workers의 실행 시간 한계**: O-G-D Agent Loop(Phase C)와 같은 장기 실행 워크플로우는 Workers의 30초/10분 제한을 초과할 가능성이 매우 높습니다. Queue/Batch Worker 아키텍처 전환은 필수적이며, 이는 기존 아키텍처의 근본적인 변경을 의미합니다.
- **CLI ↔ Web 통합의 복잡성**: CLI 스킬의 stdout/stderr를 실시간으로 Web UI에 표시하려면 WebSocket + 로그 파싱 계층이 필요합니다. 기존 CLI 스킬들이 구조화된 출력을 제공하지 않는다면, 파싱 로직이 취약하고 유지보수 비용이 높아질 것입니다.
- **다중 외부 시스템 간 일관성 보장 불가**: BdSkillExecutor(REST), CLI subprocess, D1 Database 간 원자적 트랜잭션을 보장할 수 없습니다. "best-effort 일관성"은 데이터 불일치와 복구 난이도를 크게 증가시킵니다.

### 2. 아키텍처 적합성
**이벤트 기반 상태 머신 접근법은 적절하지만, 구체적인 설계가 부족합니다.**

- **상태 관리의 복잡성**: 발굴(11단계) + 형상화(6단계) + HITL 체크포인트를 통합한 상태 머신은 상태 전이 그래프가 매우 복잡해질 것입니다. 상태 수가 기하급수적으로 증가하여 디버깅이 어려울 수 있습니다.
- **데이터 흐름 설계 미비**: "발굴 산출물 → 형상화 입력" 데이터 계약(JSON Schema)이 구체적으로 정의되지 않았습니다. 기존 스킬들의 출력 형식이 이 계약을 따르지 않는다면, 대규모 변환 레이어가 필요합니다.
- **모놀리식 파이프라인 오케스트레이터 위험**: 모든 단계를 하나의 오케스트레이터가 관리하면 단일 장애점(SPOF)이 됩니다. 단계별로 독립적인 워커/마이크로서비스로 분리하는 것이 확장성에 유리합니다.

### 3. 구현 복잡도
**과소평가된 복잡도가 다수 존재합니다.**

- **에지 케이스 처리**: 네트워크 일시적 장애, CLI 스킬의 비정상 종료(exit code ≠ 0), 외부 API rate limiting, 사용자 HITL 응답 지연/거부 등 수십 가지 에지 케이스에 대한 명확한 처리 로직이 설계에 포함되어야 합니다.
- **동시성 제어**: 동일한 아이템에 대해 사용자가 수동 개입(재시도/건너뛰기)을 시도하는 동안 파이프라인이 백그라운드에서 진행되는 상황에 대한 동시성 제어 메커니즘이 필요합니다.
- **데이터 버전 관리 및 롤포워드**: "완전한 롤백은 불가"하지만, 실패 단계부터 재시작하기 위해서는 각 단계의 입력/출력 데이터를 버전 관리하고 저장해야 합니다. 이는 기존 DB 스키마에 큰 부담을 줄 수 있습니다.

### 4. 기술 리스크
**가장 우려되는 기술적 실패 요인은 다음과 같습니다.**

1. **장기 실행 워크플로우와 서버리스 환경의 불일치**: O-G-D 루프가 Cloudflare Workers 환경에서 안정적으로 실행될지 검증되지 않았습니다. 이는 P0 기능의 핵심 위험 요소입니다.
2. **CLI 의존성으로 인한 취약성**: CLI 스킬 실행은 프로세스 생성, 권한, 환경 변수 등 플랫폼 의존성이 높아 배포와 운영을 복잡하게 만듭니다. Web API 기반으로의 점진적 마이그레이션이 장기적으로 필요합니다.
3. **E2E 테스트의 유지보수 비용**: Playwright를 이용한 15건 이상의 E2E 테스트는 UI 변경에 매우 취약합니다. 테스트 스위트의 유지보수에 상당한 리소스가 지속적으로 필요할 것입니다.
4. **운영 인프라 및 모니터링 부재**: "AI+1명" 인력으로 설계, 개발, QA, 운영까지 담당하기에는 운영 복잡도가 너무 높습니다. 24/7 파이프라인 모니터링, 알림, 장애 대응 체계가 구축되지 않으면 프로덕션 신뢰도를 확보할 수 없습니다.

### 5. 착수 판단
**착수 판단: Conditional**

**조건:**
1. **프로토타입 검증 완료**: P0 기능(형상화 자동 전환)의 엔드투엔드 흐름을 **Queue 기반 아키텍처로 프로토타입** 구현하여, 장기 실행(Phase C)이 실제 환경에서 동작 가능함을 검증해야 합니다.
2. **데이터 계약 및 상태 머신 상세 설계 확정**: 모든 단계의 입출력 데이터 형식(JSON Schema)과 상태 전이 다이어그램을 명시적으로 정의하고, 기존 스킬들과의 호환성을 검증해야 합니다.
3. **운영 및 모니터링 계획 수립**: 파이프라인 운영 담당자, 24/7 모니터링 체계(Cloudflare Alerts + Slack), 장애 대응 프로세스(Hotfix/롤백)가 문서화되고 리소스가 확보되어야 합니다.

이 PRD는 문제 인식과 목표 설정은 훌륭하지만, 기술적 구현의 복잡성과 운영 리스크를 현저히 과소평가하고 있습니다. 위 3가지 조건이 충족되지 않으면, 프로젝트는 중도에 심각한 기술적 장벽에 부딪히거나 운영 부담으로 인해 실질적인 가치를 제공하지 못할 위험이 큽니다.
