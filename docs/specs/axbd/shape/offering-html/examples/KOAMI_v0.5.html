1
복잡 쿼리 응답 지연 (max 153초)

현재 문제

  • 자연어 → LLM이 SQL 생성 → 637 테이블 탐색
  • LLM이 매번 테이블 구조를 추론 — 복잡할수록 느림
  • 일부 질의에서 SLA(90초) 초과
max 153.6초

Ontology 해결

  • 품목-산업-국가 관계가 사전 정의된 그래프로 존재
  • "어떤 테이블을 JOIN할지" 추론 → 그래프 탐색으로 대체
  • 쿼리 계획 시간 단축
그래프 탐색

한계 / 고려사항

  • DB 엔진 성능(인덱싱 등)은 별도 최적화 필요
  • 그래프 DB 도입 시 이중 운영 부담 가능
  • 데이터 양 자체의 문제는 Ontology로 해결 불가
2
품목 간 인과관계 분석 불가

현재 문제

  • 단일 테이블 / 단일 품목 조회에 최적화
  • 품목 A와 B의 관계를 시스템이 알 수 없음
  • "원유 감소 → 자동차 부품 영향?" 답변 불가

Ontology 해결

  • 공급 체인 그래프: 원유 → 나프타 → PE/PP → 부품
  • 경로 기반 설명(explainability) 가능
  • "왜 영향받는가"에 대한 근거 제시

한계 / 고려사항

  • 관계 정의 품질이 전부 — 도메인 전문가 참여 필수
  • 2,443개 전체 관계 구축은 수개월 — PoC는 50~100개 서브그래프
  • 구조적 연결이지, 정량적 인과 모델(X% 영향)은 아님
3
데이터 사일로 (6개 시스템 분산)

현재 문제

  • 무역 · 산업 · R&D · EWS · GIVC · 보고서 분산
  • 1차에서 통합한 것: 보고서 21종 + 공정도 96종
  • 아직 연결 안 된 것: 무역, R&D, 소부장넷, EWS

Ontology 해결

  • 공통 스키마 계층 — 각 소스가 같은 엔터티 참조
  • 물리적 ETL 없이 논리적 통합 가능
  • Palantir 핵심 가치: "데이터를 옮기지 않고 연결"

한계 / 고려사항

  • 온프레(GIVC) + 클라우드(KT) 분리 — 보안 정책 확인 필요
  • 외부 데이터(NICE 등)는 API 라이선스 이슈 별도
  • PoC에서는 기존 1차 데이터만으로 매핑 검증
4
"만약 ~하면?" 시나리오 예측 불가

현재 문제

  • 과거 데이터 조회만 가능
  • 가정 기반 분석("중동 분쟁 시 영향?") 불가

Ontology 해결

  • 그래프 시뮬레이션: 노드에 조건 적용 → 영향 전파
  • LLM이 탐색 결과를 단계별 자연어로 설명

한계 / 고려사항

  • 정량 예측("X% 상승")은 시계열 모델 별도 필요
  • PoC에서는 정성적 영향도(H/M/L)가 현실적
  • 엣지 가중치는 과거 데이터에서 학습해야 함

Ontology 해결 범위 — 3단계 구분

할 수 있는 것만 정확히 말합니다.

PoC 가능 구조적 연결

품목 간 공급 체인 그래프 구축, 영향 경로 시각화, 정성적 영향도(H/M/L) 산출

조건부 정량 예측

과거 데이터 품질에 의존. 시계열 모델 + 엣지 가중치 학습 필요. PoC에서는 과거 사례 기반 유사도 비교

본사업 전수 관계 구축

2,443개 품목 전체 관계 + 실시간 경보 연동 + 정책 자동 추천. 도메인 전문가 장기 참여 필요

AI에게 '이 문서 읽어봐'가 아니라 '이 품목의 관계를 봐'라고 주는 것입니다.
Palantir가 전 세계에서 하는 것과 같은 구조를, 귀사 데이터에 맞춤화합니다.
