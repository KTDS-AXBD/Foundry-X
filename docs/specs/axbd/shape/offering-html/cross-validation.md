# GAN 교차검증 — 표준 질문 풀 + 판정 로직 (가이드 §4/05-4 기반)

> **용도:** SKILL.md Step 6(교차검증)에서 참조
> **원칙:** "경영진이 안심하도록"이 아니라 **"정확히 판단하도록"** — No-Go 판정도 가능한 수준
> **컴포넌트:** [gan-verdict.html](templates/components/gan-verdict.html)
> **에이전트:** ogd-orchestrator (추진론/반대론 생성)

## 1. 표준 질문 풀 (7개)

사업기획서 전체 내용을 기반으로 아래 7개 질문을 자동 적용한다.

| # | 질문 | 유형 | 적용 조건 | 검증 섹션 |
|---|------|------|----------|----------|
| Q1 | 올해 매출·수익에 어떻게 기여하는가? | 수익성 | 항상 | 05-2, 04-3 |
| Q2 | 이 시장/기술이 정말 새로운 기회인가? | 시장성 | 항상 | 01-2, 01-3, 01-7 |
| Q3 | 필요한 데이터를 안정적으로 확보할 수 있는가? | 실행력 | 항상 | 05-1 |
| Q4 | KT와 연계 논의가 된 것인가? | KT 적합성 | 항상 | 01-1, 04-2 |
| Q5 | 기존 인프라/자산 레버리지가 실제로 가능한가? | 실행력 | 기존 자산 언급 시 | 01-5, 04-1 |
| Q6 | 금번 미진행 시 기회비용은 무엇인가? | 긴급성 | 경쟁 압박·데드라인 있을 때 | 01-8, Exec Summary |
| Q7 | 핵심 기술/역량은 외부 의존 리스크가 있는가? | 리스크 | 항상 | 02-3, 05-3 |

### 조건부 질문 적용 규칙

- **Q5**: 01-5(기존 사업 현황) 또는 04-1(차별화 요소)에서 "기존 인프라", "자산", "레버리지", "보유" 키워드가 있으면 적용. 없으면 "해당 없음 — 기존 자산 활용 전략 없음"으로 표기
- **Q6**: Exec Summary에 "긴급", "경쟁", "데드라인", "선점" 키워드가 있거나, 01-8에 경쟁사 언급이 3건 이상이면 적용. 없으면 "해당 없음 — 긴급성 낮음"으로 표기

## 2. 질문별 판정 기준

### Q1: 수익성 — "올해 매출·수익에 어떻게 기여하는가?"

| 판정 | 기준 |
|------|------|
| **유효** | 05-2에 올해 매출 목표 수치 + 산출 근거 존재. PoC 매출이 적어도 전략 자산(데이터/레퍼런스/검증) 명시 |
| **조건부** | 매출 수치 있으나 산출 근거 불충분, 또는 "전략적 가치"만 언급하고 재무 기여 미정량 |
| **주의** | 매출 계획 없음, 또는 "향후 검토" 수준. 올해 기여 방안이 전혀 없음 |

### Q2: 시장성 — "이 시장/기술이 정말 새로운 기회인가?"

| 판정 | 기준 |
|------|------|
| **유효** | 01-7 TAM/SAM/SOM 수치 + 산출 근거 존재. 01-2/01-3에서 기존 방식 대비 차별점 구체적 |
| **조건부** | 시장 규모 수치 있으나 출처 불분명, 또는 차별점이 "AI 적용"처럼 일반적 |
| **주의** | TAM 부재 또는 비현실적 추정. 기존 솔루션과의 차이가 불명확 |

### Q3: 실행력(데이터) — "필요한 데이터를 안정적으로 확보할 수 있는가?"

| 판정 | 기준 |
|------|------|
| **유효** | 05-1에 데이터 확보 전략 3계층(자체/API/구매) 구체적. 라이선스·비용 이슈 식별 완료 |
| **조건부** | 데이터 소스 명시되었으나 확보 가능성 미검증, 또는 비용·라이선스 미확인 |
| **주의** | 데이터 확보 방안 없음, 또는 "크롤링 예정" 수준 |

### Q4: KT 적합성 — "KT와 연계 논의가 된 것인가?"

| 판정 | 기준 |
|------|------|
| **유효** | 01-1 KT 적합성 축에 구체적 KT 전략 연계점 + 04-2에 그룹 시너지 2가지 이상. 현재 상태 솔직 기술 |
| **조건부** | KT 전략 방향은 맞으나 구체적 연계 채널/담당 미확인, 또는 "협의 예정" 수준 |
| **주의** | KT 연계가 명목적("KT 그룹사이므로"). 실질적 시너지 불명확 |

### Q5: 실행력(자산) — "기존 인프라/자산 레버리지가 실제로 가능한가?"

| 판정 | 기준 |
|------|------|
| **유효** | 01-5에 기존 사업/관계 구체적 기술 + 04-1에 해당 자산 기반 차별점 명시 |
| **조건부** | 기존 자산 언급은 있으나 레버리지 방법이 구체적이지 않음 |
| **주의** | "기존 역량 활용" 수준의 추상적 언급만 존재 |

### Q6: 긴급성 — "금번 미진행 시 기회비용은 무엇인가?"

| 판정 | 기준 |
|------|------|
| **유효** | 구체적 기회비용 정량화 (경쟁사 선점 시 매출 손실, 시장 진입 타이밍 등) |
| **조건부** | 경쟁 압박 언급은 있으나 기회비용 미정량 |
| **주의** | 긴급성 근거 불충분, 또는 "트렌드이므로 빨리" 수준 |

### Q7: 리스크 — "핵심 기술/역량은 외부 의존 리스크가 있는가?"

| 판정 | 기준 |
|------|------|
| **유효** | 02-3 서비스 구성에서 핵심 기술의 내재화/외부 의존 구분 명시. 05-3에 리스크 대비 비용 포함 |
| **조건부** | 외부 의존 기술 존재하나 대안/마이그레이션 전략 미비 |
| **주의** | 핵심 기술이 단일 외부 벤더에 완전 의존. 전환 비용 미산출 |

## 3. 교차검증 생성 프로세스

### Step 6 실행 흐름

```
[1] 사업기획서 전체 읽기
    └── 각 섹션의 핵심 내용을 질문별로 매핑

[2] 7개 질문 자동 적용 (조건부 질문 적용 여부 판단)
    ├── Q1~Q4, Q7: 항상 적용
    ├── Q5: 기존 자산 키워드 탐색 → 있으면 적용
    └── Q6: 긴급성 키워드 탐색 → 있으면 적용

[3] 질문별 추진론 생성
    ├── 해당 검증 섹션에서 근거 추출
    ├── 톤: 사업을 해야 하는 이유를 객관적으로 서술
    └── 분량: 2~4문장, 구체적 수치/근거 포함

[4] 질문별 반대론 생성
    ├── 해당 검증 섹션에서 약점/리스크 추출
    ├── 톤: 사업을 하면 안 되는 이유를 냉철하게 서술
    ├── "경영진 안심" 금지 — 실질적 리스크를 직시
    └── 분량: 2~4문장, 반드시 구체적 우려 포함

[5] 판정 배지 부여
    ├── §2 "질문별 판정 기준" 참조
    ├── 추진론/반대론 균형 분석
    └── 판정 + 판정 근거 1문장 작성

[6] 종합 판정
    ├── 7개 질문의 판정 결과 집계
    ├── 종합 배지 결정 (아래 §4 규칙)
    └── gan-verdict.html 컴포넌트로 렌더링
```

### ogd-orchestrator 연동

교차검증의 추진론/반대론은 **ogd-orchestrator** 에이전트를 활용하여 품질을 높인다:

1. **Generator**: 사업기획서 내용 기반으로 추진론/반대론 초안 생성
2. **Discriminator**: 초안의 냉철함 수준 평가 — "경영진 안심" 톤이면 재생성 요청
3. **Orchestrator**: Generator↔Discriminator 루프 관리 (최대 2회)

```
[ogd-orchestrator 호출]
├── 입력: 사업기획서 전문 + 7개 질문
├── Generator 지시: "각 질문에 대해 추진론과 반대론을 냉철하게 작성하라"
├── Discriminator 기준:
│   ├── 추진론이 과대 낙관이면 REJECT
│   ├── 반대론이 "그래도 괜찮다"로 끝나면 REJECT
│   ├── 판정 근거가 1문장 이하면 REJECT
│   └── "혁신적", "독보적" 등 과대 표현 포함 시 REJECT
└── 출력: 7개 질문 × (추진론 + 반대론 + 판정 + 근거)
```

## 4. 종합 판정 규칙

### 배지 정의

| 배지 | 색상 | HTML class | 의미 |
|------|------|-----------|------|
| **유효** | 초록 `#16a34a` | `valid` | 근거 충분, 리스크 낮음 |
| **조건부** | 주황 `#ea580c` | `partial` | 일부 리스크, 보완 필요 |
| **주의** | 빨강 `#dc2626` | `risk` | 근거 부족, 높은 리스크 |

### 종합 배지 결정 기준

| 조건 | 종합 판정 |
|------|----------|
| "주의" 0건 + "조건부" 2건 이하 | **Go** — 추진 권고 |
| "주의" 1건 또는 "조건부" 3건 이상 | **Conditional Go** — 조건부 추진 (보완 사항 명시) |
| "주의" 2건 이상 | **No-Go** — 재검토 권고 (핵심 보완 사항 목록 제시) |
| Q1(수익성) 또는 Q4(KT 적합성)가 "주의" | 무조건 **No-Go** — 핵심축 미충족 |

### 종합 판정 출력 형식

```
## 종합 판정: {Go / Conditional Go / No-Go}

| 질문 | 판정 | 핵심 근거 |
|------|------|----------|
| Q1 수익성 | ✅ 유효 | ... |
| Q2 시장성 | ⚠️ 조건부 | ... |
| ... | ... | ... |

### 보완 필요 사항 (Conditional Go / No-Go 시)
1. {구체적 보완 항목 + 해당 섹션 + 기대 결과}
2. ...
```

## 5. 냉철함 가이드라인

### 추진론 작성 원칙

- 사업의 강점을 객관적으로 서술하되 **과대 포장 금지**
- 수치가 있으면 수치로, 없으면 "정량 근거 미확보"로 솔직 표기
- "혁신적", "독보적", "최초" 표현 금지 (경영 언어 원칙 동일 적용)
- 추진 근거가 약해도 있는 그대로 서술 — 약한 근거는 약하다고 표현

### 반대론 작성 원칙

- **가장 중요한 원칙**: "그래도 괜찮다"로 끝나면 안 된다
- 반대론은 실질적 리스크를 직시해야 함 — 형식적 반대론 금지
- "경영진이 걱정할 수 있지만..." 형태 금지
- 구체적 실패 시나리오 제시: "만약 ~하면 ~의 손실/리스크 발생"
- 반대론이 추진론보다 짧으면 안 됨 (최소 동등한 분량)

### 판정 근거 원칙

- 1문장으로 핵심 판단 이유 작성
- "~이므로 유효" 형태 — 근거가 명확해야 함
- "종합적으로 판단하면" 같은 모호한 표현 금지

## 6. gan-verdict.html 데이터 매핑

cross-validation 결과를 gan-verdict.html 컴포넌트에 매핑:

| 컴포넌트 변수 | 데이터 소스 |
|-------------|-----------|
| `questions[].number` | 질문 번호 (1~7) |
| `questions[].question` | §1 표준 질문 텍스트 |
| `questions[].type` | 유형 (수익성/시장성/실행력/KT 적합성/긴급성/리스크) |
| `questions[].pro` | §3 Step 3에서 생성된 추진론 |
| `questions[].con` | §3 Step 4에서 생성된 반대론 |
| `questions[].verdict` | "유효" / "조건부" / "주의" |
| `questions[].reason` | 판정 근거 1문장 |

**종합 판정 표시**: gan-verdict.html의 `.gan-summary` 영역에 종합 배지 + 보완 사항 표시
