# AX BD팀 사업기획서 생성 스킬

> **목적:** AX BD팀이 2단계 발굴 결과를 KT 연계 AX 사업기획서(HTML)로 형상화하는 스킬
> **사용 시점:** KT 연계 AX 사업개발 체계의 **3. 형상화** 단계
> **산출물:** `사업기획서_{사업명}_v{버전}_{날짜}.html`

---

## 1. AX BD팀 사업개발 체계 개요

### 전체 프로세스 (6단계)

```
1. 수집 → 2. 발굴 → 3. 형상화 → 4. 검증 및 공유 → 5. 제품화 → 6. GTM
```

- **1. 수집**: KT Pipeline 연계, 현장 및 도메인 전문가 기반 수집, Agent 자동 수집, 전사 Bottom-up 등을 통한 고객 Pain points 및 사업 아이템 수집
- **2. 발굴**: Foundry-X 기반 사업 아이템 구체화·분석 (2-0 ~ 2-8 단계)
- **3. 형상화** ← **이 스킬이 담당하는 단계**, Offering Pack 제작, Prototype 제작
- **4. 검증 및 공유**: AX사업개발 본부장 검토, 고객·도메인 전문가 인터뷰, KT 내부 유관부서 의견, 본부 내 보고 → 임원 의사결정 (Go/Hold/Drop)
- **5. 제품화**: MVP 제작, 고객 타겟 Offering Pack 제작
- **6. GTM**: 대고객 선제안 (KT 경유), PoC 추진

### Pain Point와 이 스킬의 역할

| AX BD팀 Pain Point | 이 스킬이 해결하는 것 |
|---|---|
| 설득력 있는 사업기획서 및 프로토타입 공유·보고 필요 | 발굴 단계 산출물 기반 → 표준 포맷의 사업기획서 자동 생성 |
| 담당자별 사업성 분석·평가 자동화 필요 | GAN 방법론 기반 교차검증 섹션 자동 구성 |
| KT 전략/조직개편 방향을 빠르게 반영하는 구조 필요 | KT 연계 GTM 섹션 표준 패턴 제공 |
| '발굴 수가'가 아닌 '수주 가능성과 재사용 가능성' 중심 평가 체계 필요 | 수익성·KT 적합성·실행력 3축 기반 구조화 |

## 2. Input: 발굴 단계(2-0~2-8) 산출물

이 스킬의 Input은 발굴 단계에서 도출된 산출물입니다. 각 단계의 산출물이 사업기획서의 어느 섹션에 매핑되는지 아래와 같습니다.

| 발굴 단계 | 내용 | 사업기획서 매핑 섹션 |
|---|---|---|
| **2-0** 사업 아이템 구체화·분류 | 아이템 개요, 고객, 도메인 | Hero, Executive Summary, 01-4 왜 이 ‘고객/도메인’인가 |
| **2-1** 레퍼런스 분석 | 경쟁사 분석, 시장 분석 등 | 01-8 글로벌·국내 동향 |
| **2-2** 수요·시장 검증 | TAM/SAM/SOM 및 Pain Point 분석 | 01-2 왜 이 ‘문제/영역’인가, 01-3 왜 이 ’기술/접근법’인가, 01-7 시장 분석 |
| **2-3** 경쟁·자사 분석 | KT 그룹 내 중복/시너지, 경쟁 구도 등 | 01-8 글로벌·국내 동향, 04-1 차별화 요소, 04-2 그룹 시너지 |
| **2-4** 사업 아이템 도출 | AI 등 사업 아이템 체크리스트 도출 | 01 문제 정의 섹션 |
| **2-5** 핵심 아이템 선정 | KT 방향성, 수주달성 관점 숏리스트 도출 | 01-1 추진 배경 및 목적 (KT 적합성 축), 02-1 사업 개요, 02-3 서비스 구성안, 04-1 차별화 요소, 04-2 그룹 시너지, 05-1 데이터 확보 방식 |
| **2-6** 타겟 고객 정의 | 고객 페르소나, 구매력, Value 정의 | 01-4 왜 이 ‘고객/도메인’인가, 03-1 유저 시나리오 / Use Case |
| **2-7** 비즈니스 모델 정의 | BM(수익 시뮬레이션 관련) | 03-2 사업 구도, 05-2 사업화 방향 및 매출 계획, 05-3 투자 계획 |
| **2-8** 발굴 결과 패키징 | 형상화를 위한 Input Deck 종합 | 전체 섹션의 기초 자료 |

**중요:** 모든 단계의 산출물이 반드시 있어야 하는 것은 아닙니다. 사업 주제에 따라 일부 단계가 생략되거나 다른 방법론이 적용될 수 있으며, 그에 따라 사업기획서의 해당 섹션 내용이 달라집니다.

## 3. 사업기획서 표준 목차

### 설계 원칙 (스토리라인 구조)
```
Why → What → How → Proof → Next
```
청중이 왜 이게 필요한지(공감) → 뭘 하는 건지(이해) → 어떻게 하는 건지(신뢰) → 진짜 되는 건지(확신) → 다음은 뭔지(행동)의 흐름으로 전개

## 4. 섹션별 작성 가이드

### 0. Hero

- 타이틀: 사업명(2줄 이내)
- 서브타이틀: 고객명(있을 경우) + 한 줄 가치 제안
- 날짜, 버전, 조직명

### 0.5 Executive Summary

**구조:** 5~6개 불릿. 경영진이 이것만 읽고 Go/No-Go 판단 가능해야 함. 한 장에 전체 스토리를 압축

표준 불릿 풀 (사업 특성에 따라 선택·조합):

| 불릿 | 역할 | 언제 포함 |
|---|---|---|
| 기존 고객 후속 수주 / 신규 고객 확보 | What | 항상 |
| 왜 지금인가 (긴급성) | Why Now | 데드라인·경쟁 압박이 있을 때 |
| 수익성 | How much | 항상 |
| KT 적합성 | Why KT | 항상 |
| 비대칭 우위 | Why Us | 경쟁 구도가 있을 때 |
| 반복 가능한 사업 모델 | Scale | 항상 |

**원칙:**
- 기술 용어보다 경영 언어를 사용
- "~할 수 있다"가 아닌 "~를 제안 예정", "~를 추진"
- 볼드는 핵심 키워드에만 (1불릿당 2~3개)

### 01. 문제 정의

- **01-1 추진 배경 및 목적:**
　- **항상 3축 구조.** 축의 라벨은 사업 특성에 따라 조정:
　- 축1:**이 사업이 매출·수익에 어떻게 기여하는가**. 수익성, 시장 기회, 고객 가치
　- 축2:**KT 전략과 어떻게 연결되는가**. KT 적합성, KT 시너지, 그룹 전략 정합성
　- 축3: **왜 우리(kt ds)가 할 수 있는가**. 실행력, 기술 우위, 자산 레버리지
　- **핵심 원칙:** KT 연계가 없는 사업은 이 포맷으로 올리지 않음. KT와 동떨어진 독자 AX 사업은 이 기획서의 대상이 아님.
　- 하단에 **"추진 목적"** 블랙 박스: 1문장 + 3개 키워드 카드

- **01-2 (Why):** 왜 이 문제/영역인가 
　— 시장 트렌드, Pain Point, 글로벌 동향
　
- **01-3 (How):** 왜 이 기술/접근법인가 
　— 기존 방식 대비 차별점, 비교표
　
- **01-4 (What):** 왜 이 고객/도메인인가 
　— 고객 선정 근거 (판단 기준 테이블)
　
- **01-5 (선택):** 기존 사업/관계 현황 
　— 레버리지 자산 (기존 고객일 때만)
　
- **01-6 (선택):** 현황 Gap 분석 
　— 현재 vs 목표
　
- **01-7:** 시장 분석 
　— **TAM / SAM / SOM** 3개 카드 (추정 근거 명시)
  - **경쟁 비대칭 가설**: kt ds가 이 사업에서 가진 구조적 우위 3가지
　- 과대 추정 금지. 추정치에는 반드시 "약" 표기. 산출 근거 링크 삽입
　- **01-8:** 글로벌·국내 동향 — 경쟁사 비교, 벤치마크

### 02. 제안 방향

- **02-1 사업개요:** 
　- 사업명, 한줄 정의, 핵심 가치 3가지 이상
　- 엘리베이터 피치 30초 텍스트 포함.
　
- **02-2 To-Be 전환:** 
　- Before/After 시각화. 
　- 현재 상태와 목표 상태를 좌우로 대비하고, 핵심 전환 포인트를 명확히함.
　- 필요시 일러스트를 삽입하여 한 눈에 이해하기 쉽도록 표현.
　
- **02-3 서비스 구성안:** 
　- 전체 서비스 아키텍처/구성도. 
　- 모듈별 기능 설명. 
　- 데이터 흐름과 AI 적용 지점 표시. 
　- 기술 스택 간략 표기.

### 03. 실행 방안

- **03-1 유저 시나리오 / Use Case:**
　- **본사업 기준 시나리오:** 전체 데이터 확보를 전제로 설계.(최소 2개)
　- 상황(Context) → 행동(Action) → 결과(Result) 구조로 상세 서술.
　- 필요시 일러스트를 삽입하여 한 눈에 이해하기 쉽도록 표현.
　- 고객이 "분석·보고" 기관인 경우: "추천"이 아닌 **"분석 근거 제공 + 대응 옵션 비교"** 톤 
  - 고객이 의사결정 주체인 경우: "추천안 제시" 톤 가능
  - **PoC 시나리오:** 본사업의 축소판. 범위 한정, 1계층 데이터로 가능, 데이터 증강 허용, 기술 실행성 검증 목적
　- **B2C 서비스의 경우:** **페르소나 기반 Use Case**로 구성
　- **03-2 사업 구도:** 
　- KT DS 역할, 고객 역할, 파트너(있을 경우) 역할을 도식화. 
　- R&R 매트릭스. 데이터/기술/운영 각 영역별 담당
　
- **03-3 사업화 로드맵:**
　- 반드시 **단기(PoC) → 중기(본사업) → 장기(확장 방향)** 3단계
　- 단계별 범위, 산출물, GO/No-Go 기준 명시
  - 매출이 적거나 수익이 낮은 사업이라도 **"이 사업이 쌓는 전략 자산"**을 명시
  - 예: "당장 매출은 작지만, 이 PoC를 통해 ①도메인 데이터 ②고객 레퍼런스 ③기술 검증을 확보하면, 다음 단계에서 ○○억 규모 사업으로 전환 가능"

### 04. 경쟁력 및 기대효과

- **04-1 차별화 요소:** 
　- 경쟁사 대비 KT DS의 핵심 차별점 3가지. 
　- 예) SI/ITO 역량 + AI 역량 + 도메인 전문성 시너지. 
　- 포지셔닝 맵 또는 배틀카드 요약
　- **"왜 지금?"** + **"왜 kt ds?"** 질문에 대한 답을 요약하여 하단에 배치
　
- **04-2 그룹 시너지:** 
　- 경쟁사 대비 KT DS가 KT그룹사 협력으로 얻는 시너지 효과 2~3가지. 
　- **KT 조직개편 반영:** 최신 조직 구조 기반 연계 포인트 식별
　
- **04-3 기대 효과:** 
　- 정량 효과: 비용 절감율, 처리 시간 단축, 정확도 향상 등. 
　- 정성 효과: 의사결정 품질, 직원 경험 개선. ROI 시뮬레이션(보수/기본/낙관)
　- 정량 효과의 경우, 계산 및 추정 근거 표기. 과대 추정 금지. 산출 근거 링크가 있을 경우 첨부.

### 05. 추진 계획

- **05-1 데이터 확보 방식:**
  - **데이터는 매우 중요.** 모든 사업기획서에 데이터 확보 전략을 명시
  - 표준 구조: 계층별 (자체 보유 / 공개 API·연계 / 유료 구매·파트너십)
  - PoC vs 본사업 구분 명시
  - 크롤링 이슈, 라이선스, 비용 이슈 사전 식별

- **05-2 사업화 방향 및 매출 계획:**
  - 3단계: 기술 검증 → 패키지 → 확장
  - 매출 계획: 보수/기본/낙관 3개년 시나리오
  - kt ds의 SI·ITO 역량이 어디에 결합되는지 명시

- **05-3 투자 계획:**
　- 비용: 항목별 산출 (인건비, 인프라, 라이선스, 예비비)
  - 금액에는 반드시 "약" 표기

- **05-4 사업성 교차검증:**
  - **방법론:** GAN(Generator-Discriminator) 기반 추진론/반대론 교차 검증
　- **원칙:** 냉철하게. No-Go를 하더라도 정확한 근거를 제공하는 것이 목적. 
　- "경영진이 안심하도록"이 아니라 "정확히 판단하도록"
  - **표준 질문 풀** (사업 특성에 따라 선택·조합·추가):

  | # | 표준 질문 | 유형 |
  |---|---------|------|
  | 1 | 올해 매출·수익에 어떻게 기여하는가? | 수익성 |
  | 2 | 이 시장/기술이 정말 새로운 기회인가? | 시장성 |
  | 3 | 필요한 데이터를 안정적으로 확보할 수 있는가? | 실행력 |
  | 4 | KT와 연계 논의가 된 것인가? | KT 적합성 |
  | 5 | 기존 인프라/자산 레버리지가 실제로 가능한가? | 실행력 |
  | 6 | 금번 미진행 시 기회비용은 무엇인가? | 긴급성 |
  | 7 | 핵심 기술/역량은 외부 의존 리스크가 있는가? | 리스크 |

  - 발굴 단계(2-0~2-8)에서 사용된 방법론에 따라 질문이 달라질 수 있음. 위는 표준 풀이며, 주제별로 추가·삭제·변형 가능
  - 각 질문에 대해 **추진론 / 반대론 / 판정(배지)** 구조
  - 판정 배지: 초록(유효) / 주황(조건부·부분) / 빨강(리스크·주의)

## 6. 작성 원칙

### 경영 언어 원칙

- **"~할 수 있다", "~가능하다"** → **"~를 제안 예정", "~를 추진"**
- **기술 용어 최소화.** 기술 설명이 필요하면 비교표(LLM vs RDB vs Ontology 등)로
- **금액에는 반드시 "약" 표기.** 확정되지 않은 모든 금액
- **"최초", "첫" 표현 사용 금지.** "선도적 사례", "초기 시장" 수준으로 완화
- **과대 표현 금지.** 검증할 수 없는 주장은 삭제
- 근거가 되는 URL 링크는 반드시 포함

### AX BD팀 포지셔닝

- AX BD팀의 역할: "KT 전략과 고객 수요를 연결해 kt ds가 수주·사업화할 수 있게 만드는 사업화 추진 사내 기능"
- "발굴팀"이 아닌 "사업화 추진" 기능
- "이끌기" → "추진"

### 고객 유형별 톤 조정

| 고객 유형 | 시나리오 톤 |
|---|---|
| 분석·보고 기관 (정책 지원) | "추천" 아닌 **"분석 근거 제공 + 대응 옵션 비교"** |
| 의사결정 주체 (기업, 금융 등) | **"추천안 제시"** 가능 |
| B2C 서비스 대상 | 시나리오 대신 **"페르소나 기반 Use Case"** |

### KT 연계 원칙

- KT 연계가 없는 사업은 이 포맷의 대상이 아님
- KT와의 현재 상태를 솔직하게 기술 (협약 없으면 "없음"이라고 명시)
