## Part 1: 왜 (목적/문제)

**Q: Prototype Builder 고도화의 가장 핵심적인 동기는?**
A: 3가지 모두 해결 필요 — 디자인 품질 부족, 판별 정확도 한계, 비용 부담

**Q: 우선순위는?**
A: 3가지 병렬 진행. 각각 독립적이라 동시 진행 가능

**배경 정리:**
- Phase 19(Builder Evolution)에서 5차원 스코어링 PoC 완료 (96% 구현)
- max-cli 통합은 PoC만 완료, 본격 적용 미완
- Discriminator가 체크리스트 수준 (LLM 판별 미도입)
- impeccable 같은 체계적 디자인 품질 가이드 없음

## Part 2: 누구를 위해 (사용자/이해관계자)

**Q: 고도화 후 주 사용자와 사용 시나리오에 변화가 있나요?**
A: 기존과 동일

- 주 사용자: BD팀 7명
- 부 사용자: 고객 이해관계자, 임원
- 시나리오: PRD 입력 → 프로토타입 자동 생성 → 고객 데모

## Part 3: 무엇을 (범위/기능)

### 트랙 A: impeccable 반영 (디자인 품질)

**Q: impeccable을 어떻게 반영할 것인지?**
A: Skill 직접 적용 — 7도메인 참조문서 + 안티패턴을 생성 시스템 프롬프트에 주입

**impeccable 분석 결과 (github.com/pbakaus/impeccable):**
- Paul Bakaus 제작, AI 코딩 어시스턴트용 디자인 스킬 시스템
- 7개 도메인: 타이포그래피, 색상/대비, 공간, 모션, 인터랙션, 반응형, UX 라이팅
- 20개 스티어링 커맨드: /audit, /critique, /normalize, /polish 등
- 안티패턴 명시: 과용 폰트(Arial, Inter), 회색 텍스트, 순수 흑색, 카드 중첩 등

### 트랙 B: LLM 판별 고도화

**Q: LLM 판별 고도화 범위?**
A: 3가지 모두 포함
1. PRD 정합성 LLM 판별 (prd 차원 25% → 키워드 매칭에서 LLM 비교로)
2. 5차원 전체 LLM 판별 (체크리스트 → LLM 평가)
3. Vision API 활용 (스크린샷 기반 시각적 품질 평가)

### 트랙 C: max-cli 본격 통합

**Q: max-cli 통합 시 운영 환경?**
A: WSL 로컬 실행, 단일 머신, API fallback 유지

## Part 4: 성공 기준

**Q: 이 고도화가 성공했다는 걸 어떻게 판단?**
A: 생성된 프로토타입을 추가 수정 없이 고객에게 바로 보여줄 수 있는 수준

**Q: "고객 데모 바로 가능"의 구체적 기준?**
A: 전부 다 — 시각적 완성도 + PRD 반영률 90%+ + 모바일 반응형
