### 1. 논리적 완결성

#### 평가
- **문제-해결책-성공기준**의 기본적 연결은 잘 잡혀 있습니다. As-Is와 To-Be의 대조, 구체적 한계 → 기능 → 성공 기준의 흐름도 논리적입니다.
- **문제 정의**에서 ‘실제 체감 품질’의 중요성을 인지하고 있으나, ‘정성적 평가’의 구체적 방법론(예: 고객 피드백의 정량화 방식, 내부 검증 기준) 설명이 다소 모호합니다.
- **기능 범위 → 성공 기준**에서 자동화 점수와 수동 검증을 모두 언급하지만, 실제로 두 지표가 불일치할 경우(예: 자동 기준은 통과했으나 BD팀/고객이 불만족) 어떤 우선순위로 판단할지 명확하지 않습니다.
- **오픈 이슈**와 **리스크**가 따로 정리되어 있으나, 일부 핵심 전제(비용, rate limit 등)가 미확정 상태임에도 마일스톤·일정·범위가 사실상 확정적으로 서술된 점은 논리상 불안정 요소입니다.

#### 개선 포인트
- 수동 검증과 자동 평가의 불일치 발생 시 의사결정 플로우를 명확히 정의해야 합니다.
- ‘고객 데모 가능’의 기준을 좀 더 구체적으로(예: NPS, 5점 척도 만족도 등) 정의할 필요가 있습니다.
- 기능별로 “이전 단계 선행조건 미달 시 어떻게 할 것인가(Go/No-Go)” 분기 로직이 부족합니다.

### 2. 실행 가능성

#### 평가
- **범위**는 실질적으로 엔드-투-엔드(디자인 고도화, LLM 판별, Vision API 평가, CLI 통합, 운영/모니터링/보안/피드백 루프)로 매우 넓은 편입니다.
- **인력**은 “AI + BD팀 1명(실질적으로 1명)”이며, 외부 리소스는 “필요시” 투입 검토라고만 되어 있습니다. 5~7스프린트 내 완료는 실무상 과소평가로 판단됩니다(설계, 개발, QA, 운영, 리팩터링, 커뮤니케이션 모두 한 명이 감당 불가).
- **외부 연동/비용/성능** 등 핵심 인프라가 사전 실증 전제로 되어 있지만, 실제로는 이 부분이 전체 일정/범위/예산을 결정하므로, “실증 후 범위 확정”이라는 불확실성이 큽니다.
- **운영/모니터링/보안** 등은 기능은 명시됐으나, 구체적 설계 및 리소스 배정이 빠져 있습니다.
- **WSL 기반 CLI 파이프라인**은 초기 PoC에서는 실현 가능하나, 실제 반복적 실서비스 환경(여러 명 사용, 장애 복구, 이중화, 확장/유지보수)에는 취약할 수 있습니다.

#### 개선 포인트
- 최소 2~3인 이상의 개발/QA/운영 인력(파트타임이라도) 현실적 투입/배분 계획이 필요합니다.
- Vision API/CLI 등 외부 서비스 비용, rate limit, 성능이 확정되기 전에는 전체 범위/일정을 “확정”이라 표현하지 말고, 명확히 “조건부”로 표시해야 합니다.
- 운영/보안/모니터링 등은 실제 시스템 운영 경험이 있는 담당자(또는 외주) 지정이 필요합니다.
- WSL CLI 기반 파이프라인의 한계(로컬 환경 불안정, 개발자 환경 종속성)에 대해 중장기 대안(원격 빌드/이중화) 고려가 필요합니다.

- **운영 담당자 지정**: 장애 대응, 모니터링, 보안, 데이터 관리 등 실질적 운영 담당자/역할분담이 없습니다.
- **교육/온보딩**: BD팀, 신규 운영자 등을 위한 매뉴얼, 교육 자료 언급이 없습니다.

### 4. 리스크

#### 가장 우려되는 실패 요인
- **외부 서비스 불확실성**: Vision API/CLI의 비용, rate limit, 품질, 장애 시 대안이 실제로 명확하지 않음. 이 부분이 풀리지 않으면 프로젝트 전체가 지연·중단될 수 있음.
- **인력 과소배치**: 1명+AI로 실질적 운영/품질/보안 확보는 불가능. 검증/운영/QA 리소스 미확보는 중대한 리스크.
- **품질 측정 불일치**: 자동 평가와 실제 사용자/BD팀 평가가 지속적으로 괴리될 경우, 자동화 루프가 무력화되고 반복적인 수동 개입(=비용 증가, 일정 지연)으로 이어질 수 있음.
- **운영/보안 체계 미흡**: 실제로 장애·보안 이슈 발생 시 즉각적 대응이 불가할 가능성. 이력 관리, 감사 로그, 키 관리 등 실운영에 필수적인 부분이 설계만 있고, 담당자/프로세스가 없다면 실질적 운영 리스크 큼.
- **WSL CLI 환경 한계**: 개발/테스트 단계에서는 문제없지만, 실서비스/여러 명 동시 사용 시 확장성·안정성 문제 발생 가능.

## Prototype Builder v2 PRD 검토 의견

### 1. 시장 적합성: 이 솔루션이 현재 시장/트렌드와 맞닿아 있나요?
**매우 적합합니다.** 본 솔루션은 현재 IT 시장의 가장 뜨거운 트렌드 중 하나인 **생성형 AI 기반의 개발 생산성 향상 및 디자인 자동화** 흐름에 정확히 부합합니다.

*   **주요 트렌드와의 연결성:**
    *   **AI-Powered Development (AIDev):** PRD 입력만으로 프로토타입을 자동 생성하고, O-G-D(Optimize-Generate-Discriminate) 루프를 통해 품질을 고도화하는 접근 방식은 AIDev의 핵심 목표인 개발 주기 단축과 비용 절감에 기여합니다.
    *   **Generative Design/UI:** 텍스트 프롬프트(PRD)에서 시각적으로 완성도 높은 UI/UX를 생성하려는 시도는 Adobe Firefly, Midjourney, DALL-E의 이미지 생성, 그리고 Figma의 AI 플러그인 등 디자인 업계 전반에서 나타나는 흐름입니다. 특히 "AI가 만든 느낌"을 벗어나 전문가 수준의 디자인 품질을 목표로 하는 점이 차별화됩니다.
    *   **Low-Code/No-Code Platform의 진화:** 단순한 드래그 앤 드롭을 넘어, 자연어 기반으로 복잡한 애플리케이션 프로토타입을 생성하는 것은 Low-Code/No-Code 플랫폼의 궁극적인 지향점입니다.
    *   **Developer Experience (DX) 및 Productivity Enhancement:** 개발자뿐만 아니라 기획자, BD팀 등 비개발 직군의 생산성을 극대화하여 제품 기획-검증-개발 파이프라인 전체의 효율을 높입니다.

*   **유사 서비스/제품 및 차별점:**
    *   **Vercel AI SDK / ChatGPT Code Interpreter / GitHub Copilot Workspace:** 이들은 주로 개발자를 위한 코드 생성 및 보조 도구입니다. UI/UX 디자인보다는 기능 구현이나 코드 스니펫 생성에 초점을 맞춥니다. Prototype Builder는 'PRD → 고객 데모 가능한 프로토타입'이라는 명확한 비즈니스 목표와 BD팀이라는 특정 사용자를 타겟팅하며, 코드 품질보다는 **디자인 품질과 PRD 정합성**에 강한 초점을 맞춥니다.
    *   **Galileo AI / Uizard / Microsoft Copilot Studio:** 이들은 텍스트 프롬프트에서 UI를 생성하는 기능을 제공하며, Figma 등 디자인 툴과 연동되기도 합니다.
        *   **차별점:**
            *   **'impeccable' 디자인 스킬 통합:** 특정 고품질 디자인 시스템을 명시적으로 통합하여 "AI가 만든 느낌"을 벗어나려는 시도는 독특하고 강력한 차별점이 될 수 있습니다. 대부분의 AI 생성 UI는 아직까지는 기본적이고 정형화된 패턴을 따르는 경향이 있습니다.
            *   **Claude Max CLI (비용 효율성):** API 호출 비용 문제를 해결하기 위해 max-cli를 통합하려는 시도는 반복적인 실험과 개선 루프가 필수적인 AI 개발 과정에서 중요한 비용 경쟁력을 제공할 수 있습니다. 대부분의 상용 AI 서비스는 사용량에 비례하는 비용 모델을 가집니다.
            *   **LLM 기반 다차원 판별 및 Vision API 시각 평가:** 정량적/정성적 평가를 결합하여 "데모 가능 수준"을 정교하게 판별하고 개선 루프를 돌리는 시스템은, 단순히 UI를 생성하는 것을 넘어 **품질 보증 및 자동 개선**에 대한 깊은 고민이 담겨 있습니다.
            *   **PRD 정합성 (의미 비교):** 키워드 매칭을 넘어 LLM의 의미 비교를 통해 PRD 반영률을 높이려는 시도는 생성된 프로토타입이 비즈니스 요구사항을 정확히 충족하는 데 핵심적인 역할을 합니다.

### 2. 사실 확인: PRD에 포함된 가정이나 수치 중 검증이 필요한 것들은?
v2 PRD는 많은 가정을 오픈 이슈와 리스크 관리 섹션에 명시하여 투명성을 높였지만, 여전히 핵심적인 검증이 필요합니다.

*   **1) Claude Max CLI (max-cli)의 비용 및 성능 가정 (가장 중요):**
    *   **가정:** "빌드 비용 $0/회 (구독 내)", "무제한 (구독 내)" 반복 가능.
    *   **검증 필요:** max-cli의 실제 구독 모델, rate limit, 그리고 구독 계정의 '무제한'이 어느 정도의 한도를 의미하는지 명확한 실측이 필수적입니다. 만약 rate limit이 예상보다 낮거나, 구독 비용이 예상보다 높다면 프로젝트의 경제적 타당성이 크게 흔들릴 수 있습니다. (PRD에서 오픈 이슈 2번, 리스크 관리 섹션에서 잘 인지하고 있음)
    *   **추가 검증:** WSL 환경에서의 안정성 및 성능 (빌드 시간, 자원 소모) 역시 충분히 검증되어야 합니다.

*   **2) Vision API의 비용 및 효과 가정:**
    *   **가정:** "Vision API 스크린샷 평가"로 "UI 차원(25%) 정확도 대폭 향상" 및 "시각적 완성도 Vision API 0.8+".
    *   **검증 필요:** 스크린샷 1장당 실제 토큰 사용량 및 비용 실측이 필요합니다. 또한, Vision API가 단순히 시각적 요소를 인식하는 것을 넘어, 'impeccable' 디자인 시스템의 복잡한 규칙(폰트 페어링, 모듈러 스케일, OKLCH 색상 등)을 이해하고 '전문가 수준'으로 평가하며 구체적인 개선 피드백을 제공할 수 있는지에 대한 PoC가 필요합니다. 단순히 "점수"를 주는 것을 넘어 "어떻게 개선해야 하는지"를 알려주는 것이 핵심입니다. (PRD에서 오픈 이슈 1번, 리스크 관리 섹션에서 잘 인지하고 있음)

*   **3) 'impeccable' 디자인 스킬 통합의 실현 가능성:**
    *   **가정:** 'impeccable 7도메인 참조문서 + 안티패턴'을 Generator에 주입하면 "impeccable 7도메인 적용, 전문가 수준"의 디자인이 나온다.
    *   **검증 필요:** LLM이 복잡하고 추상적인 디자인 원칙(예: 시각적 계층구조, 공간감, 유동적 레이아웃)을 얼마나 정확하게 이해하고 출력물에 반영할 수 있는지에 대한 심도 깊은 프롬프트 엔지니어링 및 반복적인 실험이 필요합니다. 단순히 문서를 주입하는 것을 넘어, 실제 디자인 전문가의 피드백을 통해 퀄리티를 검증하고 미세 조정하는 과정이 중요합니다.
    *   **추가 검증:** 참조문서의 크기가 시스템 프롬프트 토큰 한도를 초과할 경우, 효율적인 주입 전략(요약, 동적 주입, 도메인별 우선순위)에 대한 검토가 필요합니다. (PRD에서 오픈 이슈 3번에서 인지하고 있음)

*   **4) LLM 기반 5차원 정밀 판별 및 재현성:**
    *   **가정:** LLM이 5개 차원(build/ui/functional/prd/code)을 "정밀 판별"하고, PRD 정합성을 "의미 비교 90%+"로 산출하며, "5라운드 내 80점 도달" 목표를 달성한다.
    *   **검증 필요:** LLM의 평가 재현성(±20점 내) 확보가 매우 중요합니다. 동일한 프로토타입에 대해 일관된 평가를 내릴 수 없다면 개선 루프 자체가 불안정해집니다. 이를 위해 프롬프트 엔지니어링, 평가 지표의 구체화, 모델 선택 등에 대한 전략이 필요합니다. (PRD에서 오픈 이슈 6번에서 인지하고 있음)

### 3. 사용자 관점: 목표 사용자가 실제로 이 솔루션을 받아들일 것 같나요?
**높은 잠재력을 가지고 있으며, v2 PRD는 사용자 관점의 장벽을 낮추기 위한 노력이 돋보입니다.**

*   **긍정적인 요소 (수용 가능성 높음):**
    *   **BD팀의 명확한 Pain Point 해결:** "PRD 입력 → 고객 데모 가능한 프로토타입 자동 생성"이라는 목표는 BD팀의 핵심 니즈를 직접적으로 해결합니다. 수동으로 디자이너/개발자에게 요청하고 기다리는 시간, 초기 데모의 낮은 품질 등의 문제를 해소합니다.
    *   **높은 품질 목표:** "impeccable 디자인 스킬 통합" 및 "전문가 수준"의 디자인 품질 목표는 BD팀이 고객에게 자신 있게 보여줄 수 있는 결과물을 기대하게 합니다. "AI가 만든 느낌"을 벗어나려는 노력이 사용자 신뢰에 중요합니다.
    *   **피드백 루프 반영 (v2 개선):** BD팀 자체 검수, 파일럿 고객 피드백, 수동 품질 검증 절차를 KPI 및 MVP 기준으로 포함한 것은, 자동화된 시스템이 놓칠 수 있는 '실제 사용자 경험'을 반영하려는 의지로, 사용자 수용도를 높이는 데 결정적인 역할을 할 것입니다.

*   **도입 장벽 및 고려사항:**
    *   **초기 품질에 대한 기대치:** 'impeccable'이라는 단어가 주는 기대치가 매우 높습니다. 만약 초기 결과물이 기대에 미치지 못한다면, BD팀은 시스템에 대한 신뢰를 잃고 기존 수동 방식으로 회귀할 가능성이 있습니다. 첫인상이 매우 중요합니다.
    *   **제어 및 수정의 용이성:** AI가 생성한 프로토타입이 90% 만족스럽지만 10%를 수정해야 할 때, BD팀이 직접 쉽게 수정할 수 있는 방법이 없다면 (예: 디자인 툴 연동, 쉬운 코드 수정), 결국 재요청 루프나 수동 작업으로 이어질 수 있습니다. PRD는 '추가 수정 없이 바로 데모 가능'을 목표로 하지만, 미세 조정의 필요성은 항상 존재합니다.
    *   **PRD 작성의 학습 곡선:** AI가 고품질 프로토타입을 생성하려면 PRD가 매우 명확하고 구체적이어야 할 것입니다. BD팀이 AI가 잘 이해할 수 있는 방식으로 PRD를 작성하는 방법에 대한 가이드라인이나 교육이 필요할 수 있습니다.
    *   **속도와 반복성:** '무제한 반복'이 가능하다 해도, 한 번의 O-G-D 루프가 너무 오래 걸린다면(예: 수십 분) BD팀의 생산성을 저해할 수 있습니다. 빠른 피드백 루프는 AI 도구의 핵심 장점이어야 합니다.
    *   **운영 안정성 및 장애 대응:** max-cli나 Vision API의 장애, rate limit 초과 등으로 시스템이 불안정하거나 지연된다면, BD팀은 중요한 고객 데모를 앞두고 큰 불편을 겪을 것입니다. F423(운영 장애 대응·모니터링)이 P0에 포함된 것은 매우 적절합니다.

### 4. 누락된 시장 관점: 시장/경쟁/트렌드 측면에서 고려되지 않은 것은?
v2 PRD는 많은 부분을 보완했지만, 시장 관점에서 다음과 같은 추가적인 고려가 필요합니다.

*   **1) 디자인 시스템 유연성 및 확장성:**
    *   현재는 'impeccable'이라는 특정 디자인 시스템에 강하게 의존하고 있습니다. 하지만 시장에는 다양한 디자인 시스템(예: Material Design, Ant Design, Tailwind UI)이 존재하며, 기업들은 자체적인 브랜드 가이드라인과 디자인 시스템을 가지고 있습니다.
    *   **고려사항:** 'impeccable'을 레퍼런스 삼아 고품질 디자인을 학습하되, 장기적으로는 사용자가 **자신들의 디자인 시스템을 주입(Few-shot learning, Fine-tuning)하여 프로토타입을 생성**할 수 있는 기능이 강력한 경쟁 우위가 될 것입니다. 이는 고객사의 실제 디자인 시스템과 일치하는 프로토타입을 제공하여 도입 장벽을 크게 낮출 수 있습니다.
*   **2) 멀티모달 입력의 확장:**
    *   현재 PRD 텍스트가 주 입력이지만, 실제 기획 과정에서는 손 스케치, 와이어프레임 이미지, 기존 서비스 스크린샷, 브랜드 로고/색상 팔레트 이미지 등 다양한 시각적 자료가 활용됩니다.
    *   **고려사항:** PRD 텍스트 외에 **이미지 형태의 디자인 레퍼런스(예: Figma 이미지, 스케치 사진)를 함께 입력**하여 AI가 이를 참고하도록 하는 기능은 훨씬 더 풍부하고 정확한 결과물을 기대하게 할 것입니다. Claude Vision API를 활용하여 이 기능을 구현할 수 있는 잠재력이 있습니다.
*   **3) 프로토타입의 후처리 및 연동:**
    *   생성된 프로토타입이 Cloudflare Pages에 배포되어 데모에 활용되는 것은 좋지만, 이후의 워크플로우에 대한 고려가 부족합니다.
    *   **고려사항:**
        *   **디자인 툴 연동:** 생성된 프로토타입을 Figma, Sketch, Adobe XD 등 디자이너가 주로 사용하는 툴로 **내보내는 기능**은 디자이너와의 협업 및 수정 프로세스를 원활하게 할 것입니다.
        *   **프론트엔드 프레임워크/라이브러리 호환성:** 생성된 코드가 React, Vue, Angular 등 특정 프론트엔드 프레임워크에 최적화된 형태로 제공되거나, 혹은 Storybook 같은 컴포넌트 라이브러리에 바로 통합될 수 있다면 개발팀의 생산성까지 높일 수 있습니다. (현재는 '프로덕션 코드 전환'이 제외 범위이지만, '개발자가 쉽게 활용 가능한 코드' 수준으로의 발전은 필요)
*   **4) 윤리적 AI 및 접근성 (Accessibility):**
    *   생성형 AI는 학습 데이터의 편향을 반영하여 특정 디자인 패턴이나 시각적 요소를 반복하거나, 접근성(예: 색상 대비, 키보드 내비게이션)을 고려하지 않은 결과물을 생성할 수 있습니다.
    *   **고려사항:** 생성된 프로토타입에 대한 **자동 접근성 검사(WCAG 기준 등)** 및 편향성 완화를 위한 디자인 가이드라인 주입 등 윤리적 AI 관점에서의 품질 관리도 중요합니다.
*   **5) 경쟁 환경의 빠른 변화 대응:**
    *   AI 디자인/개발 도구 시장은 매주 새로운 기능과 서비스가 쏟아져 나오는 초고속 성장 시장입니다.
    *   **고려사항:** 경쟁사 동향을 지속적으로 모니터링하고, 본 솔루션의 핵심 차별점(impeccable 디자인, 비용 효율성, PRD 정합성)을 계속 강화하며 신속하게 새로운 트렌드를 통합할 수 있는 로드맵이 필요합니다.

### 5. 착수 판단: Conditional

PRD v2는 핵심적인 리스크와 운영상의 고려사항, 사용자 피드백 루프를 대폭 강화하여 프로젝트의 현실성을 크게 높였습니다. 그러나 프로젝트의 성공 여부를 결정할 **핵심 전제 조건들(max-cli의 비용/성능/제한, Vision API의 비용/효과)**이 아직 검증되지 않았습니다. PRD 자체에서도 "Vision API/CLI 등 외부 서비스의 실제 한도/비용 실측 없이는 전체 범위/일정 산정이 불가하여, 사전 실험 및 조건부 착수가 필요함"이라고 명시하고 있습니다.

이러한 불확실성이 해소되기 전까지는 본격적인 개발 착수를 권장하기 어렵습니다.

**조건:**
1.  **Claude Max CLI (max-cli)의 실제 구독 비용, rate limit, WSL 환경에서의 안정성 및 성능에 대한 PoC 완료 및 결과 공유.**
2.  **Claude Vision API의 스크린샷 1장당 실제 비용, 그리고 'impeccable' 디자인 가이드라인에 따른 시각적 품질 평가의 유효성 및 구체적인 피드백 제공 가능성에 대한 PoC 완료 및 결과 공유.**
3.  **위 두 가지 PoC 결과가 PRD의 목표값(비용, 성능, 정확도)을 충족하거나, 목표 미달 시 스코프 및 예산 조정 계획 수립.**

이 조건들이 충족된다면, Prototype Builder v2는 시장에서 강력한 경쟁력을 가질 수 있는 잠재력이 충분합니다.

## 기술 검토 의견: Prototype Builder v2 PRD

### 1. 기술적 실현 가능성

**긍정적 요소:**
- LLM 기반 평가 체계(Vision API, 의미적 PRD 비교)는 기술적으로 실현 가능하며, 기존 정적 분석보다 품질 판별 정확도를 획기적으로 향상시킬 수 있습니다.
- WSL 환경의 max-cli 통합은 API 비용을 제로화할 수 있는 합리적인 접근법입니다.

**주요 기술적 제약 및 병목:**
- **WSL CLI 의존성의 불안정성**: 로컬 CLI 환경은 확장성과 이식성에 심각한 제약을 가집니다. 다중 사용자/동시 요청 시나리오에서 단일 머신의 큐 관리(F420)만으로는 운영 부하를 견디기 어렵습니다. 장애 발생 시 복구 시간이 길어질 수 있습니다.
- **토큰 한도 초과 위험**: impeccable 7도메인 참조문서를 시스템 프롬프트에 통합(F414) 시, Claude 모델의 컨텍스트 윈도우(현재 200K 토큰)를 초과하거나, 남은 토큰이 실제 생성에 부족할 수 있습니다. 동적 주입/요약 전략이 필수적이며, 이는 추가적인 설계/구현 복잡도를 유발합니다.
- **Vision API 비용 및 성능 불확실성**: 스크린샷 평가는 토큰 소모가 크고(고해상도 이미지), 응답 지연이 발생할 수 있습니다. 빌드당 비용이 $1 이상으로 추정되면 비용 절감 효과가 상쇄됩니다. 또한, 평가 결과의 재현성(동일 입력에 대한 점수 변동)이 낮을 수 있어 품질 수렴 루프에 악영향을 줄 수 있습니다.
- **외부 서비스 장애 전파**: max-cli, Vision API, Anthropic API 등 3개의 외부 서비스에 의존합니다. 하나의 서비스 장애가 전체 빌드 파이프라인을 마비시킬 수 있으며, F423의 fallback 설계가 매우 복잡해집니다.

### 2. 아키텍처 적합성

**현재 아키텍처의 문제점:**
- **단일 지점 장애(SPOF)**: WSL 로컬 CLI는 명백한 SPOF입니다. 해당 머신의 하드웨어 장애, 네트워크 문제, OS 업데이트 시 전체 서비스 중단이 불가피합니다.
- **이벤트 기반 비동기 처리 미흡**: "큐 관리(F420)"가 설명되었지만, Cloudflare Workers/D1 기반의 분산 큐, 작업 상태 추적, 재시도 메커니즘에 대한 명확한 설계가 부족합니다. 빌드 요청이 누락되거나 중복 실행될 위험이 있습니다.
- **데이터 파이프라인 일관성 보장 어려움**: O-G-D 루프 내에서 LLM 평가, Vision API 호출, CLI 생성 등 여러 외부 단계를 거칩니다. 중간 단계 실패 시 데이터 불일치가 발생하고, 전체 트랜잭션을 롤백하거나 보상 작업을 구현하기가 매우 복잡합니다.
- **모니터링 및 관측성 부재**: 외부 서비스 호출 지연, 토큰 사용량, CLI 프로세스 상태 등을 실시간으로 모니터링할 수 있는 체계가 PRD에 구체적으로 명시되지 않았습니다.

**대안 아키텍처 제안:**
- **로컬 CLI의 제한적 사용**: 초기 MVP에서는 로컬 CLI를 사용하되, 장기적으로는 **원격 실행 에이전트(소규모 VM/컨테이너) 풀**을 도입하는 아키텍처로의 진화 경로를 명시해야 합니다. 이를 통해 확장성과 내결함성을 확보할 수 있습니다.
- **이벤트 드리븐 워크플로우 엔진 도입**: 빌드 작업을 상태 머신으로 모델링하고, 각 단계를 이벤트로 트리거하는 방식을 고려해야 합니다. (예: Cloudflare Queues + D1 for state persistence). 이는 복잡한 O-G-D 루프와 재시도 로직을 관리하는 데 훨씬 유리합니다.
- **평가 결과 캐싱 계층**: 동일한 스크린샷/프로토타입에 대한 Vision API 평가 결과를 캐시하여 비용과 지연 시간을 줄여야 합니다.

### 3. 구현 복잡도

**과소평가된 난이도:**
- **LLM 평가의 재현성 확보(F417)**: "±20점 범위 내 재현성"은 매우 도전적인 목표입니다. 프롬프트 엔지니어링, 온도(Temperature) 설정, 평가 기준의 객관화에 상당한 실험과 반복이 필요하며, 한 명의 담당자로는 부족할 수 있습니다.
- **운영 장애 대응(F423)**: max-cli 장애 시 API로의 "자동 전환"은 단순히 엔드포인트를 바꾸는 것이 아닙니다. 두 방식은 요청 형식, 응답 구조, 비용 구조가 완전히 다릅니다. 상태를 유지하면서 전환하는 로직은 상당히 복잡합니다.
- **데이터 일관성(F425)**: 빌드 → 평가 → 피드백의 전 과정에서 발생하는 모든 메타데이터(로그, 점수, 중간 산출물)를 원자적으로 저장하고, 실패 시 정리하는 것은 분산 트랜잭션 문제에 가깝습니다.
- **보안 관리(F426)**: WSL 로컬 환경에 API 키를 안전하게 저장하고, CLI 프로세스가 해당 키에 접근하도록 하는 것은 생각보다 까다롭습니다. 하드코딩이나 평문 저장을 피하기 위해 로컬 키 관리 도구 도입이 필요합니다.

### 4. 기술 리스크

**가장 우려되는 실패 요인 (우선순위 순):**
1.  **로컬 CLI 운영 리스크**: 단일 머신 의존성으로 인한 확장 불가, 장애 복구 시간 과다, 팀 멤버 간 개발/운영 환경 불일치가 발생할 가능성이 매우 높습니다. 이는 서비스 신뢰도의 근본적 위협입니다.
2.  **비용/성능 실증 실패**: Sprint 0에서의 실증 실험 결과, Vision API 비용이 예상보다 높거나, max-cli rate limit이 매우 낮게 확인될 경우, 전체 프로젝트의 가치 제안이 붕괴됩니다.
3.  **품질 평가의 유효성 위기**: LLM과 Vision API가 매기는 "고점수"가 BD팀이나 실제 고객이 느끼는 "데모 가능 품질"과 상관관계가 낮을 수 있습니다. 이 경우 모든 기술적 고도화 노력이 무의미해집니다. F424(수동 검증)을 강제하는 것은 좋지만, 이조차 주관적일 수 있습니다.
4.  **인력 리소스 부족**: 5~7 Sprint에 해당하는 복잡한 다중 트랙 작업을 1명의 담당자와 AI 에이전트만으로 수행하는 것은 현실적으로 불가능에 가깝습니다. 품질 저하와 일정 지연이 불가피하며, 이는 기술 부채로 직결됩니다.

### 5. 착수 판단

**착수 판단: Conditional**

**착수 조건:**
1.  **Sprint 0 실증 완료**: 오픈 이슈 #1, #2에 명시된 **Vision API 비용**과 **max-cli rate limit**에 대한 정량적 실측 데이터가 확보되고, 그 결과가 프로젝트의 경제성/실용성 목표(빌드당 $0 근접, 하루 N회 이상 실행 가능)를 충족할 때.
2.  **아키텍처 설계 보완**: 현재의 **WSL SPOF 아키텍처**에 대한 구체적인 위험 완화 계획(예: 빠른 장애 복구 절차, 모니터링 방법, 단기 대체안)과 **중장기 이전 로드맵**이 수립되어야 합니다.
3.  **핵심 리소스 확보**: 최소 **1명의 추가 풀타임 개발자 또는 경험 있는 파트타임 개발자**가 프로젝트에 투입될 수 있어야 합니다. 특히 LLM 평가 재현성, 이벤트 기반 워크플로우 구현, 운영 인프라 구축에 전문성이 필요합니다.
4.  **품질 검증 프로토타입 완성**: impeccable 통합 전, **1~2개의 샘플 PRD**로 현재 시스템과 향후 시스템(수동 프롬프트 개선)의 출력 품질을 BD팀과 잠재 고객이 평가하는 **블라인드 테스트**를 수행하여, 기술적 개선의 효과에 대한 신뢰를 확보해야 합니다.

위 조건이 충족되지 않을 경우, 프로젝트는 높은 확률로 예산 초과, 일정 지연, 또는 기술적 실패를 맞이할 것입니다. 특히 **Conditional**의 핵심은 **로컬 CLI의 운영 리스크**와 **인력 과소배정** 문제를 해결하는 방향으로 PRD와 실행 계획을 수정하는 데 있습니다.
