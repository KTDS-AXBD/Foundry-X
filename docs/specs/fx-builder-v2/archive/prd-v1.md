## 1. 요약 (Executive Summary)

**한 줄 정의:**
Prototype Builder의 디자인 품질·판별 정확도·비용 구조를 동시에 고도화하여, PRD 입력만으로 고객 데모가 바로 가능한 수준의 프로토타입을 자동 생성하는 시스템.

**배경:**
Phase 19(Builder Evolution)에서 5차원 품질 스코어링 체계(build/ui/functional/prd/code)와 O-G-D 자동 개선 루프를 구축하고, Claude Max CLI PoC를 완료했다(96% 구현). 그러나 3가지 핵심 한계가 남아 있다:

1. **디자인 품질**: LLM 기본 생성 결과는 "AI가 만든 느낌"이 강함. 과용 폰트(Inter/Arial), 회색 계열 단조로움, 카드 중첩 등 전형적인 AI 디자인 패턴에서 벗어나지 못함
2. **판별 정확도**: Discriminator가 정적 체크리스트(h1 존재, CTA 버튼, 기능 2개+) 수준으로, 실제 품질 향상 루프가 의미 있게 작동하지 않음
3. **비용 구조**: API 호출당 ~$2.40 비용으로 반복 개선 실험이 예산 제약을 받음. max-cli PoC는 완료했으나 본격 통합 미완

**목표:**
impeccable 디자인 스킬 시스템 통합 + LLM 기반 다차원 판별 + max-cli 본격 통합으로, 추가 수정 없이 고객 데모가 가능한 프로토타입을 자동 생성한다.

## 2. 문제 정의

### 2.1 현재 상태 (As-Is)

**디자인 품질:**
- 생성 프롬프트에 디자인 가이드라인 없음 → LLM 기본 스타일 그대로 출력
- 과용 폰트(Inter, Arial, system-ui), 순수 흑색/회색, 낮은 시각적 계층구조
- 단조로운 레이아웃: 카드 반복, CTA 버튼만 강조, 섹션 간 구분 부족
- 모바일 반응형이 깨지거나 부자연스러운 경우 다수

**판별 정확도:**
- Discriminator 체크리스트: h1 존재, 기능 2개+, CTA 버튼, 반응형, 시각 계층구조 — 5개 항목
- 체크리스트 통과해도 "조잡한" 프로토타입이 80점 이상 나올 수 있음
- PRD 정합성(25%)은 키워드 매칭 수준 — 의미적 반영도 미평가
- 자동 개선 루프의 피드백이 구체적이지 않아 품질이 수렴하지 않음

**비용 구조:**
- API 모드: Sonnet $3/$15 per 1M tokens → 빌드당 ~$2.40
- 월 $20 예산으로 ~8회만 실행 가능 → 반복 실험 제약
- max-cli PoC 완료(poc-cli.md): rate limit, 응답 시간 측정 완료
- 본격 통합 미완 — WSL 로컬 실행 환경 미구축

### 2.2 목표 상태 (To-Be)

| 차원 | As-Is | To-Be |
|------|-------|-------|
| 디자인 품질 | LLM 기본 스타일 | impeccable 7도메인 적용, 전문가 수준 |
| PRD 반영률 | 키워드 매칭 (체감 60~70%) | LLM 의미 비교 90%+ |
| 시각적 평가 | 정적 DOM 분석만 | Vision API 스크린샷 평가 |
| 판별 정확도 | 5항목 체크리스트 | LLM 5차원 정밀 판별 |
| 빌드 비용 | ~$2.40/회 (API) | $0/회 (max-cli 구독) |
| 반복 가능성 | 월 ~8회 | 무제한 (구독 내) |
| 데모 가능성 | 추가 수정 필요 | 바로 데모 가능 |

### 2.3 시급성

- Phase 22(Offering Skill v2)에서 프로토타입이 핵심 산출물 — 품질 미달 시 파이프라인 전체 효용 저하
- API 비용 누적이 반복 실험을 제약하고 있어 빠른 max-cli 전환이 필요
- impeccable이 공개된 시점에서 디자인 품질 기준이 시장에서 상향 중

### 4.3 제외 범위 (Out of Scope)

- **실시간 협업**: 여러 사용자가 동시에 프로토타입을 편집하는 기능
- **프로토타입 → 프로덕션 코드 전환**: 생성된 코드를 실서비스로 직접 배포
- **원격 빌드 서버**: VPS/EC2/GitHub Actions 환경 — WSL 로컬만 지원
- **LLM 외 ML 모델**: 커스텀 학습된 디자인 품질 모델 (LLM API만 사용)
- **Lighthouse 통합**: Phase 19 로드맵에 있었으나 이번 범위에서 제외 (Vision API로 대체)

### 4.4 외부 연동

| Claude Code CLI (max-cli) | WSL subprocess | 필수 |

| impeccable 참조문서 | 정적 파일 (빌드 시 번들) | 필수 |

## 5. 성공 기준

### 5.1 정량 지표 (KPI)

| 지표 | 현재값 | 목표값 | 측정 방법 |
|------|--------|--------|-----------|
| 5D 품질 점수 | 체감 30~40점 | 80점+ 자동 도달 | prototype_quality 테이블 |
| PRD 반영률 | ~60~70% (키워드) | 90%+ (LLM 비교) | prd 차원 점수 |
| 시각적 완성도 | DOM 분석만 | Vision API 0.8+ | ui 차원 점수 |
| 빌드 비용 | ~$2.40/회 | $0/회 (구독 내) | prototype_usage_logs |
| 자동 개선 루프 수렴 | 미수렴 | 5라운드 내 80점 도달 | ogd_rounds 평균 |
| 안티패턴 검출률 | 0% | 95%+ | Discriminator 로그 |

### 5.2 MVP 최소 기준

- [ ] impeccable 4개 핵심 도메인(타이포, 색상, 공간, 반응형)이 Generator 프롬프트에 적용
- [ ] LLM 판별기가 5차원 중 최소 prd + ui 차원에서 작동
- [ ] max-cli가 WSL에서 빌드 파이프라인에 연결되어 실제 빌드 1건 이상 성공
- [ ] 1개 이상의 PRD로 생성한 프로토타입이 고객 데모 가능 수준 (팀 내부 확인)

### 5.3 실패/중단 조건

- max-cli rate limit이 하루 5건 미만으로 제한되어 실용성 없음
- Vision API 비용이 빌드당 $1 이상이어서 비용 절감 효과 상쇄
- LLM 판별 재현성이 ±20점 이상으로 불안정
- 3 Sprint 투입 후에도 80점 자동 도달이 불가능

## 6. 제약 조건

### 6.1 일정

- Phase 22 범위에 포함 또는 별도 Phase로 편성
- 예상 Sprint: 5~7개 (3트랙 병렬)

### 6.2 기술 스택

- 프론트엔드: React 18 + Vite 8 (기존 유지)
- 백엔드: Hono + Cloudflare Workers (기존 유지)
- 인프라: Cloudflare D1 + Pages (기존 유지)
- 신규: Claude Code CLI (WSL subprocess), Claude Vision API
- 참조: impeccable 정적 참조문서 (빌드 시 번들)

### 6.3 인력/예산

- 투입: AI 에이전트 + BD팀 1명 (Sinclair)
- 예산: Claude Max 구독 $100/월 범위 내
- Vision API 비용: 별도 확인 필요 (Anthropic API 과금)

### 6.4 컴플라이언스

- impeccable: Apache 2.0 라이선스 → 상업적 사용 가능
- Claude API 이용약관 준수
- 고객 데이터(PRD)가 외부 API로 전송됨 — 기존과 동일한 보안 수준 유지
