## 기술 검토 의견

### 1. 기술적 실현 가능성
**기본적으로 실현 가능하지만 몇 가지 기술적 제약이 존재합니다:**

- **Cloudflare Workers 환경 제약**: Phase B의 `phase-progress.sh` 실행이 주요 문제입니다. Workers는 쉘 스크립트 실행을 지원하지 않으므로, **사전 계산된 JSON 캐싱 전략**이 필수적입니다. GitHub Actions에서 주기적으로 스크립트 실행 후 결과를 JSON으로 저장하고 Workers에서 fetch하는 방식이 필요합니다.
- **파서 호환성 리스크**: 16개 기존 스크립트(board-sync-spec.sh 등)가 SPEC.md 구조 변경에 민감합니다. **Dual Operation 기간(1주)이 불충분**할 수 있으며, 정규식 기반 파싱은 미묘한 형식 변화에도 쉽게 깨집니다.
- **상태 동기화 문제**: F-item 세부 상태(괄호 표기)는 SPEC.md에만 존재하고 D1 캐시와의 실시간 동기화 메커니즘이 명확하지 않습니다. **상태 불일치 가능성**이 있습니다.

### 2. 아키텍처 적합성
**문서 중심 아키텍처의 한계가 노출됩니다:**

- **SSOT(Single Source of Truth) 분산 위험**: Blueprint, Roadmap, SPEC.md로 문서가 분리되면서 진실의 원천이 흐려집니다. 변경 시 **3개 문서 간 일관성 유지**가 새로운 부담으로 작용합니다.
- **API 계층의 임시성**: velocity, phase-progress 등 기존 쉘 스크립트 결과를 단순히 노출하는 API는 **데이터 파이프라인 아키텍처가 부재**합니다. 지속적인 데이터 신선도 유지 메커니즘이 명확하지 않습니다.
- **테스트 아키텍처 미비**: "Regression catch coverage 80%" 목표는 좋으나, **F509의 핵심 로직이 테스트 없이 운영된 지 오래**되어 테스트 작성 자체가 높은 복잡도를 가질 수 있습니다. Mocking 전략이 필요합니다.

### 3. 구현 복잡도
**과소평가된 복잡도 요소:**

- **에이전트 호환성 보장**: Claude 에이전트가 새 문서 구조를 이해하도록 프롬프트/컨텍스트를 조정해야 합니다. 이는 **단순 문서 변경 이상의 작업**이며, 에이전트 행동 회귀 테스트가 필요합니다.
- **링크 무결성 유지**: 아카이브 이동 시 `docs/INDEX.md` 및 기존 마크다운 내 상대 경로가 깨질 가능성이 높습니다. **자동 리다이렉트 생성기**가 필수적이며, edge case가 많습니다.
- **상태 머신 구현 누락**: `shared/task-state.ts`의 10-상태 머신과 SPEC.md 표기법의 매핑이 구체화되지 않았습니다. **상태 전이 규칙 검증 로직**이 필요합니다.

### 4. 기술 리스크
**가장 우려되는 3가지 리스크:**

1. **파서 연쇄 고장(Cascading Parser Failure)**: 하나의 스크립트가 깨지면 의존하는 다른 자동화(예: 아카이브, 리포트 생성)가 연쇄적으로 실패할 수 있습니다. **격리 및 회복 메커니즘이 미흡**합니다.
2. **1인 개발 병목의 극대화**: 모든 변경(문서, API, 테스트, 자동화)을 동시에 진행하면서 **품질 관리와 일정 관리가 동시에 무너질 위험**이 높습니다. Phase A의 "코드 변경 0건"은 사실상 불가능합니다(파서 수정 필요).
3. **모니터링/알림 체계의 실질적 공백**: PRD에는 모니터링 요구사항이 명시되었지만, **기존 인프라에 알림 시스템이 존재하지 않습니다**. 새로운 운영 부담이 추가됩니다.
