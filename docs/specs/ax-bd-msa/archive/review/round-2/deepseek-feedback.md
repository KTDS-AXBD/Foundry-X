## 검토 의견: AX BD MSA 재조정 PRD v2

### 1. **기술적 실현 가능성**
**기본적으로 실현 가능하나, 몇 가지 심각한 기술적 제약이 존재합니다.**

*   **긍정적 요소:**
    *   Cloudflare Workers + D1 + Hono 스택은 MSA에 적합한 경량 런타임 환경을 제공합니다.
    *   `harness-kit`을 통한 표준화된 서비스 템플릿은 신속한 서비스 생성과 일관성 유지에 도움이 될 것입니다.
    *   REST + EventBus의 하이브리드 통신 방식은 실용적인 접근입니다.

*   **주요 기술적 제약 및 병목:**
    1.  **D1의 분산 데이터베이스 한계:** 현재 단일 D1 인스턴스에 모든 테이블이 혼재된 상태에서의 "논리적 분리"는 운영 복잡성과 성능 위험을 초래합니다. 특히 **"테이블 태깅"만으로는 실제 서비스 경계가 보장되지 않습니다.** 한 서비스의 쿼리가 다른 서비스의 테이블을 직접 조인하는 것을 방지할 기술적 메커니즘이 부재합니다. 이는 데이터 결합도와 캡슐화 위반으로 이어질 수 있습니다.
    2.  **분산 트랜잭션과 데이터 일관성:** PRD에서 지적한 대로 FK와 데이터 무결성은 가장 큰 리스크입니다. EventBus 기반의 eventual consistency는 해결책이지만, **"분리 but 유지" 단계에서 동일 DB를 공유하는 모듈 간에도 이 패턴을 적용해야 하는지 명확하지 않습니다.** 이로 인해 일시적 데이터 불일치와 복잡한 동기화 로직이 발생할 수 있습니다.
    3.  **Cold Start 및 메모리 제한:** 118개 라우트가 60~70개로 줄어도, Foundry-X 자체는 여전히 다수의 모듈을 포함한 "작은 모놀리스"가 될 것입니다. Workers의 메모리 제한(128MB)과 Cold Start 시간은 모듈 수와 초기화 로직에 따라 오히려 악화될 가능성이 있습니다.

### 2. **아키텍처 적합성**
**현재 방향은 타당하나, 몇 가지 근본적인 설계 결정이 필요합니다.**

*   **적절한 부분:**
    *   Strangler Fig 패턴과 "분리 → 이관"의 단계적 접근은 위험을 분산시킵니다.
    *   공통 인프라 컴포넌트(인증, 로깅, 이벤트)를 `harness-kit`으로 표준화하는 것은 MSA의 핵심 원칙에 부합합니다.

*   **개선이 필요한 부분:**
    1.  **서비스 경계의 정의가 너무 기능적(Functional)입니다.** 데이터 소유권(Data Ownership)과 도메인 경계에 기반한 설계가 부족합니다. "발굴", "형상화"라는 비즈니스 프로세스 단계가 독립적인 서비스의 경계로 적절한지 재고해야 합니다. **도메인 주도 설계(DDD)의 Bounded Context 개념을 도입하여 서비스 경계를 재정의할 것을 강력히 권고합니다.**
    2.  **"분리 but 유지"는 위험한 중간 상태입니다.** 이 기간 동안 Foundry-X는 모놀리스와 MSA의 단점을 모두 가질 수 있습니다(복잡한 내부 의존성 + 분산 시스템의 복잡성). 이 상태를 최소화하기 위해 **이관 대상 모듈에 대한 신규 서비스(Gate-X, Launch-X)의 프로토타입을 M1 단계에서 병행 개발하고, 가능한 빠르게 완전 이관하는 전략을 고려해야 합니다.**
    3.  **API Gateway의 부재:** 여러 서비스가 생겼을 때, 라우팅, 인증/인가 집중, 로깅, Rate Limiting을 처리할 진입점이 필요합니다. 현재 PRD에는 이에 대한 명시적 계획이 없습니다. `harness-kit`이 이를 담당할 수 있도록 확장되어야 합니다.

### 3. **구현 복잡도**
**현재 평가보다 훨씬 높은 복잡도가 예상됩니다. 과소평가된 요소가 많습니다.**

*   **숨겨진 복잡도:**
    1.  **데이터 분리의 진정한 비용:** "테이블 태깅"은 시작에 불과합니다. 공유 테이블을 분리하기 위한 **데이터 마이그레이션 스크립트, 이중 기록(Dual-write) 전략, 데이터 검증 도구, 롤백을 위한 백업/복원 절차**의 설계와 구현이 막대한 작업입니다. 이는 PRD에 간략히 언급되었지만, 전체 작업량의 40% 이상을 차지할 가능성이 있습니다.
    2.  **분산 시스템의 관측 가능성(Observability):** 여러 서비스로 분리되면, 하나의 요청이 거치는 경로를 추적하기 어려워집니다. **분산 트레이싱(예: OpenTelemetry), 중앙화된 로그 집계, 서비스 간 호출 지연 시간 모니터링** 체계가 필수적이며, 이는 `harness-kit`과 배포 파이프라인에 통합되어야 합니다. 현재 PRD에는 모니터링 요구사항이 추상적으로만 명시되어 있습니다.
    3.  **의존성 관리 지옥:** `harness-kit`이 여러 서비스에서 공유 라이브러리가 됩니다. 이 `harness-kit`의 버전 업데이트는 **모든 서비스의 동시 배포 또는 정교한 하위 호환성 관리**를 요구합니다. SemVer 정책만으로는 부족하며, 자동화된 호환성 검사 도구와 체계적인 업그레이드 프로세스가 필요합니다.

### 4. **기술 리스크**
**가장 크게 우려되는 실패 요인은 다음과 같습니다.**

1.  **최우선 리스크: 데이터 일관성 파괴 및 무결성 손실**
    *   **시나리오:** 이벤트 전송 실패 또는 순서 오류로 인해 서비스 간 데이터가 불일치하게 되고, 이는 비즈니스 로직 오류로 직결됩니다. 특히 "분리 but 유지" 단계에서의 복잡한 데이터 접근 패턴은 디버깅이 극히 어렵습니다.
    *   **완화 방안:** M1 단계에서 **이벤트 기반 아키텍처의 상세 설계(Outbox 패턴, 이벤트 순서 보장, 멱등성 처리)를 반드시 완료하고, 핵심 비즈니스 흐름에 대한 엔드투엔드 데이터 일관성 테스트 시나리오를 작성**해야 합니다.

2.  **차순위 리스크: 운영 복잡도 급증으로 인한 장애 대응 불능**
    *   **시나리오:** Production 장애 발생 시, 여러 서비스의 로그를 수집하고, 의존 관계를 파악하여 근본 원인을 찾는 데 과도한 시간이 소요되며, Mean Time To Recovery(MTTR)가 크게 증가합니다.
    *   **완화 방안:** **11.2항의 운영 체계 구축을 M1~M2와 병행하여 선제적으로 완료해야 합니다.** 장애 시나리오(예: D1 장애, 이벤트 버스 장애, 서비스 다운)별 **런북(Runbook)을 사전에 작성**하고 팀이 숙지해야 합니다.

3.  **인력/일정 리스크: 1인 개발의 현실적 불가능성**
    *   PRD에 지적된 대로, 이 규모의 아키텍처 전환을 1명의 개발자(비록 AI Agent 지원을 받더라도)가 2~2.5개월 내에 완성하는 것은 **기술적, 관리적, 정신적 부하 측면에서 현실적으로 불가능에 가깝습니다.** 설계, 구현, 테스트, 문서화, 커뮤니케이션 부담을 고려할 때 **일정이 최소 2배 이상 지연되거나, 품질이 심각하게 저하될 위험이 매우 높습니다.**
