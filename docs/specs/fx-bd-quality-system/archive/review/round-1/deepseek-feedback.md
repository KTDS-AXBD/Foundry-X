## PRD 검토 의견

### 1. 기술적 실현 가능성

**긍정적 측면:**
- Cloudflare Workers + D1 스택은 에이전트 기반 비동기 작업에 적합
- 기존 O-G-D 인프라를 활용하므로 신규 개발 부담 감소

**우려사항:**
1. **LLM 평가의 신뢰성 문제**: Workers AI의 Llama-3.1-8b 모델이 "전문가 수준" 판별을 수행할 수 있을지 의문
   - 디자인 품질, 보안 검증 등은 전문 도메인 지식이 필요
   - False Positive/False Negative 비율이 높을 경우 시스템 신뢰도 하락

2. **CSS 정적 분석의 한계**: "AI 느낌"을 정량적으로 검출하는 알고리즘이 명확하지 않음
   - 폰트, 색상, 레이아웃 패턴의 주관적 판단을 객관화하기 어려움
   - 오탐지로 인해 유효한 디자인 선택이 차단될 위험

3. **PPTX 처리의 기술적 제약**: PRD에서 언급한 대로 PPTX 자동 수정은 제외되었으나, 판별 자체도 복잡함
   - PPTX의 이진 포맷과 복잡한 구조 분석에 LLM만으로는 부족

### 2. 아키텍처 적합성

**개선 필요 사항:**

1. **에이전트 간 의존성 관리 부재**
   - 3개의 QSA 에이전트와 Sentinel이 서로 통신하는 메커니즘이 명시되지 않음
   - 순환 의존성이나 데드락 가능성 존재
   - **제안**: 명확한 이벤트 기반 아키텍처 정의 필요 (예: Cloudflare Queues 활용)

2. **데이터 일관성 보장 미흡**
   - 여러 에이전트가 동시에 동일 산출물을 평가할 경우 데이터 충돌 가능
   - `prototype_quality` 테이블과 `ogd_rounds` 간 동기화 메커니즘이 추상적
   - **제안**: 낙관적 락 또는 트랜잭션 범위 명시 필요

3. **확장성 문제**
   - 모든 판별 로직이 LLM 호출에 의존 → 비용과 지연 시간이 산출물 증가에 따라 선형 증가
   - **제안**: Rule-based 필터링 레이어 도입 (LLM 호출 전 간단한 검사 수행)

### 3. 구현 복잡도

**과소평가된 요소:**

1. **"자율 복구"의 숨겨진 복잡성**
   - 파이프라인 단절 감지는 비교적 쉬우나, 복구는 시스템 전반적 변경 필요
   - 예: `DesignTokenService`와 `prototype-styles.ts` 연결 실패 시, 어떤 기준으로 수정할지 명확하지 않음

2. **에이전트 안정성 보장**
   - LLM의 비결정적 출력으로 인해 에이전트가 불안정한 상태에 빠질 수 있음
   - 재시도, 회로 차단기(Circuit Breaker), 폴백 메커니즘 필요

3. **피드백 루프의 무한 재생성 가능성**
   - Generator → Discriminator 간 불완전한 기준 정렬 시 무한 루프 발생
   - 최대 재시도 횟수와 실패 시 처리 로직 필수

### 4. 기술 리스크

**상위 3개 리스크:**

1. **LLM 평가 품질 리스크 (위험도: 높음)**
   - 판별 기준의 주관성을 LLM이 일관되게 이해하지 못할 가능성
   - 고객 대면 산출물의 품질 저하로 이어질 수 있음

2. **시스템 복잡도 증가로 인한 유지보수성 저하 (위험도: 중간)**
   - 4개의 자율 에이전트와 기존 시스템 간 상호작용이 디버깅을 매우 어렵게 만듦
   - 문제 발생 시 근본 원인 분석이 복잡

3. **비용 통제 불가능성 (위험도: 중간)**
   - LLM 호출이 예측 불가능하게 증가할 수 있음 (무한 루프, 큰 파일 처리 등)
   - "산출물 1건당 $1" 제한은 사후 측정 지표일 뿐, 실시간 제어 메커니즘 부재

### 5. 착수 판단

**착수 판단: Conditional**

**조건:**
1. **PoC(Proof of Concept) 선행 필요**: Prototype QSA의 5차원 평가를 실제 샘플로 검증
   - 최소 50개 다양한 품질의 Prototype에 적용하여 정확도 측정
   - False Positive/False Negative 비율이 10% 미만 확인

2. **아키텍처 상세 설계 확정**: 현재 PRD는 개념 수준
   - 에이전트 간 통신 프로토콜 명시
   - 데이터 일관성 보장 메커니즘 설계
   - 에러 처리 및 복구 시나리오 정의

3. **비용 제어 메커니즘 도입**: LLM 호출에 대한 하드 리밋 설정
   - 토큰 수 기반 예산 제한
   - 비정상 패턴 감지 시 자동 중단

4. **점진적 롤아웃 계획**: Big Bang 방식보다는
   - Phase 1: Prototype QSA만 구현 및 검증
   - Phase 2: Sentinel 기본 감시 기능 추가
   - Phase 3: 전체 시스템 통합
