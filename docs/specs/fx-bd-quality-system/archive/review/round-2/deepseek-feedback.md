## 검토 의견: BD Quality System PRD

### 1. 기술적 실현 가능성
**부분적으로 가능하나 중대한 제약 존재**

**긍정적 요소:**
- Cloudflare Workers + D1 스택은 기존 시스템과의 일관성 유지에 적합
- LLM 기반 판별(Discriminator)과 생성(Generator)의 GAN 구조는 이론적으로 타당
- 기존 O-G-D 인프라를 활용하므로 신규 개발 부담 감소

**기술적 제약/병목:**
- **LLM 평가 품질의 불확실성**: Workers AI의 Llama-3.1-8b가 "전문가 수준" 판별에 충분할지 매우 의문. 특히 디자인 미적 판단, 논리적 일관성 평가는 대규모 모델(Claude 3.5+, GPT-4) 수준의 이해도 필요
- **PPTX 처리의 본질적 한계**: PPTX는 이진 포맷으로, HTML 대비 구조/시각 요소 추출 및 판별이 훨씬 복잡. "판별만 수행"도 상당한 엔지니어링 비용 발생
- **CSS 정적 분석의 표면성**: "AI 느낌"을 CSS Anti-pattern만으로 포착하는 것은 얕은 접근. 레이아웃 의미론, 콘텐츠 독창성, 어조 일관성 등 심층 분석 필요
- **실시간 자율 복구의 신뢰성**: Sentinel이 7개 GAP을 자동 복구하는 것은 DB 스키마 변경, 코드 병합 등 되돌리기 어려운 변경을 수반할 수 있어 위험

### 2. 아키텍처 적합성
**중앙집중식 Sentinel에 과도한 책임 부여, 단일 장애점(Single Point of Failure) 우려**

**문제점:**
- **모놀리식 오케스트레이터**: BD Sentinel이 모든 감시, 판별, 복구를 총괄하면 복잡도가 급증하고 장애 전파 위험 높음
- **데이터 일관성 보장 미흡**: `prototype_quality` ↔ `ogd_rounds` 간 자동 동기화 시 race condition, 부분 실패 시 데이터 불일치 가능성
- **에이전트 간 의존성 관리 부재**: PRD QSA → Offering QSA → Prototype QSA의 파이프라인 흐름에서 실패 전파 및 재시도 정책 불명확

**대안 제안:**
- **이벤트 기반 분산 아키텍처**: 각 QSA를 독립 서비스로 분리, Kafka/PubSub로 이벤트 스트림 연결. Sentinel은 감시만 수행, 복구는 각 서비스의 자율 복구 모듈에 위임
- **판별 결과 캐시 계층 도입**: 동일 산출물에 대한 중복 LLM 호출 방지, 비용 절감 및 응답 시간 개선
- **Rubric 버전 관리 시스템**: Rubric 변경 시 롤백 가능한 버전 관리 및 A/B 테스트 체계 필요

### 3. 구현 복잡도
**과소평가된 복잡도 다수 존재**

**숨겨진 복잡도:**
- **동시성 제어**: 여러 BD 담당자가 동시에 산출물 제출 시, 동일 자원(LLM API, DB)에 대한 경합 발생. Rate limiting, 큐잉 전략 필요
- **Edge Case 처리**:
  - LLM이 비결정적(non-deterministic) 판별을 내릴 경우(동일 입력 → 다른 결과)
  - 외부 참조(impeccable, awesome-design-md) 사이트 다운 시 Fallback
  - 매우 큰 HTML/PPTX 파일(100MB+) 처리 시간 초과
- **피드백 루프의 순환 참조**: 고객 피드백 → Rubric 업데이트 → 생성 품질 변화 → 새로운 고객 피드백 간 피드백 루프가 불안정해질 수 있음
- **보안 판별의 한계**: 정적 분석만으로 기밀 노출을 100% 탐지 불가. 정규식 패턴 매칭은 쉽게 회피 가능, 동적 콘텐츠(JavaScript) 분석 필요

### 4. 기술 리스크
**상위 3개 리스크:**

1. **LLM 평가의 허위 양성/음성**: "전문가 수준"이라는 주관적 기준을 LLM이 일관되게 판별하지 못해, 좋은 산출물을 거부하거나 나쁜 산출물을 통과시킬 위험. 이는 BD팀의 신뢰를 즉각 붕괴시킴.

2. **자율 복구의 예측 불가능성**: Sentinel이 DB 스키마 변경, 코드 수정 등 irreversible change를 시도하다 시스템을 손상시킬 위험. "되돌릴 수 있는 변경"의 경계가 모호.

3. **비용 폭주 가능성**: 산출물당 5차원 판별 + CSS 분석 + Sentinel 감시로 LLM 호출수가 기하급수적으로 증가. Workers AI 무료 티어 한도(일일 10만 토큰)를 쉽게 초과할 수 있음.

**추가 우려사항:**
- **1인 개발-운영의 부담**: Sinclair 1인이 설계, 구현, QA, 운영, 예외 처리를 모두 담당하기엔 범위가 과도함. 특히 24/7 자율 운영 시스템의 장애 대응 부담 큼.
- **고객 피드백 수집의 실용성**: 고객이 설문, 이슈 등록에 적극 참여할지 불확실. 피드백 데이터가 부족하면 루프가 공전.

### 5. 착수 판단
**Conditional**

**착수 조건:**
1. **LLM 판별력 PoC 선행**: 실제 샘플 산출물 50개로 Llama-3.1-8b와 Claude 3.5 Sonnet의 판별 정확도를 인간 평가와 비교. 정확도 85% 이상 달성 시에만 진행.
2. **아키텍처 단순화**: Sentinel의 자율 복구 범위를 "감지 및 알림"으로 제한, 복구는 수동 개입을 통하도록 조정.
3. **비용 제어 장치 명시적 구현**: 산출물당 LLM 호출 수 상한, 일일 비용 한도, 자동 중단 메커니즘을 MVP에 반드시 포함.
4. **단계적 롤아웃 계획 수립**: 모든 BD팀원에게 한꺼번에 적용하지 말고, 1명의 파일럿 사용자에게 2주간 적용 후 문제점 보완.

**권고사항:**
- MVP 범위를 **Prototype QSA에만 집중** (가장 통제된 환경)
- Offering QSA의 PPTX 지원은 P1으로 후순위 조정
- "AI 느낌" 판별을 CSS 분석 + **템플릿 문구 검출(N-gram 분석)** 로 확장
- Manual Intervention을 위한 웹 대시보드(Override, 롤백, 이슈 추적)를 MVP 필수 기능으로 상향

**결론:** 비전은 훌륭하지만, 현재 명세는 기술적 낙관주의에 기반합니다. 자율성의 범위를 축소하고, LLM의 한계를 인정한 보수적 접근이 필요합니다. 위 조건들이 충족되면 착수 가능합니다.
