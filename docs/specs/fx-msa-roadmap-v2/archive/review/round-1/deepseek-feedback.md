## 검토 의견

### 1. 기술적 실현 가능성
**기본적으로 실현 가능하나 몇 가지 기술적 제약이 존재합니다.**

- **ESLint 커스텀 룰 구현의 복잡성**: `no-cross-domain-import` 룰은 AST 분석이 필요하며, 특히 `types.ts` 예외 처리와 모노레포 내 상대 경로 해석에 미묘한 복잡성이 있습니다. TypeScript 경로 매핑(`@/*` 등)이 사용된다면 추가 처리 필요.
- **Hono sub-app 패턴의 런타임 오버헤드**: 각 도메인 sub-app이 독립된 미들웨어 체인을 가질 경우, 글로벌 미들웨어(인증, 로깅) 적용이 복잡해집니다. `app.route()` 시 글로벌 미들웨어가 하위 app에 자동으로 상속되지 않아 중복 등록 필요.
- **D1 테이블 prefix 규약의 한계**: 단일 D1 인스턴스(`foundry-x-db`)를 공유하므로, 실제 데이터 격리는 불가능합니다. 향후 물리적 분리 시 마이그레이션 비용이 발생할 수 있습니다.

### 2. 아키텍처 적합성
**점진적 전략은 합리적이지만, 몇 가지 아키텍처적 결함이 있습니다.**

- **계층형 아키텍처의 한계**: `core/{domain}/services/` 구조는 도메인 내에서도 계층형 아키텍처를 유도할 수 있습니다. 이는 도메인 주도 설계(DDD)의 애그리게이트 경계와 다를 수 있으며, 향후 서비스 분리 시 내부 리팩토링이 필요할 수 있습니다.
- **공유 타입(contract)의 관리 부재**: `types.ts`만 예외로 허용하면, 도메인 간 공유해야 할 복합 타입(예: 공통 DTO)이 증가할 경우 타입 중복이나 순환 의존성 위험이 있습니다. 명시적인 `shared/contracts/` 레이어 도입이 더 안전합니다.
- **API Gateway의 역할 미비**: `fx-gateway`가 프로덕션 트래픽을 처리하지 않는 현재 상태에서는, 실제 MSA 환경에서의 라우팅, 로드 밸런싱, 회로 차단기(circuit breaker) 검증이 이루어지지 않습니다.

### 3. 구현 복잡도
**과소평가된 복잡성이 있습니다.**

- **기존 코드와의 경계 관리**: "신규 파일만 적용"은 git 히스토리 기반으로 CI에서 판단해야 하며, `git diff`와 ESLint 통합이 추가 복잡성을 유발합니다. false positive/negative 가능성이 높습니다.
- **도메인 경계 정의의 주관성**: "Agent/Harness 도메인 분리"를 제외했지만, 신규 F-item이 기존 도메인과 밀접하게 연관될 경우 경계를 명확히 정의하기 어렵습니다. 예를 들어, F535 Graph API가 `agent`와 `ontology` 모두에 걸쳐 있다면?
- **병렬 Sprint 충돌**: AI 에이전트가 독립적으로 작업하더라도, 공유 DB 스키마 변경이나 글로벌 설정 변경에서 충돌이 발생할 수 있습니다. 이는 도메인 경계만으로 해결되지 않습니다.

### 4. 기술 리스크
**가장 우려되는 리스크는 다음과 같습니다.**

- **R1 (개발 속도 저하)가 실제로 발생할 가능성 높음**: 신규 도메인 구조 설계, sub-app 구성, 타입 정의에 초기 Sprint당 1-2일 추가 부하가 예상됩니다. 특히 AI 에이전트가 새로운 패턴을 학습하는 동안 생산성이 일시적으로 하락할 수 있습니다.
- **가짜 모듈화(Fake Modularity) 위험**: 단일 코드베이스 내 디렉토리 분리만으로는 실제 서비스 분리의 어려움(네트워크 지연, 분산 트랜잭션, 독립 배포)이 가려집니다. 이는 향후 분리 시 예상치 못한 복잡성을 드러낼 수 있습니다.
- **ESLint 룰의 유지보수 부담**: 커스텀 룰은 코드베이스 진화에 따라 지속적으로 수정해야 합니다. 룰이 너무 엄격하면 생산성을 저해하고, 너무 느슨하면 목적을 상실할 수 있습니다.
