### 1. **논리적 완결성**

#### 장점
- 문제 → 해결책 → 성공기준의 플로우가 전반적으로 잘 연결되어 있음.
- “기존 코드는 건드리지 않고 신규 F-item에만 적용”이라는 전략이 명확히 정의됨.
- 성공 기준(KPI, MVP 기준, 실패 조건 등)도 구체적으로 제시되어 있음.
- 리스크와 완화책, 테스트, 변경관리, 후속 검증 등도 비교적 잘 체계화되어 있음.

#### 개선 포인트
- **핵심 가정(A1 등)이 실제 현실에서 성립되는지 실증적 검토가 부족**  
  → 신규 F-item이 과연 기존 모놀리식과 완전히 분리 가능한지, 실질적으로 어떤 의존성이 남는지에 대한 사전 분석/파일럿의 결과가 없다. PRD상 ‘독립 가능’이라고 가정하지만, 실제로는 예상치 못한 결합이 있을 수 있음.
- **비즈니스/기술적 목표의 우선순위가 불분명**  
  → “GTM 일정 보호”가 최우선이라면서도, Sprint 일정 30% 이상 딜레이 시에만 fallback한다는 기준이 “실제 현장 압박”을 반영하지 못할 수 있음.
- **실제 서비스 분리(물리적 MSA)와 디렉터리 분리(논리적 분리) 간 괴리**  
  → 가짜 모듈화에 대한 리스크는 언급됐으나, “실제 서비스 분리”로의 자연스러운 전환 계획 또는 현 시점에서의 한계가 더 명확히 드러날 필요 있음.

### 2. **실행 가능성**

#### 장점
- 단일 인력(Sinclair) + AI 에이전트라는 투입 리소스 명확, 일정도 단계별로 잘게 쪼개어 현실적임.
- 기존 코드 리팩토링을 범위에서 명확히 제외하여 부담을 줄임.
- ESLint 룰, 디렉터리 구조, 경계 테이블 등 비교적 구체적으로 실행 방안이 제시됨.

#### 개선 포인트
- **AI 에이전트의 실질적 생산성/신뢰성 과대평가**  
  → AI 에이전트의 병렬 개발 기여가 KPI로 등장하지만, 실제로는 사람이 직접 명확히 정의하고 관리하지 않으면, 오히려 충돌이 더 늘거나 예기치 못한 품질 저하 가능성 있음. 실제로 어느 정도까지 “자동화 Sprint”가 가능한지, PoC 혹은 제한적 범위에서의 실험이 필요함.
- **ESLint 커스텀 룰의 유지보수 및 false positive 관리**  
  → 대규모 파일 트리 구조에서 신규 코드에만 룰을 엄격히 적용하는 것이 쉬운 일은 아님 (PR 기준, 파일 path 기준, commit 시점 등에서 예외처리 복잡성 증가). 룰 유지보수에 드는 비용/노력이 과소평가된 감이 있음.
- **SKU 경계 테이블의 합의와 활용도**  
  → “W+3에 초안, W+4에 확정”이라고 하나, 실제로 조직 내 주요 이해관계자들의 의견을 얼마나 빠르고 강하게 끌어낼 수 있는지가 미지수. 이 부분 지연 시 전체 구조 설계가 흔들릴 수 있음.

- **릴리즈/배포/Hotfix 구체적 프로세스**  
  → “Hotfix시 기존 모놀리식 패턴으로 복귀”라고 하나, 이 경우의 기술적/운영적 절차가 구체적으로 정의되어 있지 않음(예: 롤백 스크립트, 데이터 마이그레이션, 장애 대응 등).
- **관계자 커뮤니케이션/합의 절차의 구체성**  
  → “주간 미팅, 이해관계자 보고”는 있으나, 실제로 SKU 경계 등 핵심 사안에서 의사결정이 지연되거나 의견 불일치가 발생할 때 escalation/결정 프로세스가 부족함.

### **1. 시장 적합성**

본 PRD의 핵심 목표인 모놀리식 아키텍처의 점진적 MSA(Microservices Architecture) 하드닝은 현재 IT 시장의 주류 트렌드와 매우 잘 맞닿아 있습니다.

*   **MSA 전환 가속화**: 클라우드 네이티브 환경이 보편화되면서, 기업들은 빠르게 변화하는 비즈니스 요구사항에 대응하고, 확장성, 탄력성, 개발 속도를 확보하기 위해 MSA로의 전환을 가속화하고 있습니다. `packages/api`의 모놀리식 구조에서 발생하는 기술 부채와 유지보수 비용 증가는 많은 기업들이 직면하는 현실적인 문제입니다.
*   **점진적 전환 전략 (Strangler Fig Pattern)**: 기존 모놀리식 코드를 건드리지 않고 신규 기능부터 MSA 친화적으로 만드는 "점진적 하드닝" 접근 방식은 'Strangler Fig Pattern'으로 불리는 검증된 마이그레이션 전략입니다. 이는 대규모 리팩토링의 위험을 최소화하면서도 장기적인 아키텍처 목표를 달성하는 효과적인 방법으로 널리 인정받고 있습니다.
*   **AI 기반 개발 (AI-Assisted Development)**: "AI 에이전트 Sprint 병렬성 확보"는 매우 선진적이고 혁신적인 접근 방식입니다. 최근 GitHub Copilot, ChatGPT, Claude 등 AI 기반 코드 생성 및 분석 도구의 발전으로 개발 생산성을 높이려는 시도가 활발합니다. AI 에이전트가 도메인 경계를 명확히 인지하여 충돌 없이 병렬 작업을 수행하도록 하는 것은 AI 기반 개발의 다음 단계이자 핵심적인 트렌드입니다. 이는 제한된 인력(Sinclair 단독)으로 개발 생산성을 극대화하려는 전략으로 해석됩니다.

**유사 서비스 및 차별점:**

*   **유사 서비스/사례**:
    *   **Netflix, Amazon**: 모놀리식에서 MSA로 성공적으로 전환한 대표적인 기업들입니다. 이들은 서비스 분리, API Gateway, 분산 데이터 관리, 옵저버빌리티(Observability) 등 MSA의 전반적인 생태계를 구축했습니다.
    *   **Google Cloud Anthos, AWS App Mesh**: 클라우드 벤더들은 MSA 환경 구축 및 운영을 위한 서비스 메시, API Gateway, 컨테이너 오케스트레이션 도구 등을 제공하여 MSA 도입을 돕습니다.
    *   **사내 모듈화 프레임워크**: 많은 기업들이 모놀리식 내부에서 도메인별 모듈화를 강제하기 위해 사내 ESLint 룰, 코드 생성기, 아키텍처 가이드라인 등을 활용합니다.
*   **본 PRD의 차별점**:
    *   **AI 에이전트 중심 개발**: 가장 큰 차별점은 AI 에이전트가 개발의 주체 중 하나로 명시되어 있으며, 이들의 병렬성 확보를 위해 MSA 원칙을 적용한다는 점입니다. 이는 단순히 기술적인 MSA 전환을 넘어, 'AI와 인간 개발자의 협업 모델'을 아키텍처 설계에 반영하는 선구적인 시도입니다.
    *   **초기 단계의 엄격한 규율**: GTM 이전 신규 기능부터 `core/{domain}/` 구조, ESLint 룰, Hono sub-app 패턴을 강제하는 것은 향후 대규모 리팩토링 비용을 선제적으로 줄이겠다는 강력한 의지를 보여줍니다. 많은 기업들이 MSA 전환 시점에 도달해서야 문제를 인지하고 큰 비용을 지불하는 것과 대비됩니다.

### **2. 사실 확인**

PRD에 포함된 가정이나 수치 중 검증이 필요한 부분은 다음과 같습니다.

*   **"W+1(지금) = 0, W+8+ = 수 Sprint 재작업"**: 원칙 적용의 한계비용을 명확히 제시한 점은 훌륭하나, '0'과 '수 Sprint'라는 수치는 다소 추상적입니다. 초기 원칙 적용에 드는 학습/구현 비용이 '0'이라는 것은 현실적으로 어렵습니다. 최소한의 초기 투자(학습, 룰 구현, 시행착오)는 발생할 것입니다. W+8+의 재작업 비용 또한 신규 기능의 복잡도와 의존성에 따라 '수 Sprint'를 넘어설 수 있습니다.
*   **"신규 F-item이 기존 모놀리식 코드의 핵심 로직과 강한 결합 없이 독립적인 도메인으로 설계될 수 있음 (A1)"**: F534~F536의 기능 분석이 충분히 이루어져 이들이 독립적인 도메인으로 분리 가능하다고 가정하고 있습니다. 만약 이 기능들이 기존 모놀리식 로직과 예상보다 강하게 결합되어야 한다면, `no-cross-domain-import` 룰이 개발 속도를 심각하게 저해하거나, 잦은 예외 처리가 필요해질 수 있습니다. 이는 R1 리스크와 직결됩니다.
*   **"AI 에이전트가 새로운 MSA 원칙과 패턴을 학습하고 적용하여 Sprint 병렬성에 기여할 수 있음 (A4)"**: AI 에이전트(Claude, autopilot)의 현재 코드 이해도, 추론 능력, 그리고 복잡한 아키텍처 원칙(도메인 경계, import 룰)을 정확히 학습하고 준수할 수 있는지가 핵심적인 가정입니다. 특히 "0건의 병렬 충돌" 목표는 매우 도전적입니다. AI 에이전트가 생성한 코드가 ESLint 룰을 준수하고, 다른 AI 에이전트나 Sinclair의 작업과 충돌 없이 merge될 수 있는지는 지속적인 검증이 필요합니다.
*   **ESLint 커스텀 룰 구현 난이도 (R8)**: `no-cross-domain-import`와 `no-direct-route-register` 같은 커스텀 룰은 AST(Abstract Syntax Tree) 분석을 기반으로 해야 하므로, 초기 구현에 예상치 못한 복잡성이나 시간이 소요될 수 있습니다. W+1 내 PoC 완성 목표는 타이트할 수 있습니다.
*   **SKU 경계 테이블 확정 가능성 (A3)**: W+3에 초안, W+4에 확정을 목표로 하는데, "핵심 이해관계자(서비스 오너/PM/아키텍트)"와의 합의가 순조롭게 이루어질지는 미지수입니다. 특히 '손해사정 반제품'의 'Ontology 미완성'과 같은 조건은 논의를 복잡하게 만들 수 있습니다.

### **4. 누락된 시장 관점**

본 PRD는 기술 부채 감소와 개발 생산성 향상에 초점을 맞추고 있으나, 장기적인 MSA 전환 관점에서 다음과 같은 시장/경쟁/트렌드 측면의 고려가 추가될 수 있습니다.

*   **데이터 전략의 부재**: `D1 분리 실행`이 제외 범위이며 `foundry-x-db` 단일 유지를 명시했습니다. 이는 점진적 하드닝의 초기 단계에서는 합리적이나, 진정한 MSA는 데이터의 독립적인 소유권(Data per Service)을 강조합니다. 향후 6개 *-X 서비스로의 완전 분리 시 데이터베이스 분리 및 분산 트랜잭션, 데이터 일관성 유지 전략(Saga 패턴, Eventual Consistency 등)에 대한 계획이 필요합니다. 이 부분은 MSA 전환의 가장 큰 난관 중 하나입니다.
*   **옵저버빌리티 (Observability)**: MSA 환경에서는 분산된 서비스 간의 트랜잭션 추적, 중앙 집중식 로깅, 분산 모니터링이 필수적입니다. `fx-gateway`가 프로덕션 트래픽을 처리하지 않는 현재는 중요도가 낮지만, 향후 완전 분리 및 프로덕션 전환 시 APM(Application Performance Monitoring), 분산 트레이싱(Distributed Tracing, Jaeger/Zipkin 등), 중앙 로깅 시스템(ELK Stack/Loki 등) 도입이 필수적입니다.
*   **DevOps 및 배포 파이프라인**: 6개 *-X 서비스로 완전 분리될 경우, 각 서비스는 독립적으로 배포되고 운영되어야 합니다. 이는 CI/CD 파이프라인, 컨테이너 오케스트레이션(Kubernetes 등), 서비스 메시(Istio, Linkerd 등) 도입 등 DevOps 전략의 전면적인 재검토를 요구합니다. 현재 Cloudflare Workers 단일 Worker 유지라는 제약 조건 하에서는 이 부분이 간과될 수 있습니다.
*   **조직 구조 (Conway's Law)**: MSA는 종종 'Conway's Law'에 따라 조직 구조와 밀접하게 연결됩니다. 6개 *-X 서비스로 분리된다면, 이 서비스들을 담당할 팀/조직 구조가 어떻게 변화할지에 대한 전략적 고민이 필요합니다. 현재 Sinclair 단독 + AI 에이전트 모델은 초기 단계의 특수성이지만, 장기적인 관점에서 고려되어야 합니다.
*   **API Gateway의 고급 기능**: `fx-gateway`는 Walking Skeleton 상태로 유지되지만, 실제 MSA 환경에서는 API Gateway가 라우팅, 로드 밸런싱, 인증/인가, 회로 차단기(Circuit Breaker), 속도 제한(Rate Limiting) 등 다양한 고급 기능을 수행하며 시스템의 안정성과 보안을 책임집니다. 이 PRD의 범위는 아니지만, GTM 이후 `fx-gateway`의 프로덕션 전환 시 이러한 기능들에 대한 검증 및 구현 계획이 필요합니다. (R7에서 부분적으로 언급됨)

### **5. 착수 판단: Conditional**

본 PRD는 현재 당면한 모놀리식의 기술 부채 문제 해결과 장기적인 MSA 전환 로드맵의 첫걸음을 매우 전략적이고 구체적으로 제시하고 있습니다. 특히 AI 에이전트와의 협업을 명시한 점은 시장 트렌드를 선도하는 혁신적인 시도로 높이 평가합니다.

하지만 다음과 같은 조건들이 충족되고 지속적으로 모니터링되어야 성공적인 착수가 가능할 것으로 판단됩니다.

1.  **AI 에이전트의 학습 및 적용 능력 검증**: AI 에이전트(Claude, autopilot)가 MSA 원칙과 ESLint 룰을 정확히 이해하고, 코드 생성 및 수정 과정에서 이를 일관되게 준수할 수 있는지에 대한 철저한 PoC 및 초기 Sprint 결과 검증이 필수적입니다. "0건의 병렬 충돌" 목표는 매우 도전적이므로, 초기에는 충돌 발생 시 신속한 해결 및 프롬프트 개선 프로세스 마련이 중요합니다.
2.  **초기 개발 속도 저해 리스크 (R1) 관리**: 새로운 패턴과 엄격한 룰 적용으로 인한 초기 개발 속도 저하를 최소화하기 위한 R1 완화 전략(warn 수준 fallback, 학습 시간 할당)이 계획대로 실행되어야 합니다. 특히 W+2 F534 적용 시점의 생산성 지표를 면밀히 모니터링해야 합니다.
3.  **SKU 경계 테이블 합의 (M4)**: W+4까지 SKU 경계 테이블 확정이 매우 중요합니다. 이는 향후 서비스 분리의 근간이 되므로, 핵심 이해관계자들과의 조기 커뮤니케이션과 합의된 최소 버전을 확정하는 노력이 필요합니다.
4.  **ESLint 커스텀 룰의 안정적 구현 (R8)**: W+1 내에 ESLint 커스텀 룰의 PoC를 성공적으로 완료하고, 신규 파일에만 정확히 적용되도록 구현하여 기존 코드에 false positive를 발생시키지 않도록 해야 합니다. 룰의 완성도가 낮으면 개발자 생산성에 심각한 영향을 미칠 수 있습니다.

이러한 조건들이 충족되고 리스크가 효과적으로 관리된다면, 본 PRD는 향후 6개 *-X 서비스로의 성공적인 전환을 위한 견고한 기반을 마련할 것입니다.

## 기술 검토 의견: fx-msa-roadmap-v2 PRD

### 1. **기술적 실현 가능성**: **대체로 가능하나 몇 가지 제약 존재**

**긍정적 요소:**
- ESLint 커스텀 룰 구현은 기술적으로 가능 (TypeScript AST 분석)
- Hono sub-app 패턴은 공식 지원 기능으로 안정적
- 단일 코드베이스 내 디렉토리 구조 변경은 비교적 저위험

**기술적 제약/병목:**
- **가짜 모듈화(Fake Modularity) 위험**: 물리적 분리 없이 디렉토리만 분리하면 실제 MSA 복잡성(네트워크 지연, 분산 트랜잭션, 독립 배포)을 검증할 수 없음
- **D1 단일 데이터베이스 제약**: 테이블 prefix만으로는 데이터 격리 불가. 향후 실제 분리 시 데이터 마이그레이션 복잡도가 높아짐
- **Cloudflare Workers 제한**: 단일 Worker 크기 제한(1MB)을 고려할 때, 모놀리스가 계속 커지면 빌드 실패 가능성

### 2. **아키텍처 적합성**: **점진적 접근은 합리적이나 구조적 한계 존재**

**적절한 점:**
- 점진적 하드닝 전략은 GTM 일정 보호 관점에서 현실적
- 도메인 경계 명확화는 향후 분리를 위한 필수 전제조건

**아키텍처적 우려사항:**
- **계층형 아키텍처의 한계**: `core/{domain}/services/` 구조는 DDD의 애그리게이트 경계와 다를 수 있음. 비즈니스 로직이 서비스 계층에 집중되면 도메인 모델이 약화될 위험
- **API Gateway 역할 미비**: `fx-gateway`가 프로덕션 트래픽을 처리하지 않아 실제 MSA 환경에서의 라우팅, 로드 밸런싱, 회로 차단기 검증 불가
- **공유 라이브러리 관리 부재**: `types.ts`만 허용하는 접근은 제한적. 공통 유틸리티, 인증, 로깅 등 횡단 관심사에 대한 전략이 명확하지 않음

**대안 고려사항:**
- **Nx 또는 Turborepo 도입**: 모노레포 도구를 활용하면 실제 서비스 경계를 더 명확히 정의 가능
- **Contract-first 접근**: OpenAPI 스펙을 먼저 정의하고 코드 생성하는 방식이 도메인 간 결합도 감소에 더 효과적

### 3. **구현 복잡도**: **과소평가된 부분 다수 존재**

**숨겨진 복잡도:**
- **ESLint 룰 유지보수 부담**: 코드베이스 진화에 따라 룰을 지속적으로 업데이트해야 함. false positive/negative 관리에 예상보다 많은 시간 소요
- **의존성 관리 복잡성**: 신규 도메인 간 불가피한 의존성 발생 시, `shared/contracts/`만으로 해결하기 어려운 경우가 많음. 이벤트 기반 아키텍처나 API 조합 패턴이 필요할 수 있음
- **테스트 전략 부재**: 도메인 격리 상태에서 통합 테스트를 어떻게 구성할지 명확하지 않음. 모의(Mock) 관리가 복잡해질 수 있음
- **빌드/배포 파이프라인 변경**: 현재는 단일 Worker 배포지만, 향후 분리 시 각 서비스별 빌드, 테스트, 배포 파이프라인이 필요

### 4. **기술 리스크**: **상위 3개 우려사항**

1. **가짜 모듈화로 인한 기술 부채 증가** (R10)
   - 디렉토리 분리만으로 만족하다가 실제 분리 시 예상치 못한 결합도 발견
   - 네트워크 호출로 전환 시 성능 저하가 critical할 수 있는 부분 식별 불가

2. **ESLint 룰의 유지보수성 악화** (R11)
   - 커스텀 룰이 코드베이스 변화에 따라 자주 깨질 수 있음

   - 프롬프트 엔지니어링에 지속적인 투자 필요

### 5. **착수 판단**: **Conditional**

**조건:**
