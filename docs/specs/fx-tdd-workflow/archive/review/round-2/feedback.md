### 1. **논리적 완결성**

**장점**
- 문제(테스트-먼저 커밋 미분리, shared 패키지 테스트 부재 등)를 비교적 명확하게 정의하고, 도입할 TDD 사이클/커밋 컨벤션/적용 등급 등 구체적 해결책을 제시하고 있음.
- 성공기준(정량/정성), 리스크, 예외 정책, 자동화 방안, 피드백 루프 등까지 논리적 흐름 자체는 일관성 있게 기술됨.

**결함/개선 포인트**
- **문제의 우선순위와 영향도**: TDD 커밋 분리 미흡이 실제로 어떤 품질/생산성 문제를 유발했는지, 조직 내에서 얼마나 시급한 pain point인지 근거가 약함. 단순히 "테스트와 구현이 섞여있다"가 왜 당장 개선해야 할 문제인지, 구체적 사례(예: QA/리뷰 실패, 버그 재발 등)에 기반한 논거 보강 필요.
- **성공기준↔실행 사이 브릿지**: 정량 목표(커밋 분리율 85% 이상 등)는 있으나, 실제로 이 수치가 왜 의미 있는지, 조직/팀의 KPI와 연결되는지 논리적 설득력이 약함. 예를 들어 "TDD 커밋 분리율이 높으면 어떤 문제가 줄어드는가"가 명확히 뒷받침되어야 함.
- **shared 패키지 전략**: shared에 대해 "smoke 수준 런타임 테스트를 도입"한다지만, 그 구체적 기준(어떤 함수/로직에, 어느 수준까지, 누가/어떻게 트리거하는가)이 애매함. 이로 인해 나중에 shared 관련 관리 혼선 가능성 높음.

### 2. **실행 가능성**

**강점**
- 적용 등급(필수/권장/선택/면제)으로 점진적 도입 구조는 현실적.
- Hotfix/Legacy 예외 정책, 자동화 도구(커밋/PR linter), 교육/온보딩 루프, Rollback 등 실행 관점의 위험도 일부 고려함.

**과소평가/위험**
- **현장 Change Fatigue**: TDD 커밋 분리, Red-먼저 원칙을 조직 전체 신규 F-item에 강제할 때, 실제 개발/리뷰 속도가 얼마나 저하될지에 대한 정량적 시뮬레이션/PoC 결과 부재. 현업에서 feature 브랜치 커밋 관리, squash merge, Red/Green 컨벤션을 잘못 이해하거나, 실무 일정이 빠듯할 때 패스하는 사례가 빈번히 발생할 수 있음(특히 단기/외주/신규 입사자).
- **자동화 도구의 성숙도**: PR Linter, /tdd check, T-NN Dashboard 등 자동화가 주요 통제 장치인데, 이들의 실제 도입/운용 난이도, 기존 CI/CD 파이프라인과의 통합 가능성, false positive/negative 대응 등이 불명확함.
- **교육/온보딩 현실성**: 신규 입사자/팀원에게 실질적 온보딩이 되려면 예제/실습/FAQ 수준이 아니라, 실전 코드리뷰/멘토링 등 구체적 실행 책임자가 지정되어야 함. 단순 문서 기반으로는 내재화가 잘 안될 수 있음.
- **shared/특수 패키지 적용관리**: shared smoke test를 "점진 도입"만 선언하고, 실제로 누가, 언제, 어떤 기준으로 추가/리뷰/검증하는지(예: 신규 유틸 PR 시 Reviewer 체크 등) 구체 제도가 부재함.

### 3. **누락된 핵심 요소**

- **실행 책임/Owner 명확화**: 각 Phase(규칙 배포, 첫 적용, 회고, 교육, 예외 승인 등)별 담당자, 책임자, Reviewer 명확화가 부족. "팀장/PL/Reviewer" 범용적 언급 대신, 실명 or 롤(예: FE팀 리드, QA팀, DevOps 담당 등) 지정 필요.
- **실제 적용 PoC/Trial Sprint**: 전사/전체 신규 F-item에 일괄 도입하기 전, 1~2개 팀/아이템에서 Pilot Sprint를 먼저 돌려보고, 문제/저항/자동화 도구/회고 결과를 반영하는 Step이 없음.
- **교육/온보딩의 실전화**: 단순 Kick-off 세션, 온보딩 문서 제공만으로는 부족. "실제 PR 리뷰/코드리뷰에서 Red/Green/Refactor 구분, 예외 승인, 자동화 도구 사용법" 등 실무 중심의 워크샵/실습 기반 교육 계획이 필요.
- **테스트 품질 검증/계량화**: "커밋 분리율" 등 형식적 지표 외에, 실제 테스트가 무의미하거나 형식적으로 작성되는(=실질적 가치 없는 assertion) 문제에 대한 방지책 부족. Reviewers가 assertion의 실질적 의미/품질을 검증하는 구체적 체크리스트/가이드 필요.
- **테스트 데이터/Mock 관리**: make*() 팩토리 패턴 언급은 있으나, 실질적으로 테스트 데이터/Mock이 복잡해질 경우(특히 E2E/동시성/대규모 시나리오) 관리 체계/자동화 방법에 대한 계획이 부족.
- **CI/CD 통합/Fail Fast**: TDD 커밋 분리 미준수 시 CI 단계에서 어떻게 Fail Fast(즉시 PR 차단 등) 또는 Soft Warning(통과는 하되 이슈화) 처리할지 세부 정책 부재. PR linter가 어떤 수준에서 차단하거나, Reviewer가 override할 수 있는지 등의 디테일이 미흡.

### 4. **리스크**

**가장 큰 실패요인**
1. **문화적 저항/현장 실행력 저하**: 실무 일정 압박, 신규 입사자/외주/Legacy 병행 상황에서 "Red-먼저→Green→커밋 분리"가 실제로 실행/정착되지 않을 위험. (→ 형식적 분리→테스트 품질 저하→팀별 편차 심화)
2. **자동화 도구 미성숙/운영 오류**: PR linter, TDD Dashboard 등 통제장치가 불안정할 경우, 오히려 개발 효율 저하/혼란 유발.
3. **shared/특수 영역 관리 혼선**: shared smoke test 도입/제외 기준 불명확→실무 혼선→PR 리뷰/커버리지 통계 왜곡.
4. **형식적 지표만 추구**: 커밋 분리율/커버리지 등 수치만 맞추고, 실제로는 의미 없는 테스트/커밋이 남발될 위험(=실질적 품질 개선 실패).
5. **교육/온보딩 부실**: 문서/FAQ만으로 실전 내재화 미흡→팀별/개발자별 이해 편차.

## PRD 검토 의견

### 1. 시장 적합성: 이 솔루션이 현재 시장/트렌드와 맞닿아 있나요?
이 PRD는 현재 IT 시장의 주요 트렌드와 매우 잘 맞닿아 있으며, 특히 Anthropic이라는 AI 기업의 특성을 잘 반영하고 있습니다.

*   **TDD (Test-Driven Development)의 지속적인 중요성**: TDD는 소프트웨어 품질, 유지보수성, 개발 효율성 향상을 위한 검증된 방법론으로, Agile, DevOps, Shift-Left Testing 트렌드와 함께 여전히 강조되고 있습니다. 코드 복잡성 증가, 배포 주기 단축 등 현대 소프트웨어 개발 환경에서 TDD의 체계적인 도입은 필수적인 경쟁력으로 작용합니다.
*   **AI/LLM 기반 개발 환경의 확산**: Claude Code를 활용하여 TDD 워크플로우를 강제하고 가이드하는 방식은 현재 가장 뜨거운 트렌드 중 하나인 LLM 기반 개발 도구(예: GitHub Copilot, Cursor, CodeWhisperer)의 활용을 극대화하는 사례입니다. 단순히 코드 생성/완성을 넘어, 특정 개발 방법론(TDD)을 *지원하고 유도하는* 방식으로 AI를 활용하는 것은 매우 진보적인 접근입니다. 이는 AI가 개발자의 생산성 도구를 넘어, 개발 프로세스 자체를 개선하는 코치 역할까지 확장될 수 있음을 보여줍니다.
*   **WSL + Modern Testing Frameworks**: WSL 환경과 Vitest, Playwright 같은 최신 테스트 프레임워크를 사용하는 것은 현대 웹/백엔드 개발 스택의 표준에 부합합니다. 이는 개발 환경의 효율성과 테스트의 신뢰성을 확보하는 데 기여합니다.

**유사 서비스/제품 및 차별점:**

*   **유사 서비스**:
    *   **일반적인 TDD 도입 가이드라인**: 많은 기업들이 TDD를 도입하기 위해 내부 가이드라인, 교육, 코드 리뷰 등을 활용합니다.
    *   **AI 기반 코드 어시스턴트**: GitHub Copilot, Cursor, Amazon CodeWhisperer 등은 코드와 테스트 코드 생성을 지원하며, 개발자가 TDD를 적용하는 데 간접적으로 도움을 줍니다.
    *   **CI/CD 파이프라인의 품질 게이트**: 커밋 메시지 컨벤션 검사, 코드 커버리지 검사 등을 통해 개발 프로세스 품질을 강제하는 도구들은 이미 널리 사용됩니다.
*   **이 PRD의 차별점**:
    1.  **AI(Claude Code)의 능동적 활용**: 단순히 AI가 코드/테스트를 생성하는 것을 넘어, "Red→Green→Commit"과 같은 특정 TDD 사이클 및 커밋 컨벤션을 *실질적으로 유도하고 검증하는* 역할로 Claude Code를 명시적으로 활용하려는 시도가 독특합니다. `/.claude/rules/tdd-workflow.md` 파일 생성, Claude Code 지시 패턴 명시 등은 AI를 개발 프로세스의 능동적인 파트너로 격상시킵니다.
    2.  **종합적인 Change Management 전략**: TDD 도입은 단순한 기술 적용을 넘어 개발 문화 변화를 수반합니다. 이 PRD는 교육, 온보딩, 예외 정책, 롤백, 성공 기준, 리스크 관리 등 Change Management에 필요한 거의 모든 요소를 상세하게 포함하고 있어, 단순히 TDD를 '하라'고 지시하는 것이 아니라 '성공적으로 도입하고 안착시키기 위한' 전략이 매우 구체적입니다. 이는 시장의 많은 TDD 도입 실패 사례를 학습한 결과로 보입니다.
    3.  **단계적/유연한 적용 범위**: "필수/권장/선택/면제" 등급을 명확히 하여 무리한 일괄 적용을 피하고, Hotfix, 레거시 코드 등 예외 상황을 구체적으로 명시한 점은 현실적인 도입 전략으로, 개발자들의 저항을 줄이고 실제 업무 효율을 높이는 데 기여할 것입니다.

### 2. 사실 확인: PRD에 포함된 가정이나 수치 중 검증이 필요한 것들은?

이 PRD는 v2 업데이트를 통해 상당한 수준의 구체성과 현실성을 확보했습니다. 대부분의 수치는 목표치이며, 검증이 필요한 '사실'이라기보다는 '가정' 또는 '목표'에 가깝습니다.

*   **가정 (검증 필요성 중간):**
    *   **Claude Code의 TDD 지원 능력**: `.claude/rules/tdd-workflow.md`를 통해 Claude Code가 TDD 사이클을 얼마나 효과적으로 이해하고, 개발자에게 올바른 프롬프트를 제공하며, 결과물의 TDD 원칙 준수 여부를 검증할 수 있을지에 대한 실제적인 역량 검증이 필요합니다. PRD는 "PR Linter", "/tdd check 자동화" 등의 자동화 지원 도구를 언급하여 이를 보완하려 하지만, Claude Code 자체의 내장된 'TDD 코칭/검증' 능력에 대한 깊이 있는 테스트가 필요합니다.
    *   **Anthropic 내부 TDD 패턴의 적합성**: "Anthropic 내부 TDD 패턴(Red→Green→Commit)"이 Foundry-X 프로젝트의 특성 및 기존 개발 문화와 얼마나 잘 조화될 수 있는지에 대한 내부적인 심층 분석이 선행되었는지 확인이 필요합니다. (PRD상에서는 이미 검증된 패턴으로 가정하고 있습니다.)
    *   **생산성 지표 목표 (±10% 이내 유지)**: TDD 도입 초기에는 생산성 저하가 흔히 발생합니다. ±10% 이내 유지는 매우 도전적인 목표이며, 이를 달성하기 위해 추가적인 지원(자동화, 교육, 멘토링)이 충분히 계획되어야 합니다.
*   **수치 (목표치, 지속적인 모니터링 필요):**
    *   **테스트 총 파일 수 (514 + 54)**: 현재 상태를 나타내는 지표로, 기준점으로서 유효합니다.
    *   **T-NN 테스트 계약 이행률 90% 이상**: 'T-NN 테스트 계약'에 대한 명확한 정의와 측정 방법이 내부적으로 공유되어야 합니다.
    *   **TDD 커밋 분리율 85% 이상**: 이 또한 측정 시스템(PR Linter, git log 분석 도구 등)의 정확성이 중요합니다.
    *   **테스트 커버리지 기존 대비 95% 이상 유지**: TDD 도입이 커버리지 감소로 이어지지 않음을 확인하는 중요한 지표입니다.
    *   **개발자 설문 4점 이상**: 정성적 지표로, 설문 문항 구성 및 익명성 보장이 중요합니다.

### 3. 사용자 관점: 목표 사용자가 실제로 이 솔루션을 받아들일 것 같나요? 도입 장벽이 있다면?

이 PRD는 개발자 관점의 도입 장벽을 매우 깊이 이해하고 있으며, 이를 완화하기 위한 전략들이 훌륭하게 포함되어 있어 **높은 확률로 사용자들이 받아들일 것**으로 예상됩니다.

**사용자 수용 가능성이 높은 이유:**

*   **점진적/유연한 적용**: '필수/권장/선택/면제' 등급 분류와 Hotfix, 레거시 등 예외 정책은 개발자들이 TDD를 강제적으로 느끼기보다, 필요한 곳에 유용하게 적용할 수 있도록 유도합니다. 이는 'TDD는 좋은 것'이라는 막연한 당위성을 넘어, '나의 업무에 어떻게 도움이 되고, 언제 예외를 적용할 수 있는가'에 대한 현실적인 답을 제공합니다.
*   **명확한 가이드라인 및 교육**: `.claude/rules/tdd-workflow.md`, `testing.md`, `sdd-triangle.md` 등 문서화와 Kick-off 세션, 온보딩 자료 제공은 개발자들이 새로운 워크플로우를 학습하고 적용하는 데 필요한 정보를 충분히 제공합니다.
*   **AI(Claude Code)의 지원**: Claude Code가 TDD 프로세스를 가이드하고 커밋 메시지를 생성하는 등의 지원을 해준다면, 개발자의 인지 부하를 줄이고 TDD 적용을 더 쉽고 편리하게 만들 수 있습니다. 특히, '레드 페이즈'에서 테스트 코드만 작성하고 실패를 확인하는 과정에서 AI의 피드백은 큰 도움이 될 것입니다.
*   **리스크 관리 및 피드백 루프**: 생산성 저하, 팀/조직 저항 등의 리스크를 명시하고, 이에 대한 대응 방안(완화 정책, 회고, 피드백 루프)을 마련한 점은 개발자들의 우려를 해소하고, 실제 문제 발생 시 빠르게 개선할 수 있는 기반을 제공합니다.
*   **자동화 지원 도구**: PR Linter, T-NN/TDD Rule Dashboard 등은 수동 검토의 부담을 줄이고, 규칙 준수 여부를 객관적으로 확인할 수 있게 하여 개발자들의 수용도를 높입니다.

**도입 장벽 (및 PRD의 대응):**

1.  **초기 생산성 저하 (가장 큰 장벽)**: TDD는 초기 학습 곡선과 추가적인 테스트 작성 시간으로 인해 생산성 저하를 유발할 수 있습니다.
    *   **PRD 대응**: "Sprint별 Story Point 소화량 변화, PR cycle time... ±10% 이내로 유지"라는 도전적인 목표를 설정하고, "팀/조직 저항, 생산성 저하"를 리스크로 명시하며, "단계적 적용 등급 운영, 예외/완화 정책 명시" 등으로 대응합니다. 이 부분이 가장 중요하며, 실제 도입 후 지속적인 모니터링과 유연한 조정이 필요합니다.
2.  **학습 곡선 및 익숙하지 않은 워크플로우**: TDD에 익숙하지 않은 개발자들에게는 새로운 사고방식과 작업 방식이 요구됩니다. 특히 '테스트 먼저'라는 역발상적 접근이 어렵게 느껴질 수 있습니다.
    *   **PRD 대응**: "교육 및 온보딩" 섹션에서 Kick-off 세션, 온보딩 자료, FAQ/피드백 루프 등을 통해 적극적인 교육과 지원을 약속합니다.
3.  **오버헤드 및 형식적인 테스트 작성 우려**: 모든 코드에 TDD를 적용하려는 강박이 생기거나, 'Red-Green-Commit' 규칙을 맞추기 위한 형식적인 테스트를 작성하게 될 수 있습니다.
    *   **PRD 대응**: "과적합 검증 패턴"을 제시하고, "테스트 품질의 형식화"를 리스크로 명시하며, "Reviewers의 assertion 의미 검토"를 대응 방안으로 제시합니다.
4.  **AI 도구의 의존성 및 신뢰성**: Claude Code의 TDD 코칭 및 검증 기능이 기대만큼 강력하지 않거나, 오작동할 경우 개발자들의 불신을 초래할 수 있습니다.
    *   **PRD 대응**: "자동화 지원 도구" 섹션에서 PR Linter 등 보조 도구를 명시하고, "Rollback/Fail-safe" 정책을 통해 극단적인 경우 TDD 적용을 일시 중단할 수 있음을 명시합니다.

### 4. 누락된 시장 관점: 시장/경쟁/트렌드 측면에서 고려되지 않은 것은?

이 PRD는 매우 상세하고 포괄적이지만, 시장/경쟁/트렌드 관점에서 몇 가지 추가적으로 고려해볼 만한 사항이 있습니다.

*   **TDD의 다양한 접근 방식 (School of TDD)**: PRD는 'Red-Green-Commit'이라는 고전적인 TDD 사이클을 강조합니다. 하지만 TDD에는 'London School (Outside-in)'과 'Chicago School (Inside-out)' 같은 다양한 접근 방식이 존재합니다. 이 PRD는 'Chicago School'에 가까운 방식인데, 이 선택의 배경이나 Foundry-X 프로젝트에 더 적합한 이유를 간략하게 언급한다면, TDD 방법론에 대한 더 깊은 이해를 보여줄 수 있습니다. (필수 사항은 아님)
*   **AI 기반 테스트 생성/최적화 트렌드**: Claude Code를 TDD 워크플로우의 '가이드' 및 '검증' 도구로 활용하는 것은 매우 혁신적입니다. 하지만 현재 시장에서는 AI가 단순히 테스트를 보조하는 것을 넘어, 테스트 케이스 자체를 *생성*하거나, 기존 테스트 스위트를 *최적화*하고, 테스트 데이터 관리까지 자동화하는 솔루션들이 등장하고 있습니다. 이 PRD에서 Claude Code가 'Red Phase'에서 초기 테스트 케이스 제안이나 'Green Phase'에서 구현 코드 제안을 넘어, 이러한 테스트 생성/최적화 영역까지 확장될 가능성을 암시한다면 더욱 미래지향적인 관점을 보여줄 수 있습니다. (예: `/tdd` 스킬이 단순 가이드를 넘어 테스트 케이스를 직접 제안하는 방향으로 발전)
*   **TDD와 보안/규정 준수의 연관성**: 특히 Anthropic과 같이 AI 모델을 다루는 기업은 보안과 규정 준수(Compliance)가 매우 중요할 수 있습니다. TDD는 코드 품질을 높이고 버그를 줄임으로써 간접적으로 보안 취약점을 감소시키고, 요구사항에 대한 정확한 구현을 보장하여 규정 준수에도 기여할 수 있습니다. 이러한 부가적인 가치를 시장 관점에서 강조한다면, TDD 도입의 중요성을 더욱 부각할 수 있습니다.
*   **경쟁사 TDD/품질 관리 사례 벤치마킹**: Anthropic과 유사한 규모나 도메인을 가진 선도 기업들이 TDD를 어떻게 도입하고 성공시켰는지, 또는 어떤 실패를 겪었는지에 대한 간략한 벤치마킹 사례를 언급한다면, 이 PRD의 전략이 시장의 'Best Practice'를 반영하고 있음을 더욱 설득력 있게 보여줄 수 있습니다. (예: Google, Microsoft 등 대형 테크 기업들의 테스트 문화)
*   **테스트 데이터 관리의 고도화**: PRD는 `make*()` 팩토리 패턴을 언급하며 테스트 데이터 관리에 대한 인식을 보여줍니다. 하지만 대규모 시스템에서 테스트 데이터는 매우 복잡한 문제가 될 수 있습니다. 시장에는 테스트 데이터 생성(TDM), 가상화(Test Data Virtualization), 마스킹(Masking) 등을 전문적으로 다루는 솔루션이나 패턴들이 존재합니다. 장기적으로 이러한 영역까지 고려할 계획이 있음을 시사한다면 더욱 완벽한 그림이 될 것입니다.

### 5. 착수 판단: Ready / Not Ready / Conditional

**착수 판단: Ready**

이 PRD는 TDD 도입이라는 중요한 문화적/기술적 변화를 성공적으로 이끌기 위한 모든 핵심 요소를 매우 상세하고 현실적으로 담고 있습니다. 특히 v2 업데이트를 통해 성공 기준, Change Management, 자동화 지원, 예외 처리, 리스크 관리 등 실제 도입 과정에서 발생할 수 있는 거의 모든 문제점과 우려 사항에 대한 구체적인 대응 방안을 제시하고 있습니다.

Claude Code를 활용한 AI 기반 TDD 가이드/검증이라는 차별화된 접근 방식과 함께, 체계적인 교육 및 온보딩, 단계적 적용 범위, 그리고 강력한 피드백 루프를 통해 개발자들의 수용도를 높이고 실제적인 성과를 창출할 가능성이 매우 높습니다.

몇 가지 누락된 시장 관점은 PRD의 완성도를 더욱 높일 수 있는 '개선 제안'에 가깝지, '도입 지연'을 야기할 만한 '결함'으로 보이지 않습니다. 지금 바로 착수하여 PoC 또는 초기 스프린트에 적용해도 무방하다고 판단합니다.

## 검토 의견

### 1. 기술적 실현 가능성
**부분적 가능성 있으나 상당한 제약 존재**

- **Vitest/Playwright 스택**: 기존 인프라와 호환되므로 기술적 실현 가능함
- **E2E 테스트의 Red Phase 한계**: PRD에서 인정한 대로 Playwright E2E에서 "구현 zero"는 사실상 불가능. Route mocking, response stubbing 등은 이미 구현의 일부임. 이는 TDD 원칙의 근본적 훼손 가능성 있음
- **Ink-testing-library 신뢰성**: 터미널 렌더링 테스트의 flaky 특성으로 인해 Red Phase의 실패 검증이 신뢰할 수 없음. "논리상 실패" 판단은 주관적 기준이 될 수 있음
- **shared 패키지 테스트 부재**: TypeScript strict mode만으로 런타임 오류 방지 불가능한 점은 정확한 지적. 그러나 "smoke test 수준"이라는 모호한 기준은 실효성 의문

### 2. 아키텍처 적합성
**규칙 분산과 책임 경계 불명확**

- **규칙 파일 분산**: `.claude/rules/tdd-workflow.md`, `testing.md`, `sdd-triangle.md`에 TDD 관련 규칙이 분산됨. 일관성 유지 어려움
- **테스트 계층 구조 명확성 부족**: Unit/Integration/E2E 테스트의 책임 경계와 TDD 적용 수준이 혼재됨
- **Git Workflow와의 통합**: Squash merge로 인해 Red→Green 커밋 이력이 master에는 보존되지 않음. 브랜치 내부만 참조 가능하다는 점은 협업 효율성 저하 가능성
- **모노레포 구조 영향**: 여러 패키지 간 의존성 있는 변경 시 TDD 사이클 적용이 복잡해짐

### 3. 구현 복잡도
**심각하게 과소평가됨**

- **동시성/상태 관리 테스트**: 비동기 작업, 동시 요청, 상태 공유 시나리오에 대한 TDD 적용 방법이 전혀 고려되지 않음
- **테스트 데이터 관리**: make*() 팩토리 패턴이 복잡한 도메인 객체 생성 시 충분하지 않을 수 있음. 테스트 간 독립성 보장 어려움
- **레거시 코드와의 통합**: 신규 F-item이 기존 시스템과 연동될 때 TDD 사이클 유지 어려움. Mocking 복잡도 급증
- **과적합 검증의 주관성**: "의미 있는 assertion" 판단 기준이 모호하여 리뷰어 간 의견 차이 발생 가능

### 4. 기술 리스크
**가장 우려되는 요소 3가지**

1. **생산성 저하 리스크**: Red→Green 커밋 분리로 인한 컨텍스트 스위칭 오버헤드가 Sprint velocity에 10% 이상 영향 줄 가능성 높음. 특히 복잡한 비즈니스 로직에서 더 심각
2. **테스트 품질 형식화**: 규칙 준수에 집중하여 무의미한 테스트(assertion만 있는) 증가 가능성. 테스트 커버리지 수치는 유지되나 실제 결함 발견율 저하
3. **팀 저항과 규칙 유지 실패**: 강제적 규칙에 대한 개발자 저항으로 "예외" 사례가 점차 확대될 가능성 높음. PR 리뷰어의 부담 증가로 인한 병목 발생

### 5. 아키텍처 개선 제안
**점진적 도입과 명확한 책임 분리**

1. **계층별 TDD 적용 수준 명확화**
   - Unit 테스트: 엄격한 Red→Green→Commit
   - Integration 테스트: Red Phase에서 최소한의 스텁 구현 허용
   - E2E 테스트: 구현과 병행 개발, 커밋 분리 선택적

2. **테스트 계약(Test Contract) 명세화**
   - Design 단계에서 테스트 시나리오, edge case, 성공/실패 조건을 명시적 계약으로 정의
   - T-NN 표기법을 실제 실행 가능한 테스트 스펙으로 발전

3. **자동화 인프라 선행 구축**
   - PR Linter, 커밋 메시지 검증 도구를 TDD 도입 전에 먼저 구현
   - 테스트 커버리지 리포트와 TDD 준수율 대시보드 병행 운영

4. **피드백 루프 단축**
   - 2주 Sprint가 아닌 1주 단위로 TDD 적용 효과 측정 및 조정
   - 개발자 설문과 실제 생산성 지표를 실시간 모니터링

---

**착수 판단: Conditional**

**조건:**
1. **Pilot 프로젝트 선정**: 1~2개 F-item에 대해 소규모 팀으로 2주간 시범 적용 후 효과 측정
2. **자동화 도구 선구축**: PR Linter, 커밋 규칙 검증, TDD 준수율 리포트 도구를 먼저 구현
3. **명확한 롤백 기준 수립**: Story Point 소화량 20% 이상 감소 또는 PR Cycle Time 30% 이상 증가 시 즉시 중단할 명확한 메트릭 정의
4. **shared 패키지 테스트 전략 재정의**: "smoke test" 대신 핵심 유틸리티에 대한 실제 단위 테스트 필수화 방안 수립

**권고사항:**
현재 PRD는 규칙과 프로세스에 집중되어 있으나, 실제 기술적 구현의 난이도를 과소평가하고 있습니다. 특히 테스트 환경의 제약(E2E, Ink)과 기존 코드베이스와의 통합 복잡도를 고려할 때, 점진적 도입과 강력한 피드백 루프가 필수적입니다. "모든 신규 F-item에 필수 적용"이라는 접근보다는 "위험도 기반 적용 등급"을 도입하는 것이 현실적입니다.
