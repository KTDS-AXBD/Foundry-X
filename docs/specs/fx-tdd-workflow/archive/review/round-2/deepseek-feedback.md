## 검토 의견

### 1. 기술적 실현 가능성
**부분적 가능성 있으나 상당한 제약 존재**

- **Vitest/Playwright 스택**: 기존 인프라와 호환되므로 기술적 실현 가능함
- **E2E 테스트의 Red Phase 한계**: PRD에서 인정한 대로 Playwright E2E에서 "구현 zero"는 사실상 불가능. Route mocking, response stubbing 등은 이미 구현의 일부임. 이는 TDD 원칙의 근본적 훼손 가능성 있음
- **Ink-testing-library 신뢰성**: 터미널 렌더링 테스트의 flaky 특성으로 인해 Red Phase의 실패 검증이 신뢰할 수 없음. "논리상 실패" 판단은 주관적 기준이 될 수 있음
- **shared 패키지 테스트 부재**: TypeScript strict mode만으로 런타임 오류 방지 불가능한 점은 정확한 지적. 그러나 "smoke test 수준"이라는 모호한 기준은 실효성 의문

### 2. 아키텍처 적합성
**규칙 분산과 책임 경계 불명확**

- **규칙 파일 분산**: `.claude/rules/tdd-workflow.md`, `testing.md`, `sdd-triangle.md`에 TDD 관련 규칙이 분산됨. 일관성 유지 어려움
- **테스트 계층 구조 명확성 부족**: Unit/Integration/E2E 테스트의 책임 경계와 TDD 적용 수준이 혼재됨
- **Git Workflow와의 통합**: Squash merge로 인해 Red→Green 커밋 이력이 master에는 보존되지 않음. 브랜치 내부만 참조 가능하다는 점은 협업 효율성 저하 가능성
- **모노레포 구조 영향**: 여러 패키지 간 의존성 있는 변경 시 TDD 사이클 적용이 복잡해짐

### 3. 구현 복잡도
**심각하게 과소평가됨**

- **동시성/상태 관리 테스트**: 비동기 작업, 동시 요청, 상태 공유 시나리오에 대한 TDD 적용 방법이 전혀 고려되지 않음
- **테스트 데이터 관리**: make*() 팩토리 패턴이 복잡한 도메인 객체 생성 시 충분하지 않을 수 있음. 테스트 간 독립성 보장 어려움
- **레거시 코드와의 통합**: 신규 F-item이 기존 시스템과 연동될 때 TDD 사이클 유지 어려움. Mocking 복잡도 급증
- **과적합 검증의 주관성**: "의미 있는 assertion" 판단 기준이 모호하여 리뷰어 간 의견 차이 발생 가능

### 4. 기술 리스크
**가장 우려되는 요소 3가지**

1. **생산성 저하 리스크**: Red→Green 커밋 분리로 인한 컨텍스트 스위칭 오버헤드가 Sprint velocity에 10% 이상 영향 줄 가능성 높음. 특히 복잡한 비즈니스 로직에서 더 심각
2. **테스트 품질 형식화**: 규칙 준수에 집중하여 무의미한 테스트(assertion만 있는) 증가 가능성. 테스트 커버리지 수치는 유지되나 실제 결함 발견율 저하
3. **팀 저항과 규칙 유지 실패**: 강제적 규칙에 대한 개발자 저항으로 "예외" 사례가 점차 확대될 가능성 높음. PR 리뷰어의 부담 증가로 인한 병목 발생

### 5. 아키텍처 개선 제안
**점진적 도입과 명확한 책임 분리**

1. **계층별 TDD 적용 수준 명확화**
   - Unit 테스트: 엄격한 Red→Green→Commit
   - Integration 테스트: Red Phase에서 최소한의 스텁 구현 허용
   - E2E 테스트: 구현과 병행 개발, 커밋 분리 선택적

2. **테스트 계약(Test Contract) 명세화**
   - Design 단계에서 테스트 시나리오, edge case, 성공/실패 조건을 명시적 계약으로 정의
   - T-NN 표기법을 실제 실행 가능한 테스트 스펙으로 발전

3. **자동화 인프라 선행 구축**
   - PR Linter, 커밋 메시지 검증 도구를 TDD 도입 전에 먼저 구현
   - 테스트 커버리지 리포트와 TDD 준수율 대시보드 병행 운영

4. **피드백 루프 단축**
   - 2주 Sprint가 아닌 1주 단위로 TDD 적용 효과 측정 및 조정
   - 개발자 설문과 실제 생산성 지표를 실시간 모니터링
