아래는 Foundry-X TDD Workflow 도입 작업 계획서(PRD)에 대한 실무 중심의 검토 의견입니다.

## 1. 논리적 완결성

### 장점
- **문제-해결-성공기준**의 논리 흐름이 비교적 명확합니다.  
  - 기존의 "테스트-먼저 커밋 분리 미흡"이라는 문제를 정의하고,
  - Anthropic 권장 TDD 사이클(Red→Green→Commit) 도입을 해법으로 제시,
  - PDCA 및 TDD 계약 이행률 등으로 성공기준을 구체화했습니다.
- **단계별 절차**와 **커밋 메시지 컨벤션** 등 실무에 필요한 세부 가이드가 구체적입니다.
- **실제 예시 및 Claude Code 세션 흐름**을 제공하여 추상적이지 않고, 실제 업무 적용을 상정하고 있습니다.

### 결함 및 개선점
- **성공기준이 정량적으로 충분히 명시되어 있지 않습니다.**  
  - 예: 'T-NN 이행률 90% 이상'이 측정 방법과 함께 반복적으로 언급되나, 전체 적용 범위별로 어떤 수준(팀/모듈/스프린트 단위 등)에서 달성해야 성공인지 명확하지 않습니다.
  - **"생산성 저하 없음" 등 부정적 영향에 대한 기준선**도 필요합니다(예: 스프린트별 Story Point 소화량 변화폭, PR cycle time 등).
- **실제 현업에서의 예외적 상황(급한 hotfix, legacy 대규모 변경 등)에 대한 예외 처리 기준**이 부족합니다.
- **shared 패키지(테스트 0개)의 개선 방안 언급 부족**: 왜 shared는 테스트가 0개인지, 향후 방침은 무엇인지 모호합니다.

## 2. 실행 가능성

### 장점
- **도입 범위가 '신규 F-item' 등으로 한정되어 있어, 초기 롤아웃 부담을 줄이고 있습니다.**
- **구체적 파일 경로, 명령어, 커밋 정책** 등 상세한 실행 방안이 있어, 실무자 입장에서 즉시 착수 가능하다는 인상을 줍니다.
- **기존 `/tdd` 스킬과의 호환성**을 언급하여, 중복 작업이나 혼란을 예방하려는 의도가 보입니다.

### 과소평가된 부분/개선점
- **교육/온보딩·저항 관리에 대한 고려 부족**:  
  - TDD 경험이 적은 팀원, 외주/신입 개발자, QA 등 모든 참여자에게 명확하게 전파할 계획이 없습니다.
  - "Claude Code 세션 흐름"은 구체적이지만, 실제로 인력 전체가 이 워크플로우를 일관되게 사용할 수 있을지 검증된 바가 없습니다.
- **도입 후 검증/모니터링 체계 미흡**:  
  - 실전 적용 이후, 실제로 커밋 분리가 얼마나 잘 이루어지고 있는지, 생산성 저하가 있는지, TDD 효과(버그 감소 등)가 있는지 정기적 리뷰/지표 수집 체계가 미흡합니다.
- **스프린트 일정상의 추가 작업량 평가 부족**:  
  - Red→Green→Refactor로 커밋을 분리하는 프로세스가 실제로 개발 속도에 어떤 영향을 미치는지 구체적 산출 근거가 부족합니다.
- **기존 코드베이스와의 경계 정의 미흡**:  
  - '신규 F-item'과 '리팩토링/버그픽스'의 경계가 모호하고, 점진적 도입 시 혼란 가능성이 있습니다.
- **자동화(예: PR bot, pre-commit hook 등) 적용 방안 부족**:  
  - 사람이 실수로 커밋 순서를 어기는 경우를 자동 감지/경고할 방안이 없습니다.

## 3. 누락된 핵심 요소

- **Change Management/교육/문서화 방안**  
  - 실무자, 신입, 외주 인력 모두가 TDD 워크플로우와 커밋 컨벤션을 일관성 있게 따르도록 하는 교육/가이드/FAQ/워크숍 계획이 필요합니다.
- **자동화 지원 도구/Fail-safe**  
  - 커밋 메시지 검사(pre-commit hook), PR 체크리스트, Red/Green 커밋 체크 bot 등 자동화된 Fail-safe 장치가 필요합니다.
- **성과 측정/피드백 루프**  
  - 도입 효과(테스트 커버리지 변화, PR cycle time, 버그 건수 변화 등)를 측정할 정량적 지표와, 이를 바탕으로 정책을 수정할 수 있는 피드백 루프가 필요합니다.
- **Roll-back/예외 상황 대응**  
  - 도입 실패 시 복귀 플랜, 급한 패치나 장애 대응 시 TDD 스킵 등 예외 상황을 어떻게 처리할지 명확한 방침이 없습니다.
- **Stakeholder Alignment**  
  - PO/QA/디자인 등 타 직군과의 연계(예: API 명세 변경, QA 커버리지 연동) 방안이 없습니다.

## 4. 리스크

- **팀/조직 단위 저항 및 일관성 붕괴**  
  - 일부 팀원만 TDD를 엄격히 적용하고, 다른 팀원은 예전 방식대로 진행 시 정책 무력화 리스크가 큽니다.
- **생산성 저하 및 일정 지연**  
  - TDD 미숙련 팀원, 급한 일정, 복잡한 비즈니스 로직 등에서 실제로 Red-Green-Commit을 엄격히 분리하면 개발 사이클이 느려질 수 있습니다.
- **커밋/PR 규칙 미준수**  
  - 사람이 직접 커밋을 승인하는 과정이 많아, 누락/오타/순서 오류 등 실수 가능성이 상존합니다.
- **테스트 품질의 형식화(무의미한 assertion 남발, 과적합 검증 형식화)**  
  - Red phase에서 의미 없는 테스트(실질적 계약이 부족한 assertion)만 늘어나면, TDD 자체가 형식적 루틴이 될 수 있습니다.
- **기존 코드와 신규 코드 사이의 관리 혼란**  
  - 점진적 도입 시, 어떤 부분에서 TDD를 엄격히 적용/면제할지 경계가 모호해져 혼란이 발생할 수 있습니다.

Foundry-X TDD Workflow 도입 작업 계획서 PRD를 IT 시장 트렌드와 유사 서비스 분석 관점에서 평가합니다.

### 1. 시장 적합성: 이 솔루션이 현재 시장/트렌드와 맞닿아 있나요?
   - 유사한 서비스나 제품이 있다면 어떤 차별점이 필요한가요?

**시장 적합성**: **매우 높음.**
이 PRD는 현재 IT 시장의 핵심 트렌드와 깊이 맞닿아 있으며, 미래 지향적인 접근 방식을 보여줍니다.

*   **AI 기반 개발 (AI-Assisted Development)의 확산**: GitHub Copilot, Google Gemini (Code), OpenAI ChatGPT 등 AI 코딩 어시스턴트의 도입은 이제 선택이 아닌 필수가 되어가고 있습니다. 이 PRD는 Claude Code를 단순한 코드 생성 도구가 아닌, TDD라는 특정 개발 방법론을 체계적으로 지원하고 가이드하는 '지능형 조력자'로 활용하려는 전략을 보여줍니다. AI를 개발 프로세스에 깊이 통합하고 제어하려는 시도는 현재 가장 선도적인 트렌드 중 하나입니다.
*   **Shift-Left Testing 및 품질 확보 강화**: 개발 초기 단계에서 테스트를 강화하여 버그를 조기에 발견하고 수정하는 'Shift-Left' 패러다임은 소프트웨어 품질과 개발 효율성을 높이는 데 필수적입니다. TDD는 이 패러다임의 핵심 방법론이며, PRD는 TDD를 체계화하고 강제함으로써 품질 확보에 기여합니다.
*   **개발 프로세스 자동화 및 최적화**: CI/CD 파이프라인 구축을 넘어, 개발자의 일상적인 코딩 경험을 개선하고 워크플로우를 자동화하려는 노력은 계속되고 있습니다. TDD 사이클을 AI와 Git Workflow, PDCA 프로세스에 통합하는 것은 이러한 최적화 노력의 정점입니다.
*   **개발자 경험(DX) 향상**: AI 어시스턴트가 TDD 사이클을 가이드하고, 테스트 작성 부담을 줄여주며, 일관된 워크플로우를 제공하는 것은 개발자 경험을 긍정적으로 변화시킬 수 있습니다.

**유사 서비스/제품 및 차별점**:

*   **유사 서비스/제품**:
    *   **AI 코딩 어시스턴트**: GitHub Copilot, Google Gemini (Code), OpenAI ChatGPT, Anthropic Claude Code 등. 이들은 모두 코드 생성, 테스트 스텁 생성, 리팩토링 제안 등을 수행할 수 있습니다.
    *   **TDD 프레임워크/도구**: Vitest, Playwright, Jest 등 다양한 테스트 프레임워크와 IDE 통합 기능.
    *   **개발 워크플로우 관리**: Jira, GitHub, GitLab 등의 SCM 및 프로젝트 관리 도구.

*   **차별점**:
    1.  **AI 코딩 어시스턴트와의 심층 통합 및 제어**: 단순히 AI에게 코드를 생성하라고 지시하는 것을 넘어, TDD의 Red→Green→Commit이라는 특정 "절차"와 "규칙"을 AI가 이해하고 따르도록 명확한 프롬프트와 가이드라인을 제시합니다. 특히 기존 `/tdd` 스킬을 활용하여 AI의 행동을 세밀하게 제어하려는 시도는 일반적인 AI 어시스턴트 활용을 넘어선 차별점입니다.
    2.  **PDCA 개발 프로세스 전반으로의 확장**: TDD를 단순히 코딩 방법론으로 국한하지 않고, Plan, Design, Analyze, Iterator, Report 등 PDCA의 모든 단계에 걸쳐 어떻게 통합되고 영향을 미치는지 상세하게 설명합니다. '테스트 계약'을 Design 단계에 명시하고, 'T-NN'이라는 새로운 지표를 Analyze 단계에 도입하는 것은 매우 체계적이며, 일반적인 TDD 도입 계획에서는 보기 드문 수준의 깊이입니다.
    3.  **내부 개발 문화 및 특정 도구 스택에 최적화**: Foundry-X라는 특정 조직의 WSL + Claude Code 환경, Vitest/Playwright 프레임워크, 기존 `make*()` 팩토리 패턴, `.claude/rules`, `.claude/skills` 등 내부 시스템에 맞춰 커스터마이징된 워크플로우라는 점이 강력한 차별점입니다. 이는 다른 범용 솔루션이 제공하기 어려운 조직 특화된 가치입니다.

**결론적으로, 이 PRD는 TDD라는 검증된 방법론을 최신 AI 기술 및 기존 개발 프로세스에 깊이 통합하여, 개발 효율성과 소프트웨어 품질을 동시에 높이려는 매우 전략적인 접근을 보여줍니다. 이는 현재 시장에서 AI 기반 개발의 다음 단계로 나아가려는 시도로 해석될 수 있습니다.**

### 2. 사실 확인: PRD에 포함된 가정이나 수치 중 검증이 필요한 것들은?

PRD는 전반적으로 구체적이고 사실 기반의 현황과 계획을 제시하고 있어 가정이 적지만, 몇 가지 검증 또는 추가적인 고려가 필요한 부분이 있습니다.

*   **"Anthropic 내부 TDD 패턴(Red→Green→Commit)을 Foundry-X에 체계적으로 도입"**: Anthropic이 Claude Code 개발에 이 패턴을 사용하고 권장한다는 것이 사실인지, 그리고 그 패턴이 Foundry-X의 환경(WSL, Vitest 등)에 그대로 적용 가능하며 효과적일지에 대한 검증이 필요합니다. Anthropic의 내부 TDD 패턴 관련 공개 자료를 참고하여 Foundry-X 환경에 맞게 최적화하는 과정이 필요할 수 있습니다.
*   **Claude Code의 TDD 지시 준수 일관성**: "Claude Code에게 지시할 때: 'TDD Red phase — 테스트만 작성하고 구현은 하지 마'"와 같은 구체적인 프롬프트가 Claude Code에서 항상 의도대로 작동하는지, 또는 다른 LLM과 비교했을 때 Claude Code가 TDD 패턴을 따르는 데 특별히 더 강점을 보이는지(예: 제약 조건을 더 잘 준수하는지)에 대한 사전 테스트와 검증이 필요합니다. LLM의 특성상 일관된 결과가 나오지 않을 수도 있습니다.
*   **AI 기반 과적합 검증의 신뢰성**: "과적합 검증 (선택): Green 통과 후 서브에이전트로 검증: '이 테스트가 구현에 과적합되어 있지 않은지...'"와 같은 요청에 대해 서브에이전트(아마도 Claude Code의 다른 인스턴스 또는 모드)가 얼마나 정확하게 '과적합' 여부를 판단하고 유용한 피드백을 줄 수 있는지에 대한 검증이 필요합니다. 이는 LLM의 추론 능력과 관련이 깊으며, 아직 완벽하게 신뢰하기 어려운 영역일 수 있습니다.
*   **T-NN 지표의 구체적인 측정 기준**: "테스트 계약 이행률: 9/10 (90%)"는 예시이지만, 실제 도입 후 'T-NN' 지표를 측정할 때, "PASS" 또는 "계약 이행"의 기준이 명확해야 합니다. 예를 들어, 단순히 Red 커밋이 존재하고 Green 커밋이 통과하면 '계약 이행'으로 볼 것인지, 아니면 테스트 커버리지나 특정 품질 기준까지 충족해야 하는지 등.
*   **생산성 저하 판단 기준**: "생산성에 부정적 영향이 있는 영역 → 등급 조정 (필수→권장 등)"이라는 계획은 좋지만, 실제 저하가 발생했을 때 이를 어떻게 측정하고, 어떤 기준으로 '부정적 영향'을 판단하여 '등급 조정'을 할 것인지에 대한 구체적인 기준이 필요합니다(예: 특정 기능 개발에 평균 시간 20% 증가 시 등급 조정 고려).

전반적으로 계획이 상세하여 검증할 부분이 많지는 않지만, AI 어시스턴트의 실제 성능과 일관성에 대한 검증은 중요합니다.

### 3. 사용자 관점: 목표 사용자가 실제로 이 솔루션을 받아들일 것 같나요?
   - 도입 장벽이 있다면?

**사용자 수용 가능성**: **높음 (단, 초기 교육 및 성공 사례 공유가 중요)**

*   **긍정적 요인**:
    *   **AI 어시스턴트 활용**: 개발자들은 이미 AI 코딩 어시스턴트에 대한 높은 관심을 가지고 있으며, 이를 활용하여 작업 효율을 높이고 싶어 합니다. TDD라는 다소 번거로울 수 있는 방법론을 AI가 도와준다는 점은 매력적으로 다가올 것입니다.
    *   **명확한 가이드라인**: PRD에 제시된 상세한 워크플로우, 커밋 메시지 컨벤션, PDCA 연동 등은 개발자들이 TDD를 어떻게 적용해야 할지 명확하게 이해하는 데 도움을 줍니다. 불확실성이 줄어들면 도입 저항도 줄어듭니다.
    *   **품질 향상 기대**: 개발자들은 자신이 작성하는 코드의 품질이 높아지고, 버그가 줄어드는 것을 원합니다. TDD가 이러한 목표 달성에 기여한다는 인식이 있다면 수용도는 높아질 것입니다.
    *   **기존 `/tdd` 스킬의 존재**: 이미 TDD 스킬이 존재한다는 것은 기존에 TDD에 대한 관심이나 시도가 있었음을 의미하며, 이를 공식화하는 과정이므로 완전히 새로운 것을 도입하는 것보다 저항이 적을 수 있습니다.

*   **도입 장벽**:
    *   **초기 학습 곡선 및 생산성 저하 우려**: TDD는 익숙하지 않은 개발자에게는 초기 학습 곡선이 존재하며, "테스트를 먼저 작성해야 한다"는 생각 자체가 심리적 장벽이 될 수 있습니다. 특히 "구현 코드 zero" 상태에서 테스트를 작성하는 Red Phase는 익숙해지기까지 시간이 걸릴 수 있으며, 이로 인한 일시적인 생산성 저하에 대한 우려가 있을 수 있습니다.
    *   **AI 어시스턴트의 완벽성 부족**: Claude Code가 항상 완벽하게 TDD 원칙을 따르는 코드를 생성하지 못할 경우, 개발자가 AI가 생성한 코드를 수정하거나 다시 지시하는 데 시간을 소모하게 되어 피로도가 높아질 수 있습니다. AI에 대한 과도한 의존이나 불신이 생길 수 있습니다.
    *   **기존 습관과의 충돌**: "구현과 테스트가 같은 커밋에 섞이는 경우가 다수"라는 현황에서 알 수 있듯이, 기존 개발 습관을 바꾸는 것은 쉽지 않습니다. 특히 마감 압박이 있을 때 Red-Green-Commit 사이클을 철저히 지키는 것이 어려울 수 있습니다.
    *   **오버헤드 인식**: TDD, PDCA 연동, 과적합 검증 등 상세한 절차가 일부 개발자에게는 불필요한 오버헤드로 인식될 수 있습니다. 특히 "선택" 또는 "면제" 영역의 개발자들은 "왜 나까지?"라는 생각을 할 수 있습니다.

**장벽 완화 방안**:
*   **충분한 교육 및 온보딩**: TDD의 원칙, Claude Code와의 연동 방법, PDCA 단계별 역할 등에 대한 상세한 교육과 워크숍이 필수적입니다.
*   **성공 사례 공유**: 초기 도입 팀이나 F-item에서 TDD를 통해 얻은 구체적인 이점(버그 감소, 리팩토링 용이성, 코드 품질 향상 등)을 전사에 공유하여 긍정적인 인식을 확산시켜야 합니다.
*   **점진적 도입 및 유연성**: "필수", "권장", "선택", "면제" 등급을 명확히 한 것은 좋은 전략입니다. 초기에는 '필수' 영역에 집중하고, 점차 확대해나가는 유연한 접근이 필요합니다.
*   **AI 피드백 루프 구축**: Claude Code가 TDD 패턴을 더 잘 따르도록 사용자 피드백을 수집하고, 프롬프트 엔지니어링을 지속적으로 개선하는 노력이 필요합니다.

### 4. 누락된 시장 관점: 시장/경쟁/트렌드 측면에서 고려되지 않은 것은?

PRD는 매우 상세하고 깊이 있는 분석을 담고 있지만, 몇 가지 시장/경쟁/트렌드 관점에서 추가적으로 고려해볼 사항이 있습니다.

*   **AI 코딩 어시스턴트의 발전 속도 및 변화에 대한 유연성**: Claude Code는 현재 특정 시점의 AI 모델입니다. 하지만 AI 기술은 매우 빠르게 발전하고 있으며, 몇 달 안에 더 강력하고 새로운 기능(예: 자체적으로 TDD 사이클을 더 잘 이해하고 실행하거나, 더 복잡한 테스트 시나리오를 제안하는 등)을 가진 AI 모델이 등장할 수 있습니다. 이 PRD는 현재의 Claude Code에 최적화되어 있지만, 미래의 AI 변화에 대한 유연성이나 적응 전략에 대한 언급은 부족합니다(예: 다른 LLM으로의 전환 가능성, 새로운 AI 기능 통합 계획).
*   **데이터 보안 및 프라이버시**: AI 코딩 어시스턴트를 내부 코드베이스에 사용하는 경우, 코드 데이터가 외부로 유출될 가능성이나 AI 모델 학습에 사용될 가능성에 대한 고려가 필요합니다. 특히 Anthropic의 Claude Code를 사용하는 만큼, 데이터 처리 정책에 대한 명확한 이해와 내부 보안 가이드라인 준수 여부가 중요합니다. (PRD에는 언급이 없지만, 내부적으로는 이미 고려되었을 수 있습니다.)
*   **비용 효율성 (AI 토큰 사용)**: Claude Code와 같은 AI 서비스는 API 호출 시 토큰 사용량에 따라 비용이 발생합니다. TDD 사이클을 AI와 함께 반복적으로 실행하고, 과적합 검증 등 추가적인 AI 호출이 많아질 경우, 예상되는 AI 사용 비용에 대한 분석이나 최적화 방안(예: 불필요한 호출 최소화, 캐싱 전략 등)에 대한 고려가 있다면 좋습니다.
*   **AI에 대한 윤리적/사회적 책임**: AI가 생성하는 코드의 품질, 편향성, 보안 취약점 등에 대한 책임은 최종적으로 개발자에게 있습니다. AI가 TDD를 돕는 과정에서 발생할 수 있는 잠재적 문제점(예: AI가 생성한 테스트가 특정 케이스를 놓치거나, 오히려 과도하게 복잡한 테스트를 생성하는 경우)에 대한 대응 전략이나 개발자의 최종 검토의 중요성 강조가 필요합니다.
*   **오픈소스 및 커뮤니티 트렌드와의 연계**: TDD, Vitest, Playwright 등은 모두 활발한 오픈소스 커뮤니티를 가지고 있습니다. 이 커뮤니티에서 TDD와 AI를 결합하는 새로운 도구나 방법론이 등장할 수 있습니다. 이러한 외부 트렌드를 모니터링하고, 필요시 내부 워크플로우에 통합할 계획에 대한 고려가 있다면 더 좋습니다.

이러한 요소들은 PRD의 직접적인 범위는 아닐 수 있지만, 장기적인 관점에서 AI 기반 개발 전략을 수립하는 데 중요한 고려사항이 될 수 있습니다.

## 기술 검토 의견

### 1. 기술적 실현 가능성
**부분적으로 가능하나 심각한 운영 리스크 존재**

- **기술 스택 측면**: Vitest + Playwright 조합은 기존 환경과 호환되며 TDD 사이클 자체는 구현 가능
- **주요 제약사항**:
  1. **shared 패키지 테스트 부재(0개)**가 치명적 리스크. 타입 의존성만으로는 런타임 오류를 방지할 수 없음. TDD 도입 전 shared에 대한 기본 테스트 프레임워크 구축이 선행되어야 함
  2. **E2E 테스트의 Red Phase 실현 불가능**: Playwright spec 작성 시 이미 mock/route 설정이 구현 로직을 포함. "구현 zero" 원칙 위반
  3. **컴포넌트 테스트 환경 불안정**: Ink-testing-library + Vitest 조합에서 실제 터미널 렌더링과의 차이로 인해 Red Phase 실패 검증이 신뢰할 수 없음

### 2. 아키텍처 적합성
**과도한 규칙 중첩으로 인한 아키텍처 부채 발생 우려**

- **규칙 파일 분산 문제**: 
  - `tdd-workflow.md` (신규)
  - `testing.md` (수정) 
  - `sdd-triangle.md` (수정)
  - `git-workflow.md` (참조)
  - `/tdd` 스킬 (기존)
  
  **5개 파일에 TDD 규칙이 분산되어 일관성 유지가 어려움**. 단일 진실 공급원(SSOT) 원칙 위반.

- **PDCA 연동의 구조적 문제**:
  - Design 단계에 "테스트 계약"을 추가하는 것은 **책임 분리 원칙 위반**. Design은 "무엇을"에 집중해야 하며, 테스트 시나리오는 별도의 Test Design 단계로 분리되어야 함
  - Analyze 단계의 T-NN 메트릭은 **테스트 품질을 양적으로만 측정**하여 false sense of security 유발 가능

### 3. 구현 복잡도
**심각하게 과소평가된 복잡도**

- **Git Workflow와의 충돌**:
  - Red/Green 커밋 분리는 **CI/CD 파이프라인과 충돌** 가능성 높음
  - Red 커밋(테스트만)은 빌드 실패를 유발하여 PR 자동 검사 실패
  - "squash merge로 해결"은 팀 협업에서 혼란 가중

- **동시성 및 상태 관리 테스트 누락**:
  - 제시된 API 예제(`F510`)는 **낙관적 동시성 제어(optimistic concurrency) 테스트가 완전히 누락됨**
  - 동일 초대에 대한 병렬 요청 시 데이터 정합성 테스트 필요
  - 분산 트랜잭션 시나리오(초대 생성 + 알림 발송)에 대한 통합 테스트 고려 안 됨

- **테스트 데이터 관리 복잡도**:
  - `make*()` 팩토리 패턴이 Red Phase에서 **의존성 주입 없이 동작할 수 있는지 불확실**
  - 테스트 간 상태 오염(state leakage) 방지 메커니즘이 계획에 포함되지 않음

### 4. 기술 리스크
**상위 3개 우선순위 리스크:**

1. **Red Phase의 실패 검증 신뢰성 문제**
   - "테스트만 작성하고 구현 zero" 원칙이 실제로 검증 가능한지 불확실
   - TypeScript 컴파일 단계에서 이미 타입 오류 발생 가능
   - Stub 구현(`export {}`)으로는 모듈 의존성 해결 불가

2. **팀 생산성 저하 및 저항 가능성**
   - 기존 1커밋 워크플로우에서 2-3커밋으로 증가하는 오버헤드
   - Red 커밋 검토에 대한 명확한 기준 부재 ("테스트 계약" 품질 검증 책임 불분명)
   - Claude Code에 대한 과도한 의존성으로 인한 벤더 락인(vendor lock-in) 리스크

3. **테스트 피라미드 불균형 가속화**
   - E2E 테스트에 TDD 강제 시 **테스트 실행 시간 기하급수적 증가**
   - Red Phase에서의 Playwright mock 설정이 실제 통합 테스트 가치를 훼손
   - 단위 테스트보다 통합/E2E 테스트 비중이 불필요하게 증가할 위험

### 5. 착수 판단
**Conditional - 다음과 같은 선행 조건 필요**

1. **shared 패키지 테스트 프레임워크 구축 완료**
   - 최소 70% 이상의 단위 테스트 커버리지 확보
   - 타입 안전성과 런타임 안전성 모두 검증 가능한 구조 마련

2. **단계적 롤아웃 전략 수립**
   - "필수 적용" 범위를 API 서비스 로직으로 제한 (E2E 제외)
   - Pilot 프로젝트(1-2개 F-item)로 실증 후 확장
   - Red 커밋의 CI 실패를 허용하는 브랜치 전략 수립

3. **통합 테스트 아키텍처 개선**
   - 테스트 데이터 관리 전략 명확화 (트랜잭션 격리 보장)
   - 동시성 테스트 시나리오를 TDD Red Phase에 포함하는 방법 정의
   - 컴포넌트 테스트 환경 안정성 검증

4. **규칙 통합 및 단순화**
   - TDD 관련 규칙을 최대 2개 파일로 통합
   - PDCA 단계와의 명확한 책임 분리 (Test Design 단계 명시적 추가 고려)

**권고사항**: 현재 상태에서 전체 적용 시 실패 확률이 높습니다. 먼저 shared 패키지 테스트 기반을 구축하고, API 서비스 레이어에만 제한적으로 Pilot 실행하여 실제 효과와 비용을 측정하는 것이 안전합니다.
