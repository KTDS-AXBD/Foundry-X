아래는 Foundry-X TDD Workflow 도입 작업 계획서(PRD)에 대한 실무 중심의 검토 의견입니다.

---

## 1. 논리적 완결성

### 장점
- **문제-해결-성공기준**의 논리 흐름이 비교적 명확합니다.  
  - 기존의 "테스트-먼저 커밋 분리 미흡"이라는 문제를 정의하고,
  - Anthropic 권장 TDD 사이클(Red→Green→Commit) 도입을 해법으로 제시,
  - PDCA 및 TDD 계약 이행률 등으로 성공기준을 구체화했습니다.
- **단계별 절차**와 **커밋 메시지 컨벤션** 등 실무에 필요한 세부 가이드가 구체적입니다.
- **실제 예시 및 Claude Code 세션 흐름**을 제공하여 추상적이지 않고, 실제 업무 적용을 상정하고 있습니다.

### 결함 및 개선점
- **성공기준이 정량적으로 충분히 명시되어 있지 않습니다.**  
  - 예: 'T-NN 이행률 90% 이상'이 측정 방법과 함께 반복적으로 언급되나, 전체 적용 범위별로 어떤 수준(팀/모듈/스프린트 단위 등)에서 달성해야 성공인지 명확하지 않습니다.
  - **"생산성 저하 없음" 등 부정적 영향에 대한 기준선**도 필요합니다(예: 스프린트별 Story Point 소화량 변화폭, PR cycle time 등).
- **실제 현업에서의 예외적 상황(급한 hotfix, legacy 대규모 변경 등)에 대한 예외 처리 기준**이 부족합니다.
- **shared 패키지(테스트 0개)의 개선 방안 언급 부족**: 왜 shared는 테스트가 0개인지, 향후 방침은 무엇인지 모호합니다.

## 2. 실행 가능성

### 장점
- **도입 범위가 '신규 F-item' 등으로 한정되어 있어, 초기 롤아웃 부담을 줄이고 있습니다.**
- **구체적 파일 경로, 명령어, 커밋 정책** 등 상세한 실행 방안이 있어, 실무자 입장에서 즉시 착수 가능하다는 인상을 줍니다.
- **기존 `/tdd` 스킬과의 호환성**을 언급하여, 중복 작업이나 혼란을 예방하려는 의도가 보입니다.

### 과소평가된 부분/개선점
- **교육/온보딩·저항 관리에 대한 고려 부족**:  
  - TDD 경험이 적은 팀원, 외주/신입 개발자, QA 등 모든 참여자에게 명확하게 전파할 계획이 없습니다.
  - "Claude Code 세션 흐름"은 구체적이지만, 실제로 인력 전체가 이 워크플로우를 일관되게 사용할 수 있을지 검증된 바가 없습니다.
- **도입 후 검증/모니터링 체계 미흡**:  
  - 실전 적용 이후, 실제로 커밋 분리가 얼마나 잘 이루어지고 있는지, 생산성 저하가 있는지, TDD 효과(버그 감소 등)가 있는지 정기적 리뷰/지표 수집 체계가 미흡합니다.
- **스프린트 일정상의 추가 작업량 평가 부족**:  
  - Red→Green→Refactor로 커밋을 분리하는 프로세스가 실제로 개발 속도에 어떤 영향을 미치는지 구체적 산출 근거가 부족합니다.
- **기존 코드베이스와의 경계 정의 미흡**:  
  - '신규 F-item'과 '리팩토링/버그픽스'의 경계가 모호하고, 점진적 도입 시 혼란 가능성이 있습니다.
- **자동화(예: PR bot, pre-commit hook 등) 적용 방안 부족**:  
  - 사람이 실수로 커밋 순서를 어기는 경우를 자동 감지/경고할 방안이 없습니다.

## 3. 누락된 핵심 요소

- **Change Management/교육/문서화 방안**  
  - 실무자, 신입, 외주 인력 모두가 TDD 워크플로우와 커밋 컨벤션을 일관성 있게 따르도록 하는 교육/가이드/FAQ/워크숍 계획이 필요합니다.
- **자동화 지원 도구/Fail-safe**  
  - 커밋 메시지 검사(pre-commit hook), PR 체크리스트, Red/Green 커밋 체크 bot 등 자동화된 Fail-safe 장치가 필요합니다.
- **성과 측정/피드백 루프**  
  - 도입 효과(테스트 커버리지 변화, PR cycle time, 버그 건수 변화 등)를 측정할 정량적 지표와, 이를 바탕으로 정책을 수정할 수 있는 피드백 루프가 필요합니다.
- **Roll-back/예외 상황 대응**  
  - 도입 실패 시 복귀 플랜, 급한 패치나 장애 대응 시 TDD 스킵 등 예외 상황을 어떻게 처리할지 명확한 방침이 없습니다.
- **Stakeholder Alignment**  
  - PO/QA/디자인 등 타 직군과의 연계(예: API 명세 변경, QA 커버리지 연동) 방안이 없습니다.

## 4. 리스크

- **팀/조직 단위 저항 및 일관성 붕괴**  
  - 일부 팀원만 TDD를 엄격히 적용하고, 다른 팀원은 예전 방식대로 진행 시 정책 무력화 리스크가 큽니다.
- **생산성 저하 및 일정 지연**  
  - TDD 미숙련 팀원, 급한 일정, 복잡한 비즈니스 로직 등에서 실제로 Red-Green-Commit을 엄격히 분리하면 개발 사이클이 느려질 수 있습니다.
- **커밋/PR 규칙 미준수**  
  - 사람이 직접 커밋을 승인하는 과정이 많아, 누락/오타/순서 오류 등 실수 가능성이 상존합니다.
- **테스트 품질의 형식화(무의미한 assertion 남발, 과적합 검증 형식화)**  
  - Red phase에서 의미 없는 테스트(실질적 계약이 부족한 assertion)만 늘어나면, TDD 자체가 형식적 루틴이 될 수 있습니다.
- **기존 코드와 신규 코드 사이의 관리 혼란**  
  - 점진적 도입 시, 어떤 부분에서 TDD를 엄격히 적용/면제할지 경계가 모호해져 혼란이 발생할 수 있습니다.

## 5. 착수 판단

### **착수 판단: Conditional**

#### 조건
1. **Change Management 및 교육/온보딩 플랜 수립**
   - 모든 실무자(신입, 외주, PO, QA 포함)가 반드시 알아야 할 최소 TDD 정책, 커밋 규칙, 예외 상황 대응법, FAQ, 워크플로우 예시를 문서화/교육할 것
2. **실행 후 검증 체계 추가**
   - 도입 효과(테스트 커버리지, PR cycle time, Red/Green 커밋 분리율 등)를 Sprint 단위로 측정하는 KPI 및 리뷰/회고 프로세스 포함
3. **자동화 지원 도구 설계**
   - pre-commit hook, PR bot, Red-Green 커밋 체크 등 자동화된 Fail-safe 장치 설계/우선순위화
4. **예외/롤백/Hotfix 정책 명문화**
   - 급한 장애대응, legacy 변경 등 예외 상황에서 TDD 적용 면제 기준과, 정책 실패/철회 시 복귀 방안 명시
5. **적용 범위/우선순위 명확화**
   - 신규 F-item, 리팩토링, shared 패키지 등 각 영역별 TDD 적용/면제 경계 명확화
