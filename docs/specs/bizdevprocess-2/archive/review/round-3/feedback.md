### 1. 논리적 완결성

**장점 및 개선 포인트**
- 문제 → 해결책 → 성공 기준의 흐름이 전반적으로 잘 구성되어 있습니다.  
- 문제(분석 방법론 부재, 품질 편차, 누락 등) → 해결책(표준화된 프로세스, 자동화, 가이드, 검증 등) → 성공 기준(KPI, PoC, 사용자 평가 등)으로 이어지는 구조는 설득력이 있습니다.
- 하지만 아래와 같은 결함이 있습니다.

**결함**
- 문제 정의와 성공 기준(특히 KPI) 사이의 연결이 약함  
  - ‘분석 소요 시간’의 목표값이 미확정 상태로 남아 있습니다. As-Is 측정 필요성을 명시했지만, 이 수치가 확정되지 않으면 개선 효과를 측정하기 어렵습니다.
  - "산출물 품질 등급(팀장 평가)"의 평가 기준이 주관적일 수 있습니다. 체크리스트는 있지만, 등급 산정 방식(예: 9개 항목별 가중치/점수 합산)이 좀 더 명확해야 합니다.
- 자동화의 한계 및 수동 보완 범위가 다소 애매하게 서술됨  
  - "자동화 한계 구간 명확화"라고만 하고 실제로 어느 단계/케이스에서 수동 전환되는지, 예시가 부족합니다.  
- "실제 의사결정 활용성" (Go/Hold/Drop 등)이 PRD 품질의 최종 판별 기준인데, 이를 수치화하거나, 현업에서 어떻게 반영/기록할지 구체성이 부족합니다.

**개선 제안**
- 현행 KPI 목표값을 최소한의 추정치라도 제시(예: 평균 2주→1주 단축 목표 등)
- 산출물 품질 평가표(9개 항목별 구체 점수, 가중치, 등급 산정 방식) 부록으로 명시
- 자동화 한계(플러그인 기술적 한계, 수동 입력 필요 구간 등)를 구체적으로 표/흐름도로 시각화
- 실무에서 "Go/Hold/Drop" 의사결정에 실제로 얼마나 활용되는지 기록·피드백 루프를 명확히

### 2. 실행 가능성

**장점 및 개선 포인트**
- 기능 범위(Must/Should/Out of Scope)와 주요 일정(MVP 기준), 투입 인력(1인, 파트타임) 등은 비교적 명확히 제시됨
- PoC 단계에서 실사용자 2인 이상 확보를 명시한 점은 실효성 확보에 도움이 됨

**과소평가/비현실적 부분**
- 1인 파트타임 체계로 초기 구축, 운영, QA, 피드백, 교육, 개선까지 모두 병행한다는 점이 비현실적  
  - PRD에서는 2인 이상 운영/QA 분담의 필요성을 언급했으나, 실제로 초기에 1인으로 착수하면 업무 과부하 및 품질 저하가 불가피
- 교육/온보딩, 현장 피드백, QA 등 Change Management에 필요한 리소스(시간, 인력)가 구체적으로 산정되지 않음
- ‘기술적 한계’에 대한 현실적인 상세 예시 및 담당자(예: 플러그인/AI 변경시 누가 어떤 방식으로 대응하는지) 역할 분담이 더 명확히 필요
- BDP-001/002/pm-skills 등 이전 산출물·가이드와 실제 연동(적용/호환/변경 관리 등)이 어떻게 관리되는지, 실무적 프로세스가 모호

**개선 제안**
- 초기 단계부터 최소 2인(운영+QA 담당) 확보를 착수 조건으로 명시하고, 실제 투입 가능 인력/시간을 산정(예: 주 2~4시간 이상 확보 등)
- Change Management, 온보딩, QA에 필요한 투입 인력·시간을 구체적으로 산정해 일정/범위 재조정 필요
- AI/플러그인 변경 등 주요 변경 상황에 대한 담당 역할 분담, 책임자 지정, Escalation 체계를 명확히

## PRD 검토 의견

제공해주신 PRD는 매우 상세하고 체계적으로 작성되어 있으며, 특히 AI 검토 의견을 반영하여 여러 차례 개선된 점이 돋보입니다. 문제-해결책의 직접 연결, 예외 처리, 현장 적용성, QA 및 교육 전략, 보안 리스크 관리 등 제품 개발의 여러 측면을 깊이 있게 고려한 흔적이 분명합니다. 이는 높은 수준의 PRD라고 평가할 수 있습니다.

하지만 IT 시장 트렌드와 유사 서비스 분석 전문가의 관점에서 몇 가지 추가적인 평가와 제언을 드리고자 합니다.

---

### 1. 시장 적합성

**평가:** 이 솔루션은 현재 IT 시장의 주요 트렌드와 매우 높은 적합성을 가집니다.

*   **생성형 AI를 활용한 업무 자동화 및 효율화:** 최근 가장 뜨거운 트렌드 중 하나로, 지식 노동자의 업무 생산성을 향상시키기 위한 AI 코파일럿(Copilot) 도입이 활발합니다. 특히 사업 개발(BD)이나 제품 관리(PM)와 같이 정형화되지 않은 사고와 분석을 요구하는 영역에서 AI를 활용하여 초벌 분석 및 문서 작성을 돕는 시도는 매우 시의적절합니다.
*   **프로세스 표준화 및 품질 일관성 확보:** 조직 내에서 업무 프로세스의 편차로 인한 비효율과 품질 저하는 고질적인 문제입니다. AI를 활용하여 'pm-skills'와 같은 체계적인 프레임워크를 강제하고 가이드함으로써, 경험 수준과 무관하게 일관된 분석 품질을 달성하려는 목표는 많은 기업에서 추구하는 바입니다.
*   **데이터 기반 의사결정 지원:** Discovery 9개 완료기준과 같은 체크리스트를 통해 누락 없이 데이터를 수집하고, 이를 바탕으로 PRD를 자동 생성하는 것은 보다 객관적이고 신뢰할 수 있는 의사결정을 지원하는 방향과 일치합니다.
*   **Low-code/No-code 및 플러그인 생태계 활용:** Claude Desktop과 pm-skills 플러그인을 활용하는 방식은 별도의 대규모 개발 없이 기존 도구의 기능을 확장하여 빠르게 솔루션을 구축하는 전략으로, IT 개발 자원이 제한적인 팀에게 매우 효율적인 접근 방식입니다.

**유사 서비스 및 차별점:**

*   **유사 서비스 (내부 지향):**
    *   **Notion AI, Coda AI 등 통합 협업 도구 내 AI 기능:** 이들은 문서 작성, 요약, 아이디어 발상 등을 지원하지만, 'pm-skills'처럼 특정 방법론에 기반한 구조화된 분석 가이드를 제공하지는 않습니다.
    *   **ChatGPT/Gemini 등 범용 LLM:** 프롬프트 엔지니어링을 통해 유사한 분석을 시도할 수 있지만, PRD 자동 생성이나 9개 기준 체크리스트와 같은 특정 업무 흐름에 대한 체계적인 연동은 사용자가 직접 구축해야 합니다.
    *   **일부 'AI Product Manager' SaaS 솔루션:** Use case.ai, Producter.co 등은 요구사항 관리, 로드맵 작성 등을 AI로 지원하지만, 'pm-skills'처럼 비즈니스 아이템 Discovery 단계의 깊이 있는 분석 가이드를 제공하는 경우는 아직 드뭅니다.
*   **차별점:**
    *   **'pm-skills' 프레임워크와의 깊은 연동:** 이 PRD의 핵심적인 차별점은 특정 프레임워크(pm-skills)를 AI 기반으로 자동화하여 사업 개발 프로세스에 녹여냈다는 점입니다. 이는 단순히 텍스트를 생성하는 것을 넘어, 특정 방법론에 따라 사고하고 분석하는 과정을 '하네싱(Harnessing)'하여 표준화하려는 시도입니다.
    *   **KT DS AX BD팀의 특정 맥락에 최적화:** 5가지 시작점 분류, 9개 Discovery 완료기준, 내부 페르소나 평가 연동 등은 외부 범용 솔루션이 제공하기 어려운 커스터마이징된 가치입니다.
    *   **PRD 자동 생성까지의 End-to-End 프로세스 지원:** 아이디어 발상부터 체계적인 분석을 거쳐 최종 산출물(PRD)까지 자동화하는 흐름은 일반적인 AI 도구보다 높은 수준의 통합성을 제공합니다.

### 2. 사실 확인

PRD는 많은 부분에서 현행 상태(As-Is)에 대한 객관적인 데이터 수집의 필요성을 명시하고 있어, 작성자가 이러한 중요성을 인지하고 있음을 보여줍니다. 하지만 몇 가지 핵심적인 가정과 수치에 대한 검증이 필요합니다.

*   **KPI 현행 값 (As-Is Baseline)의 부재:**
    *   **분석 소요 시간:** "측정 필요"로 명시되어 있습니다. 이 시스템의 핵심 목표 중 하나가 시간 단축이므로, 현재 담당자들이 사업 아이템 하나를 분석하여 PRD 수준의 정보를 도출하는 데 걸리는 **평균 시간(As-Is)**을 최소 3건 이상의 실제 사례로 측정하여 목표값 설정의 근거를 마련해야 합니다.
    *   **산출물 품질 등급 (팀장 평가):** 현재 "미확인" 상태입니다. 기존 산출물에 대한 팀장/동료 평가표를 활용하여 **평균 품질 등급(As-Is)**을 측정하고, 이를 목표값(80점/100점 이상)과 비교할 수 있어야 합니다.
    *   **사용자 입력 오류/루프백 빈도:** "미확인" 상태입니다. 현재 수동 프로세스에서 발생하는 오류나 재작업 빈도에 대한 정성적/정량적 데이터가 필요합니다.
    *   **→ 제언:** PRD §15.1 "As-Is 데이터/프로세스 샘플" 섹션의 실행을 최우선 과제로 삼아, 최소 3건 이상의 실제 사례에 대한 정량/정성적 As-Is 데이터를 반드시 확보해야 합니다. 이는 시스템 도입 효과를 객관적으로 측정하고 ROI를 입증하는 데 필수적입니다.

*   **핵심 로직의 타당성 검증 (오픈 이슈):**
    *   **Discovery 9개 완료기준의 타당성:** (오픈 이슈 #1) 이 시스템의 핵심 로직이므로, 이 기준들이 실제 사업 개발 과정에서 PRD를 작성하는 데 충분하고 적절한지 내부 팀원 및 관련 전문가의 광범위한 합의와 검증이 필요합니다.
    *   **pm-skills 스킬 프롬프트의 최적화:** (오픈 이슈 #2) `pm-skills` 플러그인의 기본 프롬프트가 AX BD팀의 특정 맥락과 요구사항에 얼마나 잘 맞는지, 추가적인 커스터마이징이나 가이드라인이 필요한지 검증해야 합니다.
    *   **5가지 시작점 경로와 9개 완료기준의 매핑 정합성:** (오픈 이슈 #3) 각 시작점별 분석 경로를 따라갈 때 9개 완료기준이 모두 자연스럽게 커버되는지, 특정 시작점에서는 특정 기준이 누락되거나 불필요한 과정이 생기지 않는지 면밀한 매핑 검증이 필요합니다.

*   **보안 정책 확정:** (오픈 이슈 #4) "KT DS 보안 정책상 외부 AI 사용 가능 범위 확정"은 시스템 착수에 앞서 반드시 선행되어야 할 사실 확인입니다. 민감 정보의 정의, 마스킹/추상화 가이드라인, 데이터 보존 정책, 외부 AI 사용 승인 여부 등을 명확히 해야 합니다. 이는 시스템의 생존 여부를 결정하는 핵심 리스크입니다.

### 3. 사용자 관점

**평가:** 목표 사용자(AX BD팀 담당자)는 이 솔루션의 도입을 긍정적으로 받아들일 가능성이 높습니다. PRD에 명시된 문제점(분석 방법론 부재, 시간 낭비, 품질 편차, 아이템 누락, 팀 신뢰도 저하)은 실제 업무에서 느끼는 강력한 Pain Point이며, 이 시스템이 제시하는 해결책(표준화된 경로, 일관된 품질, PRD 자동 생성)은 이들의 업무 부담을 크게 줄여줄 것으로 기대되기 때문입니다. 특히 "담당자 경험 수준과 무관하게 동일한 분석 수준 달성"이라는 목표는 초급자에게 큰 매력으로 작용할 것입니다.

**도입 장벽:**

1.  **새로운 도구 학습 곡선:**
    *   **문제:** Claude Desktop과 pm-skills 플러그인이라는 새로운 도구 조합에 익숙해지는 데 시간이 필요합니다. 특히 "비개발자~중급 혼합"이라는 사용자 기술 수준을 고려할 때, AI 도구 사용 경험이 적은 사용자에게는 진입 장벽이 될 수 있습니다.
    *   **PRD 반영 및 제언:** PRD §3.1, §3.3, §9.1에서 온보딩 및 교육의 중요성을 명시하고 있어 긍정적입니다. 초기 사용자를 위한 **상세한 Step-by-step 가이드, 비디오 튜토리얼, FAQ, 그리고 실제 AX BD팀의 사례를 활용한 실습 자료**를 풍부하게 제공하는 것이 중요합니다. Q&A 채널 운영도 필수적입니다.

2.  **AI 생성 결과물에 대한 신뢰 및 검증 부담:**
    *   **문제:** AI가 생성한 PRD나 분석 결과가 얼마나 정확하고 신뢰할 수 있는지에 대한 초기 의구심이 있을 수 있습니다. 사용자가 AI 결과물을 맹신하지 않고, 비판적으로 검토하고 보완해야 한다는 인식이 필요합니다. 이는 오히려 검토 시간을 늘리거나, AI 결과물이 틀렸을 때의 책임 소재 문제로 이어질 수 있습니다.
    *   **PRD 반영 및 제언:** PRD §1에 "자동화 한계 구간 명확화," §4.1 #7에 "수동 전환/안내," §10.2에 "교차 검증"을 명시한 점은 매우 좋습니다. 초기 단계에서는 AI가 초안을 만들되, **사용자가 반드시 확인하고 수정해야 하는 부분을 명확히 가이드**하고, **팀장/동료의 교차 검증 프로세스를 강력하게 권장**해야 합니다. '다중 AI 검토' 기능(P1)도 신뢰도 향상에 기여할 것입니다.

3.  **데이터 입력의 부담 및 보안 우려:**
    *   **문제:** 9개 Discovery 기준을 충족하기 위한 데이터 입력 과정이 번거롭거나, 민감한 내부 정보를 외부 AI에 입력하는 것에 대한 보안 우려가 있을 수 있습니다.
    *   **PRD 반영 및 제언:** PRD §6.4, §13.1에서 보안 정책 준수 및 민감 정보 마스킹/추상화 원칙을 명시한 점은 중요합니다. **명확한 보안 가이드라인과 함께, 어떤 정보는 입력하면 안 되는지, 어떤 정보는 어떻게 추상화해야 하는지 구체적인 예시**를 제공하여 사용자의 혼란을 줄여야 합니다. 또한, 입력 과정을 최대한 직관적이고 효율적으로 설계하여 입력 부담을 최소화해야 합니다.

4.  **자동화의 완벽성에 대한 기대치 관리:**
    *   **문제:** 사용자들이 AI가 모든 것을 완벽하게 해줄 것이라는 비현실적인 기대를 가질 수 있습니다. 현실은 AI가 초벌 작업을 돕고, 인간이 최종 검토하고 보완하는 '코파일럿' 역할이라는 점을 명확히 해야 합니다.
    *   **PRD 반영 및 제언:** "완전 자동화의 한계는 분명히 존재함을 명확히 한다"는 문구(§1)는 매우 중요합니다. PoC 및 온보딩 과정에서 이러한 **현실적인 기대치를 지속적으로 소통**하여 사용자 만족도 저하를 방지해야 합니다.

### 4. 누락된 시장 관점

PRD는 내부 솔루션으로서 AX BD팀의 특정 니즈에 집중하고 있어, 시장 관점의 분석이 일부 생략되어 있습니다.

1.  **대체재/경쟁 솔루션 (상업용)에 대한 면밀한 비교 분석:**
    *   이 시스템은 내부 효율성 향상을 목표로 하지만, 유사한 기능을 제공하는 상업용 AI 기반 PM/BD 도구들이 빠르게 발전하고 있습니다. 예를 들어, AI 기반 시장 조사 도구, 경쟁사 분석 도구, 비즈니스 모델 캔버스 자동 생성 도구 등이 있습니다.
    *   **제언:** 이러한 상업용 솔루션들이 AX BD팀의 요구사항을 얼마나 충족시킬 수 있는지, 그리고 이 내부 시스템을 구축하는 것과 비교했을 때의 TCO(총소유비용) 및 ROI를 비교하는 분석이 있다면 더 설득력이 높을 것입니다. 'pm-skills'와의 연동이라는 차별점이 있지만, 그 외 기능들은 외부 솔루션으로 대체 가능성이 있는지 검토가 필요합니다.

2.  **AI 기술 스택의 장기적인 리스크 및 유연성:**
    *   **문제:** Claude Desktop (Anthropic)과 pm-skills (GitHub 플러그인)에 대한 의존도가 높습니다. AI 시장은 매우 빠르게 변동하며, 특정 LLM 제공업체의 정책 변경, 가격 인상, 서비스 중단 또는 특정 플러그인의 유지보수 중단 등의 리스크가 있습니다.
    *   **PRD 반영 및 제언:** PRD §11.2, §13.2에서 Plan B와 버전/변경 관리를 언급한 점은 좋으나, **장기적인 관점에서 이러한 기술 스택의 유연성을 확보하기 위한 전략**이 필요합니다. 예를 들어, 여러 LLM을 쉽게 교체할 수 있는 추상화 계층을 고려하거나, 오픈소스 LLM을 자체 호스팅하는 방안에 대한 장기 로드맵 검토 등이 있을 수 있습니다. 이는 "사내망 내 AI 대체 솔루션" (§13.2) 검토와도 연결됩니다.

3.  **데이터 연동 범위 및 확장성:**
    *   현재는 Claude Desktop 내에서 분석이 이루어지지만, 장기적으로는 BD팀의 업무가 CRM, 프로젝트 관리 시스템, 재무 시스템 등 다른 사내 시스템과 연동될 필요가 있습니다.
    *   **PRD 반영 및 제언:** "사내망 연동"이 Out of Scope (§4.3)인 것은 이해하나, 향후 **Foundry-X 플랫폼과의 연동 계획(§4.4)과 더불어, 다른 핵심 사내 시스템과의 데이터 연동 확장 로드맵**이 제시된다면 시스템의 가치가 더욱 커질 것입니다. '시장/트렌드/경쟁 데이터 자동 연동' (§4.2) 기능은 좋은 시작점입니다.

### 5. 착수 판단

**착수 판단: Conditional**

PRD는 매우 상세하고 치밀하게 작성되었으며, 많은 잠재적 리스크와 도입 장벽에 대한 해결책을 제시하고 있습니다. 특히 AI 검토 의견을 성실히 반영하여 PRD의 완성도를 크게 높인 점은 높이 평가합니다.

그러나 몇 가지 핵심적인 "Open Issues"와 "측정 필요" 항목이 현재 "미정" 또는 "미확인" 상태이며, 이들이 시스템의 성공적인 착수와 효과 검증에 필수적이기 때문에 즉각적인 "Ready" 판단은 어렵습니다.

**조건:**

1.  **핵심 KPI의 As-Is 베이스라인 측정 완료:** 분석 소요 시간, 산출물 품질 등급, 사용자 입력 오류/루프백 빈도 등 주요 KPI의 현재 값(As-Is)을 최소 3건 이상의 실제 사례로 확보해야 합니다. (PRD §5.1, §15.1 참조)
2.  **주요 오픈 이슈 해결:**
    *   **Discovery 9개 완료기준 및 5가지 시작점 경로의 타당성/정합성 검증 완료:** 시스템의 핵심 로직에 대한 팀 내 합의와 검증이 필요합니다. (PRD §7 #1, #3 참조)
    *   **KT DS 보안 정책상 외부 AI 사용 가능 범위 확정:** 민감 정보 처리 방안 및 외부 AI 사용 승인 여부가 명확히 결정되어야 합니다. (PRD §7 #4 참조)
    *   **pm-skills 스킬 프롬프트의 AX BD팀 맥락 최적화 확인:** 핵심 플러그인의 적합성 검증이 필요합니다. (PRD §7 #2 참조)
3.  **운영/유지보수 인력 확보 계획 구체화:** 현재 '1명 (파트타임)'이라는 제약 조건은 시스템의 성공적인 운영, QA, 교육, 피드백 반영, 장애 대응에 심각한 리스크입니다. 최소 2인 이상의 운영/백업 체계에 대한 구체적인 계획과 인력 확보 방안이 마련되어야 합니다. (PRD §6.3, §9.3, §11.1 참조)
4.  **PoC 실사용자 2인 이상 확보 및 현장 사례 수집:** MVP 최소 기준(§5.2)에 명시된 2인 이상 실사용자 PoC를 위한 참여자 확보 및 실제 현장 사례(Pain Point, 도입 수용성) 수집 계획이 구체적으로 진행되어야 합니다. (PRD §14.1, §15.2 참조)

위 조건들이 충족된다면, 이 PRD는 매우 강력한 기반 위에서 시스템 개발을 착수할 수 있을 것입니다.

**부분적 가능성 있으나 심각한 제약 존재**

**긍정적 요소:**
- Claude Desktop + pm-skills 조합은 프로토타이핑에 적합한 저비용 접근법
- 마크다운 기반 PRD 생성은 기술적 복잡도 낮음
- 단일 사용자 플로우는 초기 구현 용이

**심각한 기술적 제약:**
1. **데이터 흐름 자동화의 환상**: §4.1 #7에서 언급했지만 실제 구현 불가능에 가까움
   - pm-skills 플러그인이 단계 간 컨텍스트 전달 API를 제공하지 않음
   - Claude Desktop 세션 간 데이터 지속성 없음 → 매번 수동 복붙 필요
   - "자동화 불가 구간은 수동 전환"이 대부분의 워크플로우가 될 것

2. **외부 AI 의존성 리스크**:
   - Claude API 정책 변경 시 전체 시스템 마비
   - pm-skills 플러그인 유지보수 불확실성 (개인 GitHub 프로젝트)
   - 다중 AI 검토(§4.2 #1)는 API 비용 폭발 가능성

3. **보안 격리와 사용성 충돈**:
   - 사외망 PC 제한 → 협업 도구(슬랙, 깃허브) 연동 불가
   - 민감 데이터 마스킹이 분석 품질 저하로 이어질 수 있음

### 2. 아키텍처 적합성
**현재 구조는 확장성/유지보수성 관점에서 부적절**

**문제점:**
1. **모노리식 프롬프트 아키텍처**:
   - 모든 로직이 프롬프트에 내장 → 변경 관리 어려움
   - 버전 컨트롤 불가능 (프롬프트 히스토리 추적 불가)

2. **상태 관리 부재**:
   - Discovery 9개 기준 진행 상태를 저장할 중앙 저장소 없음
   - 사용자 입력값의 영구적 보관 체계 미정의

3. **확장 경로 차단**:
   - Claude Desktop에 Lock-in → 다른 AI 모델로 전환 불가능
   - API 기반 자동화(시장 데이터 연동 등)와 데스크톱 앱의 괴리

**대안 아키텍처 제안:**
```
[웹 인터페이스] ←→ [백엔드 API 서버] ←→ [AI Gateway (Claude/GPT/등)]
       ↓                    ↓                    ↓
[상태 DB]          [워크플로우 엔진]      [프롬프트 템플릿 관리]
```
- 장점: 멀티 AI 지원, 상태 추적, 협업 기능, 보안 통제 가능
- 단점: 개발 리소스 10배 이상 필요

### 3. 구현 복잡도
**과소평가된 복잡도 요소:**

1. **예외처리 로직의 폭발적 증가** (§10):
   - 시작점 분류 모호성 → 휴리스틱 규칙 수십 개 필요
   - 9개 기준 간 의존성 관리 (예: 시장 규모 없이 수익 구조 계산 불가)
   - 부분 완료 상태의 복구 로직

2. **품질 검증의 주관성 문제** (§12):
   - "80점/100점 이상"의 정량화 불가능
   - AI 생성 PRD의 표면적 완성도 vs 실제 분석 깊이 괴리

3. **동시성 문제** (비록 1인이지만):
   - 동일 아이템에 대한 병렬 분석 세션 관리
   - 프롬프트 캐싱 및 일관성 유지

4. **데이터 일관성**:
   - 수동 입력과 AI 생성 내용 간 충돌 해결
   - 이전 단계 수정 시 하위 단계 자동 갱신 필요

### 4. 기술 리스크
**상위 3개 실패 요인:**

1. **pm-skills 플러그인 종속성 리스크** (확률: 높음, 영향도: 치명적)
   - 오픈소스 프로젝트로 유지보수 보장 없음
   - 프롬프트 변경 시 하위 호환성 없을 가능성

2. **Claude Desktop 정책 변경** (확률: 중간, 영향도: 치명적)
   - 플러그인 지원 중단, 사용량 제한, 가격 인상 가능성
   - 대체제 없음 (사내망 AI는 기능 미흡)

3. **데이터 흐름 단절** (확률: 매우 높음, 영향도: 높음)
   - 사용자가 단계 간 데이터 전달을 수동으로 해야 함
   - 이로 인한 피로도 증가 → 도구 포기
