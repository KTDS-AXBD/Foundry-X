## 기술 검토 의견

### 1. 기술적 실현 가능성

**긍정적 요소:**
- Claude Desktop + pm-skills 플러그인 조합은 기술적으로 검증된 접근 방식
- 마크다운 기반 PRD 생성은 구현 복잡도가 낮음
- 기존 ax-req-interview 스크립트 재사용 가능

**주요 기술적 제약:**

**1.1 보안 및 컴플라이언스 병목**
- **심각한 리스크**: KT DS 보안 정책하에서 외부 AI(Claude, ChatGPT 등)에 내부 데이터 입력은 사실상 불가능할 수 있음
- 민감 정보 마스킹/추상화 자동화가 기술적으로 복잡함
- 사외망 PC 사용 제약은 협업과 버전 관리에 큰 장애

**1.2 pm-skills 플러그인 의존성 리스크**
- 타사 GitHub 플러그인에 대한 의존도가 과도함
- 플러그인 업데이트 시 호환성 문제 발생 가능
- 프롬프트 최적화 권한이 제한적

**1.3 다중 AI 검토 구현 복잡도**
- ChatGPT/Gemini/DeepSeek API 연동은 각기 다른 인증/레이트 리밋/응답 포맷 처리 필요
- 비용 관리와 에러 핸들링이 과소평가됨

### 2. 아키텍처 적합성

**현재 설계의 문제점:**

**2.1 단일 실패점(Single Point of Failure)**
- Claude Desktop에 모든 프로세스가 종속됨
- Anthropic 서비스 장애 시 전체 시스템 마비

**2.2 확장성 부재**
- "1명 파트타임" 리소스로 설계되었으나, 팀 전체 확산 시:
  - 대화 이력 관리 불가능
  - 템플릿/프롬프트 버전 관리 체계 없음
  - 협업 기능 전무

**2.3 데이터 흐름의 취약성**
- 분석 중간 결과물의 구조화된 저장소 없음
- PRD 생성 시 모든 데이터를 Claude 컨텍스트에 재로드 필요
- 토큰 제한으로 대규모 분석 데이터 처리 불가

**대안 아키텍처 제안:**
```
[로컬 에이전트] ↔ [구조화된 저장소(JSON/DB)] ↔ [AI 게이트웨이] ↔ [다중 AI 백엔드]
       ↑
[웹 인터페이스/CLI]
```
- 로컬 에이전트: pm-skills 로직 내장
- 구조화된 저장소: 분석 단계별 결과 저장
- AI 게이트웨이: 프롬프트 최적화, 비용 관리, 폴백 처리

### 3. 구현 복잡도 (과소평가된 요소)

**3.1 Edge Cases 처리**
- 5가지 시작점 간 중복/모호한 아이템 처리 로직 없음
- Discovery 9개 기준 중 불충분한 데이터로 판단 불가한 경우
- 사용자 실수(잘못된 입력)에 대한 복구 메커니즘 부재

**3.2 pm-skills 실행 컨텍스트 관리**
- 18개 스킬 간 데이터 흐름 자동화 필요
- 이전 단계 결과를 다음 스킬에 적절히 주입하는 로직 복잡
- 비개발자 사용자를 위한 에러 메시지/가이드 필요

**3.3 PRD 자동 생성의 실제 복잡도**
- 템플릿 매핑은 단순해 보이지만:
  - 자연어 분석 결과 → 구조화된 PRD 항목 매핑 불완전
  - 일관된 톤/포맷 유지 어려움
  - 누락된 항목 자동 감지 및 사용자 재요청 필요

### 4. 기술 리스크 (우선순위별)

**P0 (치명적):**
1. **보안 정책 차단**: 외부 AI 사용 자체가 금지될 가능성
2. **Claude Desktop 의존성**: Anthropic 정책 변경/서비스 중단 리스크
3. **데이터 유실**: Claude 대화 이력만으로 분석 과정 복원 불가능

**P1 (심각):**
4. **사용자 채택 실패**: 비개발자 대상 CLI/플러그인 인터페이스 학습 곡선 가파름
5. **프롬프트 드리프트**: pm-skills 업데이트로 분석 품질 불안정화

**P2 (관리 가능):**
6. **다중 AI 검토 비용**: API 호출 비용 기하급수적 증가 가능
7. **버전 관리 부재**: PRD 템플릿/프롬프트 변경 이력 추적 불가

### 5. 착수 판단

**착수 판단: Conditional**

**전제 조건(Pre-conditions):**

1. **보안/컴플라이언스 승인 확보**
   - KT DS 보안팀으로부터 외부 AI 사용에 대한 명시적 서면 승인
   - 민감 정보 처리 가이드라인 문서화 완료

2. **아키텍처 재설계**
   - 단일 실패점 해소를 위한 하이브리드 아키텍처 채택
   - 로컬 데이터 저장소 도입으로 Claude 의존성 완화
   - MVP 범위를 P0 기능으로 제한 (다중 AI 검토 등 P1 기능 제외)

3. **리소스 재평가**
   - 파트타임 1명으로는 기술적 검증 단계만 가능
   - 실제 팀 확산을 위한 전담 리소스(0.5FTE 이상) 필요

**권고사항:**
1. **Phase 1: 기술 검증 스프린트(2주)**
   - 보안 승인 전제하에 PoC 수준 구현
   - 가장 간단한 시작점(아이디어) 1개에 대한 엔드투엔드 흐름 검증
   - 실제 사용자(팀원 1명) 피드백 수집

2. **Phase 2: 아키텍처 재설계(3주)**
   - 로컬 에이전트 + 구조화 저장소 기반으로 재설계
   - 웹 기반 인터페이스 프로토타입 개발
   - 협업 및 버전 관리 기능 추가

3. **Phase 3: 팀 확산 준비(4주)**
   - 사용자 교육 자료 개발
   - 모니터링/관리 도구 구축
   - 점진적 롤아웃 계획 수립

**결론:** 개념은 훌륭하지만 현재의 기술 설계는 실전 운영에 심각한 리스크를 내포하고 있습니다. 보안 승인과 아키텍처 재설계 없이는 프로젝트 실패 가능성이 높습니다.
