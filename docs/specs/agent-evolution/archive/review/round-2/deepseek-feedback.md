## 검토 의견: Agent Evolution PRD v2

### 1. 기술적 실현 가능성
**기본적으로 실현 가능하나, 상당한 기술적 제약과 병목 지점이 존재합니다.**

- **OpenRouter 통합 및 멀티모델 라우팅**: 기술적으로 명확합니다. OpenAI 호환 API이므로 기존 `AgentRunner` 추상화에 통합 가능합니다. 그러나 **병목은 외부 API 의존성과 지연 시간**입니다. 300+ 모델 라우팅 시, 모델별 성능/가용성 변동성이 높아 시스템 안정성을 저해할 수 있습니다. Fallback 체인이 복잡해질수록 사용자 경험(지연)이 나빠집니다.
- **Evaluator-Optimizer 패턴**: 구현 가능하지만 **컨텍스트 관리와 토큰 폭발이 주요 위험**입니다. 반복 시마다 이전 컨텍스트(코드+피드백)를 유지해야 하며, 장기 메모리/버전 관리가 없으면 품질이 저하됩니다. 또한 무한 루프 방지와 비용 통제 메커니즘이 설계에 명시적으로 포함되어야 합니다.
- **에이전트 역할 확장**: 시스템 프롬프트와 도구 권한을 DB로 관리하는 접근은 합리적입니다. 그러나 **도구 간 상호 의존성과 동시성 문제**가 발생할 수 있습니다. 예: SecurityAgent가 파일을 스캔하는 동안 TestAgent가 동일 파일을 수정하려 할 때의 락 관리.
- **워크플로우 엔진 확장**: 기존 DAG 기반 엔진에 동적 루프(Evaluator-Optimizer)를 추가하는 것은 복잡합니다. **사이클 감지와 상태 지속성**이 새로운 요구사항입니다. 현재 설계에는 워크플로우 상태를 장기 저장하고 복구하는 메커니즘이 명시되지 않았습니다.

### 2. 아키텍처 적합성
**확장성에는 적합하지만, 유지보수성과 복잡도 관리 측면에서 우려됩니다.**

- **장점**: Runner 추상화, 역할 레지스트리(DB), 규칙 기반 라우팅은 확장성 좋은 설계입니다. 관심사 분리가 명확합니다.
- **우려사항**:
    - **단일 게이트웨이 의존**: OpenRouter를 유일한 멀티모델 게이트웨이로 삼는 것은 **단일 실패 지점(SPOF)**을 만듭니다. OpenRouter 장애 시 전체 시스템이 마비됩니다. Anthropic API를 Fallback으로 유지하는 것은 좋으나, 라우팅 로직이 복잡해집니다.
    - **데이터 흐름 분산**: 에이전트가 증가하면 메시지 버스(이벤트 드리븐) 아키텍처가 더 적합할 수 있습니다. 현재의 직렬/병렬 오케스트레이션은 에이전트 간 통신이 많아질수록 조정자(Orchestrator)에 부하를 집중시킵니다.
    - **모니터링과 관측 가능성**: 제안된 대시보드는 필수적이지만, **분산 트레이싱**이 없으면 멀티모델/멀티에이전트 호출 체인의 지연 원인을 추적하기 어렵습니다. OpenTelemetry 통합이 고려되지 않았습니다.

### 3. 구현 복잡도
**PRD에 명시된 것보다 훨씬 높은 복잡도가 예상됩니다.**

- **숨겨진 복잡도 1: 동적 컨텍스트 관리**
    - Evaluator-Optimizer 루프와 Sprint 워크플로우는 장기 실행 프로세스입니다. **세션 상태, 중간 결과물, 에이전트 간 공유 컨텍스트**를 안정적으로 저장하고 버전 관리해야 합니다. 현재 Cloudflare D1로는 세션 상태 관리에 한계가 있을 수 있습니다.
- **숨겨진 복잡도 2: 에이전트 간 협업과 충돌 해결**
    - 여러 에이전트(Architect, Security, Test)가 동일 PR을 분석할 때, **결과를 어떻게 통합하고 충돌하는 피드백을 어떻게 해결**할지 명확하지 않습니다. 메타 에이전트(Arbiter) 또는 투표 메커니즘이 필요할 수 있습니다.
- **숨겨진 복잡도 3: 비용 및 속도 최적화**
    - 모델 라우팅 규칙이 정적(DB)이라면, 실시간 모델 가격/성능 변동에 대응하지 못합니다. **동적 라우팅(ML 기반)은 복잡도가 급증**하는 영역입니다. 초기 룰 기반으로 가자는 결정은 현명하지만, 향후 개선 기대치를 관리해야 합니다.
- **숨겨진 복잡도 4: 보안 및 규정 준수**
    - 코드 자동 마스킹은 간단하지 않습니다. 정규식 기반 마스킹은 오탐/미탐이 많고, **AST(Abstract Syntax Tree) 기반 분석이 필요**할 수 있어 구현 부담이 큽니다.

### 4. 기술 리스크
**가장 우려되는 상위 3가지 기술 리스크:**

1.  **외부 API 의존성과 SPOF**: OpenRouter의 SLA, 가격 변동성, 정책 변경에 완전히 종속됩니다. 대체 공급자(다른 게이트웨이 또는 직접 통합)에 대한 **B 계획이 구체적이지 않습니다**.
2.  **비용 통제 실패 및 토큰 폭발**: Evaluator-Optimizer 루프와 멀티모델 호출은 예측하기 어려운 비용을 발생시킵니다. **사전 예산 한도 설정, 실시간 비용 차단, 비용 효율성 모니터링**이 핵심 기능으로 설계에 반영되어야 합니다. 현재는 모니터링 대시보드만 언급되어 있습니다.
3.  **시스템 복잡도 증가로 인한 운영 부담**: 에이전트, 모델, 워크플로우가 증가하면 **디버깅, 장애 추적, 성능 튜닝이 기하급수적으로 어려워집니다**. 강력한 로깅, 분산 트레이싱, 에이전트 실행 재현 기능이 없으면 운영이 불가능할 수 있습니다.

### 5. 착수 판단
**착수 판단: Conditional**

**조건:**
1.  **핵심 블로커 해결**: KT DS 보안/법무팀으로부터 **"외부 LLM API 전송 가능 데이터 범위"에 대한 공식 승인**을 반드시 확보해야 합니다. 가이드라인 없이는 개발을 진행할 수 없습니다.
2.  **범위 조정 및 인력 계획 수립**: 현재 1인 개발 계획은 비현실적입니다. **MVP 범위를 P0 기능(A1~A5)으로 제한**하고, 이를 수행할 **추가 인력(최소 1명의 백엔드 엔지니어)을 반드시 확보**해야 합니다. 또는 Phase를 세분화하여 Track B(개발 도구)만 즉시 시작합니다.
3.  **운영 인프라 선구축**: 모니터링 대시보드, Feature Flag, Canary 배포/롤백 체계, 비용 알람 시스템을 **MVP 개발과 병행하거나 선행**하여 구축해야 합니다. 복잡한 시스템은 운영 체계 없이 출시하면 실패합니다.

**권고사항:**
- **Phase 1 (즉시)**: Track B(개발 도구) 도입과 보안 정책 협의를 병행.
- **Phase 2 (조건 충족 후)**: P0 기능 중 **OpenRouter 통합(A1)과 모델 라우팅(A2)을 최소 MVP**로 삼아 비용/성능 실증 실험 진행.
- **Phase 3**: 실험 결과를 바탕으로 Evaluator-Optimizer(A3) 및 추가 에이전트(A4, A5) 확장 여부를 결정.

이 PRD는 비전과 방향성은 훌륭하지만, 기술적 실행 난이도와 운영 리스크가 상당히 과소평가되어 있습니다. 조건이 충족되지 않으면 프로젝트는 중도에 중단되거나 기술 부채가 쌓인 채 불안정한 시스템이 될 위험이 높습니다.
