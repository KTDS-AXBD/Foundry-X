## 1. 요약

**한 줄 정의:**
Foundry-X는 사람과 AI 에이전트가 동등한 팀원으로서 함께 소프트웨어를 만드는 조직 협업 **통합 플랫폼**이다.

**배경:**
AI 코딩 에이전트의 역량이 급격히 발전하면서, 엔지니어의 역할이 "코드 작성"에서 "환경 설계 + 의도 명시 + 피드백 루프 구축"으로 전환되고 있다. AX BD팀은 이 전환을 위해 3개의 서비스를 병렬로 개발해왔다:

- **Discovery-X**: 관찰→실험→기록 사이클의 신사업 발굴 플랫폼 (80% 완성)
- **AI Foundry**: 요구사항→스펙→코드 자동 생성 파이프라인 (90% 완성)
- **AXIS Design System**: 팀 공통 UI/UX 체계 (준비 완료, 구현 상태 점검 필요)

이 세 서비스가 독립적으로 존재할 때의 한계가 드러났다. 사용자는 서비스 간 전환이 필요하고, 데이터가 분산되며, 하나의 프로젝트가 여러 서비스에 걸쳐 있을 때 통합된 뷰를 얻을 수 없다.

**목표:**
Foundry-X를 중심 플랫폼(루트 앱)으로 하여, Discovery-X(탐색)·AI Foundry(구축)·AXIS DS(UI 체계)를 단일 대시보드 안의 서브 앱으로 통합한다. 하네스 엔지니어링과 SDD Triangle은 핵심 축으로 유지하되, "조직이 탐색하고, 구축하고, 동기화하는 전 과정"을 하나의 플랫폼에서 수행할 수 있게 한다.

## 3. Background

### 왜 지금인가?

v4에서 기술한 하네스 엔지니어링과 SDD의 부상은 여전히 유효하다. v5에서 추가된 배경:

**통합의 필요성이 대두된 이유:**

1. **사용자 경험 단절**: Discovery-X에서 발견한 기회를 AI Foundry로 넘기고, 그 결과를 Foundry-X에서 관리하는 흐름이 3개 서비스를 오가며 이루어짐
2. **데이터 분산**: 같은 프로젝트의 데이터가 3개 시스템에 분산되어 단일 뷰 불가
3. **운영 비용**: 3개 독립 서비스의 인프라·인증·배포를 각각 관리하는 오버헤드
4. **기술 스택 파편화**: 서비스마다 다른 기술 스택으로 인한 학습 비용과 유지보수 부담

### Discovery-X → AI Foundry → Foundry-X → **통합 플랫폼**

- **Discovery-X**: 관찰→실험→기록 사이클의 신사업 발굴 플랫폼. "탐색과 발견"의 정신.
- **AI Foundry**: 요구사항→스펙→코드 자동 생성 파이프라인. "구축"의 정신.
- **AXIS Design System**: 팀 공통 UI/UX 체계. "일관된 경험"의 정신.
- **Foundry-X (v5 비전)**: 세 프로젝트의 합류. **탐색하고, 구축하고, 동기화하는 — 그 전 과정**을 동료와 에이전트가 함께 하나의 플랫폼에서 수행.

### 경쟁 환경

| 도구 | 핵심 가치 | Foundry-X 차별점 |
|------|----------|-----------------|
| GitHub Spec Kit | 개발자 개인 SDD 워크플로우 | Foundry-X는 조직 단위 협업 + 하네스 + 통합 플랫폼 |
| Kiro (AWS) | IDE 내 스펙 관리 | Foundry-X는 Git 중심 SSOT + CLI + 웹 대시보드 |
| Plumb | SDD Triangle PoC 도구 | Foundry-X는 Plumb을 핵심 엔진으로 확장 |
| Augment Code Intent | 에이전트 오케스트레이션 | Foundry-X는 SDD + 하네스 + 협업 + 탐색/구축 통합 |
| synthnoosh/agentic-harness-bootstrap | 하네스 bootstrap 레퍼런스 구현 | Foundry-X는 SDD Triangle + 조직 협업 + 통합 플랫폼 |

**Foundry-X의 독자적 위치:** 코드 생성 도구가 아니라 "스펙과 하네스를 자산화"하는 조직 협업 **통합 플랫폼**. 개별 개발자 도구와 달리 팀/조직 단위 운영을 전제로 설계하며, 탐색→구축→동기화의 전 생명주기를 커버.

조직 내에서 사람과 AI 에이전트가 동등한 팀원으로 협업하며, **탐색(Discovery-X)→구축(AI Foundry)→동기화(Foundry-X)의 전 과정**이 하나의 통합 플랫폼에서 이루어지는 환경을 제공한다. 명세↔코드↔테스트가 항상 동기화된 상태를 유지한다.

## 6. Value Proposition(s)

### 에이전트 지휘자 (Phase 1~2, 완료)

- **What before**: 에이전트를 쓰면 코드는 나오지만, 명세와 코드가 점점 어긋남
- **How**: `foundry-x init`으로 하네스를 구축하면, 매 커밋마다 결정사항이 자동 추출되어 명세에 반영
- **What after**: 명세가 항상 코드와 일치

### 의도 설계자 (Phase 2, 완료)

- **What before**: 요구사항을 PPT와 구두로 전달하면 의도가 왜곡됨
- **How**: 자연어로 의도를 입력하면 구조화된 명세로 자동 변환
- **What after**: 내가 말한 것이 그대로 동작하는 프로토타입

### 탐색자 (Phase 4, v5 신규)

- **What before**: 시장 기회를 발굴해도 구현까지 별도의 프로젝트 설정이 필요하고, 탐색 맥락이 유실됨
- **How**: Discovery-X에서 발견한 기회를 "구축 시작" 한 번으로 Foundry-X 프로젝트로 전환, 탐색 메타데이터가 자동으로 하네스에 반영
- **What after**: 탐색→구축이 끊김 없이 연결되고, 탐색 맥락이 명세에 보존됨

### 스킬 설계자 (Phase 4, v5 신규)

- **What before**: AI 에이전트의 도구/스킬이 개인 환경에 묶여있고, 조직 차원 재사용이 불가
- **How**: AI Foundry에서 설계한 스킬을 MCP 서버로 등록하면, Foundry-X의 에이전트가 즉시 활용
- **What after**: 한 번 만든 스킬이 조직 전체 에이전트에게 공유되어 역량이 누적

## 7. Solution

### 7.1 설계 원칙

**"Git이 진실, Foundry-X는 렌즈"** — v4에서 확립, v5에서도 유지

- 모든 명세, 코드, 테스트, 결정 이력은 **Git 리포에 존재** (SSOT)
- Foundry-X는 이를 읽고, 분석하고, 시각화하고, 동기화를 강제하는 **레이어**
- DB(Cloudflare D1)에는 메타데이터만 저장, **Git이 항상 승리**
- 에이전트는 Git 리포에 직접 작업하고, Foundry-X는 그 결과를 관찰

v5 추가 원칙:
- **통합 플랫폼의 SSOT는 여전히 Git**: 통합되는 서비스들도 Git 기반 데이터 관리를 따른다
- **점진적 통합**: UI 먼저, API는 검증 후. Big Bang 마이그레이션 금지
- **기술 스택 통일**: Cloudflare 생태계(Workers + D1 + Pages)로 수렴

### 7.2 핵심 5축 + 1축

v4에서 정의한 5축을 유지하고, v5에서 **축 6(통합 플랫폼)**을 추가한다.

### Phase 4 판정: Go / Pivot / Kill

```
Go 조건 (하나 이상):
  - 통합 플랫폼 NPS 6+ (K12)
  - WAU 60%+ (K7 — 5명 중 3명 이상 주간 활성)
  - "개별 서비스로 돌아가고 싶지 않다" 피드백 2명 이상

Pivot:
  - 일부 서비스만 통합 (예: AXIS DS + Foundry-X만, Discovery-X는 독립 유지)
  - 통합 대신 API 연동으로 회귀 (F73 C1/C2/C3 경로)

Kill:
  - 통합이 오히려 복잡성만 증가시키고 가치를 제공하지 못함
  - 학습 문서화 후 개별 서비스 독립 운영으로 전환
```
