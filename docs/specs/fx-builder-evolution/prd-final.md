## 1. 요약 (Executive Summary)

**한 줄 정의:**
Prototype Builder의 품질을 계량화·자동 향상하고, Claude Max 구독을 활용해 API 비용을 제거하여 고객 데모 가능한 수준의 프로토타입을 지속 생성하는 시스템.

**배경:**
Phase 16에서 Prototype Auto-Gen 시스템을 구축하여 PRD → 코드 생성 → Vite 빌드 → Cloudflare Pages 배포 파이프라인을 완성했다. Railway 클라우드에서 5종 PRD의 E2E 실행에 성공했으나, 생성된 프로토타입의 품질이 고객에게 보여줄 수준에 미달한다. 현재 Discriminator는 `vite build` 성공 여부만 판정(0.9/0.4)하여 "빌드는 되지만 조잡한" 결과물이 통과한다.

동시에 Railway에서 Anthropic API를 직접 호출하는 구조로 인해 토큰 비용이 지속 발생하며, Claude Max 구독($100/월)에 포함된 Claude Code CLI를 활용하면 이 비용을 제거할 수 있다.

**목표:**
1. 다차원 품질 스코어링 체계를 구축하여 프로토타입 수준을 정량적으로 측정한다
2. 품질 점수 기반 자동 개선 루프로 80점 이상 달성을 자동화한다
3. Claude Max 구독의 CLI를 Builder에 통합하여 API 비용을 제거한다

**추가 검증 목표:**
- 5차원 품질 스코어링이 실제 '고객 데모 가능'과 높은 상관관계를 갖는지 사전 PoC(Proof of Concept)를 통해 검증한다.
- CLI, Vision API, Lighthouse 등 핵심 외부 도구의 환경 호환성, rate limit, 성능 및 비용을 프로젝트 착수 전 반드시 PoC로 확인한다.
- 자동 개선 루프(MVP)의 수렴율, 평가 재현성(±10점 이내), 실제 비용 절감 효과의 사전 검증 결과를 기반으로 본 프로젝트의 단계별 진행 여부를 결정한다.

## 2. 문제 정의

### 2.1 현재 상태 (As-Is)

**품질 문제:**
- 5종 프로토타입 모두 "빌드 성공"이지만 고객 데모 불가 수준
- Discriminator가 1차원(빌드 성공 0.9 / 실패 0.4)만 측정 — 디자인, 기능, PRD 반영도 미평가
- 생성된 코드의 UI가 깨져 있고, 인터랙션이 미구현된 상태로 배포됨
- 품질을 객관적으로 측정할 기준이 없어 "충분히 좋은지" 판단 불가

**비용 문제:**
- Railway에서 Anthropic API 직접 호출 → 토큰당 과금 (Sonnet: $3/$15 per 1M tokens)
- 5종 PRD × 평균 2라운드 × ~16K output tokens = 약 $2.40/회
- 반복 개선 시 비용 누적 (목표 80점까지 5+ 라운드 예상 → $6~12/회)
- CostTracker 월 $20 예산이 10회 미만 실행으로 소진

**환경 문제:**
- Railway Docker 환경에서 Claude CLI가 hang (SKIP_CLI=true로 우회 중)
- API 모드와 CLI 모드가 완전히 분리되어 통합 운영 어려움

<!-- CHANGED: 문제-해결 매핑 및 외부 도구 환경 호환성, PoC 필요성 명시 -->
**외부 도구 및 환경 리스크:**
- Claude Max CLI의 `--bare` 모드 지원 및 rate limit, 비용 정책 등 실제 운영 제약에 대한 공식적 확인 및 PoC 미흡
- Vision API, Lighthouse 등 외부 평가 도구의 환경 호환성·과금 여부·실행속도 미확인

**평가 및 성공 기준 보완:**

### 5.2 MVP 최소 기준
- [ ] 5차원 품질 스코어링이 동작하고 점수가 산출됨
- [ ] 점수 < 80일 때 자동 개선 루프가 1회 이상 실행됨
- [ ] Claude Code CLI로 코드 생성이 가능함 (API 비용 $0)
- [ ] 기존 5종 PRD 중 3종 이상이 80점+ 달성
- [ ] CLI, Vision API, Lighthouse의 환경 호환성 및 rate limit, 비용이 실사용에 충분함이 PoC로 검증됨
- [ ] 동일 입력에 대해 5차원 평가 점수의 편차가 ±10점 이내임이 자동화 테스트로 증명됨

### 5.3 실패/중단 조건
- Claude Max 구독의 CLI가 subprocess 모드(`--bare`)를 지원하지 않거나 rate limit이 실용적이지 않을 때
- 5차원 평가의 재현성이 낮아(±20점 이상 편차) 신뢰할 수 없을 때
- 5라운드 이내 80점 수렴율이 30% 미만일 때

## 7. 기술 설계 핵심

### 7.1 5차원 품질 스코어카드

| 차원 | 가중치 | 측정 방법 | 설명 |
|------|:------:|-----------|------|
| **빌드 성공** | 20% | `vite build` exit code | 빌드 가능 여부 (현재 Discriminator) |
| **UI 완성도** | 25% | Vision API (스크린샷 평가) | 레이아웃, 디자인 일관성, 스타일링 |
| **기능 동작** | 20% | 정적 분석 + 빌드 워닝 | onClick 핸들러, 라우팅, 상태 관리 |
| **PRD 반영도** | 25% | LLM 평가 (PRD vs 코드 비교) | Must Have 기능 구현 여부 |
| **코드 품질** | 10% | ESLint + TypeScript strict | 타입 에러, 린트 경고 수 |

**총점 = Σ(차원 점수 × 가중치) × 100**

각 차원은 0.0~1.0으로 평가되며, 가중 평균을 100점 환산한다.

<!-- CHANGED: 5차원 자동 평가의 신뢰성 확보(재현성, 상관관계) 및 독립 실행/결과 병합으로 내결함성 강화 -->
**평가 체계 신뢰성 확보:**
- 동일 코드 입력에 대해 반복 평가 시 점수 편차가 ±10점 이내임을 자동화 테스트로 검증(M1 착수 전 PoC)
- 각 차원별 평가(예: LLM, Vision, 정적분석)를 독립적으로 실행하고, 결과 병합 시 장애 발생 차원만 리트라이/대체(내결함성)
- 품질 점수와 실제 사용자(고객사, BD팀)의 주관적 평가(데모 가능 여부) 간 상관관계(예: 0.8+)를 파일럿으로 검증

### 7.2 듀얼 모드 아키텍처

```
┌─────────────────────────────────────────────┐
│               Builder Server                 │
│                                              │
│  ┌──────────┐                                │
│  │ CLI Mode │                                │
│  │ (구독)   │                                │
│  └────┬─────┘                                │
│       │                                      │
│       ▼                                      │
│  ┌─────────────────────────┐                 │
│  │    Code Generator       │                 │
│  └────────────┬────────────┘                 │
│               ▼                              │
│  ┌─────────────────────────┐                 │
│  │  5-Dimension Scorer     │                 │
│  │  ┌───┬───┬───┬───┬───┐ │                 │
│  │  │빌드│UI │기능│PRD│코드│ │                 │
│  │  └───┴───┴───┴───┴───┘ │                 │
│  └────────────┬────────────┘                 │
│               ▼                              │
│  ┌─────────────────────────┐                 │
│  │  Improvement Loop       │                 │
│  │  (score < 80 → retry)   │                 │
│  └────────────┬────────────┘                 │
│               ▼                              │
│  ┌─────────────────────────┐                 │
│  │  Deploy (Pages)         │                 │
│  └─────────────────────────┘                 │
└─────────────────────────────────────────────┘
```
<!-- CHANGED: 듀얼 모드 복잡성 감소, CLI 모드 primary로 고정, API 모드 완전 별도 파이프라인 분리 -->
**아키텍처 단순화:**
- **CLI Mode (Primary):** Claude Max 구독의 `claude --bare` CLI만 사용. CLI 미동작 시 전체 파이프라인 중단 후 장애 알림. API 모드는 완전히 별도(disaster recovery) 파이프라인으로 분리 관리.
- CLI, Vision API, Lighthouse 등 외부 도구 환경 테스트 및 rate limit 자동 모니터링 포함
- 각 평가 차원별 독립 실행 및 결과 병합, 장애 발생 시 해당 차원만 리트라이/대체(내결함성 강화)
- 결정 로그(deterministic logging) 및 피드백 체인 추적(품질 개선 경로 투명성 확보)

### 7.3 자동 개선 루프 (Enhanced O-G-D)

```
Round 1: Generator → 5-Dim Score → (score < 80?)
  ↓ Yes
Round 2: Generator + [가장 낮은 차원 타겟 피드백] → 5-Dim Score → (score < 80?)
  ↓ Yes
Round N: ... (최대 5회)
  ↓ 수렴 실패
Best Score 채택 + 미달 리포트 생성
```
<!-- CHANGED: 개선 루프의 수렴 불가시 장애 리포트 및 상태 복구 메커니즘, 피드백 체인 추적 -->
- 각 라운드에서 가장 낮은 차원을 식별하고, 해당 차원에 특화된 개선 프롬프트를 생성한다:
  - **UI 낮음** → "레이아웃을 정리하고, Tailwind 유틸리티를 사용하여 일관된 간격/색상을 적용하세요"
  - **기능 낮음** → "모든 버튼에 onClick 핸들러를 연결하고, 라우팅이 동작하도록 하세요"
  - **PRD 낮음** → "PRD의 Must Have 기능 중 미구현 항목: {목록}. 이를 추가하세요"
- 5라운드 이내 80점 미달 시, Best Score 결과와 미달 사유 리포트 자동 생성 및 담당자 알림
- 피드백 체인·결정 로그 저장, 장애 발생시 자동 복구(재시도/롤백) 및 이력 관리
