### 1. **논리적 완결성**

#### 강점
- 문제 → 해결책 → 성공기준의 구조가 명확하게 드러나 있습니다.
- 품질 문제(1차원 평가, 고객 데모 불가), 비용 문제(API 과금), 환경/외부 도구 문제 등 현상 진단이 구체적입니다.
- 이를 해결하기 위한 5차원 품질 평가, 자동 개선 루프, 비용 절감(CLI 활용)이라는 솔루션, 그리고 정량적 KPI와 PoC 기반의 실효성 검증까지 논리적 흐름이 잘 이어집니다.

#### 개선 포인트
- **품질 측정과 실제 고객 만족 간 상관관계**: "5차원 평가 점수와 실제 데모 가능성의 상관관계"를 파일럿으로 검증한다고는 하나, 실패 시 구체적 액션 플랜(예: 평가항목 개정, 추가 인터뷰 등)이 부족합니다. 만약 상관관계가 낮으면 어떻게 할지(예, 평가 차원 구조를 바꾼다, 사용자 평가를 보완지표로 쓴다 등)까지 명확히 명시해야 합니다.
- **자동 개선 루프의 실패 처리**: 5라운드 후 미달이면 결과 리포트만 생성하는데, 근본적 원인분석 및 개선(예: 프롬프트 템플릿 개정, LLM 파라미터 튜닝 등)에 대한 후속 액션이 빠져 있습니다.
- **외부 도구 장애 시 fallback**: 예를 들어 Vision API 장애 시 대체 평가 방법(수동 평가, 임시 점수, 평가 skip 등)이 구체적으로 정의되어야 합니다.

**요약**: 논리적 구조는 전반적으로 완결성이 높으나, 실패 시 옵션과 평가-실사용 상관관계 저조 시의 구체적 후속조치가 더 명확히 정의되어야 합니다.

### 2. **실행 가능성**

#### 강점
- 각 마일스톤별 주요 목표와 산출물이 세분화되어 있고, PoC를 통해 외부 의존성 및 평가 신뢰성 검증을 전제하고 있습니다.
- 인력/일정 과소평가 리스크를 인지하고, 마일스톤별 PoC 결과에 따라 공수/일정을 확장할 수 있다고 명시되어 있습니다.

#### 개선 포인트
- **초기 투입 인력 과소**: 1명, 2~3주로 기획되어 있으나 실제로는 M1(5차원 평가기, 반복 테스트, 피드백 루프 파일럿)만 해도 2~3주 소요될 수 있습니다. CLI 통합, 장애 대응, 대시보드 등까지 감안할 때 최소 2인 이상, 4~6주로 잡는 것이 현실적입니다.
- **외부 도구 환경 리스크**: Vision API, Lighthouse, CLI 등 도구들의 docker/wsl 환경에서의 안정성, 과금, rate limit의 복합 위험이 큽니다. 초기에 반드시 독립형 PoC로 집중 검증이 필요하고, 그 결과에 따라 전체 일정이 크게 변동될 수 있습니다.
- **테스트/QA 공수 누락**: 평가 자동화, 장애 시나리오, 피드백 루프 등 QA/테스트 자동화를 위한 코드, 케이스 설계, 반복 테스트의 공수가 과소평가되어 있습니다.
- **사용자 피드백 루프의 현실성**: 실제 고객사 담당자, BD팀의 평가를 정기적으로 수집하고 분석하는 과정(설문, 인터뷰, 피드백 반영)도 상당한 시간과 조율이 필요할 수 있습니다.

**요약**: 목표는 실무적으로 타당하나, 초기 인력·일정 산정이 상당히 과소평가되어 있습니다. PoC 이후 현실 반영이 필수입니다.

- **롤백/파이프라인 복구 시나리오**: 장애 발생 시 자동 중단 이후 롤백 및 재시작 절차, 히스토리 관리 등 운영 안정성 확보 방안이 상세화되어야 합니다.

## fx-builder-evolution PRD 검토 의견

IT 시장 트렌드와 유사 서비스 분석을 전문으로 하는 애널리스트로서 `fx-builder-evolution PRD`를 검토했습니다. 본 PRD는 LLM 기반 코드 생성 및 자동화된 품질 개선이라는 최신 트렌드를 잘 반영하고 있으며, 특히 "고객 데모 가능한 수준"의 프로토타입 생성이라는 목표가 매우 구체적이고 도전적입니다. 다만, 몇 가지 핵심적인 리스크와 가정에 대한 철저한 사전 검증이 필요합니다.

### 1. 시장 적합성

본 솔루션은 현재 IT 시장의 주요 트렌드와 매우 밀접하게 맞닿아 있습니다.

*   **LLM 기반 코드 생성 및 AI 에이전트**: GitHub Copilot, AWS CodeWhisperer, Cursor 등 LLM 기반 코드 생성 도구가 이미 널리 사용되고 있으며, 최근 Devin과 같은 자율형 AI 에이전트가 코딩 작업을 넘어 전체 개발 프로세스를 자동화하려는 시도가 활발합니다. `fx-builder-evolution`은 PRD 입력부터 코드 생성, 빌드, 배포, 평가, 그리고 **자동 개선 루프**까지 포함하여 단순 코드 생성을 넘어선 **End-to-End AI 기반 프로토타이핑 에이전트** 시스템으로 진화하려는 명확한 방향성을 보여줍니다.
*   **개발 생산성 및 비용 효율성 증대**: 개발 인력 부족과 비용 상승 압력 속에서 AI를 활용한 개발 생산성 향상은 기업의 핵심 과제입니다. Claude Max 구독을 활용한 API 비용 제거 목표는 이러한 비용 효율성 트렌드에 부합합니다.
*   **MLOps/DevOps for AI**: AI 모델(여기서는 Generator LLM)의 성능을 지속적으로 모니터링하고 개선하는 `Observe-Generate-Discriminate (O-G-D)` 루프는 MLOps의 개념을 개발 프로세스에 적용한 사례로 볼 수 있습니다. 품질 스코어링을 통해 모델의 "성능"을 측정하고 피드백을 통해 "개선"하는 과정은 AI 기반 시스템의 안정적 운영 및 발전에 필수적입니다.
*   **로우코드/노코드의 진화**: 기존 로우코드/노코드 툴이 제공하는 한계(디자인/기능의 유연성 부족, 복잡한 요구사항 대응의 어려움)를 AI가 직접 코드를 생성하고 개선함으로써 극복하려는 시도로 해석됩니다.

**유사 서비스 및 차별점:**

*   **유사 서비스**:
    *   **LLM 기반 코드 생성 도구**: GitHub Copilot, AWS CodeWhisperer, Google Gemini Code Assistant 등. 이들은 주로 개발자의 코드 작성 속도를 높이는 데 초점을 맞춥니다.
    *   **AI 기반 웹사이트/앱 빌더**: Durable.co, Framer AI, Mixo.io 등. 이들은 주로 특정 템플릿 기반으로 웹사이트를 빠르게 생성하지만, PRD를 입력받아 복잡한 프론트엔드 로직까지 생성하고 품질을 자동 개선하는 수준에는 미치지 못합니다.
    *   **UI/UX 프로토타이핑 툴**: Figma, Sketch, Adobe XD 등. 이들은 디자이너가 수동으로 프로토타입을 만들며, 코드를 자동 생성하지 않습니다.
*   **필요한 차별점**:
    1.  **"고객 데모 가능" 수준의 품질**: PRD에 명시된 가장 중요한 차별점입니다. 단순한 코드 생성을 넘어, **사람이 만든 것과 구별하기 어려울 정도의 높은 완성도**를 AI가 자동으로 달성할 수 있다면, 이는 시장에서 매우 강력한 경쟁 우위가 될 것입니다. 이를 위해 5차원 스코어링의 정확성과 자동 개선 루프의 효율성이 핵심입니다.
    2.  **자동 개선 루프의 효율성**: "가장 낮은 차원 타겟 피드백"을 통한 반복 개선은 강력한 컨셉입니다. 이 루프가 실제로 얼마나 빠르고 정확하게 목표 품질에 도달하는지가 중요합니다. (예: 5라운드 이내 80점+ 달성 목표)
    3.  **비용 효율성**: Claude Max CLI를 통한 API 비용 $0/회 목표는 운영 비용 측면에서 매우 매력적인 차별점입니다. 단, Vision API 등 숨겨진 비용에 대한 면밀한 검토가 필수적입니다.
    4.  **다차원 품질 스코어링의 투명성과 신뢰성**: 빌드 성공 여부를 넘어 UI 완성도, 기능 동작, PRD 반영도, 코드 품질까지 종합적으로 평가하는 시스템은 AI 생성 결과물에 대한 신뢰도를 높이고, 개선 방향을 명확히 제시하는 데 기여합니다.

### 2. 사실 확인 (검증이 필요한 가정/수치)

PRD는 PoC의 중요성을 여러 차례 강조하고 있으며, 이는 매우 적절한 접근입니다. 다음은 특히 검증이 필요한 가정과 수치입니다.

*   **Claude Max CLI `--bare` 모드 지원 및 Rate Limit**: (오픈 이슈 #1) PRD의 핵심 전제인 "API 비용 제거"를 위해 CLI 모드의 안정적인 동작과 실제 운영 환경에서의 제약 사항(분당/시간당 토큰 제한, 동시 실행 수 등)을 반드시 확인해야 합니다. Anthropic의 공식 문서를 통한 확인과 실제 테스트가 필수입니다.
*   **Vision API (스크린샷 평가) 비용**: (오픈 이슈 #2) Discriminator의 핵심인 UI 완성도 평가를 위한 Vision API 호출 비용이 월 $5 이내로 유지될 수 있는지 확인해야 합니다. 평가 빈도와 스크린샷 크기/개수에 따른 과금 정책을 면밀히 분석하고 PoC를 통해 검증해야 합니다. 이외 Lighthouse CI 등 다른 외부 도구의 잠재적 비용도 함께 고려되어야 합니다.
*   **Lighthouse CI의 Railway Docker 환경 호환성**: (오픈 이슈 #3) Headless Chrome이 필요한 Lighthouse CI가 Railway Docker 환경에서 안정적으로 실행되는지 검증이 필요합니다. 환경 설정의 복잡성 및 성능 오버헤드도 고려해야 합니다.
*   **5차원 평가의 재현성 및 '고객 데모 가능' 상관관계**: (오픈 이슈 #5, 추가 검증 목표)
    *   **재현성**: 동일한 코드 입력에 대해 5차원 평가 점수의 편차가 ±10점 이내로 유지되는지 자동화된 테스트로 증명해야 합니다. LLM 기반 평가(UI 완성도, PRD 반영도)는 비결정적일 수 있어 특히 중요합니다.
    *   **상관관계**: 5차원 점수 80점+가 실제 '고객 데모 가능'과 높은 상관관계(예: 0.8 이상)를 갖는지 PoC를 통해 실제 BD팀 및 고객사의 주관적 평가와 비교하여 검증해야 합니다. 이 상관관계가 낮으면 평가 체계 자체가 무의미해질 수 있습니다.
*   **자동 개선 루프의 수렴율**: "70%+ (5라운드 이내) 80점+ 달성" 목표는 매우 도전적입니다. LLM의 특성상 반복 개선이 항상 선형적으로 이루어지지 않거나, 특정 차원에서 병목 현상이 발생할 수 있습니다. 실제 PRD와 생성된 코드를 대상으로 한 파일럿 테스트를 통해 이 수렴율의 현실성을 검증해야 합니다.
*   **인력/일정 산정**: "투입 인원: 1명, 목표 완료일: 2~3주"는 매우 공격적인 일정입니다. 특히 다차원 평가기 구현, CLI 통합, 자동 개선 루프, 대시보드 구현 등 광범위한 기능을 1명이 2~3주 내에 안정적으로 구현하고 검증하는 것은 높은 리스크를 수반합니다. PoC 결과에 따른 인력 증원 및 일정 연장 가능성을 명시한 것은 긍정적이나, 초기 산정 자체가 과소평가되었을 가능성이 높습니다.
*   **초기 품질 점수 (~35점 추정)**: 현재 품질 수준에 대한 추정치입니다. 실제 측정 기준을 확립하고 정확한 베이스라인을 설정하는 것이 중요합니다.

### 3. 사용자 관점

**목표 사용자가 이 솔루션을 받아들일 것 같나요?**

네, 현재 "빌드는 되지만 조잡한" 프로토타입에 대한 불만이 명확하고, BD팀의 "높은 품질의 프로토타입을 빠르게 생성" 니즈가 강하기 때문에, 목표가 달성된다면 사용자들은 이 솔루션을 매우 긍정적으로 받아들일 것으로 예상됩니다. 특히 "고객 데모 가능" 수준에 도달한다면, BD팀의 업무 효율성과 고객 제안의 성공률에 직접적인 기여를 할 것입니다.

**도입 장벽:**

1.  **신뢰성 부족**: 초기에는 AI가 생성한 프로토타입의 품질과 5차원 스코어링의 객관성에 대한 불신이 있을 수 있습니다. "빌드는 되지만 조잡한" 경험이 지속된다면 사용자는 시스템을 외면할 것입니다.
    *   **대응**: 5차원 평가의 재현성 및 '데모 가능' 상관관계 검증 결과를 투명하게 공유하고, 초기 성공 사례를 적극적으로 홍보하며, 사용자 피드백 루프를 통해 지속적으로 시스템을 개선해야 합니다.
2.  **통제감 상실**: AI가 자동으로 코드를 생성하고 개선하는 과정에서, BD팀이나 디자이너가 원하는 미묘한 디자인/기능적 요구사항이 반영되지 않거나, 수정이 어려울 수 있다는 우려가 있을 수 있습니다.
    *   **대응**: 품질 대시보드에서 개선 과정을 시각적으로 보여주고, 어떤 피드백으로 어떻게 개선되었는지 투명하게 공개하여 통제감 상실 우려를 완화해야 합니다. 또한, 필요시 개발자가 개입하여 수동으로 수정할 수 있는 여지를 남겨두는 것이 좋습니다.
3.  **학습 곡선**: 새로운 품질 스코어링 방식, 자동 개선 루프의 작동 원리, 대시보드 사용법 등을 BD팀이 학습하는 데 시간이 필요할 수 있습니다.
    *   **대응**: 직관적인 UI/UX 설계, 명확한 사용 가이드 제공, 초기 교육 및 온보딩 지원이 필요합니다.
4.  **외부 도구 의존성 및 안정성**: Claude CLI, Vision API, Lighthouse 등 외부 도구의 장애나 불안정성이 시스템 전체의 신뢰성에 영향을 줄 수 있습니다.
    *   **대응**: 장애/실패 대응 및 복구 메커니즘을 견고하게 구축하고, 외부 도구의 안정성을 지속적으로 모니터링하며, 문제가 발생했을 때의 대체 플랜(예: API 모드 fallback, 수동 평가 전환)을 마련해야 합니다.

### 4. 누락된 시장 관점

PRD는 기술적 구현과 내부 목표에 집중하고 있으나, 더 넓은 시장 관점에서 다음과 같은 사항들이 추가로 고려되면 좋습니다.

1.  **LLM 공급망 다변화 및 벤더 락인 리스크**: 현재 Claude Max에 대한 의존도가 매우 높습니다. Anthropic의 정책 변경, 가격 인상, 서비스 종료 등의 리스크에 대비하여 GPT-4o, Gemini, 혹은 오픈소스 LLM (예: Llama 3, Code Llama) 등 다른 LLM으로의 확장 가능성 또는 멀티 LLM 전략을 고려하는 것이 장기적인 관점에서 중요합니다.
2.  **보안 취약점 및 라이선스 문제**:
    *   **코드 보안**: AI가 생성한 코드에 잠재적인 보안 취약점(XSS, CSRF, 불완전한 입력 유효성 검사 등 프론트엔드 취약점)이 포함될 가능성에 대한 심층적인 검증 프로세스가 필요합니다. `ESLint + TypeScript strict`는 기본적인 코드 품질을 높이지만, 보안 취약점을 완전히 커버하지 못할 수 있습니다. 추가적인 정적 분석 도구(SAST)나 보안 전문가의 검토가 필요할 수 있습니다.
    *   **저작권/라이선스**: LLM이 학습한 데이터셋의 저작권 문제나 생성된 코드의 라이선스 귀속 문제에 대한 법적/윤리적 검토가 필요합니다. 특히 오픈소스 라이선스 코드가 생성될 경우, 해당 라이선스 조건을 준수해야 합니다.
3.  **데이터 거버넌스 및 학습 데이터 활용**: PRD, 생성된 코드, 평가 데이터, 사용자 피드백 등 시스템 내에서 생성/활용되는 모든 데이터의 소유권, 보관 정책, 재활용 방안에 대한 명확한 정책이 필요합니다. 특히 고객사 PRD에 민감 정보가 포함될 경우, 데이터 익명화/비식별화, 접근 제어 등 보안 및 컴플라이언스 강화가 필수적입니다. 이 데이터는 향후 Generator LLM의 파인튜닝 또는 새로운 LLM 개발의 귀중한 자산이 될 수 있습니다.
4.  **개발자 경험 (DX) 및 유지보수성**: BD팀이 주 사용자이지만, 생성된 프로토타입 코드를 실제 제품 개발 시 참조하거나, 필요에 따라 개발자가 수정해야 할 수도 있습니다. AI가 생성한 코드의 가독성, 유지보수성, 표준 코딩 컨벤션 준수 여부 등 개발자 경험 측면도 고려되면 좋습니다. (예: 생성된 코드에 대한 자동화된 코드 리뷰 시스템 도입)
5.  **경쟁 환경 변화에 대한 대응**: AI 기술은 매우 빠르게 발전하고 있습니다. 경쟁사들이 유사하거나 더 발전된 AI 기반 개발 도구를 출시할 가능성에 대비하여, 지속적인 기술 스택 업데이트 및 기능 고도화 전략이 필요합니다. (예: 3D/VR 프로토타입 생성, 백엔드 로직 자동 생성 확장 등)

### 5. 착수 판단

**착수 판단: Conditional**

PRD는 프로젝트의 목표, 배경, 문제 정의, 기능 범위, 성공 기준 등을 매우 상세하고 명확하게 기술하고 있으며, 특히 "추가 검증 목표", "외부 도구 및 환경 리스크", "오픈 이슈" 섹션에서 PoC의 중요성과 잠재적 리스크를 솔직하게 명시하고 있다는 점에서 매우 긍정적입니다.

그러나, 프로젝트의 성공에 결정적인 영향을 미치는 몇 가지 핵심적인 가정과 기술적 리스크에 대한 **사전 PoC 결과 없이는 착수하기 어렵습니다.**

**다음 조건들이 충족될 경우 착수 가능합니다:**

1.  **핵심 외부 도구 PoC 완료**:
    *   Claude Max CLI `--bare` 모드의 실제 지원 여부, 운영 환경에서의 rate limit, 성능, 안정성, 비용 정책이 PRD의 목표(`$0/회`)에 부합하는지 **성공적으로 검증**되어야 합니다.
    *   Vision API 및 Lighthouse CI의 비용(월 $5 이내), 환경 호환성(Railway Docker), 성능, 안정성이 **성공적으로 검증**되어야 합니다.
2.  **5차원 평가 체계의 신뢰성 검증**:
    *   동일 입력에 대한 5차원 평가 점수의 **재현성(±10점 이내)**이 자동화된 테스트를 통해 **입증**되어야 합니다.
    *   5차원 평가 점수(80점+)와 실제 BD팀 및 고객사의 '고객 데모 가능' 여부 간 **높은 상관관계(예: 0.8 이상)**가 파일럿 PoC를 통해 **확인**되어야 합니다.
3.  **자동 개선 루프의 효율성 검증**:
    *   기존 5종 PRD 중 최소 3종 이상에 대해 자동 개선 루프를 실행하여, **5라운드 이내 80점+ 달성 수렴율이 70% 이상**임이 파일럿 PoC를 통해 **확인**되어야 합니다.
4.  **인력/일정 산정의 현실성 재검토**:
    *   위 PoC 결과를 바탕으로 각 마일스톤별 공수 산정을 재실시하고, **1명 2~3주 일정이 현실적으로 달성 가능한지 재확인**해야 합니다. 만약 PoC 결과에 따라 인력 증원(최소 0.5~1인 추가) 또는 일정 연장(최소 4주)이 필요하다면, 이에 대한 명확한 계획과 경영진의 승인이 있어야 합니다.

이러한 사전 검증을 통해 프로젝트의 리스크를 최소화하고, 성공 가능성을 높인 후 본 프로젝트에 착수하는 것이 바람직합니다.

## 기술 검토 의견: fx-builder-evolution PRD v2

### 1. 기술적 실현 가능성
**부분적 가능성 있으나 중대한 제약 존재**

**긍정적 요소:**
- 5차원 평가 체계는 합리적이며 기존 기술(빌드 성공, ESLint, 정적 분석)과의 통합은 실현 가능
- CLI 모드 통합으로 API 비용 제거는 명확한 비용 절감 효과 기대

**주요 기술적 제약/병목:**
- **Claude CLI `--bare` 모드의 운영 안정성**: subprocess 호출 시 세션 관리, timeout 처리, 메모리 누수 리스크가 높음. 장시간 실행 시 hang 가능성(현재 SKIP_CLI=true 상태)이 검증되지 않음.
- **Vision API 의존도**: 스크린샷 평가를 위한 Vision API는 별도 과금이며, 품질 평가 품질이 LLM의 주관적 판단에 크게 의존. 재현성 확보가 매우 어려움.
- **Lighthouse CI 환경 호환성**: Railway Docker에서 headless Chrome 실행 가능성 미검증. 성능 측정 시 변동성(variance)이 커 신뢰성 있는 기준점 설정이 필요.
- **평가 체계의 재현성**: 5개 차원 중 3개(UI, PRD 반영도, 기능 동작)가 LLM/Vision에 의존 → 동일 입력에 대해 ±10점 편차 보장이 기술적으로 매우 어려움.

### 2. 아키텍처 적합성
**단일 실패점(Single Point of Failure)이 과다하고 복잡도가 높음**

**문제점:**
- **듀얼 모드 아키텍처의 복잡성**: CLI 모드 primary, API 모드 fallback으로 설계했으나, 두 모드의 출력 품질/특성이 상이할 경우 비교/통합이 매우 어려움.
- **외부 서비스 의존도 과중**: Claude CLI, Vision API, Lighthouse, Cloudflare Pages 등 4개 이상의 외부 서비스에 의존. 하나의 장애가 전체 파이프라인 중단으로 이어짐.
- **상태 관리 부재**: 5라운드 자동 개선 과정에서의 상태 저장/복구 메커니즘이 추상적. 중간 실패 시 롤백이나 재개가 불가능할 가능성 높음.

**개선 제안:**
- **평가 체계 단순화**: 초기 MVP에서는 LLM 의존 평가를 최소화(예: PRD 반영도는 키워드 매칭으로 대체)하고, 점진적으로 정교화.
- **장애 격리 설계**: 각 평가 차원을 독립적인 마이크로태스크로 분리, 하나의 차원 실패 시 기본값 부여로 전체 평가 중단 방지.
- **결정론적 로깅 강화**: 입력→출력→점수의 완전한 추적 체인을 로깅하여 디버깅 가능성 확보.

### 3. 구현 복잡도
**현재 일정(2-3주)으로는 MVP 달성 불가능할 것으로 예상**

**과소평가된 복잡도:**
- **동시성 및 상태 관리**: 여러 프로토타입이 동시에 생성/평가/개선될 경우 리소스 경합(특히 CLI의 rate limit)과 상태 충돌이 발생.
- **에지 케이스 처리**: 
  - 빌드는 성공했으나 런타임 오류(예: 무한 루프)가 있는 경우
  - Vision API가 스크린샷에서 텍스트를 오인식하는 경우
  - CLI 출력이 부분적이거나 손상된 경우
- **데이터 일관성**: 라운드별 점수, 개선 피드백, 최종 결과의 관계를 보장하는 트랜잭션 메커니즘이 설계되지 않음.

**추가 개발 요소:**
- CLI 세션 관리 및 health check
- 평가 결과 캐싱 및 재현성 보장 메커니즘
- 외부 서비스 장애 시 fallback/retry 정책

### 4. 기술 리스크
**상위 3개 리스크:**

1. **Claude CLI 운영 불가능성 (가장 높은 리스크)**
   - `--bare` 모드의 실제 가용성과 안정성이 검증되지 않음.
   - 구독 계정의 rate limit(분당/시간당 요청 수)이 공개되지 않아 대량 생성 시 차단 가능성.

2. **평가 체계 신뢰성 부재**
   - LLM 기반 평가(UI, PRD 반영도)는 높은 변동성(variance)을 가짐.
   - "고객 데모 가능" 80점 기준이 실제 사용자 피드백과 상관관계가 낮을 경우, 전체 시스템의 유용성 무효화.

3. **외부 서비스 비용 누출**
   - Vision API 비용이 예상치를 초과할 가능성(월 $5 한계는 매우 낮은 할당량).
   - Lighthouse CI의 지속적 실행으로 인한 Railway 리소스 초과로 추가 비용 발생 가능.

### 5. 착수 판단
**착수 판단: Conditional**

**착수 조건:**
1. **사전 PoC 필수 수행**: 다음 3가지를 반드시 검증해야 함.
   - Claude CLI `--bare` 모드로 50회 이상 연속 코드 생성 테스트 (성공률 95% 이상, 평균 응답 시간 < 2분)
   - 동일 코드에 대한 5차원 평가 10회 반복 실행 (표준편차가 전체 점수의 5% 이내)
   - Vision API + Lighthouse 실제 비용 측정 (프로토타입 1개 평가 시 < $0.10)

2. **범위 조정**: MVP를 다음과 같이 축소.
   - 5차원 → 3차원 평가로 단순화 (빌드 성공, 코드 품질, 정적 분석 기반 기능 점검)
   - 자동 개선 루프 최대 라운드 3회로 제한
   - 품질 대시보드는 기본 테이블 형태로 단순 구현

3. **일정/인력 재조정**: 
   - 최소 4주 일정으로 확장
   - 외부 서비스 통합(CLI, Vision, Lighthouse) 담당 인력 0.5명 추가

**결론:** 기술적 비전은 타당하나, 현재 PRD의 범위와 일정은 현실성 없음. 핵심 리스크 요소에 대한 사전 검증 없이는 프로젝트 실패 가능성이 매우 높습니다. PoC를 통한 검증 후 범위와 일정을 조정하는 접근이 필수적입니다.
