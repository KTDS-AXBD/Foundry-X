### 1. 논리적 완결성

  - 품질을 "정량화 → 자동 개선 → 비용 절감"으로 명확히 문제-해결-성과로 연결한 점은 좋습니다.

- **개선 포인트**  
  - **문제-해결 매핑 부족:**  
    - "품질 점수 기반 자동 개선"이 실제로 어느 정도 효과가 있을지, 어떤 방식으로 개선되는지 구체적 메커니즘(예: 개선 지시의 프롬프트가 LLM에 어떻게 전달되고, 그 피드백이 어떻게 반영되는지)에 대한 설명이 부족합니다.  
    - 5차원 스코어링이 실제로 "고객 데모 가능"을 보장한다고 볼 수 있을지, 각 차원의 임계점(예: UI 0.7, PRD 0.8 등)이 달성되면 진짜 ‘데모 가능’인지에 대한 설계적 근거가 약합니다.
  - **성공 기준의 구체성 부족:**  
    - "고객 데모 가능 비율"을 품질 점수 80점+로 대체하고 있는데, 실제 사용자(고객/B2B 영업 담당)의 주관적 평가와 정량 점수와의 상관관계가 불명확합니다.  
    - "자동 개선 수렴율"이 무엇을 의미하는지, 목표(70% 이상)가 현실적으로 왜 가능한지 논리적 근거(파일럿 테스트 결과 등)가 필요합니다.
  - **비용 절감 효과 검증:**  
    - CLI 기반으로 완전히 전환할 경우 실제로 API 비용이 0이 맞는지(숨은 비용, 예: Vision API 등) 언급이 미약합니다.

### 3. 누락된 핵심 요소

- **사용자 피드백 루프**  
  - 품질 점수의 기준이 LLM/Vision 기반 자동화에 치우쳐 있습니다.  
    - 실제 BD팀/고객이 보는 "데모 가능"의 기준과 gap이 있을 수 있으므로, 사용자(내부/외부) 평가 피드백을 최초 1~2회 라운드에 반영하는 프로세스가 필요합니다.
- **장애/실패 대응 시나리오**  
  - CLI/WSL/tmux 등에서 hang, 세션 drop, Vision API timeout 등 장애 발생 시 fallback, 재시도, 알림(예: Slack, Email) 등 운영 대응 방안 미정의.
- **품질 데이터 관리/로그 정책**  
  - 각 라운드별 점수, 개선 내역, 실패 사유 등 결과물의 저장·관리·분석 체계가 명확하지 않습니다.
- **보안/운영 관리**  
  - CLI 구독 키, API 키, 배포 URL 등 민감 정보의 저장·노출 방지 대책 언급이 필요합니다.
- **테스트/QA 프로세스**  
  - 자동화된 품질 평가의 신뢰도를 높이기 위해, 샘플 PRD에 대한 수작업 검증(휴먼 QA) 병행 계획이 있어야 합니다.

*   **유사 서비스:**

### 2. 사실 확인 (Assumptions & Verifications)

PRD에 포함된 주요 가정 및 검증이 필요한 사항은 다음과 같습니다.

*   **Claude Max CLI의 `--bare` 모드 기능 및 Rate Limit:**
    *   **가정:** Claude Max 구독에 CLI 사용이 포함되며, `subprocess --bare` 모드에서 API 비용 없이 무제한 또는 충분한 rate limit으로 코드 생성이 가능할 것이라는 가정.
    *   **검증 필요:** Anthropic의 공식 문서 또는 지원 채널을 통해 **1) `--bare` 모드의 존재 여부, 2) 구독 플랜에 따른 비용 정책 (정말로 $0인지), 3) 분당/시간당/토큰당 rate limit이 실제 운영에 충분한지**를 반드시 확인해야 합니다. 이 부분이 프로젝트의 핵심 전제이므로 가장 중요합니다.
*   **Vision API (스크린샷 평가) 비용:**
    *   **가정:** Discriminator용 Vision API 호출 비용이 월 $5 이내로 유지될 것이라는 가정.
    *   **검증 필요:** Vision API의 이미지 처리 단위(예: 이미지당, 픽셀당) 및 가격 정책을 기반으로, 예상되는 스크린샷 평가 횟수(5종 PRD × N라운드)에 대한 **정확한 비용 시뮬레이션**이 필요합니다. 예상보다 높을 경우, 다른 경량화된 UI 평가 방식(예: DOM 구조 분석, CSS 유효성 검사)을 고려하거나, Vision API 호출 빈도를 최적화해야 합니다.
*   **Lighthouse CI의 Railway Docker 환경 실행 가능성:**
    *   **가정:** Railway Docker 환경에서 headless Chrome을 사용하여 Lighthouse CI를 실행할 수 있다는 가정.
    *   **검증 필요:** Docker 환경에서 `puppeteer` 또는 `playwright`와 같은 headless 브라우저 구동 라이브러리의 **호환성 및 자원 소모량**을 테스트해야 합니다. 일부 클라우드 환경에서는 headless 브라우저 실행에 제약이 있거나, 예상보다 많은 CPU/메모리 자원을 요구할 수 있습니다.
*   **"1명 (AI 에이전트 활용)" 인력 및 2~3주 일정의 현실성:**
    *   **가정:** 1명의 인력이 AI 에이전트의 도움을 받아 2~3주 내에 5차원 스코어링 엔진, CLI 통합, 자동 개선 루프, 품질 대시보드 구축이라는 광범위한 목표를 달성할 수 있다는 가정.
    *   **검증 필요:** 이는 **매우 공격적인 일정 및 인력 계획**입니다. 특히, Vision API 연동, LLM 기반 PRD 반영도 평가 로직 구현, CLI 안정화, 복잡한 자동 개선 루프 설계 및 디버깅, 그리고 웹 대시보드 구축까지 고려하면 1인 개발로는 **최소 2~3개월**은 소요될 것으로 보입니다. 'AI 에이전트 활용'이 어느 정도의 생산성 향상을 의미하는지 구체적인 근거가 필요합니다. 이 부분은 프로젝트의 **가장 큰 위험 요소**입니다.
*   **5차원 평가의 재현성 및 신뢰도:**
    *   **가정:** LLM 기반의 UI 완성도, 기능 동작, PRD 반영도 평가가 ±10점 이내의 일관성을 가질 것이라는 가정.
    *   **검증 필요:** LLM의 비결정성(non-determinism)으로 인해 동일한 코드에 대한 평가가 달라질 수 있습니다. **다양한 프롬프트 엔지니어링 기법과 평가 기준의 명확화**를 통해 재현성을 확보해야 합니다. 초기 M1 단계에서 재현성 테스트를 철저히 수행하고, 필요시 사람의 검증을 보조적으로 활용하는 방안도 고려해야 합니다.
*   **WSL tmux 세션의 안정성:**
    *   **가정:** CLI 모드에서 WSL tmux 세션이 장시간 안정적으로 유지될 것이라는 가정.
    *   **검증 필요:** 장시간 실행되는 백그라운드 프로세스의 경우, WSL 환경에서 리소스 관리나 세션 유지에 문제가 발생할 수 있습니다. **스트레스 테스트 및 모니터링**을 통해 안정성을 확인해야 합니다.

### 3. 사용자 관점 (User Adoption)

목표 사용자인 BD팀은 이 솔루션을 적극적으로 받아들일 가능성이 높습니다. 현재 "빌드는 되지만 조잡한" 프로토타입으로 인해 고객 데모가 불가능하다는 문제 인식이 명확하고, 이 솔루션은 그들의 핵심 니즈인 "높은 품질의 프로토타입을 빠르게 생성"하는 것을 직접적으로 해결해주기 때문입니다.

**도입 장벽:**

*   **실제 품질과 점수의 괴리:** 가장 큰 장벽은 "80점 이상"이라는 점수가 BD팀의 눈높이에서 "고객 데모 가능한 수준"과 일치하지 않을 때 발생할 수 있습니다. 점수는 높지만 디자인이 여전히 어색하거나, 중요한 인터랙션이 누락되었다고 느껴진다면 신뢰를 잃을 수 있습니다.
    *   **해결 방안:** M1 단계에서 BD팀과 함께 **"80점+" 프로토타입의 실제 시각적/기능적 기준**을 명확히 정의하고, 스코어링 로직이 이를 잘 반영하는지 지속적으로 검증해야 합니다. 초기에는 BD팀의 수동 평가를 통해 스코어링 모델을 튜닝하는 과정이 필요할 수 있습니다.
*   **개선 루프의 투명성 및 제어:** 자동 개선 루프가 진행될 때, BD팀이 "왜 이렇게 개선되었는지", "어떤 피드백이 전달되었는지"를 명확히 이해하지 못하면 답답함을 느낄 수 있습니다. 또한, 자동 개선이 특정 부분에서 막히거나 원치 않는 방향으로 진행될 때, 사용자가 개입하여 피드백을 수동으로 조정할 수 있는 기능이 없다면 불만이 생길 수 있습니다.
    *   **해결 방안:** M4 품질 대시보드에서 **각 라운드별 자동 생성된 타겟 피드백과 그 결과**를 시각적으로 보여주고, 필요시 사용자가 추가 피드백을 입력하여 다음 라운드에 반영할 수 있는 **'수동 피드백 주입' 기능**을 고려해볼 수 있습니다 (이는 Should Have 또는 P2로 시작하여 점진적으로 추가).
*   **속도 및 반응성:** 프로토타입 생성 및 개선에 소요되는 시간이 너무 길면, BD팀은 차라리 수동으로 코드를 수정하는 것이 빠르다고 느낄 수 있습니다.
    *   **해결 방안:** 각 라운드의 **최대 소요 시간을 명확히 제시**하고, 이 시간 내에 수렴하도록 최적화해야 합니다. 특히 CLI 모드의 응답 속도 및 WSL 환경의 성능 최적화가 중요합니다.

### 4. 누락된 시장 관점 (Missing Market Perspectives)

*   **다양한 LLM 지원 및 유연성:** 현재 Claude Max CLI에 강력하게 의존하고 있습니다. Anthropic의 정책 변경이나, 다른 LLM(예: GPT-5, Gemini Ultra)이 훨씬 더 뛰어난 코드 생성 능력을 보일 경우, 이 시스템이 유연하게 다른 LLM으로 전환할 수 있는 아키텍처적 고려가 부족합니다. (현재는 Claude Max 비용 최적화가 우선이므로 이해는 되지만, 장기적인 관점에서는 중요)
*   **보안 및 취약점 분석 (프로토타입 코드):** 생성된 프로토타입 코드가 비록 데모용일지라도, 잠재적인 보안 취약점(예: XSS, 인젝션 공격에 취약한 코드 패턴)을 포함할 수 있습니다. 이를 감지하고 개선하는 기능은 현재 코드 품질 차원(ESLint, TypeScript)에서 다루기 어렵습니다.
    *   **고려 사항:** 향후 '코드 품질' 차원에 SAST(정적 애플리케이션 보안 테스트) 도구를 통합하여 기본적인 보안 취약점을 검사하는 기능을 추가하는 것을 고려할 수 있습니다.
*   **환경 변화 대응 (프레임워크/라이브러리 업데이트):** 프론트엔드 생태계는 빠르게 변화합니다. React, Vite, shadcn/ui, Tailwind 등의 버전 업데이트에 따라 생성된 코드가 더 이상 유효하지 않거나, 빌드 문제가 발생할 수 있습니다.
    *   **고려 사항:** 이러한 환경 변화에 대응하여 Generator의 프롬프트 및 내부 로직을 주기적으로 업데이트하고, 주요 라이브러리 버전업에 대한 테스트 전략이 필요합니다.
*   **LLM의 편향성 및 윤리적 측면:** LLM은 학습 데이터의 편향성을 반영하여 특정 디자인 패턴, 기능 구현 방식, 심지어는 비즈니스 로직에 편향된 결과를 생성할 수 있습니다. 이는 "UI 완성도"나 "PRD 반영도" 평가 시에도 영향을 미칠 수 있습니다.
    *   **고려 사항:** 생성된 프로토타입이 특정 사용자 그룹에게만 친숙하거나, 접근성(Accessibility) 측면에서 문제가 발생하지 않도록, `Lighthouse` 점수(P1)를 P0로 격상하거나, 'UI 완성도' 평가에 접근성 가이드라인을 포함하는 것을 고려할 수 있습니다.

### 5. 착수 판단: Conditional (조건부 착수)

이 PRD는 현재 시장의 니즈와 트렌드에 매우 잘 부합하며, 제시된 문제 해결 방안과 기술 설계는 매우 혁신적이고 매력적입니다. 그러나 몇 가지 **핵심적인 '사실 확인'과 '제약 조건'에 대한 해결 또는 현실적인 조정**이 선행되어야 합니다.

2.  **"1명 (AI 에이전트 활용)" 인력 및 2~3주 일정의 현실적인 재조정:** 현재 제시된 인력과 일정은 **매우 비현실적**입니다. M1(5차원 스코어링)만 하더라도 Vision API, LLM 평가, 정적 분석 통합 등 상당한 개발 공수가 필요합니다. 최소한 M1~M4까지의 각 마일스톤에 대한 **세부 공수 산정 및 리스크 평가를 재수행**하고, 그에 따라 **일정을 현실적으로 재조정**하거나, **추가 인력 투입**을 고려해야 합니다.
3.  **5차원 평가의 재현성 확보 계획 구체화:** LLM 기반 평가의 비결정성 문제를 해결하기 위한 구체적인 **프롬프트 엔지니어링 전략, 평가 기준 명확화, 그리고 재현성 테스트 계획**이 M1 착수 전에 더욱 상세하게 수립되어야 합니다.

이러한 조건들이 해결되거나 현실적인 계획으로 조정된다면, 이 프로젝트는 BD팀의 효율성을 혁신하고 회사의 AI 기반 개발 역량을 한 단계 끌어올릴 잠재력이 충분합니다.

## 기술 검토 의견

### 1. 기술적 실현 가능성

**주요 우려사항:**

1. **Claude CLI `--bare` 모드의 운영 리스크**
   - 문서에서 언급된 rate limit 확인이 미흡합니다. 구독 기반 CLI도 사용량 제한이 있을 가능성이 높으며, batch 처리 시 throttling 문제 발생 가능
   - `subprocess` 실행 환경에서의 안정성: 장시간 실행 시 메모리 누수, 프로세스 고아화, timeout 관리 미흡
   - WSL/tmux 의존성은 Railway Docker 환경과의 호환성 문제 야기 가능

2. **5차원 평가의 재현성과 객관성**
   - Vision API 기반 UI 평가는 주관적 요소가 크며, 동일 코드에 대해 일관된 점수 보장 어려움
   - LLM 기반 PRD 반영도 평가는 프롬프트 엔지니어링에 민감하며, 평가 비용 누적 가능성
   - 각 차원별 가중치(20%, 25% 등)가 임의적이며, 실제 고객 데모 품질과의 상관관계 검증 필요

3. **자동 개선 루프의 수렴 보장 불가**
   - "가장 낮은 차원 타겟 피드백" 접근법은 로컬 최적화(local optimization) 문제 야기
   - 5라운드 제한 내 80점 달성은 과도한 낙관적 가정
   - 개선 프롬프트의 효과 측정 메커니즘이 부재

### 2. 아키텍처 적합성

**개선 필요 사항:**

1. **듀얼 모드 아키텍처의 복잡성**
   - CLI/API 전환 로직이 운영 복잡도 증가시킴
   - **제안**: CLI 모드를 primary로 고정하고, API 모드는 완전 별도 파이프라인으로 분리
   - 상태 관리와 에러 핸들링이 모드별로 분기되어 유지보수 어려움

2. **평가 파이프라인의 단일 포인트 실패**
   - 5개 평가 차원이 직렬 실행되어 하나의 실패가 전체 평가 실패로 이어짐
   - **제안**: 평가 차원별 독립적 실행과 결과 병합으로 내결함성 향상

3. **데이터 흐름의 투명성 부족**
   - 라운드별 개선 결정 과정이 블랙박스화되어 디버깅 어려움
   - **제안**: 결정 로그(deterministic logging)와 추적 가능한 피드백 체인 구현

### 3. 구현 복잡도

**과소평가된 요소:**

1. **Vision API 평가의 실제 비용**
   - 스크린샷당 $0.0025~$0.01 (해상도에 따라) × 5차원 평가 × 여러 라운드 = 월 $5 예산 초과 가능성 높음
