# Foundry-X 커스터마이징 범위 정의서

## 1. 목적

이 문서는 Foundry-X를 고객사에 배포할 때 **커스터마이징 가능 범위와 책임 주체**를 정의해요.
PRD v8 Conditional 조건 #3 (Q12: 고객사별 커스터마이징 범위 정의)을 해소하기 위한 공식 정책 문서예요.

## 2. 5-레이어 커스터마이징 모델

### Layer 1: 배포 환경 (Deployment)

| 항목 | 설명 | 커스터마이징 방식 |
|------|------|-------------------|
| 클라우드 플랫폼 | Cloudflare Workers / Azure Functions / On-premise | 어댑터 교체 (`src/azure.ts`, `src/index.ts`) |
| DB 엔진 | Cloudflare D1 (SQLite) / Azure SQL / PostgreSQL | 마이그레이션 스크립트 변환 |
| CDN/정적 호스팅 | Cloudflare Pages / Azure Static Web Apps / Nginx | 빌드 출력 + 환경변수 |
| 환경 변수 | `.env` / `wrangler.toml` / Azure App Configuration | 환경별 설정 파일 |

**책임 주체**: SI 파트너 (배포 가이드 제공, 내부 개발자 지원)

### Layer 2: 인증/SSO (Authentication)

| 항목 | 설명 | 커스터마이징 방식 |
|------|------|-------------------|
| 인증 방식 | JWT / SAML 2.0 / OIDC | 인증 미들웨어 설정 변경 |
| IdP 연동 | Google OAuth / Azure AD / Okta / 자체 LDAP | OAuth 프로바이더 설정 |
| SSO Hub | 기존 SSO Hub Token 연동 | Hub Token 엔드포인트 설정 |
| RBAC 역할 | admin / user / guest / 커스텀 역할 | `CustomRoleManager` 서비스 활용 |

**책임 주체**: SI 파트너 (고객사 IdP 정보 필요, 내부 개발자 기술 지원)

### Layer 3: DB 스키마 (Data Schema)

| 항목 | 설명 | 커스터마이징 방식 |
|------|------|-------------------|
| 커스텀 필드 | 기존 테이블에 고객사 전용 컬럼 추가 | 마이그레이션 스크립트 추가 |
| 확장 테이블 | 테넌트별 전용 테이블 생성 | `tenant_id` FK + 네이밍 컨벤션 |
| 데이터 보존 정책 | 로그/감사 데이터 보존 기간 설정 | 테넌트 설정 테이블 |
| 초기 데이터 | 시드 데이터 (SR 유형, 워크플로우 템플릿 등) | 시드 스크립트 커스터마이징 |

**책임 주체**: 내부 개발자 (스키마 변경은 리뷰 필수, SI 요청 접수)

### Layer 4: UI 테마/레이아웃 (UI Customization)

| 항목 | 설명 | 커스터마이징 방식 |
|------|------|-------------------|
| 브랜딩 | 로고, 파비콘, 앱 이름 | 환경변수 + 에셋 교체 |
| 색상 테마 | Primary/Secondary/Accent 색상 | CSS 변수 (`--primary`, `--accent` 등) |
| 레이아웃 | 사이드바 메뉴 구성, 대시보드 위젯 배치 | 설정 JSON + 컴포넌트 슬롯 |
| 언어/로케일 | 한국어 / 영어 / 일본어 | i18n 리소스 파일 추가 |

**책임 주체**: 고객사/SI 파트너 (CSS 변수는 비개발자도 변경 가능)

### Layer 5: 워크플로우/에이전트 (Business Logic)

| 항목 | 설명 | 커스터마이징 방식 |
|------|------|-------------------|
| 에이전트 역할 | 기본 6종 + 커스텀 에이전트 | `CustomRoleManager` API |
| 에이전트 프롬프트 | 시스템 프롬프트 오버라이드 | `systemPromptOverride` 설정 |
| 워크플로우 DAG | SR 유형별 에이전트 실행 순서 | `SrWorkflowMapper` 설정 |
| SR 분류 규칙 | SR 유형 분류 규칙 추가/수정 | `SrClassifier` 규칙 DB |
| 모델 라우팅 | 태스크별 AI 모델 선택 규칙 | `ModelRouter` 라우팅 테이블 |

**책임 주체**: 내부 개발자 (워크플로우 변경은 QA 검증 필수)

## 3. Tier 매트릭스 (고객 제안서용)

| 기능 | Standard | Professional | Enterprise |
|------|:--------:|:------------:|:----------:|
| L1: 배포 환경 선택 | ✅ | ✅ | ✅ |
| L2: SSO/IdP 연동 | ✅ (1개 IdP) | ✅ (3개 IdP) | ✅ (무제한) |
| L3: 커스텀 DB 필드 | - | ✅ (10필드) | ✅ (무제한) |
| L4: UI 브랜딩 | 로고만 | 테마 전체 | 테마 + 레이아웃 |
| L5: 에이전트 커스터마이징 | - | 프롬프트만 | 전체 (역할+DAG) |
| 플러그인 설치 | - | - | ✅ |
| 기술 지원 | 이메일 | 이메일+채팅 | 전담 엔지니어 |
| 초기 세팅 지원 | 가이드 문서 | 원격 지원 | 온사이트 지원 |
| SLA | 99.0% | 99.5% | 99.9% |

## 4. 커스터마이징 프로세스

```
1. 요청 접수  →  고객사가 커스터마이징 요청서 제출 (표준 양식)
2. 범위 판정  →  PM이 레이어/Tier 기준으로 가능 여부 판정
3. 견적 산출  →  레이어별 공수 산출 (L1~L2: 1~3일, L3~L4: 3~5일, L5: 5~10일)
4. 구현      →  책임 주체가 구현 + 테스트
5. 검증      →  QA + 고객 확인
6. 배포      →  고객사 환경에 배포
```

## 5. 커스터마이징 제한 사항 (Out of Scope)

- **코어 엔진 변경 불가**: SDD Triangle, Plumb 엔진, 에이전트 오케스트레이터 핵심 로직
- **보안 모듈 변경 불가**: 인증/인가 핵심 로직, 감사 로그 비활성화
- **무제한 확장 불가**: 플러그인이 코어 서비스 DB를 직접 변경하는 것은 차단
- **테스트 우회 불가**: 커스터마이징 코드도 PDCA 검증 의무 적용
- **라이선스 위반**: 오픈소스 라이선스 오염이 발생하는 커스터마이징 거부
