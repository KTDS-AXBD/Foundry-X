# AI 생성 코드 법적/윤리적 가이드라인

## 1. 목적

이 문서는 Foundry-X 플랫폼에서 AI 에이전트가 생성하는 코드, 테스트, 문서에 대한 **법적 권리, 윤리적 책임, 품질 관리** 정책을 정의해요.
PRD v8 Conditional 조건 #5 (Q13: AI 생성 코드 법적/윤리적 정책 수립)을 해소하기 위한 공식 정책이에요.

## 2. 저작권 귀속 정책

### 2.1 기본 원칙

| 상황 | 저작권 귀속 | 근거 |
|------|-------------|------|
| 내부 개발 (KT DS 자체) | KT DS BD팀 | 업무상 저작물 (저작권법 §9) |
| 고객사 프로젝트 (SI) | 계약에 따름 | SI 계약서 저작권 조항 |
| 오픈소스 기여 | 해당 프로젝트 라이선스 | 기여자 라이선스 동의(CLA) |

### 2.2 AI 생성물 특수 규정

- AI가 생성한 코드는 **도구로 생성한 저작물**로 취급 (AI 자체에 저작권 없음)
- 프롬프트 작성자(사람)가 창작적 기여를 한 경우, 해당 사람이 저작자
- AI 생성 코드의 저작권 분쟁 방지를 위해 **감사 로그에 생성 이력 기록** 의무

### 2.3 고객사 납품 시

- 계약서에 "AI 도구 활용" 명시 의무
- 납품 코드에 AI 생성 비율 보고 (감사 로그 기반)
- 고객사 요청 시 AI 미사용 옵션 제공 (수동 개발 전환)

## 3. 오픈소스 라이선스 관리

### 3.1 라이선스 오염 방지 절차

```
AI 코드 생성
  │
  ▼
[자동 검사] 라이선스 스캐너 (향후 자동화)
  │
  ├─ GPL/AGPL 감지 → ❌ 차단 + 알림
  ├─ MIT/Apache/BSD → ✅ 허용 (출처 표기)
  └─ 불명확 → ⚠️ 사람 검토 필요
  │
  ▼
[사람 검토] 코드 리뷰 시 라이선스 확인
  │
  ▼
승인/거부
```

### 3.2 금지 패턴

- AI에게 "특정 오픈소스 프로젝트의 코드를 복사해달라" 요청 금지
- 라이선스 불명확한 코드 스니펫을 프롬프트에 포함하여 AI에 전달 금지
- GPL 계열 코드와 혼합 가능성이 있는 프롬프트 패턴 주의

### 3.3 허용 라이선스 목록

| 라이선스 | 허용 여부 | 조건 |
|----------|:---------:|------|
| MIT | ✅ | 저작권 고지 유지 |
| Apache 2.0 | ✅ | NOTICE 파일 유지 |
| BSD 2/3-Clause | ✅ | 저작권 고지 유지 |
| ISC | ✅ | 저작권 고지 유지 |
| GPL 2.0/3.0 | ❌ | 전염성 라이선스 |
| AGPL | ❌ | 네트워크 전염성 |
| SSPL | ❌ | 서비스 전염성 |
| 라이선스 없음 | ⚠️ | 사람 검토 후 결정 |

## 4. 코드 리뷰 의무

### 4.1 사람 검토 의무 범위

| AI 생성물 유형 | 자동 커밋 | 사람 검토 필요 |
|----------------|:---------:|:--------------:|
| 코드 (production) | ❌ | ✅ 필수 |
| 테스트 코드 | ❌ | ✅ 필수 |
| 문서/주석 | ❌ | ✅ 권장 |
| 설정 파일 | ❌ | ✅ 필수 |
| 프롬프트/시스템 프롬프트 | N/A | ✅ 필수 |

> **핵심 원칙**: Foundry-X의 "자동 커밋 절대 금지" 원칙은 AI 생성물 전체에 확장 적용.

### 4.2 리뷰 체크리스트

- [ ] AI 생성 코드가 의도한 기능을 정확히 수행하는가?
- [ ] 보안 취약점 (XSS, SQL Injection, CSRF 등)이 없는가?
- [ ] 하드코딩된 비밀번호, API 키, 개인정보가 없는가?
- [ ] 오픈소스 라이선스 오염 가능성이 없는가?
- [ ] 기존 코드 컨벤션과 일관성이 유지되는가?
- [ ] 테스트가 충분히 작성되었는가?

## 5. 책임 분장

### 5.1 AI 생성 코드로 인한 장애 시

| 역할 | 책임 |
|------|------|
| AI 에이전트 | 책임 없음 (도구) |
| 프롬프트 작성자 | 프롬프트 품질 + 리뷰 승인 책임 |
| 코드 리뷰어 | 리뷰 통과시킨 코드에 대한 공동 책임 |
| PM | 전체 프로세스 관리 책임 |
| 운영팀 | 장애 대응 + 롤백 책임 |

### 5.2 고객사 환경 장애 시

- 1차: 운영팀 장애 대응 (SLA 기준)
- 2차: 내부 개발팀 근본 원인 분석
- 3차: AI 생성 코드 여부 감사 로그 확인
- 4차: 재발 방지 대책 수립 (프롬프트 개선, 리뷰 강화 등)

## 6. 감사 로그 관리

### 6.1 기록 항목

모든 AI 생성 활동은 다음 정보를 감사 로그에 기록:

- 이벤트 유형 (코드 생성, 리뷰, 테스트 생성 등)
- 사용 모델 (claude-sonnet-4-6, gpt-4 등)
- 프롬프트 해시 (SHA-256, 원본 미저장)
- 입력 데이터 분류 등급
- 출력 유형 (코드, 테스트, 문서 등)
- 승인자 ID + 승인 시간

### 6.2 보존 정책

- **기본 보존 기간**: 1년
- **고객사 프로젝트**: 계약 종료 후 3년 (또는 계약서 명시 기간)
- **삭제 불가**: 감사 로그는 soft delete만 허용 (법적 요구)
- **프롬프트 원본 미저장**: 개인정보 보호를 위해 해시만 저장

## 7. 윤리적 사용 원칙

1. **투명성**: AI 사용 사실을 이해관계자에게 투명하게 공개
2. **인간 주도**: AI는 보조 도구, 최종 의사결정은 반드시 사람
3. **품질 동등성**: AI 생성물도 수동 작성물과 동일한 품질 기준 적용
4. **편향 주의**: AI 생성물의 편향(기술적, 문화적) 가능성 인식 및 검토
5. **책임 명확화**: AI 생성물에 대한 책임은 사용자(사람)에게 귀속
