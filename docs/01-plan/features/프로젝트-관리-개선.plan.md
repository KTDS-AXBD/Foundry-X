| **Solution** | Git MD 문서(SSOT) + GitHub Projects 보드(뷰/추적) 복합 체계 구축. 표준 문서 구조·템플릿·리뷰 프로세스 정립 |

## 1. Overview

### 1.1 Purpose

AX BD팀은 현재 프로젝트 관리에 두 가지 핵심 공백이 있어요:

1. **WBS/일정 부재**: 작업 분해와 일정 추적이 체계적이지 않아 진행 상태 파악이 어려움
2. **팀 리뷰 프로세스 부재**: 산출물(문서, 코드) 리뷰·승인 절차가 없거나 비공식적

이 Plan은 두 공백을 해소하기 위한 **팀 수준 프로젝트 관리 체계**를 설계해요.

### 1.2 Background

- Foundry-X에서 SPEC.md + MEMORY.md + PDCA 프로세스로 관리 체계를 운영 중
- 하지만 이는 단일 프로젝트·단일 개발자 중심 — 팀 협업·리뷰 메커니즘이 부족
- AX BD팀에는 Foundry-X 외 다수 프로젝트가 존재 (KTDS-AXBD org)
- GitHub KTDS-AXBD org이 이미 존재하므로 GitHub Projects 활용 가능

### 1.3 Related Documents

- SPEC: [[FX-SPEC-001]] (`SPEC.md`)
- PRD: [[FX-SPEC-PRD-V4]] (`docs/specs/prd-v4.md`)
- 문서 관리 표준: `~/.claude/standards/doc-governance.md`
- 요구사항 관리 표준: `~/.claude/standards/requirements-governance.md`

## 2. Scope

### 2.1 In Scope

- [ ] **현행 점검**: 각 프로젝트별 BluePrint/WBS/요구사항/산출물 현황 조사
- [ ] **문서 체계 표준화**: 프로젝트별 표준 문서 구조 + 템플릿 정의
- [ ] **WBS/일정 관리**: GitHub Projects 보드 + 타임라인 뷰 설정
- [ ] **리뷰 프로세스**: PR 기반 문서·코드 리뷰 워크플로우 정립
- [ ] **현행화 가이드**: 기존 산출물을 새 체계로 마이그레이션하는 가이드

### 2.2 Out of Scope

- 외부 도구 도입 (Notion, Jira 등) — Git + GitHub으로 통일
- 자동화 도구 개발 (CLI/봇) — Phase 2에서 Foundry-X가 담당
- 조직 구조 변경, 인사 관련 프로세스
- CI/CD 파이프라인 변경

## 4. 개선 방안 (To-Be)

### 4.1 문서 체계 표준 (Git MD — SSOT)

각 프로젝트 리포에 통일된 문서 구조를 적용:

```
{project}/
├── CLAUDE.md                    # 프로젝트 개요 + AI 협업 가이드
├── SPEC.md                      # 프로젝트 사양 (요구사항, F-items, Tech Debt)
├── CHANGELOG.md                 # 변경 이력
├── docs/
│   ├── INDEX.md                 # 문서 인덱스
│   ├── blueprint/               # BluePrint (아키텍처, 기술 스택, 시스템 구성)
│   │   └── overview.md
│   ├── specs/                   # PRD, 기능 명세
│   ├── 01-plan/                 # PDCA Plan 문서
│   │   └── features/
│   ├── 02-design/               # PDCA Design 문서
│   │   └── features/
│   ├── 03-analysis/             # PDCA 분석 결과
│   ├── 04-report/               # PDCA 완료 보고서
│   │   └── features/
│   ├── review/                  # 리뷰 기록
│   └── archive/                 # 아카이브
└── .github/
    └── PULL_REQUEST_TEMPLATE.md # PR 템플릿 (리뷰 체크리스트 포함)
```

### 4.2 WBS/일정 관리 (GitHub Projects)

| 구성요소 | 도구 | 설명 |
|---------|------|------|
| **팀 보드** | GitHub Projects (org-level) | AX BD팀 전체 프로젝트 Overview |
| **프로젝트 보드** | GitHub Projects (repo-level) | 개별 프로젝트 WBS |
| **타임라인 뷰** | GitHub Projects Timeline | 일정/마일스톤 시각화 |
| **이슈 연동** | GitHub Issues | F-item → Issue 1:1 매핑 |

**보드 컬럼 구성**:
| 컬럼 | 대응 상태 | 설명 |
|------|----------|------|
| Backlog | 📋 PLANNED | 백로그 |
| Sprint | 🔧 IN_PROGRESS | 현재 스프린트 작업 |
| In Review | 🔍 REVIEW | 리뷰 대기 |
| Done | ✅ DONE | 완료 |

**커스텀 필드**:
- Priority (P0~P3)
- REQ코드 (FX-REQ-NNN)
- Sprint (v0.x)
- 담당자
- 산출물 경로 (docs/ 링크)

### 4.3 팀 리뷰 프로세스

#### 4.3.1 코드 리뷰

| 단계 | 행위 | 담당 |
|------|------|------|
| 1 | Feature branch에서 작업 | 담당자 |
| 2 | PR 생성 (템플릿 기반) | 담당자 |
| 3 | 리뷰어 지정 (최소 1명) | 담당자 |
| 4 | 리뷰 코멘트 | 리뷰어 |
| 5 | 수정 반영 + Re-request | 담당자 |
| 6 | Approve + Merge | 리뷰어 |

#### 4.3.2 문서 리뷰

| 산출물 유형 | 리뷰 방식 | 승인 기준 |
|------------|----------|----------|
| BluePrint / PRD | PR 리뷰 + 팀 미팅 | 팀원 과반 동의 |
| Plan / Design (PDCA) | PR 리뷰 | 리뷰어 1명 Approve |
| 분석 보고서 | PR 리뷰 | 자동 (AI Check ≥ 90%) + 리뷰어 확인 |
| 코드 변경 | PR 리뷰 | 리뷰어 1명 Approve + CI 통과 |

#### 4.3.3 PR 템플릿

```markdown
## 변경 요약
-

## 관련 요구사항
- [ ] REQ코드:
- [ ] F-item:

## 체크리스트
- [ ] SPEC.md 갱신 (해당 시)
- [ ] 문서 INDEX.md 갱신 (해당 시)
- [ ] 테스트 통과 확인
- [ ] 리뷰어 지정

## 산출물
-
```

### 4.4 현행화 가이드

기존 프로젝트를 새 체계로 전환하는 단계:

| 단계 | 작업 | 산출물 |
|:----:|------|--------|
| 1 | 프로젝트별 현황 감사 (문서·이슈·산출물 목록 작성) | 현황 체크리스트 |
| 2 | 표준 문서 구조 적용 (폴더 생성, 기존 문서 이동) | 정리된 docs/ |
| 3 | SPEC.md 생성/갱신 (요구사항 F-items 등록) | SPEC.md |
| 4 | GitHub Projects 보드 생성 + F-items → Issues 등록 | Projects 보드 |
| 5 | PR 템플릿 추가 + Branch Protection 설정 | .github/ 설정 |
| 6 | 팀원 온보딩 (가이드 공유, 첫 리뷰 사이클 실습) | 온보딩 세션 |

## 6. Success Criteria

### 6.1 Definition of Done

- [ ] AX BD팀 org-level GitHub Projects 보드 생성 및 운영
- [ ] Foundry-X 프로젝트에 표준 체계 적용 완료 (파일럿)
- [ ] PR 템플릿 + Branch Protection 규칙 적용
- [ ] 팀 리뷰 프로세스 정의 문서 작성
- [ ] 팀원 온보딩 가이드 작성
- [ ] 최소 1회 팀 리뷰 사이클 완료 (실습)

### 6.2 Quality Criteria

- [ ] 팀원 전원이 GitHub Projects 보드에 접근 가능
- [ ] 기존 SPEC.md F-items가 GitHub Issues와 동기화
- [ ] PR 리뷰 없이 main 브랜치 직접 push 차단 (Branch Protection)

## 7. Risks and Mitigation

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| 팀원 적응 부담 | Medium | High | 최소한의 프로세스만 도입, 점진적 강화 |
| GitHub Projects 기능 한계 | Medium | Medium | 복잡한 추적은 SPEC.md에 유지, Projects는 뷰 역할 |
| 문서 현행화 누락 | High | Medium | PR 체크리스트에 문서 갱신 항목 포함 |
| SPEC.md ↔ Issues 동기화 drift | Medium | Medium | 주 1회 sync 점검 (수동 또는 /ax-10-req sync) |
| 과도한 프로세스로 속도 저하 | High | Low | "가벼운 리뷰" 기본, P0/P1만 정식 리뷰 |
