### 1. 이 프로젝트가 실패한다면 가장 큰 이유 (조직적/인간적 관점)

가장 큰 이유는 **"개발자들의 조용한 보이콧 + 비기술자들의 무관심"** 조합이다.

구체적인 실패 시나리오 (2026년 9~11월쯤 현실화될 가능성 높음):

- 개발자들은 CLI를 써보더니 "pre-commit 훅이 또 깨지네", "sync 할 때마다 LLM 호출 비용 + 대기시간 때문에 오히려 느려", "내가 직접 짜는 게 더 빠르다" → 3주 만에 --no-verify 옵션으로 훅 우회하거나 아예 CLI 안 씀.
- PM/기획자들은 웹 대시보드에 들어가서 자연어로 요구사항 쓰면 → AI가 이상한 Spec JSON 만들어서 → 개발자들이 "이거 뭐임?" 하면서 무시 → 결국 다시 슬랙/이메일/지라로 돌아감.
- 중간관리자들은 "에이전트가 동등 팀원"이라는 말은 좋지만, 실제로는 에이전트가 실패한 태스크를 인간이 뒤치다꺼리하는 꼴을 보면서 "이거 왜 우리가 만드나?" 의문 시작.
- 결국 "파일럿 팀" 2~3개 돌리다가 사용률 15% 미만 → 예산 심의 때 "사용자 채택률 저조" 한 줄로 사업 종료.
- 남는 건 "KT DS AX BD팀이 AI 플랫폼 만들었다"는 PPT 한 장과, 아무도 안 쓰는 GitHub 레포 + PostgreSQL에 쌓인 죽은 작업 로그뿐.

이건 기술이 아니라 **조직 내 신뢰와 습관의 문제**다. 사람은 새로운 워크플로우를 받아들이는 데 6~18개월이 걸리는데, 그 기간 동안 "기존 방식보다 불편하다"는 인식이 박히면 끝이다.

### 2. 과대평가된 부분

**"AI 에이전트가 동등한 팀원"**이라는 비전 자체가 2026년 3월 기준으로 **완전 과대평가**다.

현재(2026년 초) Claude Code, Devin류 에이전트조차도:
- 장기 맥락 유지 실패
- 멀티파일 리팩토링 시 충돌 폭발
- 요구사항 미묘한 뉘앙스 놓침
- hallucination + workslop(저품질 AI 산출물) 문제 여전

이걸 "동등 팀원"이라고 포장하는 건 위험하다. 현실은 **고급 인턴** 수준(잘해야 junior 개발자 하위 30%)에 가깝다. SDD Triangle(명세-코드-테스트 삼각 동기화)은 이론적으로 아름답지만, 실제로는:

- 매 커밋마다 LLM 호출 → 비용 + 지연 = 개발자 짜증
- 자동 동기화가 오히려 "거짓 동기화" 만들어서 (AI가 잘못 이해한 Spec을 코드에 반영) 더 큰 혼란 초래
- 결과적으로 **관료적 오버헤드 + 검증 부담 증가** → 순수 개발 속도 **감소** 확률이 70% 이상

Drew Breunig의 원래 SDD Triangle조차 "아이디어" 단계에서 실험 도구 수준이지, 조직 전체 워크플로우로 굴릴 만큼 성숙하지 않았다.

### 3. 스코프가 너무 넓다 (당연히)

멀티리포 지원 + GitHub/GitLab 추상화 + CLI + 웹 대시보드 + 자연어→Spec + 에이전트 오케스트레이션 + SSOT Git + SDD Triangle + 하네스 + 실시간 SSE + PostgreSQL/Redis 백엔드 + 멀티테넌시 준비까지…

이걸 **소규모 팀**(KT DS AX BD팀 + 외부 SI 몇 명 수준)이 만든다는 건 불가능에 가깝다. 6개월 안에 MVP조차 제대로 못 낼 가능성이 크다.

**만약 딱 하나만 만들어야 한다면?**  
→ **Plumb 기반의 강력한 Git pre-commit 훅 + 간단 CLI (`foundry-x sync` & `foundry-x status`만)**

이것만 제대로 만들고, 개발자 10~20명이 3개월 이상 실제로 쓰게 만든 다음에 확장해도 늦지 않다. 나머지는 전부 "나중에"로 미뤄야 한다.

### 4. "일정: 유연" = 가장 큰 함정

데드라인 없는 사내 플랫폼 프로젝트가 성공한 사례는 **거의 없다**.  
유연 → 우선순위 하락 → 다른 사업에 사람 뺏김 → 개발 속도 점점 느려짐 → 모멘텀 상실 → 자연사.

2개월 후(2026년 5월 중순) 가장 현실적인 상태:
- CLI는 반쯤 돌아가지만 pre-commit 훅이 자주 깨짐
- 웹 대시보드는 Next.js로 대충 만든 로그인 + 빈 화면
- Plumb PoC는 "재밌는 데모" 수준에서 멈춤
- 팀 내에서 "이거 언제 끝나냐" 불평 시작
- 예산/인력 재검토 논의 테이블에 올라감

### 5. 내가 이 팀의 CTO라면 바꾸는 것

**가장 먼저 할 일 (첫 2주 안에)**

1. **Phase 1 범위 70% 삭감**  
   → 하네스 스캐폴딩 + Plumb 기반 SDD sync CLI (`foundry-x init`, `sync`, `status`) **단 3개 커맨드만** 구현 목표로 재설정. 웹 대시보드·에이전트 오케스트레이션·멀티리포·자연어 입력 전부 Phase 2 이후로 미룸.

2. **실제 사용자 8~12명 선정 후 강제 온보딩** (파일럿 팀 강제 편입)  
   → 매주 금요일 30분 "왜 안 쓰는지" 인터뷰 필수. 사용 안 하면 KPI에 반영.

3. **측정 가능한 성공 지표 재정의**  
   → "의도 정렬률 90%" 같은 뜬구름 지표 버리고  
   → "CLI 호출 횟수 / 주당 커밋 대비 비율"  
   → "sync 후 인간이 직접 고친 파일 수"  
   → "개발자가 --no-verify 쓰는 비율"  
   이런 **현실 지표**로 전환.

**절대 하지 않을 일**

- "AI 에이전트가 동등 팀원"이라는 문구를 외부/내부 발표 자료에서 완전히 삭제
- 에이전트 오케스트레이션, 멀티 에이전트 병렬 작업 등 Phase 2~3 기능에 1줄의 코드도 쓰지 않음
- "유연 일정" 유지 → 3개월 단위 하드 마일스톤 + 성공/실패 판가름 기준 명시

### 6. 착수 판단: Conditional

**Conditional Go** — 지금 당장 시작해도 되지만, **아래 3가지 조건 중 2개 이상 만족해야만 본격 개발 진행**.

1. 4주 내 Plumb 기반 sync PoC를 실제 개발자 5명이 1주일 이상 써보고 NPS 6점 이상 나올 것
2. Phase 1 범위를 위에서 말한 최소 범위(CLI 3개 커맨드 + 기본 하네스)로 60% 이상 축소하는 데 팀 전체 동의
3. 최소 1명의 PM/기획자가 "내가 직접 자연어로 요구사항 넣고 싶다"고 적극적으로 나설 것 (지금은 A1 가정 High Risk 그대로)

이 조건들 안 맞으면 → **Not Ready**. 바로 중단하고, Plumb 포크나 SDD 아이디어 실험 정도로 축소해서 놀아야 한다.

솔직히 말해서, 지금 PRD는 "우리가 미래를 만든다"는 꿈보다 **"조직이 우리 꿈을 믿어주지 않을 때 어떻게 죽나"**를 먼저 고민해야 하는 단계다.  
운 좋게 살아남으려면 지금 당장 스코프를 찢고, 개발자들의 진짜痛点부터 해결해야 한다.
