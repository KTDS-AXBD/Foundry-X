# REFACTOR Phase — 테스트 보호 아래 코드 개선

> "동작하는 코드를 아름답게. 테스트가 안전망이다."

## 원칙

1. **테스트 보호**: 리팩토링 중 테스트는 항상 통과해야 한다
2. **작은 단위**: 한 번에 하나의 개선만 수행하고 테스트 재실행
3. **되돌림 준비**: 리팩토링 후 테스트 실패 시 즉시 되돌린다

## 개선 항목

### 1. DRY (Don't Repeat Yourself)

- 중복 코드 → 공통 함수 추출
- 반복 패턴 → 유틸리티 함수
- 단, 2회 이하 사용이면 중복 허용 (과도한 추상화 방지)

### 2. 타입 안전성

- `any` 타입 → 구체적 타입으로 교체
- 타입 가드 추가 (런타임 안전성)
- 함수 반환 타입 명시 (추론이 복잡한 경우)

### 3. 네이밍

- 의미 있는 변수명 (약어 지양)
- 함수명은 동작을 설명 (`getData` → `fetchUserProfile`)
- 상수는 UPPER_SNAKE_CASE

### 4. 에러 핸들링

- 암묵적 에러 → 명시적 에러 클래스
- 에러 메시지에 컨텍스트 포함
- 복구 가능한 에러 vs 치명적 에러 구분

### 5. 구조

- 함수 길이: 50줄 이하 권장
- 중첩 깊이: 3단계 이하
- 조기 반환 (early return) 패턴 활용

## 검증 순서

매 개선 후 아래 순서로 검증:

```bash
# 1. 테스트 통과
pnpm test

# 2. 타입 체크
pnpm typecheck

# 3. 린트
pnpm lint
```

하나라도 실패하면 → 해당 개선 되돌리기.

## 리팩토링 하지 않는 것

- 테스트 코드 자체 (테스트는 가독성 > DRY)
- 외부 라이브러리 인터페이스
- 이 Sprint 범위 밖의 코드
- 동작이 변경되는 수정 (그건 새 RED 사이클)

## 체크리스트

- [ ] `pnpm lint` 에러 없음
- [ ] `any` 타입 미사용 (또는 정당한 사유)
- [ ] 함수 길이 50줄 이하
- [ ] 중첩 깊이 3단계 이하
