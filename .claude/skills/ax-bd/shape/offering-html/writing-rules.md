## 2. KT 연계 원칙

### 2.1 대전제

**KT 연계가 없는 사업은 이 포맷의 대상이 아니다.**

KT와 동떨어진 독자 AX 사업은 이 기획서로 올리지 않는다. KT 연계 여부를 Step 1에서 먼저 확인하고, 연계가 없으면 경고 메시지를 출력한 뒤 사용자에게 확인한다.

### 2.2 추진배경 3축 필수 체크

01-1 추진배경은 **항상 3축 구조**. 하나라도 누락 시 경고:

| 축 | 핵심 질문 | 누락 시 메시지 |
|----|----------|---------------|
| **축1: 수익성** | 이 사업이 매출·수익에 어떻게 기여하는가? | "추진배경에서 수익성 근거가 누락되었어요." |
| **축2: KT 적합성** | KT 전략과 어떻게 연결되는가? | "추진배경에서 KT 적합성이 누락되었어요." |
| **축3: 실행력** | 왜 우리(kt ds)가 할 수 있는가? | "추진배경에서 실행력 근거가 누락되었어요." |

### 2.3 KT 현재 상태 기술 원칙

- 협약 없으면 **"없음"** 명시 (애매하게 "논의 중" 금지)
- 단계적 접근 표기: kt ds 주도 → KT 사전 협의 → KT 공동 제안
- KT 조직개편 반영: 최신 조직 구조 기반 연계 포인트 식별
- **축 라벨 조정 가능**: 사업 특성에 따라 "수익성" 대신 "시장 기회", "KT 적합성" 대신 "그룹 전략 정합성" 등으로 변경 가능. 단, 3축 구조 자체는 유지

## 3. 고객 유형별 톤 자동 조정

Step 2 목차 확정 시 고객 유형을 확인하고, 이후 전체 섹션 톤에 반영한다.

### 3.1 유형 판별 (AskUserQuestion)

```
"이 사업기획서의 주요 고객(보고 대상)은 어떤 유형인가요?
 1) 분석·보고 기관 (정책 지원 — 예: 공공기관, 연구기관)
 2) 의사결정 주체 (기업, 금융 등 — 예: 대기업, 은행)
 3) B2C 서비스 대상"
```

### 3.2 유형별 톤 규칙

| 고객 유형 | 시나리오 톤 (03-1) | Exec Summary 톤 | 제안 방향 톤 |
|-----------|-------------------|-----------------|-------------|
| **분석·보고 기관** | "추천" 아닌 **"분석 근거 제공 + 대응 옵션 비교"** | 객관적, 다각적 비교 | 옵션 A/B/C 제시 |
| **의사결정 주체** | **"추천안 제시"** 가능 | 결론 선행, 근거 후속 | 명확한 추천안 |
| **B2C 서비스** | **"페르소나 기반 Use Case"** | 사용자 중심 가치 강조 | 사용 경험 중심 |

### 3.3 톤 적용 섹션

| 섹션 | 분석·보고 기관 | 의사결정 주체 | B2C 서비스 |
|------|--------------|-------------|-----------|
| Exec Summary | 배경 분석 → 옵션 → 권고 | 결론 → 근거 → 행동 | 사용자 가치 → 비즈니스 가치 |
| 01-1 추진배경 | 정책 동향 중심 | 수익/전략 중심 | 시장 트렌드 중심 |
| 03-1 시나리오 | 분석 보고서 활용 시나리오 | 의사결정 지원 시나리오 | 페르소나 Use Case |
| 04-3 기대효과 | 정책 기여도, 사회적 가치 | ROI, 비용 절감 | 사용자 만족도, 전환율 |

## 5. 섹션별 작성 규칙 (주요)

### 5.1 Executive Summary (§0.5)

- **불릿 수**: 5~6개 (너무 많으면 집중도 저하)
- **표준 불릿 풀**에서 선택·조합 (SKILL.md 참조)
- **기술 용어 금지**: 경영진이 읽는 섹션 — LLM/RAG 등 직접 언급 안 됨
- **볼드**: 키워드에만, 1불릿당 2~3개
- **가능형 금지**: "~할 수 있다" 대신 "~를 추진"

### 5.2 추진배경 (§01-1)

- **3축 카드 필수** (수익성 / KT 적합성 / 실행력)
- 하단 **추진 목적 블랙 박스**: 1문장 + 3개 키워드 카드
- 라벨 조정 가능하지만 3축 구조 유지

### 5.3 시장 분석 (§01-7)

- **TAM/SAM/SOM** 카드 3개
- 모든 수치에 **"약" 표기** + **산출 근거**
- 과대 추정 금지
- **경쟁 비대칭 가설**: kt ds가 가진 구조적 우위 3가지 명시

### 5.4 데이터 확보 (§05-1)

- **모든 사업기획서에 필수**
- 계층별 구조: 자체 보유 / 공개 API·연계 / 유료 구매·파트너십
- PoC vs 본사업 구분 명시
- 크롤링 이슈, 라이선스, 비용 이슈 사전 식별

### 5.5 교차검증 (§05-4)

- **냉철한 톤** — "경영진이 안심하도록"이 아니라 "정확히 판단하도록"
- No-Go 판정도 가능한 수준이어야 함
- cross-validation.md 참조

### 5.6 로드맵 (§03-3)

- **3단계 필수**: 단기(PoC) → 중기(본사업) → 장기(확장)
- 매출이 적어도 **"이 사업이 쌓는 전략 자산"** 명시
- 예: "당장 매출은 작지만, PoC를 통해 (1)도메인 데이터 (2)고객 레퍼런스 (3)기술 검증을 확보"

## 6. 최종 점검 체크리스트 (Step 7)

Step 7에서 아래 10항목을 자동 검토한다. 위반 시 해당 부분을 표시하고 수정을 제안한다.

```
[최종 점검 — 10항목]
├── [1] 가능형 표현 검색: "~할 수 있다", "~가능하다", "~될 수 있다"
├── [2] 최초 표현 검색: "최초", "첫", "처음으로"
├── [3] 금액 약 표기 확인: 숫자+억/만/원 앞에 "약" 존재 여부
├── [4] 과대 표현 검색: "혁신적", "독보적", "유일한", "최고의"
├── [5] 볼드 과다 확인: 한 불릿에 볼드 3개 초과 여부
├── [6] 기술 용어 확인: Exec Summary에 LLM/RAG/Embedding 등 직접 사용 여부
├── [7] KT 현재 상태 확인: 01-1, 04-2에서 협약 상태 솔직 기술 여부
├── [8] TAM 근거 확인: 01-7 수치에 산출 근거 첨부 여부
├── [9] 수익 근거 확인: 05-2 매출 수치에 산출 근거 첨부 여부
└── [10] 출처 URL 확인: 시장 데이터 인용 시 출처 링크 존재 여부
```

## 7. 피드백 반영 규칙 (Step 5)

사용자가 피드백을 주면:

1. **섹션 식별**: 피드백 내용에서 해당 섹션 자동 식별
2. **톤 유지**: 고객 유형별 톤을 유지하면서 수정
3. **마커 삽입**: 수정된 부분에 `<!-- CHANGED: {사유} -->` 마커
4. **경영 언어 재검증**: 수정 후 10항목 체크리스트 재적용
5. **버전 업**: v0.1 → v0.2 등 자동 증가
