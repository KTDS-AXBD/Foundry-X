예시 질의 경로
"희토류 영구자석 — 수입 현황, 관련 R&D, 어떤 산업에 영향?"
희토류 영구자석 — 교역된다 → 무역 — 중국 의존도 → 85%
희토류 영구자석 — 연구한다 → R&D — 관련 과제 → 3건 진행 중
희토류 영구자석 — 속한다 → 산업 — 관련 산업 → 전자 · 자동차
하나의 품목에서 출발해 무역·R&D·산업 등 복수 엔티티를 동시에 조회·조합하여,
"의존도가 높고 대체 R&D도 초기 단계인 품목이 어떤 산업에 집중되어 있는가" 같은 복합 의사결정 질문에 답할 수 있습니다.
데이터를 Ontology로 연결하면, AI에게 '이 문서의 검색'이 아니라 '이 품목의 관계'를 기반으로 인과 경로, 영향 범위, 대안까지 추론할 수 있습니다.

Why Ontology

왜 Ontology / Knowledge Graph인가?

"LLM으로 되는 거 아냐?" "RDB로 충분하지 않나?"

LLM + RAG RDB (SQL) Ontology + KG
인과관계 추적
"왜 이 품목이 영향받나?"		 문서에 있으면 답함
없으면 생성 (hallucination)		 미리 설계한 JOIN만 가능
새 관계 = 새 테이블 필요		 관계를 따라가며 인과 경로 추적
+ 근거 제시

"만약 ~하면 어디까지?"		 시나리오 예측 불가
(과거 문서 기반)		 새 질문마다 재설계 필요 관계 그래프에서
영향 범위를 자동 추적
설명가능성
"이 답의 근거는?"		 왜 그 답인지 설명 어려움
(블랙박스)		 쿼리는 투명하나
복잡 관계 표현 한계		 A→B→C 인과 경로를
그대로 제시
재현성
"같은 질문, 같은 답?"		 같은 질문에 다른 답 가능 동일 쿼리 = 동일 결과
단, 관계 추론 불가		 동일 그래프 = 동일 경로
확장성
"새 데이터 추가 시?"		 문서 추가 시 재색인 스키마 변경 = 전체 재설계 엔티티/관계 추가만으로 확장
LLM은 '읽는' 도구, RDB는 '찾는' 도구, Ontology는 '연결하고 추론하는' 도구입니다.

Ontology 기반 의사결정 지원 — 두 가지 핵심 시나리오(안)

GIVC Ontology는 두 가지 의사결정 질문에 답합니다.

첫째, 외부 충격이 오면 어디까지 어떻게 영향이 퍼지고, 무엇을 먼저 해야 하는가.

둘째, 원래 취약한 품목은 무엇이며 어떤 대응이 최적인가.

A = 충격 대응형 의사결정   ·   B = 취약성 대응형 의사결정
Scenario A
외부 충격 발생 시 산업 영향과 우선 조치 도출
"이란 사태, 철강 급등, 물류 차질 같은 외부 충격이 발생하면
어떤 품목과 산업이 어떤 경로로 영향받고, 지금 무엇을 먼저 해야 하는가?"
INPUT A — 외부 이벤트
이란 사태, 수출규제, 항로 봉쇄, 자연재해 등
뉴스/글로벌 이벤트 기반 입력
INPUT B — 내부 EWS 경보
가격 급등, 수급 차질, 환율 급변 등
기산진 EWS 시스템 경보 기반 입력
Ontology가 두 입력을 동일한 충격 드라이버 체계로 변환하여 처리합니다
처리 과정
STEP 1
충격 해석
이란 사태 → 호르무즈 리스크, 원유/정제품 수급 불안, 물류 차질을 충격 드라이버로 구조화
STEP 2
영향 경로 탐색
KG에서 원유, 정제품, 물류와 연결된 GIVC 품목을 따라가며 인과 경로 추적
STEP 3
산업 영향도 산출
원가 민감도, 대체 가능성, 시차, 수입의존도를 결합해 High/Med/Low 산출
STEP 4
의사결정안 도출
적용 주체별 대응 옵션 도출 + 4축 비교(즉시성, 영향범위, 실행난이도, 정책활용성)
STEP 5
최종 권고
추천안 + 추천 근거 + 선택 시 예상 효과를 종합 제시

Scenario B
공급망 취약 품목 진단과 대응 전략 추천
"우리 산업 공급망에서 구조적으로 취약한 품목은 무엇이고,
왜 취약하며, 어떤 대안을 선택해야 하는가?"
처리 과정
STEP 1
취약 품목 식별
의존도, 대체 가능성, 전략산업 연관도 기준으로 취약 품목 후보군 추출

STEP 3
유형 판정
즉시 다변화형 / 국산화 추진형 / R&D 집중형 / 모니터링 유지형으로 최종 판정
STEP 4
의사결정안 도출
진단 데이터와 직접 연결된 대응 옵션 도출 + 추천 근거 제시
