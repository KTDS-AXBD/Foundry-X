## Part 1: 왜 (목적/문제)

### 핵심 응답
- "AI Foundry"라는 이름은 부적절. 코드 생성 도구가 아니라 **AI 시대에 조직 내 동료들 간의 협업을 위한 플랫폼**이 정체성
- PoC/프로토타입 구축 비용 절감, 요구사항↔코드 간 의도 왜곡 제거는 맞지만, **더 큰 개념의 협업 플랫폼**이 본질
- 하네스 엔지니어링 컨셉 반영 필요 (OpenAI Codex 팀 사례 참조)

### 반영된 컨셉
- **하네스 엔지니어링**: 에이전트가 안정적으로 일할 수 있는 환경(하네스)을 설계하는 것이 엔지니어의 주된 역할
- **리포지토리 = SSOT**: 에이전트 관점에서 컨텍스트 내에 접근할 수 없는 것은 존재하지 않는 것
- **의도 명시 + 피드백 루프**: 환경을 설계하고, 의도를 명시하며, 피드백 루프를 구축
- **역할 전환**: 코드 작성 → 환경 설계 + 의도 명시 + 피드백 루프 구축

### As-Is → To-Be
- As-Is: 엔지니어의 역할이 코드 작성에서 전환되는 시대에, 조직이 이 전환을 체계적으로 수행할 플랫폼이 없음
- To-Be: 사람과 에이전트가 동등한 팀원으로 함께 만드는 협업 플랫폼

## Part 3: 무엇을 (기능/범위)

### 핵심 5축 (Must Have)

| # | 축 | 설명 |
|---|---|---|
| 1 | **하네스 구축** | 리포/스키마/규칙 등 에이전트가 일할 수 있는 환경 설계 |
| 2 | **SDD Triangle** | 명세↔코드↔테스트 삼각 동기화. 코드 구현이 명세를 개선하고, 테스트가 명세를 검증하는 순환 루프. Plumb 컨셉(커밋 시 결정사항 자동 추출→명세 업데이트→커버리지 갭 보고) 반영 |
| 3 | **에이전트 오케스트레이션** | 병렬 작업 지시, 추적, 조율 |
| 4 | **지식 공유 (SSOT)** | 리포 내 모든 컨텍스트를 단일 진실 공급원으로 관리. Spec/Test/Code 모두 1급 시민 |
| 5 | **협업 워크스페이스** | 프로젝트 관리, 역할 분담, 상태 추적 |

### SDD Triangle 상세 (Part 3 수정사항)

**Spec-Driven Development Triangle** 반영:
- 기존 "의도 명시"와 "피드백 루프"를 SDD Triangle로 통합·강화
- 변경 전: 일방향 파이프라인(요구사항→코드) + 별도 리뷰 루프
- 변경 후: 명세↔코드↔테스트가 상시 동기화되는 삼각 피드백 루프
- 코드 구현이 명세를 개선하고, 테스트가 명세를 검증하고, 명세가 코드와 테스트를 구속하는 순환 구조
- Plumb 컨셉: 커밋 시 결정사항(decisions) 자동 추출 → 명세 업데이트 → 커버리지 갭 보고
- Git 훅으로 결정 검토 강제
- 리포 내에서 Spec/Test/Code가 모두 동등한 1급 시민(first-class citizen)

참고 자료:
- Drew Breunig, "Learnings from a No-Code Library: Keeping the Spec Driven Development Triangle in Sync"
- GitHub dbreunig/plumb — Spec/Test/Code 동기화 CLI 도구
- GitHub github/spec-kit — GitHub의 SDD 툴킷
- Fission-AI/OpenSpec — AI 코딩 어시스턴트용 Spec 레이어

### 기존 기능 유지 (플랫폼 하위 기능으로 편입)
- 코드 자동 생성 (Spec→코드 스캐폴딩)
- 자연어→Spec DSL 자동 변환 (LLM 연동)
- 프로토타입 미리보기/샌드박스
- 외부 도구 연동 (Jira, Slack, GitHub 등)

### 작동 단위

## Part 4: 어떻게 판단할 것인가 (성공 기준)

### KPI 4개
1. **의도 정렬률**: 요구사항 ↔ 최종 산출물 일치도
2. **PoC/프로토타입 구축 시간 단축**: 2~4주 → X
3. **팀 협업 효율성**: 커뮤니케이션 손실 감소
4. **에이전트 신뢰도**: 생성 코드 품질, 테스트 통과율

### MVP 최소 기준
- 하네스 환경이 구축되고 에이전트가 첫 작업을 완료하는 시점

### 최대 실패 리스크
- **사용자 채택 실패** (아무도 쓰지 않음)
- 기술보다 UX와 온보딩이 우선순위
